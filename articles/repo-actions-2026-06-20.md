### 1. Multi-Model Role Assignment

**Type:** Feature
**Effort:** Medium (1-2 days)
**Impact:** Issue #193 asks for separate thinking token budgets for reasoning-capable models. The natural next primitive is model-level control: let operators assign different LLM models to different agent roles. "Key opinion leaders" (high-influence agents) run on Claude Opus with extended thinking; background filler agents run on a local Ollama model or a cheap API tier. A 40-agent simulation that today costs $4 could cost $0.80 using tiered model assignment — without sacrificing quality on the agents that drive the outcome. The `influence_score` in `agents.json` gives a data-driven basis for automatic tier assignment: high-influence agents get the expensive model, low-influence get the cheap one.

**How:**
1. Extend `simulation_config.json` schema: add an optional `model_overrides: {profession: str}` map (e.g., `{"Analyst": "claude-opus-4-8", "Retail Investor": "ollama/llama3.2"}`). Add an optional `model_tier_policy: "by_influence"|"by_profession"|"flat"` field: if `by_influence`, automatically assign the top tercile of agents (by initial influence_score from the agent generation step) to `tier_high_model`, middle tercile to `tier_mid_model`, bottom to `tier_low_model` — three new optional config fields. In `wonderwall/social_agent/agent.py`, modify `_aget_model_response` to read the resolved model from `self.config.model_overrides.get(self.profession)` or from the tier policy before falling back to the global `MODEL` env var. Add `get_agent_model(agent_config, sim_config)` to a new `backend/app/services/model_routing_service.py` (~80 LoC, pure stdlib). Add `POST /api/simulation/{id}/config/model-overrides` (admin-gated; body: `{overrides: {profession: str}, tier_policy: str|null, tier_models: {high: str, mid: str, low: str}|null}`; 409 if simulation has already started). Add 10 unit tests in `backend/tests/test_unit_model_routing.py`: profession match returns overridden model, unmatched profession falls back to global model, tier_policy "by_influence" assigns top-tercile agents to tier_high_model, tier_policy "flat" ignores tiers, config with no model_overrides returns global model for every agent, influence_score=None agents assigned to mid tier, model_overrides key is case-insensitive profession match, tier_models.high is used when policy is "by_influence" and agent is in top tercile, 409 returned if sim already running, validate that model string format is accepted (non-empty string).
2. In `Step2EnvSetup.vue` (or the simulation config step), add a collapsible "Model Assignment" section below the global model selector. Two sub-modes (toggleable): **By Role** — a table listing each configured profession (from the agent list above), each row has a model dropdown (global default / Claude Opus / Claude Sonnet / GPT-4o / ollama/llama3.2 / custom). **By Influence Tier** — three dropdowns: High Influence / Mid Influence / Low Influence, each mapping to a model. The section is collapsed by default with a summary chip showing "3 model tiers" or "2 custom roles" when configured. Save the model_overrides object alongside the rest of the simulation config. Add `setModelOverrides(simId, overrides)` to `frontend/src/api/simulation.js`. In the simulation watch page, show a "Model routing" section in the operator panel: a small table of which professions/tiers ran on which model. Bilingual EN/ZH labels throughout.
3. Add `POST /api/simulation/{id}/config/model-overrides` to the OpenAPI spec under the "Operator" tag with `ModelOverridesRequest` schema. Add `GET /api/simulation/{id}/config/model-overrides` (returns current overrides or 404 if none set). Add a "Multi-Model Role Assignment" section to `docs/FEATURES.md`: explains the three policies (flat / by_profession / by_influence), the precedence order (explicit profession override > tier policy > global MODEL env var), and a worked example showing cost reduction for a 40-agent simulation using Ollama for low-influence agents. Note: model strings follow camel-ai's model identifier format — document `ollama/model-name` syntax explicitly.

**Effort:** Small (hours)

A preset library of curated agent sets for common simulation topics reduces that to a single click. "Crypto Twitter (25 agents)", "Tech Community (15 agents)", "Political Analysis Panel (30 agents)", "Academic Forum (10 agents)", "Retail Consumer Panel (20 agents)" — stored as JSON templates in `presets/agents/`. The feature is also a community contribution vector: contributors can open a PR adding a new preset for their domain, a contribution type that requires zero backend or frontend expertise beyond JSON.

**How:**

Add 5 preset JSON files, each with schema: `{name: str, description: str, total_agents: int, agents: [{profession: str, count: int, personality_template: str, initial_stance_distribution: {bullish_pct: float, bearish_pct: float, neutral_pct: float}}], tags: [str], author: str, created_at: ISO}`. Add `backend/app/services/preset_service.py` (~60 LoC, pure stdlib). `list_presets(presets_dir)` scans `presets/agents/*.json`, reads name/description/total_agents/tags. `get_preset(presets_dir, name)` returns the full preset JSON. Add `GET /api/presets/agents` (unauthenticated; returns list of presets without full agent detail). Add `GET /api/presets/agents/{name}` (unauthenticated; returns full preset including agents array; 404 if not found). Add 8 unit tests in `backend/tests/test_unit_presets.py`: list_presets returns all .json files, list_presets excludes non-json files, get_preset returns full agents array, unknown preset returns 404, preset JSON validation (each preset has name/agents/total_agents), total_agents matches sum of agent counts, tags is always an array (not null), empty presets dir returns empty list from list_presets.

### 3. Stalled Simulation Health Monitor

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** The camel-ai 0.2.90 regression ran undetected for ~2 months: simulations appeared to "complete" but wrote zero actions. The strengthened smoke test now guards that exact regression class. But a live running simulation that stalls mid-run — last activity 15 minutes ago, still marked "running" — is currently invisible to operators. There is no signal in the watch page or the API that says "this simulation is not making progress." `GET /api/health/simulations` scans all `state.json` files with `status: "running"`, reads `activity.json` for the last event timestamp, and returns a health summary per running simulation: `{sim_id, started_at, running_minutes, last_activity_at, last_activity_ago_seconds, health: "ok"|"warning"|"stalled"}`. The watch page adds a dismissible "Stalled?" banner when the monitor returns `health: "stalled"` (last_activity > 10 minutes). Operators know about stuck simulations from the UI rather than a post-mortem.

**How:**
1. Add `backend/app/services/health_service.py` (~80 LoC, pure stdlib). `scan_running_simulations(data_dir)` scans all sim dirs for `state.json` files where `status == "running"`. For each running sim: reads `started_at` from `state.json`, computes `running_minutes = (now - started_at).total_seconds() / 60`. Reads the last line of `activity.json` (JSONL format) to get `last_activity_at`; if `activity.json` does not exist or is empty, uses `started_at` as last_activity_at. Computes `last_activity_ago_seconds = (now - last_activity_at).total_seconds()`. Assigns `health`: "ok" if `last_activity_ago_seconds < 300` (5 min); "warning" if 300–600 (5–10 min); "stalled" if > 600 (10+ min). Returns `{checked_at: ISO, running_count: int, simulations: [{sim_id: str, started_at: ISO, running_minutes: float, last_activity_at: ISO, last_activity_ago_seconds: float, health: str}]}`. Add `GET /api/health/simulations` (admin-gated; `Cache-Control: no-store` — always fresh). Also expose a lightweight `GET /api/health` endpoint (existing or new) that includes `{running_simulations_stalled: count_of_stalled}` as a top-level field alongside the existing uptime/version health check. Add 10 unit tests in `backend/tests/test_unit_health.py`: sim with no activity.json uses started_at as last_activity_at, sim with activity < 5 min ago returns health="ok", sim with activity 6 min ago returns health="warning", sim with activity 11 min ago returns health="stalled", completed sims (status != "running") excluded from results, empty data_dir returns running_count=0, running_minutes computed correctly from started_at, last line of activity.json is used (not first line), admin token required (401 without), checked_at is present in response.
2. In `Step3Simulation.vue`, when the simulation `status === "running"`, poll `GET /api/health/simulations` every 60 seconds (separate from the main status poll). If the response contains an entry for the current sim_id with `health: "warning"`, show a dismissible amber banner below the simulation header: "Activity slowing — last update {N} min ago. This simulation may be running slowly." If `health: "stalled"`, show a dismissible red banner: "No activity detected for {N} minutes. This simulation may be stalled — refresh or contact your operator." The banner includes a "Dismiss" button (hides for the browser session via sessionStorage per sim_id). If the operator is logged in, show an additional "Stop simulation" action link in the stalled banner. Add `getSimulationHealth()` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels on both banner messages. Stop polling the health endpoint once the simulation status changes to "complete", "failed", or "budget_exceeded".
3. Add `GET /api/health/simulations` to the OpenAPI spec under a new "Health" tag with `SimulationHealthResponse` schema: `{checked_at: str, running_count: int, simulations: [{sim_id: str, started_at: str, running_minutes: float, last_activity_at: str, last_activity_ago_seconds: float, health: "ok"|"warning"|"stalled"}]}`. Add a "Simulation Health Monitor" section to `docs/FEATURES.md`: explains the three health tiers and their thresholds (5 min warning, 10 min stalled), notes that the health endpoint is admin-gated (requires the same token as other operator endpoints), and explains why `activity.json` is the health signal (every agent action and round completion is logged there).

### 4. Ollama One-Click Configuration

**Type:** Integration / DX
**Effort:** Small (hours)
**Impact:** PR #188 increased timeouts for non-English and local LLMs because real operators hit timeouts running MiroShark with Ollama. 52M Ollama downloads; the market is already there. But setup still requires manually setting `LLM_BASE_URL`, choosing a compatible model, and debugging timeouts. There's no feedback loop: a misconfigured Ollama setup fails silently at simulation start, not at config time. `POST /api/config/llm-provider` accepts `{provider: "ollama", base_url: "http://localhost:11434", model: "llama3.2"}` and runs a preflight check before saving: can it reach base_url? Is the model listed? Does a test generate call succeed within 30 seconds? Returns a structured result: `{status: "ok"|"unreachable"|"model_missing"|"too_slow", latency_ms: int, model_context_length: int, note: str}`. Operators know at configuration time whether their Ollama instance will work, not after a failed 20-minute simulation run.

**How:**
1. Add `backend/app/services/llm_provider_service.py` (~120 LoC, pure stdlib). `probe_ollama(base_url, model, timeout=30)` performs three checks using `urllib.request`: (1) GET `{base_url}/api/tags` — verifies Ollama is reachable and lists available models; if the model is not in `response.models[*].name`, return `{status: "model_missing", ...}`; (2) POST `{base_url}/api/generate` with `{model: model, prompt: "Reply with the word OK.", stream: false}` — times the round-trip; if > 30s return `{status: "too_slow", latency_ms: int, note: "Consider a faster model for multi-agent simulations"}`; (3) On success: read `model.details.parameter_size` and context_length from `{base_url}/api/show` (POST with `{name: model}`). Returns `{status: "ok", latency_ms: int, model_context_length: int, parameter_size: str, note: str}`. `save_provider_config(config_dir, provider_config)` writes `{provider, base_url, model, probed_at: ISO, probe_result}` to `config/llm_provider.json`. Add `POST /api/config/llm-provider` (admin-gated; body: `{provider: "ollama"|"openai-compatible", base_url: str, model: str}`; runs probe, saves config if ok; returns probe result). Add `GET /api/config/llm-provider` (admin-gated; returns current saved provider config or `{configured: false}` if absent). Add 10 unit tests in `backend/tests/test_unit_llm_provider.py`: unreachable base_url returns status="unreachable", model not in tags list returns status="model_missing", generate response > 30s returns status="too_slow", successful probe returns status="ok" with latency_ms, config saved to llm_provider.json on "ok" probe, get returns configured=false when no file, probe result included in save, admin token required for both endpoints, invalid provider string returns 400, base_url missing scheme returns 400.
2. In `Step1` (or the settings/admin page — wherever the global LLM model is configured), add an "LLM Provider" section. Default state: "Using API (env: MODEL, LLM_BASE_URL)". An "Add Ollama" button opens a compact setup panel with two fields: Base URL (default: http://localhost:11434) and Model (text input with a placeholder of "llama3.2"). A "Test connection" button fires `POST /api/config/llm-provider`. Display the result inline: a green chip "Connected (Llama 3.2 · 8B · 128K ctx · 2.1s avg)" on success, or a red chip with the specific failure reason on error ("Model not found — available: llama3.2:latest, mistral"). A "Save" button appears only after a successful probe. When saved, a banner replaces the setup panel: "Ollama active: {model} at {base_url}" with a "Change" link. Add `probeOllamaProvider(config)` and `getLlmProviderConfig()` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels throughout.
3. Add `POST /api/config/llm-provider` and `GET /api/config/llm-provider` to the OpenAPI spec under the "Configuration" tag with `LlmProviderProbeResult` schema: `{status: "ok"|"unreachable"|"model_missing"|"too_slow", latency_ms: int|null, model_context_length: int|null, parameter_size: str|null, note: str}`. Add an "Ollama Setup" section to `docs/CONFIGURATION.md`: step-by-step from `ollama pull llama3.2` through the UI configuration flow. Include a table of recommended models with their parameter sizes and whether they're fast enough for multi-agent simulations (rule of thumb: < 5s test generate = ok for 10-agent sims; < 2s = ok for 30+). Note: the Ollama base_url defaults to localhost but can be a remote server for team deployments.

### 5. Locale Auto-Detect from Browser

**Type:** DX / Feature
**Effort:** Small (hours)
**Impact:** MiroShark supports four languages as of this week (EN, ZH-CN, FR, DE). But every new visitor — including German and French users — lands in English and has to manually find and click the locale toggle. Standard UX for multilingual apps: read `navigator.language` on first visit, match to the nearest supported locale, set it automatically, persist in localStorage. A German user visiting from a German browser opens in German. No extra API calls, no configuration. For MiroShark's push toward non-English markets (French community issue #161, German landing this week, ZH-CN since launch), automatic locale detection removes the biggest friction point in the multilingual onboarding path.

**How:**
1. In `frontend/src/i18n.js` (the i18n initialization file), add a `detectBrowserLocale()` function that reads `navigator.language` and `navigator.languages[0]` (fallback). Maps browser locale tags to supported locales: `de` / `de-*` → `'de'`; `zh` / `zh-CN` / `zh-Hans-*` → `'zh-CN'`; `zh-TW` / `zh-HK` / `zh-Hant-*` → `'zh-CN'` (best match); `fr` / `fr-*` → `'fr'`; anything else → `'en'`. Add a `LOCALE_STORAGE_KEY = 'miroshark_locale'` constant. On app init: (a) check `localStorage.getItem(LOCALE_STORAGE_KEY)` — if set, use that value as the initial locale (user's explicit prior choice wins); (b) if not set, call `detectBrowserLocale()`, set that as the initial locale, and write to localStorage. Update the existing `toggleLocale()` function (or equivalent) to write the chosen locale to localStorage on each toggle, so manual overrides persist across sessions. No backend changes needed — all locale state lives in the browser. Add 6 unit tests in `frontend/src/i18n.test.js` (Vitest): `detectBrowserLocale()` maps `"de-DE"` → `"de"`, maps `"fr-CA"` → `"fr"`, maps `"zh-CN"` → `"zh-CN"`, maps `"en-US"` → `"en"`, maps `"es-MX"` → `"en"` (unsupported → default), maps `"zh-TW"` → `"zh-CN"` (best match).
2. In `LocaleToggle.vue`, add a one-time "auto-detected" visual hint: on the first visit (when the locale was set by detection rather than explicit toggle), show a brief 3-second tooltip under the locale toggle chip: "Language auto-detected from your browser. Click to change." (bilingual: same in detected language — if detected as German, the tooltip appears in German). Track "shown" state in `sessionStorage` so the tooltip only appears once per session, not on every page load. No other UI changes needed — the toggle already works and persists correctly once localStorage is updated. Ensure `LocaleToggle.vue` reads from localStorage on mount so the chip always shows the active locale (not always the default).
3. No backend changes, no OpenAPI changes. Add a "Locale Auto-Detection" section to `docs/FEATURES.md`: explains the `navigator.language` detection order, the locale-to-supported-locale mapping table (including the `zh-TW` → `zh-CN` best-match behavior), and notes that localStorage takes precedence over auto-detection (user's explicit choice always wins). Note: `navigator.language` is undefined in some embedding contexts (iframe without allow="language") — the fallback is always `'en'`. Add a comment in `i18n.js` at the `detectBrowserLocale()` function explaining why `zh-TW` maps to `zh-CN` rather than falling back to `en` (better experience for traditional Chinese users than forcing English).

## Selection Rationale

Five ideas built on what just changed this week:

- **Multi-Model Role Assignment** (#1) — Issue #193 (thinking token budgets for reasoning models) is the trigger. The natural next step isn't just separate token budgets — it's separate models per role. The `influence_score` already in `agents.json` gives an automatic basis for tiering; operators don't need to assign models manually if they set `tier_policy: "by_influence"`. The cost case is real: 40 agents on a $3 model = $120/sim; tiered assignment gets it to $15-30.

- **Agent Preset Library** (#2) — The blank agent form is the biggest new-user friction point. Five curated presets in JSON are a 1-hour build with outsized onboarding impact. The contribution path (open a PR adding a JSON file) is also the lowest-barrier way to get community pull requests from domain experts who don't know Python.

- **Stalled Simulation Health Monitor** (#3) — The camel-ai PR thread explicitly named "2 months of silent failure" as the thing the smoke test guards against. But it only guards against future regressions, not current stalls. The health monitor is the runtime equivalent: a live signal that tells operators a running sim isn't making progress, before it times out.

- **Ollama One-Click Configuration** (#4) — PR #188 is evidence of real Ollama operators hitting real pain. 52M downloads. The preflight probe is the key primitive: fail at configuration time, not mid-simulation. The "Connected / Model Missing / Too Slow" feedback loop is the thing that currently doesn't exist.

- **Locale Auto-Detect** (#5) — German landed this week. Four languages, all requiring a manual toggle to switch from English. Auto-detect is one function and a localStorage write. The one-time tooltip closes the loop so users know the detection happened. For every German user who would have bounced on an English UI and never found the toggle, this is the fix.
