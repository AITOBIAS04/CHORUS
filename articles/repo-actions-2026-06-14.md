### 1. Webhook Delivery for Simulation Events

**Type:** Integration
**Effort:** Medium (1-2 days)
**Impact:** Fourteen ecosystem projects currently poll /api/activity.json to learn when simulations complete. A webhook registry lets operators register callback URLs and receive a POST the moment a simulation's status changes — no polling, no lag, no missed events. This is the integration primitive that turns MiroShark from a pull API into a push partner. Projects like those in the ecosystem registry can subscribe once and react in real time to new publications, completion events, or simulation errors. This is the unlock that makes MiroShark composable with external pipelines (Zapier, n8n, custom webhooks).

**How:**
1. Add `backend/app/services/webhook_service.py` (~180 LoC, pure stdlib). A `webhooks.json` registry (stored at the data directory root) holds an array of `{id: uuid4, url: str, events: ["simulation.complete", "simulation.error", "simulation.published"], created_at: ISO, active: bool}`. `register_webhook(url, events)` writes a new entry (validate URL with urllib.parse — must be https:// or http://localhost for local dev). `delete_webhook(webhook_id)` removes the entry. `deliver_event(event_type, payload, data_dir)` reads the registry, filters for webhooks subscribed to `event_type`, and for each: makes a POST request via `urllib.request.urlopen` with a JSON body `{event: str, timestamp: ISO, data: payload}` and a `X-MiroShark-Signature` HMAC-SHA256 header (key = webhook secret stored alongside the webhook entry). Delivery is fire-and-forget (no retry on failure — log failure to a `.webhook-errors.jsonl` file with the error, url, timestamp, and payload). Add delivery calls to `SimulationRunner` at three points: on simulation completion (`simulation.complete`), on simulation error (`simulation.error`), and when a sim is published via the publish endpoint (`simulation.published`). Add `POST /api/webhooks` (admin-gated: requires admin token; body: `{url, events, secret}`) and `DELETE /api/webhooks/{id}` (admin-gated). Add `GET /api/webhooks` (admin-gated) to list registered webhooks without exposing secrets. Return `{id, url, events, created_at, active}` per entry.
2. In the operator dashboard (`OperatorDashboardView.vue`), add a "Webhooks" tab alongside the existing simulation list. The tab shows a table of registered webhooks (url, subscribed events, created_at, active status). A "Register webhook" form at the top: URL field, event checkboxes (simulation.complete, simulation.error, simulation.published), secret field (optional; if empty, signature header is omitted). A "Test" button next to each registered webhook sends a `POST /api/webhooks/{id}/ping` (new endpoint) that delivers a `{event: "ping", timestamp: ..., data: {}}` test payload and returns `{success: bool, status_code: int, error?: str}` to confirm the URL is reachable. A red/green "Last delivery" status indicator per webhook (populated from `.webhook-errors.jsonl`). Add `registerWebhook(url, events, secret)`, `deleteWebhook(id)`, `listWebhooks()`, and `pingWebhook(id)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels throughout the tab.
3. Add `POST /api/webhooks`, `GET /api/webhooks`, `DELETE /api/webhooks/{id}`, and `POST /api/webhooks/{id}/ping` to the OpenAPI spec under a new "Webhooks" tag with `WebhookEntry` schema: `{id: str, url: str, events: [str], created_at: str, active: bool}`. Add 14 unit tests in `backend/tests/test_unit_webhooks.py`: register writes to webhooks.json, delete removes entry, register validates URL scheme (rejects ftp://), register accepts localhost for local dev, deliver_event skips webhooks not subscribed to that event_type, deliver_event sends HMAC-SHA256 signature when secret present, deliver_event omits signature header when no secret, delivery failure logged to .webhook-errors.jsonl (not raised), list_webhooks omits secret field, get_webhooks requires admin token (401 without), register requires admin token (401 without), delete of unknown id returns 404, ping endpoint calls deliver_event with event="ping", ping returns status_code from the target response. Add a "Webhooks" section to `docs/FEATURES.md`: explains the event types, the HMAC signature verification pattern (for receivers to authenticate payloads), the fire-and-forget delivery model, and the error log format.

### 2. Simulation Data Bundle Export

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Research teams running MiroShark simulations often want to download the complete data for offline analysis — their own visualization pipelines, academic datasets, or archival. Today they have to hit 8-10 different endpoints and assemble the files manually. `GET /api/simulation/{id}/bundle.zip` streams all of a simulation's data surfaces as a single ZIP archive: trajectory.json, agents.json, signal.json, confidence.json, platform_sentiment.json, stance_flips.json, mentions.json, signed-result.json, and the raw actions.jsonl. Pure stdlib (zipfile, io.BytesIO). No new computation — just packages what's already on disk. The ZIP becomes the "simulation as a data artifact" primitive that researchers can cite, archive, and pass to colleagues without requiring MiroShark access.

**How:**
1. Add `backend/app/services/bundle_service.py` (~90 LoC, pure stdlib). `generate_bundle(sim_dir, sim_id)` constructs an in-memory ZIP using `zipfile.ZipFile(io.BytesIO(), "w", zipfile.ZIP_DEFLATED)`. It iterates a fixed manifest of known surface files: `trajectory.json`, `agents.json`, `signal.json`, `confidence.json`, `confidence_trajectory.json`, `platform_sentiment.json`, `stance_flips.json`, `mentions.json`, `signed-result.json`, `actions.jsonl`, `simulation_config.json`. For each file in the manifest, check if `<sim_dir>/<filename>` exists — if yes, add it to the ZIP at path `<sim_id>/<filename>`. If the file doesn't exist, skip silently (older sims may not have every surface). After adding all files, add a `<sim_id>/MANIFEST.json` entry that lists `{generated_at: ISO, sim_id: str, included_files: [...], excluded_files: [...], bundle_format_version: "1.0"}`. Returns the BytesIO buffer seeked to position 0. Add `GET /api/simulation/{id}/bundle.zip` to `simulation.py` blueprint: publish-gated (only published sims), admin-gated for private. Response: `Content-Type: application/zip`, `Content-Disposition: attachment; filename="miroshark-{sim_id}.zip"`, `Cache-Control: public, max-age=300` (5-min cache — data doesn't change for published sims). Stream the BytesIO buffer directly. No disk caching — the bundle is generated on demand (fast: all source files are already cached on disk). Add `bundle` to `surfaces_catalog.py` as a `utility` type surface (40th).
2. In `Step3Simulation.vue`, add a "Download Bundle" button to the existing action toolbar (alongside the Share/Embed buttons). On click, trigger `window.location.href = '/api/simulation/{id}/bundle.zip'` — the browser handles the ZIP download natively (no frontend streaming logic needed). Show a brief toast notification "Downloading simulation data bundle…" while the download initiates. In `EmbedDialog.vue`, add a "Data Bundle" section at the bottom with the download URL and a curl example: `curl -O "https://your-instance/api/simulation/{id}/bundle.zip"`. Add `getBundleUrl(simId)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH label on the button ("Download Bundle / 下载数据包").
3. Add `GET /api/simulation/{id}/bundle.zip` to the OpenAPI spec under the "Simulation" tag. Response schema is binary (application/zip) — document the included files list and the MANIFEST.json format. Add 12 unit tests in `backend/tests/test_unit_bundle.py`: ZIP contains simulation_config.json when present, ZIP contains trajectory.json when present, missing surface file skipped (no error), MANIFEST.json always present in ZIP, MANIFEST.json lists included and excluded files, bundle response has correct Content-Disposition header, bundle response has Content-Type application/zip, publish-gate enforced (unpublished sim returns 403), admin token required for private sim, unknown sim_id returns 404, ZIP is valid and extractable (zipfile.ZipFile can open the BytesIO response), bundle_format_version is "1.0" in manifest. Add a "Simulation Data Bundle" section to `docs/FEATURES.md`: explains the manifest format, lists the included surface files, notes that missing surfaces are skipped silently (not all surfaces exist for every sim), and explains the 5-minute cache-control directive.

### 3. Simulation Comparison API

**Type:** Feature
**Effort:** Medium (1-2 days)
**Impact:** Researchers who run parameter sweeps — forking the same scenario with different agent counts, topics, or platforms — have no way to compare results without manually inspecting each simulation's surfaces. `GET /api/simulations/compare?a={id1}&b={id2}` computes the semantic delta between two published simulations: direction alignment (same/opposing), confidence delta, belief drift curve correlation (Pearson r), agent count delta, platform overlap, and the top-3 diverging archetypes (agents whose final stances differ most between the two runs). A `/compare?a=...&b=...` frontend page renders both simulations in a split layout with highlighted deltas. This is the "what changed between runs?" surface — the companion to both the fork lineage graph (structural) and the full-text search (discoverability).

**How:**
1. Add `backend/app/services/compare_service.py` (~200 LoC, pure stdlib). `compare_simulations(sim_dir_a, sim_dir_b, sim_id_a, sim_id_b)` reads both sims' data: `signal.json` (direction, yes_pct), `confidence.json` (confidence_score), `trajectory.json` (per-round yes_pct series), `agents.json` (agent list with profession and final_stance). Computes: (a) `direction_match: bool` — True if both have same direction, (b) `confidence_delta: float` — B.confidence_score - A.confidence_score, signed, (c) `belief_correlation: float|null` — Pearson r between the two trajectory yes_pct series (null if series lengths differ by >50% or either is absent; otherwise resample shorter to longer using linear interpolation on round index), (d) `agent_count_delta: int` — B.total_agents - A.total_agents, (e) `platform_overlap: {shared: [...], only_a: [...], only_b: [...]}` — from each sim's `state.json` platform list, (f) `diverging_archetypes: [{profession: str, stance_a: str, stance_b: str}]` — top 3 professions where the modal final stance differs between the two sims (requires cross-referencing both agents.json arrays by profession). Returns `{sim_a: {id, title, direction, confidence_score}, sim_b: {id, title, direction, confidence_score}, direction_match: bool, confidence_delta: float, belief_correlation: float|null, agent_count_delta: int, platform_overlap: {...}, diverging_archetypes: [...]}`. No disk cache (cheap to compute, must be live). Add `GET /api/simulations/compare` to `simulation.py` blueprint (query params: `a`, `b`; both required; both must be published sims; returns 400 if `a == b`; 404 if either sim not found; 403 if either sim is unpublished and caller lacks admin token).
2. Add a `CompareView.vue` at `/compare` (route added to `router/index.js`). URL params: `?a={sim_id_a}&b={sim_id_b}`. On mount, fetch `GET /api/simulations/compare?a=...&b=...`. Render a two-column split layout: left column is sim A, right column is sim B. At the top of each column: title, direction chip, confidence score. Below the split columns: a "Delta" section showing `confidence_delta` as a signed number with color (green if B > A, red if B < A), `direction_match` as a "Aligned ✓" or "Opposing ✗" chip, `belief_correlation` as "r = {value.toFixed(2)}" with a descriptive label ("Strong alignment", "Weak", "Inverse"), `agent_count_delta` as "+N agents" or "-N agents", and `platform_overlap` as three tag groups (Shared / Only in A / Only in B). Below: a "Diverging archetypes" table showing the 3 professions where stances differed. Add a "Compare" link in the simulation card context menu and a "Compare with…" button on the watch page (opens a modal to pick a second sim from the gallery). Add `compareSimulations(simIdA, simIdB)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels throughout.
3. Add `GET /api/simulations/compare` to the OpenAPI spec under a "Comparison" tag with `CompareResponse` schema: full structure as described above. Add 14 unit tests in `backend/tests/test_unit_compare.py`: direction_match true when both BULLISH, direction_match false when opposing, confidence_delta is signed (B - A), belief_correlation computed from two equal-length series, belief_correlation null when trajectory absent, belief_correlation null when series lengths differ >50%, agent_count_delta is signed (B - A), platform_overlap correctly partitions into shared/only_a/only_b, diverging_archetypes max 3 entries, diverging_archetypes only includes professions present in both sims, `a == b` returns 400, unpublished sim returns 403, unknown sim_id returns 404, both missing returns 404 (not 400). Add a "Simulation Comparison" section to `docs/FEATURES.md`: explains the Pearson r computation and resampling approach (why series length must be within 50%), explains diverging archetypes (modal stance per profession — not individual agent comparison), and gives a worked example of the platform_overlap output.

### 4. API Rate Limiting & Usage Headers

**Type:** Performance / Security
**Effort:** Small (hours)
**Impact:** MiroShark's REST API is public-facing with 14 known ecosystem integrators and 1,270 star-driven inbound curiosity. There is no rate limiting. A misbehaving client (bot, misconfigured cronjob, aggressive scraper) can currently hammer any endpoint at unlimited frequency — degrading performance for all users and raising hosting costs. A pure-stdlib sliding-window rate limiter (no Redis, no external deps) on a per-IP basis returns standard `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` headers on every response, and a 429 Too Many Requests when the limit is exceeded. The default limit (e.g., 100 requests per IP per minute) is configurable via environment variable. This is the production hardening baseline that should exist before any SLA-aware ecosystem partner integrates against the API.

**How:**
1. Add `backend/app/middleware/rate_limit.py` (~120 LoC, pure stdlib). Implements a sliding-window counter using a module-level dict `_windows: dict[str, list[float]]` (IP → list of request timestamps). `check_rate_limit(ip, limit, window_seconds)` appends `time.time()` to `_windows[ip]`, prunes entries older than `window_seconds`, and returns `(allowed: bool, remaining: int, reset_at: float)`. `limit` defaults to `int(os.environ.get("RATE_LIMIT_RPM", "100"))` (requests per minute); `window_seconds = 60`. The dict is in-memory: resets on process restart (acceptable for a single-process deployment; add a note in docs for multi-worker setups). For multi-process (gunicorn workers), document that a shared counter requires Redis — but the middleware falls back gracefully if `RATE_LIMIT_DISABLED=true` is set. The IP is extracted from `request.remote_addr` (with X-Forwarded-For support: `request.headers.get("X-Forwarded-For", request.remote_addr).split(",")[0].strip()`). Add the middleware as a Flask `before_request` + `after_request` pair registered in `create_app()`. `before_request` runs `check_rate_limit`; if `allowed` is False, returns early with `jsonify({"error": "Rate limit exceeded", "retry_after": reset_at})`, status 429, and the three rate-limit headers. `after_request` appends `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` to every response (even 429). Exempt `/api/health` and `/api/health/status` from rate limiting (monitoring probes shouldn't be throttled).
2. No frontend changes needed — rate limiting is transparent to the browser. In `EmbedDialog.vue`, update the curl examples to include a note: "Rate limit: {RATE_LIMIT_RPM} requests/minute per IP. Check X-RateLimit-Remaining in response headers." In the OpenAPI spec, add the three `X-RateLimit-*` headers to the global response headers section for all paths, and document the 429 response schema `{error: "Rate limit exceeded", retry_after: float}` as a shared response component `RateLimitExceeded`. Also add the `RATE_LIMIT_RPM` and `RATE_LIMIT_DISABLED` env vars to `docs/CONFIGURATION.md` (or create it if absent) with default values and the multi-worker caveat.
3. Add 12 unit tests in `backend/tests/test_unit_rate_limit.py`: request within limit returns allowed=True, request at limit returns remaining=0 and allowed=True, request over limit returns allowed=False, old requests pruned from window (timestamp > window_seconds ago), X-RateLimit-Limit header present on 200 response, X-RateLimit-Remaining decrements with each request, X-RateLimit-Reset header is a future Unix timestamp, 429 response includes retry_after field, 429 response has all three X-RateLimit headers, /api/health exempt from rate limiting (no 429 even at 200+ requests), RATE_LIMIT_DISABLED=true bypasses check (all requests allowed), X-Forwarded-For respected for IP extraction (comma-separated, first IP used). Add a "Rate Limiting" section to `docs/FEATURES.md`: explains the sliding-window algorithm (timestamps in memory), the default limit, the RATE_LIMIT_DISABLED escape hatch, the multi-worker caveat (in-process dict is per-worker), and how to verify the headers with curl (`curl -I`).

### 5. 24-Hour Activity Digest Endpoint

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Ecosystem partners, newsletter operators, and Discord/Slack bots need a machine-readable summary of what happened on the platform in the last 24 hours — without building their own aggregation logic across multiple endpoints. `GET /api/digest.json` returns a pre-aggregated digest: new publications count, top simulation by confidence, top simulation by embed hits, most common final direction (BULLISH/BEARISH), average confidence across all newly published sims, and a list of the 5 most recent publications (each with id, title, direction, confidence_score, created_at). Generated on demand, cached for 1 hour. This is the single-call daily briefing that a Telegram bot, a newsletter plugin, or an ecosystem partner's analytics dashboard can call every morning to "catch up" on the platform — without hitting activity.json, leaderboard, and gallery separately.

**How:**
1. Add `backend/app/services/digest_service.py` (~130 LoC, pure stdlib). `generate_digest(data_dir, window_hours=24)` scans all published sim dirs (same scandir pattern as leaderboard_service). For each sim, reads `simulation_config.json` for `created_at` — only include sims where `created_at` is within the last `window_hours`. From the filtered set: `new_count: int` (count), `top_by_confidence: {id, title, direction, confidence_score}` (highest `confidence_score` from `confidence.json`), `top_by_embed_hits: {id, title, embed_hits}` (highest `embed_hits` from `surface-stats.json`), `avg_confidence: float` (mean confidence_score across filtered sims, null if no sims), `direction_majority: str|null` (most common direction across filtered sims, null if tie or no sims), `recent_publications: [{id, title, direction, confidence_score, created_at}]` (5 most recent by created_at). Assembles `{window_hours: 24, generated_at: ISO, new_count: int, avg_confidence: float|null, direction_majority: str|null, top_by_confidence: {...}|null, top_by_embed_hits: {...}|null, recent_publications: [...]}`. Caches to `<data_dir>/digest.json` with 1-hour mtime-based invalidation. `get_digest(data_dir)` reads cache if fresh, else calls `generate_digest`. Add `GET /api/digest.json` (no auth required — public) to `simulation.py` blueprint. Returns the payload with `Cache-Control: public, max-age=3600`. Add `digest` to `surfaces_catalog.py` as a `utility` type surface (41st). Track hit count via `surface_stats.py`.
2. In `ExploreView.vue` (the gallery), add a compact "Today at a glance" banner at the top of the page (above the search bar and filter tabs). Fetch `GET /api/digest.json` on mount. Render as a horizontal strip of 4 stat chips: "N new today", "Avg confidence: X%", "Leaning: BULLISH/BEARISH/Mixed", "Top: {top_by_confidence.title} ({direction})". Each chip is small and dismissible (localStorage flag). If `new_count === 0`, hide the banner entirely. This gives returning visitors an instant sense of platform activity without disrupting the gallery layout. Add `getDigest()` to `frontend/src/api/simulation.js`. In `EmbedDialog.vue`, add a "Platform Digest" section at the end with the endpoint URL and a curl example. Bilingual EN/ZH labels on all banner chips.
3. Add `GET /api/digest.json` to the OpenAPI spec under the "Discovery" tag with `DigestResponse` schema: full structure as described above. Add 12 unit tests in `backend/tests/test_unit_digest.py`: sim published 23h ago included, sim published 25h ago excluded, new_count reflects only within-window sims, top_by_confidence picks highest confidence_score sim, top_by_embed_hits picks highest embed_hits sim, avg_confidence is mean across filtered sims (float, not rounded), direction_majority returns most common direction (BULLISH wins if 3 BULLISH 1 BEARISH), direction_majority is null on exact tie, recent_publications sorted by created_at desc, recent_publications capped at 5, cache written on first call, cache reused when mtime < 1h. Add a "Platform Digest" section to `docs/FEATURES.md`: explains the 24-hour window (UTC midnight-to-now, not rolling), notes the 1-hour cache (so the digest lags behind by up to 1 hour), and gives a worked example of the direction_majority tie-breaking (null — explicit, not a coin flip, so downstream consumers can render "Mixed" without ambiguity).

## Selection Rationale

Five integration and platform-hardening ideas — none overlapping with the 39 push-blocked builds, the three open PRs (Contributing guide, same-origin API fix, SECURITY.md), or the five ideas from June 12 (Agent Stance Flip Report, Simulation Full-Text Search, Confidence Component Breakdown, Simulation Fork Lineage Graph, Per-Round Agent Participation Heatmap):

- **Webhook Delivery** (#1) — The push-based integration primitive the 14 ecosystem projects don't have. Activity.json polling works until you need sub-minute latency or reliable event delivery. Webhooks solve both. HMAC signature verification makes the delivery trustworthy for downstream systems.

- **Simulation Data Bundle Export** (#2) — Research teams and ecosystem partners building their own analysis pipelines need the raw data files without hitting 8 endpoints. A ZIP bundle is the "simulation as a self-contained data artifact" — citable, archivable, shareable without requiring MiroShark access. Pure stdlib (zipfile), zero new computation.

- **Simulation Comparison API** (#3) — Parameter sweep research is a primary use case for fork (already supported). But the "what changed between my baseline and fork?" question has no API surface. The comparison endpoint computes confidence delta, trajectory correlation, and diverging archetypes — the three signals that tell a researcher whether the fork produced meaningful variation or converged to the same result.

- **API Rate Limiting & Usage Headers** (#4) — Fourteen integrators and growing public exposure mean this is no longer optional. A sliding-window in-memory rate limiter with standard X-RateLimit-* headers is the minimum production baseline. Zero external dependencies, configurable via env var, exempts health probes.

- **24h Activity Digest** (#5) — A single endpoint that aggregates what happened on the platform today: new publications, average confidence, direction majority, top sims. Newsletter bots, Discord integrations, and ecosystem partner dashboards need exactly this — not a custom join across activity.json + leaderboard + gallery. Cache-Control: max-age=3600 makes it cheap to call from any downstream consumer.
