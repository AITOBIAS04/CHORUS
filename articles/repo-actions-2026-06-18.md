### 1. Agent Influence Leaderboard

**Type:** Feature
**Effort:** Small (hours)
**Impact:** `agents.json` already contains an `influence_score` field for every agent — but there is no API surface or UI panel that exposes it. `GET /api/simulation/:id/influence-rank` returns agents sorted by `influence_score` descending, with their `profession`, `final_bullish_pct`, `stance_flip_count`, and a derived `influence_tier` ("high" / "mid" / "low" based on score terciles across this simulation's agent pool). A "Key Voices" panel on the watch page shows the top-5 influencers with gold/silver/bronze rank badges. This answers the most natural post-simulation question — "who drove this outcome?" — using data that is already computed but invisible.

**How:**
1. Add `backend/app/services/influence_rank_service.py` (~100 LoC, pure stdlib). `get_influence_rank(sim_dir)` reads `agents.json`, extracts the full agent list, sorts by `influence_score` descending. Computes `influence_tier` per agent: top tercile = "high", middle = "mid", bottom = "low" (based on tercile boundaries across the full agent pool in this simulation). Returns `{generated_at: ISO, sim_id: str, total_agents: int, agents: [{rank: int, name: str, profession: str, influence_score: float, influence_tier: str, final_bullish_pct: float, stance_flip_count: int, final_stance: str}]}`. Caches to `<sim_dir>/influence_rank.json` with 1-hour mtime invalidation (the source agents.json doesn't change after completion). Add `GET /api/simulation/:id/influence-rank` to `simulation.py` blueprint (publish-gated; admin token for private sims). Response: `Cache-Control: public, max-age=3600`. Add `influence_rank` to `surfaces_catalog.py` as a `simulation` type surface. Add `influence_rank_hits` tracking to `surface_stats.py`. Add 10 unit tests in `backend/tests/test_unit_influence_rank.py`: agents sorted by influence_score desc, rank starts at 1, influence_tier "high" assigned to top tercile, influence_tier "low" to bottom tercile, ties in influence_score preserve stable sort (by name), total_agents count matches agents.json length, publish-gate enforced (unpublished sim returns 403), unknown sim_id returns 404, cache written on first call, cache reused when mtime < 1h.
2. In `Step3Simulation.vue`, add a "Key Voices" toolbar button (person icon) alongside the existing "@ Mentions" and "↻ Stance Shifts" buttons. On click, fetch `GET /api/simulation/:id/influence-rank` and render a dark-themed overlay panel. Layout: title "Key Voices" + "Top agents by influence score". Top row: the top 3 agents as large cards with gold/silver/bronze badge, name, profession chip, influence score bar (violet, relative to max in this simulation), and final stance chip (green BULLISH / red BEARISH). Below: a scrollable table of all agents ranked — rank, name, profession, influence_score, stance, flip count. Each row is clickable (no action yet — future: highlight that agent in the trajectory chart). Add `getInfluenceRank(simId)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels throughout the panel.
3. Add `GET /api/simulation/:id/influence-rank` to the OpenAPI spec under the "Simulation" tag with `InfluenceRankResponse` schema: full structure as described above. Add an "Agent Influence Leaderboard" section to `docs/FEATURES.md`: explains how `influence_score` is computed (reference the simulation engine docs for the derivation — don't restate it), what the tercile-based tier assignment means, and why the cache is set to 1 hour (agents.json is immutable post-completion). Add `influence-rank` as a surface in `surfaces.json`.

### 2. Simulation Time-to-Complete Estimator

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The MiroShark homepage promises "for $1 & less than 10 min" — but there is no pre-run estimate. Users submit a simulation configuration without knowing if they'll get a result in 2 minutes or 20, or whether it will cost $0.20 or $2.00. `GET /api/estimate?agents=N&rounds=N&platforms=P` returns `{estimated_duration_seconds: int, estimated_cost_usd: float, confidence: "low"|"medium"|"high"}` derived from median actual durations and costs across completed simulations with similar parameters (bucketed by agent count and round count). Shown in the simulation setup form before submission with a "Estimated: ~3 min · ~$0.40" label. Bridges the gap between the promise and the user's live experience.

**How:**
1. Add `backend/app/services/estimate_service.py` (~120 LoC, pure stdlib). `build_estimate_index(data_dir)` scans all completed published sim dirs. For each, reads `state.json` (`started_at`, `completed_at` — derive `duration_seconds`), `simulation_config.json` (`agent_count`, `num_rounds`, `platforms`), and `cost.json` (`total_cost_usd` if present). Returns a list of `{agent_count, num_rounds, platform_count, duration_seconds, cost_usd}` records. Caches to `<data_dir>/estimate_index.json` with 15-minute mtime invalidation. `compute_estimate(agent_count, num_rounds, platform_count, index)` filters the index to records within ±30% of the requested agent_count and ±1 of num_rounds. If ≥5 matches: compute median `duration_seconds` and median `cost_usd` from the filtered set, set confidence="high". If 2–4 matches: compute medians, confidence="medium". If <2 matches: fall back to the global median of all records, confidence="low". Returns `{estimated_duration_seconds: int, estimated_cost_usd: float, confidence: str, sample_size: int}`. If the index has 0 records (fresh install), return `{estimated_duration_seconds: null, estimated_cost_usd: null, confidence: "unavailable", sample_size: 0}`. Add `GET /api/estimate` to `simulation.py` blueprint (unauthenticated; query params: `agents` int required, `rounds` int required, `platforms` int optional default 1). Validate: agents 1–200, rounds 1–20, platforms 1–5; return 400 with field-specific error on invalid input. Response: `Cache-Control: public, max-age=900`. Add 10 unit tests in `backend/tests/test_unit_estimate.py`: ≥5 matching records returns confidence="high", 3 matching records returns confidence="medium", 0 matching records returns confidence="low" with global medians, fresh install returns confidence="unavailable", median computed correctly from odd-count list, median computed correctly from even-count list (avg of middle two), agents param validation rejects 0 and 201, rounds param validation rejects 0 and 21, estimate_index written on first call and reused within 15 min, null cost_usd records excluded from cost median (not counted as $0).
2. In the simulation setup form (`SimulationSetupView.vue` or equivalent — wherever agent count, round count, and platform selection are configured), add an `EstimateChip` component that fetches `GET /api/estimate` whenever the form values change (debounced 500ms). Render inline below the agent/round input fields: "Estimated: ~{N} min · ~${cost.toFixed(2)}" with a small confidence indicator (a circle: green=high, yellow=medium, gray=low; hide when confidence="unavailable"). On hover/tap, expand to show the full estimate card: duration range, cost range, sample size ("Based on N similar simulations"). If confidence="unavailable", show "No estimate available yet — run a simulation to build history." Add `getEstimate(agents, rounds, platforms)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels on the chip and expanded card.
3. Add `GET /api/estimate` to the OpenAPI spec under a new "Utilities" tag with `EstimateResponse` schema: `{estimated_duration_seconds: int|null, estimated_cost_usd: float|null, confidence: "high"|"medium"|"low"|"unavailable", sample_size: int}`. Add a "Pre-Run Time & Cost Estimate" section to `docs/FEATURES.md`: explains the ±30% agent_count bucketing, the ±1 round bucketing, the confidence levels, and why confidence="unavailable" means no completed simulations match (not a system error). Add a note: the estimator requires `cost.json` data (shipped in #179) — instances that never had cost.json will show null cost estimates.

### 3. Simulation Replay Stepper

**Type:** Feature
**Effort:** Medium (1-2 days)
**Impact:** `trajectory.json` contains per-round consensus data: `yes_pct`, `agent_stances`, and round metadata. The watch page's BeliefDriftChart renders the full trajectory as a static line chart. A "Replay" control adds a play/pause button and a round scrubber slider to the chart. On play, the chart renders one round at a time at configurable speed (0.5×, 1×, 2×), with the current round's `yes_pct` and majority stance label updating in real time. Users can pause, scrub to any round, and replay. The feature turns a static result into a walkthrough — operators presenting simulations can step through the consensus formation story round by round rather than pointing at a completed chart.

**How:**
1. No new backend endpoints needed — `trajectory.json` already contains all per-round data. Add a lightweight `useReplay(trajectoryData)` composable in `frontend/src/composables/useReplay.js` (~60 LoC). State: `currentRound: ref(0)`, `isPlaying: ref(false)`, `speed: ref(1)` (1 = 1 round per second). `play()` starts a `setInterval` that increments `currentRound` by 1 every `(1000 / speed.value)` ms, stops when `currentRound >= trajectory.length - 1`. `pause()` clears the interval. `seek(round)` sets `currentRound` directly. `reset()` sets `currentRound = 0` and pauses. Exposes `currentRound` (readonly), `isPlaying`, `roundData` (computed: `trajectory[currentRound]`), `progress` (computed: `currentRound / (trajectory.length - 1)`). The composable is purely UI-side — no API calls, no backend changes.
2. In `Step3Simulation.vue`, add replay controls below the BeliefDriftChart (visible only when simulation status is "complete" and `trajectory.json` has been loaded). Controls: a "Replay" button (triangle/play icon) that initializes `useReplay(trajectoryData)` and calls `play()`; a pause button (shown when `isPlaying`); a round scrubber slider (HTML range input, min=0, max=trajectory.length-1, value=currentRound); a speed selector (0.5× / 1× / 2× toggle buttons); a round counter "Round {currentRound + 1} of {trajectory.length}". When `useReplay` is active, pass `currentRound` to BeliefDriftChart as a new `replayUpTo` prop: the chart renders only rounds 0 through `replayUpTo` (slice the trajectory data series). The final `yes_pct` label and majority-stance chip above the chart update to reflect `roundData.yes_pct` and the consensus at that round. When replay reaches the last round (or is paused at the last round), the controls show a "Restart" button instead of play. Add a `replayUpTo` prop to `BeliefDriftChart.vue`: if null/undefined, render full trajectory (existing behavior — no regression for non-replay usage). Bilingual EN/ZH labels on all controls.
3. Add 8 unit/component tests for the `useReplay` composable in `frontend/src/composables/__tests__/useReplay.test.js` (using Vitest): `play()` increments `currentRound` over time (mock `setInterval`), `pause()` stops incrementing, `seek(N)` sets `currentRound` to N directly, `seek` clamps to [0, trajectory.length-1] (no out-of-bounds), `reset()` sets `currentRound` to 0 and `isPlaying` to false, `play()` stops automatically when `currentRound` reaches `trajectory.length - 1`, `speed` of 2 halves the interval duration, `progress` is 0 at round 0 and 1 at last round. Add a "Simulation Replay" section to `docs/FEATURES.md`: explains the three speed settings, notes that replay requires a completed simulation with trajectory data, and explains why there is no backend call (all data is client-side from the already-loaded trajectory).

### 4. Per-Round Cost Curve

**Type:** Feature
**Effort:** Small (hours)
**Impact:** PR #179 shipped `GET /api/simulation/:id/cost.json` — operators can now see total simulation cost. What they can't see is *when* the cost accumulates. Multi-round simulations may front-load cost in early rounds (agent reasoning) or back-load it in synthesis. `GET /api/simulation/:id/cost/rounds` returns a per-round cost breakdown: `[{round: 1, round_cost_usd: float, cumulative_cost_usd: float}]`. A small stacked-bar chart is added to the cost panel on the watch page. Operators running high-round parameter sweeps can identify which rounds are expensive and tune their configuration — number of agents per round, synthesis depth — before committing to a full run.

**How:**
1. Add `backend/app/services/cost_rounds_service.py` (~80 LoC, pure stdlib). `get_cost_rounds(sim_dir)` reads `cost.json` (the per-call cost log from #179 — assume it has a `calls: [{round: int, cost_usd: float, model: str, tokens: int}]` array structure; if it has a flat total, aggregate from per-call records if available, else return a single-round entry). Groups calls by `round`, sums `cost_usd` per round, computes cumulative. Returns `{sim_id: str, generated_at: ISO, total_cost_usd: float, rounds: [{round: int, round_cost_usd: float, cumulative_cost_usd: float, call_count: int}]}`. If `cost.json` does not contain per-round data (i.e., flat total only), returns `{rounds: null, note: "Per-round breakdown requires cost.json with per-call records."}` — this gracefully handles simulations run before the per-round cost logging is added. Add `GET /api/simulation/:id/cost/rounds` to `simulation.py` blueprint (publish-gated). Response: `Cache-Control: public, max-age=3600`. Add 10 unit tests in `backend/tests/test_unit_cost_rounds.py`: calls grouped by round correctly, cumulative computed as running sum, single-round simulation returns one entry, call_count per round is correct, total_cost_usd matches sum of round_cost_usd values, missing cost.json returns 404, flat cost.json (no per-call records) returns rounds=null with note, publish-gate enforced (403 on unpublished sim), unknown sim_id returns 404, rounds sorted by round number asc.
2. In `Step3Simulation.vue` (or the cost display component, wherever the total cost from `GET /api/simulation/:id/cost.json` is rendered), add a "Cost by round" expandable section below the total cost. When expanded, fetch `GET /api/simulation/:id/cost/rounds`. If `rounds` is non-null: render a horizontal stacked bar chart (one bar per round, height proportional to `round_cost_usd`, a thin cumulative line overlaid, round labels on the x-axis). Color: violet bars for round cost, dashed gray line for cumulative. Label each bar with `$round_cost_usd.toFixed(4)` on hover. If `rounds` is null: show the message from `note`. The section is collapsed by default (expand on click). Add `getCostRounds(simId)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels on the section header and axis labels.
3. Add `GET /api/simulation/:id/cost/rounds` to the OpenAPI spec under the "Simulation" tag. Add a `CostRoundsResponse` schema: `{sim_id: str, generated_at: str, total_cost_usd: float, rounds: [{round: int, round_cost_usd: float, cumulative_cost_usd: float, call_count: int}]|null, note: str|null}`. Add a "Per-Round Cost Breakdown" section to `docs/FEATURES.md`: explains the data source (requires per-call records in cost.json), what `call_count` represents (number of LLM API calls in that round), and why `rounds: null` is not an error (it means the simulation ran before per-call cost logging was enabled).

### 5. Topic Trend Tracker

**Type:** Feature
**Effort:** Small (hours)
**Impact:** There is no discovery surface that shows which simulation topics are gaining momentum over time. `GET /api/topics/trends` aggregates simulation title keywords across published simulations into weekly counts for the past 30 days, returning the top-10 rising topics as `[{word: str, weekly_counts: [int, int, int, int], trend: "rising"|"falling"|"stable"}]` — one count per week, four weeks. An ExploreView "Trending" tab shows these as sparkline chips. Ecosystem partners and community operators can identify which topics are getting attention on the platform without scanning individual simulations. Pure stdlib aggregation of existing data — no new computation.

**How:**
1. Add `backend/app/services/topic_trends_service.py` (~120 LoC, pure stdlib). `compute_topic_trends(data_dir, weeks=4)` scans all published sim dirs. For each, reads `simulation_config.json` for `title` and `created_at`. Groups simulations into 4 weekly buckets (week N = simulations created between `now - (N+1)*7 days` and `now - N*7 days`; week 0 = most recent). For each bucket: tokenizes all titles (lowercase, strip punctuation, exclude stopwords — same list as the Usage Analytics service if it exists, else inline: ["a","the","of","to","for","and","in","is","an","with","this","that","from","on","at","by","how","what","why","will","are","was","be","has","have","it"]). Counts word frequency per bucket. For each word that appears in at least 1 simulation across all buckets: compute `weekly_counts: [count_week3, count_week2, count_week1, count_week0]` (oldest first, most recent last). Compute `trend`: "rising" if `count_week0 > count_week3` (most recent week higher than oldest), "falling" if `count_week0 < count_week3`, else "stable". Sort all words by `weekly_counts[-1]` (most recent week) desc, cap at top 10. Return `{generated_at: ISO, window_weeks: 4, topics: [{word: str, weekly_counts: [int, int, int, int], trend: str}]}`. Cache to `<data_dir>/topic_trends.json` with 1-hour mtime invalidation. Add `GET /api/topics/trends` (unauthenticated; public) to a new `topics.py` blueprint registered in `create_app()`. Response: `Cache-Control: public, max-age=3600`. Add `topic_trends` to `surfaces_catalog.py` as a `discovery` type surface.
2. In `ExploreView.vue`, add a "Trending" tab in the filter tab strip (alongside "All", "Recent", "Top" or equivalent). On selecting the Trending tab, fetch `GET /api/topics/trends`. Render a horizontal scrollable row of topic chips at the top of the gallery. Each chip: the keyword in bold, a 4-bar mini sparkline (24px tall, bars colored violet), and a small trend arrow (↑ green for rising, ↓ red for falling, — gray for stable). Clicking a chip filters the simulation gallery to simulations whose title contains that keyword (client-side filter against the already-loaded gallery data, using a simple `title.toLowerCase().includes(word)` check). If `topics` is empty, show "Not enough data yet — run more simulations to see trends." Add `getTopicTrends()` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels on the tab and empty state. The chip count text ("N simulations this week") is optional — add if space permits.
3. Add `GET /api/topics/trends` to the OpenAPI spec under the "Discovery" tag with `TopicTrendsResponse` schema. Add 12 unit tests in `backend/tests/test_unit_topic_trends.py`: stopwords excluded from word list, weekly_counts has exactly 4 entries per topic, most recent week is index 3 (last), trend "rising" when week0 > week3, trend "falling" when week0 < week3, trend "stable" when equal, result capped at 10 topics, sorted by most recent week count desc (not total count), topics without any matches across all 4 weeks excluded, cache written on first call, cache reused when mtime < 1h, empty sim dir (no published sims) returns empty topics list.

## Selection Rationale

Five ideas that extend what just shipped rather than introducing orthogonal primitives:

- **Agent Influence Leaderboard** (#1) — `influence_score` is already in agents.json. No API surface exposes it. The "who drove this outcome?" question is the most natural post-simulation follow-up — this answers it with data that is already there.

- **Time-to-Complete Estimator** (#2) — cost.json landed in #179. The estimator closes the loop on the homepage promise: "for $1 & less than 10 min" is now verifiable before you run, not just after. The confidence=unavailable path handles fresh installs gracefully.

- **Simulation Replay Stepper** (#3) — trajectory.json has round-by-round consensus data. The chart shows the completed picture; replay shows the story. No backend endpoints required — purely a composable + BeliefDriftChart prop. The round scrubber is the right UX primitive for presenting simulations live.

- **Per-Round Cost Curve** (#4) — cost.json shows the total; the rounds endpoint shows where it accumulated. Operators running multi-round parameter sweeps need both. The `rounds: null` graceful path ensures backward compatibility with simulations run before per-round cost logging existed.

- **Topic Trend Tracker** (#5) — The 14 ecosystem projects and the ExploreView gallery have no "what's hot this week?" signal. Pure aggregation of existing titles into weekly counts. The sparkline chip is the zero-friction way to filter by rising topic without building a search query.
