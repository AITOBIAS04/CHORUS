## Context

Three things changed since June 14.

First: `GET /api/simulation/<id>/cost.json` shipped (PR #179). MiroShark can now tell you what a simulation cost. That's a read surface — it answers "what did I spend?" What doesn't exist yet is the control surface: "stop me when I'm about to spend too much." Budget enforcement is the obvious next primitive on top of the cost endpoint.

Second: SearXNG/Firecrawl websearch integration shipped (#178) and `pywebpush` was bumped to 2.3.0 via Dependabot (#173). The push library is already in the dependency tree. Web Push notification delivery for simulation completion is a natural ship — pywebpush is ready, the trigger point (SimulationRunner completion) already exists, and the UX gap (users needing to watch the tab) is real for 10-20 minute high-agent-count runs.

Third: community issue #161 is open requesting additional translations. MiroShark supports EN, ZH-CN, and FR — but there's no structured contribution path for new languages. "Please add German" as an issue comment is an invitation without a door. A translation coverage endpoint and scaffold turns it into a pull request.

The other two ideas — Operator Usage Analytics and Simulation RSS Feed — are both outside the 7-day window and still unbuilt. Usage analytics gives operators signal on what their users are actually simulating; the RSS feed is the zero-code integration primitive for every newsletter operator and Discord bot in the 14-project ecosystem.

None of today's five overlap with open PRs, the 39+ push-blocked builds, or the ideas from the last 7 days (Jun 14: Webhook Delivery, Data Bundle Export, Comparison API, Rate Limiting, 24h Digest; Jun 12: Stance Flips, Full-Text Search, Confidence Components, Fork Lineage, Participation Heatmap).

### 1. Simulation Cost Budget & Alerts

**Type:** Feature
**Effort:** Small (hours)
**Impact:** PR #179 shipped `GET /api/simulation/<id>/cost.json` — operators can now see what a simulation cost. What they can't do is *set* a spending ceiling before they run one. A budget registry (`POST /api/simulation/{id}/budget`) stores a max-cost threshold per simulation. The SimulationRunner reads the budget on each round-complete cycle; if `running_cost_usd` crosses 80% of the limit, it logs a warning and injects a budget-warning event into `activity.json`. At 100%, it halts the run cleanly (graceful stop: trajectory and agents.json are written, status is set to `budget_exceeded`). The watch page shows a live budget bar (current spend / limit) in the operator panel when a budget is set. For operators running parameter sweeps or high-agent-count stress tests, this is the difference between a $0.80 run and a $12 runaway. No new computation — just a budget ceiling over the already-flowing cost data.

**How:**
1. Add `backend/app/services/budget_service.py` (~80 LoC, pure stdlib). `set_budget(sim_dir, max_usd)` writes `{max_usd: float, warn_at_pct: 0.8, set_at: ISO}` to `<sim_dir>/budget.json`. `get_budget(sim_dir)` reads it (returns `None` if absent). `check_budget(sim_dir, current_cost_usd)` reads `budget.json` and returns `{status: "ok"|"warning"|"exceeded", current_usd: float, max_usd: float, pct: float}`. Add `POST /api/simulation/{id}/budget` (admin-gated; body: `{max_usd: float}`; validates `max_usd > 0`; 409 if simulation is already complete). Add `GET /api/simulation/{id}/budget` (admin-gated; returns current budget status with `current_cost_usd` read live from `cost.json` if present, else `0.0`). In `SimulationRunner`, after each round, call `check_budget(sim_dir, current_cost_usd)`. If `status == "warning"`, append a `{type: "budget_warning", pct: float}` entry to `activity.json`. If `status == "exceeded"`, write `status: "budget_exceeded"` to `state.json` and break the simulation loop (same cleanup path as a normal completion: write trajectory.json, agents.json, signal.json). Add 10 unit tests: `set_budget` writes file, `get_budget` returns None when absent, `check_budget` returns "ok" when below warn_at_pct, returns "warning" at 80%, returns "exceeded" at 100%, `set_budget` on completed sim returns 409, budget.json present but no cost.json returns pct=0.0, warn_at_pct default is 0.8, budget bar pct is clamped to [0,1] in output, 404 for unknown sim.
2. In `Step3Simulation.vue` (operator view only), add a compact budget bar below the simulation cost display. If `GET /api/simulation/{id}/budget` returns a budget, render: a thin progress bar (green → yellow → red as pct rises), "Current: ${current_usd.toFixed(4)} / ${max_usd.toFixed(4)}" label, and a status chip ("On track / Warning / Stopped: budget exceeded"). If no budget is set and the user is an operator, show a small "Set budget" link that opens a modal with a single number input (max $USD) and a "Set limit" button. If the simulation is in `budget_exceeded` status, show a persistent banner: "Simulation stopped at budget limit — partial results below." Add `setBudget(simId, maxUsd)`, `getBudget(simId)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels throughout. Poll `getBudget` every 5s while simulation is running (same polling interval as status).
3. Add `POST /api/simulation/{id}/budget` and `GET /api/simulation/{id}/budget` to the OpenAPI spec under the "Operator" tag with `BudgetResponse` schema: `{max_usd: float, warn_at_pct: float, current_usd: float, pct: float, status: "ok"|"warning"|"exceeded", set_at: str}`. Add a "Simulation Cost Budget" section to `docs/FEATURES.md`: explains the 80% warning threshold, the graceful-stop behavior on budget_exceeded (partial results preserved), and why a `budget_exceeded` simulation produces valid (if truncated) output — it's not an error state, it's an intentional early stop.

### 2. Web Push Notification for Simulation Completion

**Type:** Feature
**Effort:** Small (hours)
**Impact:** `pywebpush` (≥2.3.0) is already in the backend dependency tree — bumped by Dependabot PR #173. The missing piece is the subscription UI and delivery trigger. A "Notify me when done" button on the watch page calls `POST /api/push/subscribe` with the browser's PushSubscription object (endpoint + keys). When the SimulationRunner completes, it calls the push service to deliver a web push notification to all subscribed browsers for that simulation: "Simulation complete — {direction} at {confidence}% confidence." Users close the tab and get a native browser notification when their simulation finishes — no polling, no email, no staying on the page. This is the UX primitive that enables MiroShark to be used for long-running high-agent-count simulations without the user babysitting the tab. The infrastructure is already present; this is the wiring.

**How:**
1. Add `backend/app/services/push_service.py` (~100 LoC). `register_subscription(sim_dir, subscription_json)` appends the browser PushSubscription object (endpoint, keys.auth, keys.p256dh) to `<sim_dir>/push_subscriptions.jsonl` with a `subscribed_at` timestamp. `deliver_completion_push(sim_dir, sim_id, direction, confidence_pct, data_dir)` reads all subscription entries from `push_subscriptions.jsonl`, constructs the payload: `{title: "MiroShark — Simulation Complete", body: "{direction} signal · {confidence_pct.toFixed(0)}% confidence", url: "/simulation/{sim_id}"}`, and for each subscription calls `pywebpush.webpush(subscription_info=entry, data=json.dumps(payload), vapid_private_key=os.environ["VAPID_PRIVATE_KEY"], vapid_claims={"sub": "mailto:noreply@miroshark.app"})`. Failed deliveries (expired subscription, HTTP 410 Gone) are removed from the subscriptions file. Add `POST /api/simulation/{id}/push/subscribe` (unauthenticated; body: standard PushSubscription JSON `{endpoint, keys: {auth, p256dh}}`). Add `GET /api/push/vapid-public-key` (unauthenticated; returns `{public_key: str}` — the VAPID public key needed by the browser to create a subscription). In `SimulationRunner`, after the simulation completes successfully, call `deliver_completion_push`. In `create_app()`, add a startup check: if `VAPID_PRIVATE_KEY` and `VAPID_PUBLIC_KEY` env vars are set, enable the push feature; if absent, the subscribe endpoint returns `{enabled: false}`. Generate VAPID keys with `pywebpush` and document in `docs/CONFIGURATION.md`. Add 10 unit tests: `register_subscription` appends to jsonl, duplicate subscription (same endpoint) overwrites rather than appending, `deliver_completion_push` calls webpush for each entry, expired subscription (410) is removed from file, `GET /api/push/vapid-public-key` returns the env var value, push feature returns `{enabled: false}` when VAPID keys absent, push payload includes sim_id in URL, push title includes direction, delivery failure on non-410 is logged but not removed from file, empty subscriptions file is handled gracefully (no error).
2. In `Step3Simulation.vue`, if the simulation is running (status === "running") and `Notification.permission !== "denied"`, show a bell icon button labeled "Notify me when done". On click: call `Notification.requestPermission()` → if granted, call `navigator.serviceWorker.ready`, subscribe with the VAPID public key from `GET /api/push/vapid-public-key`, POST the resulting PushSubscription to `POST /api/simulation/{id}/push/subscribe`. Replace the button with a "Notified ✓" chip (persisted in localStorage per sim_id so it survives page refresh). If `push.enabled === false` from the server, hide the button entirely. Add a `sw.js` service worker (if not already present) that handles the `push` event: calls `self.registration.showNotification(data.title, {body: data.body, icon: "/logo.png", data: {url: data.url}})`. On `notificationclick`, opens `data.url`. Register the service worker in `main.js` if not already registered. Add `subscribeToPush(simId, subscription)`, `getVapidPublicKey()` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels on the bell button ("Notify me / 通知我") and chip ("Notified / 已订阅").
3. Add `POST /api/simulation/{id}/push/subscribe` and `GET /api/push/vapid-public-key` to the OpenAPI spec under a new "Notifications" tag. Add a "Web Push Notifications" section to `docs/CONFIGURATION.md`: how to generate VAPID keys (`python -c "from pywebpush import Vapid; v = Vapid(); v.generate_keys(); print(v.private_key.decode(), v.public_key.decode())"`), how to set the env vars, and why the feature silently disables when keys are absent (safe default for deployments without VAPID configured).

### 3. Operator Usage Analytics

**Type:** Feature
**Effort:** Medium (1-2 days)
**Impact:** MiroShark has 1,284 stars, 14 ecosystem projects, and a growing operator base — but no way for an operator to answer "what are my users actually simulating?" `GET /api/analytics/usage` aggregates across all published simulations: top-10 most-simulated topics (extracted from simulation titles via word frequency), most-run platforms (from simulation configs), agent-count distribution (histogram: 1–25 / 26–50 / 51–100 / 100+), peak usage hours (UTC hour of created_at, grouped into a 24-bar histogram), simulations this week/month (count), and top-5 most-embedded simulations (from surface-stats.json embed_hits). A new "Analytics" tab in the operator dashboard renders these as stat cards and bar charts. Pure stdlib, no new computation — pure aggregation of existing data. For operators deciding which features to build next, or ecosystem partners profiling the platform for integration suitability, this is the "what is this platform actually being used for?" surface.

**How:**
1. Add `backend/app/services/usage_analytics_service.py` (~200 LoC, pure stdlib). `compute_usage_analytics(data_dir)` scans all published sim dirs (same scandir pattern as leaderboard_service). For each sim, reads `simulation_config.json` (title, agent_count, platforms, created_at) and `surface-stats.json` (embed_hits). Computes: (a) `topic_frequency: [{word: str, count: int}]` — tokenize each title into words (lowercase, strip punctuation, exclude stopwords: ["a","the","of","to","for","and","in","is","an","with","this","that","from","on","at","by","how","what","why","will","are","was","be","has","have"]); count word frequency across all titles; return top-10 sorted by count desc. (b) `platform_frequency: [{platform: str, count: int}]` — for each sim, count each platform in its `platforms` array; return all platforms sorted by count desc. (c) `agent_count_distribution: {buckets: [{label: str, min: int, max: int, count: int}]}` — four buckets: "1–25", "26–50", "51–100", "100+". (d) `hourly_distribution: [{hour: int, count: int}]` — UTC hour of `created_at`, 24 entries (hours 0–23). (e) `sims_this_week: int`, `sims_this_month: int` — sims with `created_at` within last 7 days / 30 days. (f) `top_embedded: [{sim_id: str, title: str, direction: str, embed_hits: int}]` — top 5 by `embed_hits` (read from `surface-stats.json`; skip sims with no embed data). Returns `{generated_at: ISO, total_sims: int, topic_frequency: [...], platform_frequency: [...], agent_count_distribution: {...}, hourly_distribution: [...], sims_this_week: int, sims_this_month: int, top_embedded: [...]}`. Cache to `<data_dir>/usage_analytics.json` with 1-hour mtime-based invalidation. Add `GET /api/analytics/usage` (admin-gated; public-safe summary excludes sim IDs from topic_frequency). Returns payload with `Cache-Control: private, max-age=3600`. Add `usage_analytics` to `surfaces_catalog.py` as an `operator` type surface.
2. In `OperatorDashboardView.vue`, add an "Analytics" tab alongside the existing simulation list. Tab content: a 2-column grid of stat cards at the top: "Total simulations: N", "This week: N", "This month: N", "Most-used platform: {name}". Below the stat row: three charts side by side: (1) **Topic cloud** — a horizontal bar chart of the top-10 topic words (each word as a labeled bar, length proportional to frequency; sorted by count desc). (2) **Hourly activity** — a 24-bar chart (bars for hours 0–23, height proportional to sim count; bars at peak hours in violet, others in gray). (3) **Agent count distribution** — a 4-bar histogram ("1–25", "26–50", "51–100", "100+") showing how operators size their simulations. Below the charts: a "Most embedded simulations" table: rank, title, direction chip, embed hit count. Table is clickable — clicking a title navigates to the watch page. Add `getUsageAnalytics()` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels throughout. If `total_sims === 0`, show an empty state: "No published simulations yet — run your first simulation to see analytics."
3. Add `GET /api/analytics/usage` to the OpenAPI spec under a new "Analytics" tag with `UsageAnalyticsResponse` schema. Add 14 unit tests in `backend/tests/test_unit_usage_analytics.py`: topic_frequency excludes stopwords, topic_frequency is sorted by count desc, topic_frequency caps at 10 entries, platform_frequency counts across all sims (not unique), agent_count_distribution correctly bins a sim with 30 agents into "26–50", hourly_distribution has exactly 24 entries (hours 0–23 always present), sims_this_week counts correctly (boundary: created_at exactly 7 days ago is excluded), sims_this_month counts correctly, top_embedded sorts by embed_hits desc, top_embedded caps at 5, top_embedded skips sims with no surface-stats.json, cache written on first call, cache reused when mtime < 1h, admin token required (401 without). Add an "Operator Usage Analytics" section to `docs/FEATURES.md`: explains the stopword list, notes the 1-hour cache, explains why the endpoint is admin-gated (aggregate topic data could leak what private sims are about), and describes the embed_hits data source (surface-stats.json).

### 4. Translation Contribution Scaffold

**Type:** Community / DX
**Effort:** Small (hours)
**Impact:** Issue #161 is open: the community is asking for more translations. MiroShark supports EN, ZH-CN, and FR — but there's no structured path for contributors to add a new language. "Add German support" as an issue comment is an invitation without a door. A translation coverage API and scaffold turns it into a concrete contribution workflow: see which strings aren't translated, download a pre-populated template with English defaults, fill it in, open a PR. `GET /api/i18n/coverage` reads all locale files and returns per-locale stats (`{locale: "de", total_keys: 602, translated: 0, coverage_pct: 0}`). `GET /api/i18n/coverage/{locale}/export` generates a JSON file with just the missing keys for that locale, pre-filled with English values as copy targets. A `/i18n/coverage` frontend page renders a coverage grid. This closes the structural gap between "wanting to translate" and "knowing what to translate" — which is exactly what issue #161 is missing.

**How:**
1. Add `backend/app/services/i18n_coverage_service.py` (~100 LoC, pure stdlib). `get_locale_files(frontend_dir)` scans `frontend/src/locales/` for `*.js` files, reads each, and extracts the exported dict (using regex to parse the JS object literal: match `export default \{(.*)\}` in multiline mode; extract key-value pairs with `key: "value"` pattern). `compute_coverage(frontend_dir)` reads the `en.js` source-of-truth to get all canonical keys (N total); for each other locale file, counts how many canonical keys exist in that locale dict. Returns `{locales: [{locale: str, total_keys: int, translated: int, missing: int, coverage_pct: float}], canonical_locale: "en", canonical_key_count: int}`. `export_missing(frontend_dir, locale)` returns a dict of the missing keys for that locale with English values as placeholders (ready to paste into a new locale file). Add `GET /api/i18n/coverage` (unauthenticated; public — no sensitive data). Add `GET /api/i18n/coverage/{locale}/export` (unauthenticated; returns JSON with `Content-Disposition: attachment; filename="{locale}.json"`; 404 if locale not a known file in the locales dir). Add 10 unit tests in `backend/tests/test_unit_i18n_coverage.py`: `get_locale_files` finds all .js files in locales dir, `compute_coverage` for a 100% translated locale returns coverage_pct=100, `compute_coverage` for a 0% translated locale returns coverage_pct=0, partial locale counts correctly, canonical_locale is always "en", export_missing returns only missing keys (not already-translated ones), export_missing values are the English source strings, unknown locale returns 404, coverage response has one entry per locale file (excluding en.js itself), total_keys matches key count in en.js.
2. Add an `I18nCoverageView.vue` page at `/i18n/coverage`. On mount, fetch `GET /api/i18n/coverage`. Render: a header showing "Translation Coverage" and "Canonical key count: N". Below: a table with columns: Locale, Keys, Translated, Missing, Coverage %, Actions. Each row: a flag emoji (or 2-letter code), the stats, a percentage bar (green if 100%, yellow if 50–99%, red if < 50%), and an "Export missing keys" button (triggers `GET /api/i18n/coverage/{locale}/export` download). Below the table: a "Want to add a new language?" section with a step-by-step guide: (1) Fork the repo, (2) Copy `frontend/src/locales/en.js` to `{locale}.js`, (3) Use the "Export missing keys" feature from an existing partial locale to see what needs translation, (4) Add the locale to `frontend/src/i18n.js`, (5) Open a PR with the new locale file. Link to issue #161 as the tracking issue. Add a link to `/i18n/coverage` from the footer of the app (small "Translations" link). Add the route to `router/index.js`. No auth required — this page is for contributors, not operators. Bilingual EN/ZH labels.
3. Add `GET /api/i18n/coverage` and `GET /api/i18n/coverage/{locale}/export` to the OpenAPI spec under a new "Internationalization" tag with `I18nCoverageResponse` schema: `{canonical_locale: str, canonical_key_count: int, locales: [{locale: str, total_keys: int, translated: int, missing: int, coverage_pct: float}]}`. Add a "Translation Coverage" section to `docs/FEATURES.md` and link it from CONTRIBUTING.md — this is the primary onramp for new language contributions. Note: the coverage endpoint reads locale files at request time (no caching) so it reflects the current state of the repo without needing a rebuild.

### 5. Simulation RSS Feed

**Type:** Integration
**Effort:** Small (hours)
**Impact:** With 1,284 stars, 14 ecosystem partners, and a publicly accessible REST API, MiroShark has the raw material for the zero-code integration primitive: an RSS feed. `GET /api/feed.rss` returns an Atom/RSS 2.0 feed where each `<item>` is a published simulation — title, direction, confidence score, topic summary, creation timestamp, and a `<link>` to the watch page. Newsletter operators subscribe once and receive every new simulation automatically. Discord bots (via RSS-to-Discord bridges like MonitoRSS) get live simulation updates with no API integration needed. IFTTT, Zapier, and n8n can trigger on new items without writing a webhook receiver. The full-text search endpoint (#2 from Jun 12, push-blocked) enables discovery; the RSS feed enables *subscription*. Pure stdlib (xml.etree.ElementTree), no new computation — same published-sim scan as the gallery, cached 15 minutes. This is the integration primitive the 14 ecosystem projects can consume without writing a single line of code.

**How:**
1. Add `backend/app/services/rss_service.py` (~90 LoC, pure stdlib). `generate_feed(data_dir, base_url)` scans all published sim dirs (same scandir pattern as gallery/leaderboard). For each, reads `simulation_config.json` (title, description, created_at) and `signal.json` (direction, yes_pct). Sorts by `created_at` desc (newest first). Builds an RSS 2.0 XML document using `xml.etree.ElementTree`: `<rss version="2.0"><channel><title>MiroShark Simulations</title><link>{base_url}</link><description>Live simulation results from MiroShark — swarm intelligence forecasting</description><lastBuildDate>{most_recent_sim_date}</lastBuildDate>`. For each sim (cap at 50 items): `<item><title>{sim_title} — {direction}</title><link>{base_url}/simulation/{sim_id}</link><pubDate>{created_at in RFC 2822 format}</pubDate><description><![CDATA[{confidence_pct.toFixed(0)}% confidence · {yes_pct.toFixed(0)}% agreement · {description_first_200_chars}]]></description><guid isPermaLink="true">{base_url}/simulation/{sim_id}</guid></item>`. Returns the XML string (not cached on disk — in-memory generation). Add `GET /api/feed.rss` (unauthenticated, published sims only) to `simulation.py` blueprint. Response: `Content-Type: application/rss+xml; charset=utf-8`, `Cache-Control: public, max-age=900` (15-minute cache). Reads `base_url` from `PUBLIC_BASE_URL` env var (same source as `base_url_service`). If the env var is unset, uses `http://localhost:5000` as fallback. Add `feed` to `surfaces_catalog.py` as a `utility` type surface (40th or 41st depending on prior ships).
2. In the `ExploreView.vue` gallery header, add a small RSS icon link (orange, standard RSS icon SVG) pointing to `/api/feed.rss` with `target="_blank"` and `rel="noopener"`. Tooltip: "Subscribe via RSS". In the `<head>` of `index.html`, add the RSS autodiscovery `<link>`: `<link rel="alternate" type="application/rss+xml" title="MiroShark Simulations" href="/api/feed.rss">` — this makes the feed discoverable automatically by browsers and RSS readers without the user needing to find the icon. No other frontend changes. In `EmbedDialog.vue`, add a "RSS Feed" section at the end alongside the other integration sections: the feed URL and a note that it includes the 50 most recent published simulations. Bilingual EN/ZH tooltip on the RSS icon ("Subscribe via RSS / 订阅 RSS").
3. Add `GET /api/feed.rss` to the OpenAPI spec under the "Discovery" tag. Note: the response type is `application/rss+xml`, not JSON — document the schema as a prose description of the RSS 2.0 structure rather than a JSON schema. Add 10 unit tests in `backend/tests/test_unit_rss.py`: feed is valid RSS 2.0 XML (xml.etree.ElementTree can parse it without error), feed title is "MiroShark Simulations", each published sim appears as an `<item>`, items sorted by created_at desc (newest first), feed capped at 50 items, `<link>` contains base_url + sim_id path, `<pubDate>` is in RFC 2822 format (e.g. "Mon, 16 Jun 2026 09:00:00 GMT"), `<description>` contains direction and yes_pct, unpublished sims excluded, Content-Type header is application/rss+xml. Add a "Simulation RSS Feed" section to `docs/FEATURES.md`: notes the 15-minute cache-control, explains the 50-item cap (most recent publications only — use the gallery for the full history), and provides example RSS reader integrations (Feedly subscribe URL format, the RSS autodiscovery `<link>` tag).

## Selection Rationale

Five ideas building on what just shipped, responding to open community asks, and addressing integration gaps outside the 7-day window of prior suggestions:

- **Cost Budget & Alerts** (#1) — the direct follow-on to PR #179 (cost.json). Read surface exists; control surface doesn't. Budget enforcement with graceful stop is the one thing operators running parameter sweeps immediately ask for after "how much did that cost?"

- **Web Push Notifications** (#2) — `pywebpush` is already in requirements.txt (bumped to 2.3.0 by Dependabot PR #173). The wiring to SimulationRunner completion and the browser subscription UI are the only missing pieces. For long-running high-agent-count simulations, "notify me when done" is the UX primitive that removes the need to babysit the tab.

- **Operator Usage Analytics** (#3) — suggested on Jun 6 (10 days ago, outside the 7-day window) but never built (push blocked). 1,284 stars, 14 ecosystem projects, and multiple operators — but no aggregate picture of platform usage. Topic frequency, hourly distribution, agent-count histogram, and most-embedded sims are all derivable from existing data without new computation.

- **Translation Contribution Scaffold** (#4) — issue #161 is open ("Additional translations"). Structural response: a coverage API that tells contributors exactly which strings are missing per locale, with a one-click JSON export of untranslated keys pre-filled with English. Turns an ambiguous community ask into a concrete pull request workflow.

- **Simulation RSS Feed** (#5) — suggested Jun 2 (14 days ago) but never built. Pure stdlib (xml.etree.ElementTree), no new computation, 90 LoC. The autodiscovery `<link>` tag alone is worth shipping: every RSS reader that visits the gallery page will find the feed automatically. Fourteen ecosystem projects can subscribe without integrating any API.
