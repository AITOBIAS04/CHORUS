### 1. Real-Time Simulation Progress via Server-Sent Events

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** A user who launches a 30-agent, 15-round simulation today has one feedback loop: manually refreshing the page (or polling `GET /api/simulation/:id`) to watch `current_round` increment from 1 to 15. The experience is functionally equivalent to a loading spinner with a round counter that only updates on refresh. Server-Sent Events (SSE) replaces that with a push stream: the simulation runner writes round-completion events to a per-sim event queue; a `GET /api/simulation/:id/events` SSE endpoint reads from that queue and pushes `data: {round: N, total_rounds: M, progress_pct: int, latest_agent: str, latest_stance: str}` to any connected client in real time. The watch page subscribes on mount and updates the progress bar, round label, and a "live feed" strip of recent agent activity without any client-side polling. The effect: the simulation feels alive. A 10-round run goes from "click button → wait → results appear" to "click button → watch agent stances unfold round by round." Flask supports SSE natively via `Response(stream_with_context(...), mimetype='text/event-stream')` — zero new dependencies.

**How:**
1. In the simulation runner (wherever `run_round()` or the per-round simulation loop lives), write a line to `<sim_dir>/events.jsonl` after each round completes: `{event: "round_complete", round: N, total_rounds: M, progress_pct: int, agent_snapshots: [{name, stance}] (last 3 agents who posted), timestamp: ISO}`. Also write a `{event: "simulation_complete", ...}` line when the final round finishes and `{event: "simulation_error", error: str}` if the run fails. Events file is append-only during the run; after completion it's a static record. Add `GET /api/simulation/:id/events` as an SSE endpoint: opens `events.jsonl`, streams all existing lines immediately (for clients connecting mid-run), then polls the file for new lines using a tail-follow loop (`time.sleep(0.5)` between polls, max 60s inactivity timeout, send SSE `event: keepalive` every 15s to prevent browser disconnects). Returns `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `X-Accel-Buffering: no` (required for nginx/Cloud Run proxies to pass through the stream without buffering). Send `event: done` when the simulation_complete or simulation_error event is seen, then close the stream. The endpoint requires no admin token — it returns only the same status and round data already public on `GET /api/simulation/:id`.
2. In `SimulationWatchPage.vue` (or equivalent), on mount detect `status == "running"` and open an `EventSource('/api/simulation/:id/events')`. On each `message` event: update `currentRound`, `progressPct`, and append to a `liveActivityFeed` array (last 5 entries — agent name + stance chip). Render the live feed as a compact vertical strip below the progress bar: each entry fades in with a 200ms CSS transition, the oldest entry fades out when the 6th arrives (maintaining a max-5 sliding window). On `event: done`, close the `EventSource` and trigger the same "simulation complete" UI state the polling path uses today (results panel appears). On `EventSource` error (connection dropped), fall back to the existing polling path with a note "Reconnecting…". `EventSource` is native in all modern browsers — no new npm dependency. The live feed strip should be visually distinct from the results panel: `font-mono text-xs bg-gray-900/50 rounded p-2` (matches the new dark theme). Clean up the `EventSource` in `onUnmounted()` to prevent memory leaks.
3. Add `GET /api/simulation/:id/events` to the OpenAPI spec under the "Simulation" tag with `produces: text/event-stream`. Add an "SSE Event Shapes" section to `docs/API.md`: event field definitions for `round_complete`, `simulation_complete`, `simulation_error`, and `keepalive`, with a code example showing how to consume the stream with the native `EventSource` API or `curl -N`. Add 8 unit tests in `backend/tests/test_unit_sse.py`: events.jsonl written after each round (correct shape), `simulation_complete` event written on finish, `simulation_error` event written on failure, SSE endpoint streams existing events on connect, SSE endpoint sends keepalive within 15s of inactivity, `event: done` sent after complete, route accessible without admin token, `X-Accel-Buffering: no` header present. In `docs/FEATURES.md`, add a "Real-Time Progress" section: explains SSE, the tail-follow pattern, Cloud Run compatibility (note: requires HTTP/1.1 — Cloud Run supports it; do not use with chunked transfer encoding disabled).

### 2. Deployment Health & Status Endpoint

**Type:** DX / Performance
**Effort:** Small (hours)
**Impact:** Cloud Run deployment infra shipped this week. Fly.io, Render, and Railway are also viable targets. But every one of these platforms has a health check requirement: a URL that returns 200 when the service is healthy and non-200 when it isn't, polled on a configurable interval. Without one, Cloud Run considers the service unresponsive during cold starts and restarts it unnecessarily. Without one, operators monitoring their deployment have no API they can ping from an uptime monitor (UptimeRobot, Betterstack, StatusCake). A `/api/health` endpoint and a `/status` page close this gap. The health endpoint returns `{status: "ok"|"degraded", checks: {filesystem: bool, simulation_dir_accessible: bool, python_version: str}, version: str (from git describe or a VERSION file)}` — fast enough to serve in < 50ms with no simulation work required, suitable as a Cloud Run liveness probe URL. The status page at `/status` is the human-readable version: a simple page showing uptime (computed from a `<data_dir>/startup_timestamp.json` written at Flask init), recent simulation counts (last 24h), and a green/yellow/red indicator for each subsystem. For operators who share their MiroShark instance with users, `/status` is the page they can link from their README or share in a Slack channel when users ask "is it down?"

**How:**
1. Add `backend/app/services/health_service.py` (~100 LoC, pure stdlib). `check_health(data_dir)` runs three fast checks: (1) `filesystem` — attempt `os.listdir(data_dir)` and catch `OSError`; (2) `simulation_dir_accessible` — check that `data_dir` exists and is writable via `os.access(data_dir, os.W_OK)`; (3) `python_version` — `sys.version_info[:2]` formatted as `"3.N"`. Returns `{status: "ok"|"degraded", checks: {filesystem: bool, simulation_dir_accessible: bool, python_version: str}, uptime_seconds: int (from startup_timestamp.json written at Flask app init), simulations_last_24h: int (count of sim dirs with mtime within 86400s), version: str}`. Status is `"degraded"` if any check is `False`; otherwise `"ok"`. Add `GET /api/health` returning this payload with HTTP 200 when `status == "ok"` and HTTP 503 when `status == "degraded"`. Add `GET /api/status` (alias, same payload, always HTTP 200 — for dashboards that poll without caring about HTTP status). On Flask app init (`create_app()`), write `<data_dir>/startup_timestamp.json = {started_at: ISO}` (creates or overwrites — used to compute uptime_seconds). The health endpoint must respond in under 100ms — no simulation scanning, no database calls.
2. Add a `/status` page in the frontend (new `StatusView.vue`): a minimal page (dark theme consistent with the new reskin) showing: a large `● OPERATIONAL` / `● DEGRADED` indicator (green/red dot + text), four metric cards (Uptime: `Nd Nh Nm`, Simulations (24h): `N`, Storage: `OK`/`Limited`, Python: `3.N`), a last-checked timestamp auto-refreshing every 60s via `setInterval`. The page is publicly accessible (no admin token required), lightweight (no auth, no navigation chrome needed — just the status card centered on a dark background), and link-friendly (canonical URL `/status`). Add `/status` to the Vue Router with no authentication requirement. In the main navigation or footer (wherever the nav links live), add a "System Status" link → `/status` (only rendered if the health check was last `degraded` within the past 5 minutes — a subtle "something's wrong" signal visible to admins, invisible during normal operation). Frontend calls `GET /api/health` on page mount and on the 60s refresh cycle.
3. Add `GET /api/health` and `GET /api/status` to the OpenAPI spec under a new "Platform" tag (alongside `/api/stats` and `/api/surfaces.json`). Include the `HealthResponse` schema with all fields defined. Add the Cloud Run health check configuration note to `docs/DEPLOY.md` (create if absent, otherwise add a section): "Set the health check path to `/api/health` in your `app.yaml` or `cloudbuild.yaml` — Cloud Run will poll this URL every 30s and restart the service if it returns non-200 three consecutive times." Add a Docker Compose healthcheck snippet: `healthcheck: {test: ["CMD", "curl", "-f", "http://localhost:5001/api/health"], interval: "30s", timeout: "5s", retries: 3}`. Add 6 unit tests in `backend/tests/test_unit_health.py`: check_health returns ok when data_dir exists and writable, returns degraded when data_dir not writable, `/api/health` returns HTTP 200 on ok, `/api/health` returns HTTP 503 on degraded, `/api/status` always returns HTTP 200, startup_timestamp.json written on app init.

### 3. Zenodo DOI Auto-Deposit for Published Simulations

**Type:** Integration
**Effort:** Medium (1–2 days)
**Impact:** The BibTeX endpoint (PR #96) produces a citation entry. The reproduce.json (PR #75) exports the simulation config. The OriginTrail DKG (PR #84) anchors provenance on a decentralized knowledge graph. None of these produce a DOI. For a peer-reviewed journal or conference paper, a URL citation is insufficient — reviewers require a persistent identifier that resolves even if the hosting server changes. Zenodo (CERN's open-access repository) issues DOIs for free, persists deposits permanently, and is indexed by Google Scholar, CrossRef, and every major academic discovery layer. An operator who publishes a simulation and clicks "Register DOI" triggers a Zenodo deposit: the Archive Bundle ZIP lands in Zenodo's repository, Zenodo assigns a DOI in the format `10.5281/zenodo.NNNNNNN`, the DOI is written to `<sim_dir>/zenodo.json`, and the BibTeX citation is updated to include `doi = {10.5281/zenodo.NNNNNNN}`. The watch page shows a "DOI: 10.5281/zenodo.NNNNNNN" chip that opens the Zenodo record. For the "cited in a peer-reviewed paper" hyperstition, this is the structural requirement: journals that accept a BibTeX citation with a valid DOI require nothing else. This is the step that makes a MiroShark simulation formally citable.

**How:**
1. Add `scripts/postprocess-zenodo-deposit.sh` (~80 lines). Reads `.pending-zenodo/<sim_id>.json` files (each containing `{sim_id, archive_bundle_path, metadata: {title, description, scenario_excerpt, published_at, n_agents, n_rounds, model, confidence_score}}`). For each pending deposit: (1) calls `curl -X POST "https://zenodo.org/api/depositions" -H "Authorization: Bearer $ZENODO_API_TOKEN" -H "Content-Type: application/json"` to create a deposition record, capturing the `deposit_id` and `bucket_url` from the response; (2) calls `curl -X PUT "$bucket_url/$sim_id-archive.zip" --upload-file "$archive_bundle_path" -H "Authorization: Bearer $ZENODO_API_TOKEN"` to upload the archive bundle; (3) calls `curl -X POST "https://zenodo.org/api/depositions/$deposit_id/actions/publish" -H "Authorization: Bearer $ZENODO_API_TOKEN"` to publish and obtain the DOI; (4) writes `{sim_id, zenodo_deposit_id: str, doi: str, zenodo_url: str, deposited_at: ISO}` to `<sim_dir>/zenodo.json`; (5) removes the `.pending-zenodo/<sim_id>.json` file. On any curl failure (non-2xx HTTP status), moves the pending file to `.pending-zenodo/failed/` with the error response appended, and logs the failure without retrying (Zenodo has strict rate limits — manual retry is safer than automated). Add `backend/app/services/zenodo_service.py` (~80 LoC, pure stdlib): `queue_zenodo_deposit(sim_id, data_dir)` checks that (1) the simulation is published (`published_at` set in config.json), (2) an archive bundle exists at the expected path, (3) `zenodo.json` does not already exist (idempotent), then writes the `.pending-zenodo/<sim_id>.json` file. Returns `{queued: true}` or `{already_registered: true, doi: str}`. Add `POST /api/simulation/:id/zenodo` (admin-token gated): calls `queue_zenodo_deposit` and returns the result. Add `GET /api/simulation/:id/zenodo` (public): returns `zenodo.json` contents if present, or `{registered: false}`.
2. In `Step3Simulation.vue`, in the results panel's citation section (near the BibTeX copy button), add a "Register DOI" button (admin-gated, only shown when `zenodo.registered == false` and the simulation is published). Button state machine: idle → "Registering…" (spinner) → "DOI Registered ✓". On success, show the DOI as a chip: `DOI: 10.5281/zenodo.NNNNNNN` with a copy icon and an external link icon to the Zenodo record. In the BibTeX output (the textarea or copy-button content), dynamically add `  doi = {10.5281/zenodo.NNNNNNN},` as the second line after `@article{...` when `zenodo.json` is present and contains a DOI — the BibTeX is now a proper resolvable citation. In `EmbedDialog.vue`, add a "Zenodo" chip in the citations section alongside BibTeX and OriginTrail: "Permanent DOI deposit — citable in academic journals." The chip shows "Register DOI →" (links to the admin-gated register flow) or "DOI: 10.5281/zenodo.NNNNNNN" (links to the Zenodo record) depending on state.
3. Add `POST /api/simulation/:id/zenodo` and `GET /api/simulation/:id/zenodo` to the OpenAPI spec under the "Exports" tag with a `ZenodoRegistration` schema: `{registered: bool, doi: string|null, zenodo_url: string|null, deposited_at: string|null}`. Add `zenodo_doi: string|null` and `zenodo_url: string|null` to the gallery card payload (from `zenodo.json` if present). Add `zenodo.json` to the Archive Bundle manifest (if it exists — a deposited simulation's bundle includes its own DOI record). Add `ZENODO_API_TOKEN` to `.env.example` with a comment: `# Optional — get a token from zenodo.org/account/settings/applications. Required for DOI registration.` Add 10 unit tests in `backend/tests/test_unit_zenodo.py`: queue_zenodo_deposit writes correct pending file, queue_zenodo_deposit is idempotent (second call returns already_registered), unpublished sim returns error, missing archive bundle returns error, GET returns registered=false when no zenodo.json, GET returns doi when zenodo.json present, BibTeX endpoint includes doi line when zenodo.json present, pending file structure (all required fields), admin token required for POST, public access allowed for GET. In `docs/INTEGRATIONS.md`, add a "Zenodo DOI Registration" section: explains the postprocess script pattern, ZENODO_API_TOKEN setup, the sandbox test endpoint (`https://sandbox.zenodo.org/api/`), and a note that deposits are permanent — a published simulation's Zenodo record cannot be deleted, only versioned.

### 4. Multi-Metric Simulation Leaderboard Page

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The gallery at `/explore` has sorting (recency, trending) and filtering (tags, confidence, consensus). But it answers one question at a time: "what's trending?" or "what's highly confident?" A visitor who wants to understand the full landscape of what's happened on MiroShark — what's the most-embedded simulation, what's the most volatile, what has the most forks, what achieved the highest confidence — has to run five separate queries. The leaderboard page answers all five simultaneously: a compact table with 5 metric tabs (Most Embedded, Highest Confidence, Most Volatile, Most Forked, Most Discussed) showing the top 10 simulations per metric in a scannable ranked format. Each row: rank, scenario excerpt (50 chars), consensus direction chip, the metric value (embed hits / confidence score / volatility score / fork count / comment count), published date, and a "View →" link. The leaderboard is public (no admin token), cached for 1 hour, and embeddable as an `<iframe>` using the existing embed infrastructure (a `?leaderboard=true` embed param selects the compact leaderboard layout). For first-time visitors who land on MiroShark from a referral link, the leaderboard answers "what's the best content here?" in one glance. For the MIROSHARK token narrative, the leaderboard is shareable evidence that the platform has real usage with measurable engagement.

**How:**
1. Add `backend/app/services/leaderboard_service.py` (~180 LoC, pure stdlib). `get_leaderboard(data_dir, metric, limit=10)` scans all published sim dirs and computes: `embed_hits` (from `surface_stats.json`, key `embed`), `confidence_score` (from `confidence.json`, key `confidence_score`), `volatility_score` (from `volatility.json` if present — the Belief Volatility Analytics surface from PR #124 — or `null`), `fork_count` (from `fork_index.json` in `data_dir` — count of entries for this sim_id), `comment_count` (from `comments.jsonl` if present — count of non-deleted lines). Returns `{metric, generated_at: ISO, sims: [{rank: int, sim_id, scenario_excerpt (first 60 chars), consensus_direction, metric_value, published_at}]}` sorted by the requested metric descending (nulls last). Cache result in `<data_dir>/leaderboard_cache_{metric}.json` with 1-hour TTL. Add `GET /api/leaderboard` accepting `?metric=embed_hits|confidence|volatility|forks|comments` (default: `embed_hits`) and `?limit=N` (default 10, max 25). Return the leaderboard object. All 5 metrics derive from existing files — no new data collection, no new writes during simulation runs.
2. Add `LeaderboardView.vue` at `/leaderboard`. Layout: a dark-theme page header ("Leaderboard") with a 5-tab metric selector (Most Embedded · Highest Confidence · Most Volatile · Most Forked · Most Discussed). Selected tab is highlighted in orange (matches the theme). Content: a ranked table with columns — # (rank badge), Scenario (60-char excerpt, click → `/watch/<id>`), Direction (chip), Metric Value (the number — formatted with units: "847 embeds", "94 confidence", "3.2 volatility", "12 forks", "8 comments"), Published (relative date). Loading state: skeleton rows (3 rows of gray pulse animations). Empty state per tab: "No simulations with embed data yet — share a simulation to get started." In the main navigation, add a "Leaderboard" chip on the `/explore` page toolbar alongside Tags, Collections, Digest, Showcase. Add `/leaderboard` to the Vue Router config. In `frontend/src/api/simulation.js`, add `getLeaderboard(metric, limit)` helper. The leaderboard page also supports `?embed=true` for a compact iframe-friendly layout (no navigation, just the table — same iframe sizing as the simulation embed, using the existing embed iframe infrastructure).
3. Add `GET /api/leaderboard` to the OpenAPI spec under a new "Discovery" tag with `LeaderboardResponse` and `LeaderboardEntry` schemas. Include the `?metric` enum values and their descriptions. Add 8 unit tests in `backend/tests/test_unit_leaderboard.py`: embed_hits metric sorts correctly, confidence metric sorts with nulls last, volatility falls back to null when no volatility.json, fork_count reads fork_index.json correctly, comment_count excludes deleted comments, limit param respected, cache written on first call, cache read on second call within TTL. In `docs/FEATURES.md`, add a "Simulation Leaderboard" section: lists all 5 metrics, notes the 1-hour cache, and documents the `?embed=true` iframe param. Include the canonical embed snippet (`<iframe src="https://your-instance/leaderboard?embed=true&metric=confidence" width="600" height="400">`) as a copy-paste block.

### 5. Community Showcase (Curated Featured Simulations)

**Type:** Community / Growth
**Effort:** Small (hours)
**Impact:** A visitor who discovers MiroShark from a link in a paper, a tweet, or the GitHub trending page faces a critical first-impression moment: "what's the best thing this platform has produced?" The gallery shows everything chronologically; the leaderboard shows what got the most engagement; neither shows what's editorially *notable*. The Community Showcase is the curator's answer: 3–6 simulations that an operator has explicitly flagged as exemplary, each with a 1–3 sentence curator note explaining why this simulation is worth reading ("This 20-agent run on the 2026 EU AI Act vote produced the most coherent coalition structure we've seen — the risk-management agents formed a bloc by round 3 and held."). Featured simulations appear at `/showcase` and as a collapsible "Featured" section on the `/explore` page above the main gallery. For new visitors, the showcase answers "what's impressive here?" in under 10 seconds. For operators who run MiroShark for communities or research groups, the showcase is a curatorial identity layer — it says "this is what we think is worth your attention." For the "Will MiroShark be featured on a Chinese dev platform?" hyperstition, a polished showcase page is the asset worth linking to. Zero new dependencies — showcase data is a JSON file of sim references and curator notes.

**How:**
1. Add `backend/app/services/showcase_service.py` (~120 LoC, pure stdlib). Stores the showcase in `<data_dir>/showcase.json`: `{updated_at: ISO, entries: [{sim_id, curator_note (max 280 chars, stripped), featured_at: ISO, order: int}]}`. Max 8 entries (enforced at write). `get_showcase(data_dir)` reads the file and enriches each entry with live simulation metadata (scenario_excerpt, consensus_direction, confidence_score, embed_hits) by reading the sim dir — the same fields returned on gallery cards, so no new data model needed. Returns `{entries: [{sim_id, curator_note, featured_at, order, scenario_excerpt, consensus_direction, confidence_score: int|null, embed_hits: int}]}` sorted by `order` ascending. Add `GET /api/showcase` (public): returns the enriched showcase. Add `POST /api/showcase` (admin-token gated): accepts `{entries: [{sim_id, curator_note, order}]}` (full replace — not incremental). Validates: each `sim_id` must reference a published simulation, `curator_note` max 280 chars, max 8 entries, `order` values unique. Writes `showcase.json` atomically. Returns the enriched showcase. Add `DELETE /api/showcase/:sim_id` (admin-token gated): removes the entry for `sim_id` from the showcase.
2. Add `ShowcaseView.vue` at `/showcase`. Layout: a dark-theme page with header "Showcase — editor's picks" and a subtitle with the count ("N featured simulations"). Each featured simulation renders as a large card (wider than gallery cards): scenario excerpt (3 lines, `text-base leading-relaxed`), consensus direction chip (color-coded), confidence score badge (◎ NN), curator note (in `italic text-gray-300 mt-2`) with a small "— Editor" attribution, published date, and a "View simulation →" CTA. Cards are displayed in order (matching the `order` field). The page has an "Edit Showcase" button (admin-gated, shown when admin token is present in localStorage) that opens a management drawer: a list of current entries with curator notes as editable textareas, an "Add simulation" input (sim_id + curator note), drag handles for reordering (or simple "↑↓" buttons), and a "Save" button that calls `POST /api/showcase`. On `/explore`, add a "Featured" collapsible section above the main gallery grid: collapsed by default, showing 3 showcase entries as compact horizontal cards with curator notes. A "View all →" link leads to `/showcase`. Add a "Showcase" chip in the `/explore` toolbar alongside Tags, Collections, Leaderboard.
3. Add `GET /api/showcase`, `POST /api/showcase`, and `DELETE /api/showcase/:sim_id` to the OpenAPI spec under a new "Discovery" tag (alongside the Leaderboard routes). Add `is_featured: bool` and `curator_note: string|null` to the gallery card payload — `is_featured` is `true` if the sim appears in `showcase.json`; this lets the gallery cards show a ★ "Featured" chip without a separate API call. Add `has_showcase` to `GET /api/stats` (the platform stats endpoint) with the count of currently featured simulations. Add 10 unit tests in `backend/tests/test_unit_showcase.py`: get_showcase enriches with simulation metadata, max 8 entries enforced, curator_note max 280 chars enforced, sim_id must be published (unpublished → 400), full replace on POST (not append), DELETE removes single entry, GET returns entries sorted by order, is_featured true in gallery card for featured sim, admin token required for POST/DELETE, public GET requires no token. In `docs/FEATURES.md`, add a "Community Showcase" section: explains the admin-curated model, the curator note guidelines (context + insight, not marketing), and the Gallery `/explore` integration.

## Selection Rationale

Five angles distinct from all ideas suggested May 23–29, the 26 push-blocked builds, and the open surfaces-catalog PR:

- **SSE Real-Time Progress** (#1) — Every prior analytics feature improved what you see *after* a simulation finishes. SSE improves what you see *while* it's running. The live feed strips — agent names and stance chips flowing in round by round — transform the waiting experience from opaque to engaged. Flask's native streaming support means no new dependencies; the 0.5s tail-follow polling is well-established for this pattern. Distinct from all previous ideas because it addresses the creation experience, not the results experience.

- **Deployment Health Endpoint** (#2) — Cloud Run deploy infra shipped days ago. Without a `/api/health` endpoint, Cloud Run cannot perform liveness probes, Fly.io cannot health-check, and no uptime monitor can alert on outages. This is the standard ops primitive that should ship within 72 hours of any new deployment target. Small effort (100 LoC + a status page), immediate value to DYAI2025 and anyone else who deploys the just-landed Cloud Run config.

- **Zenodo DOI Auto-Deposit** (#3) — The BibTeX citation, reproduce.json, and OriginTrail DKG form the academic credibility arc. But journals require DOIs, not URLs. Zenodo is CERN's free, permanent open-access repository — indexed by Google Scholar, CrossRef, and every academic discovery layer. The postprocess script pattern (used by the completion webhook, the DKG, and the calibration prefetch) is exactly right here: deposit happens after Claude finishes, ZENODO_API_TOKEN stays in env. This is the single step that converts the academic citation hyperstition from plausible to structurally achievable.

- **Multi-Metric Leaderboard** (#4) — The gallery's trending sort answers "what's hot overall?" but mixes metrics. The leaderboard answers five specific questions simultaneously: most embedded (distribution reach), highest confidence (analytical trust), most volatile (interesting dynamics), most forked (reproducibility engagement), most discussed (community engagement). It's the first discovery surface that treats engagement signals as distinct dimensions rather than a single composite score. Embeddable as an iframe — site owners can embed the "top 5 confident simulations" on their research blog.

- **Community Showcase** (#5) — The leaderboard surfaces top simulations algorithmically. The showcase surfaces them editorially: 3–6 curator-chosen simulations with prose explanations of why they're notable. For a first-time visitor, algorithmic rankings signal scale; curator notes signal taste. The showcase is the difference between "this platform has lots of content" and "here's what this platform considers exemplary." For the Chinese dev platform hyperstition, a `/showcase` page is the link worth sending to a platform editor.
