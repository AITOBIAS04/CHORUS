### 1. Agent Belief Heatmap — Individual Stance Grid

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Every current MiroShark visualization is aggregate: the belief drift chart shows the YES/NO/NEUTRAL price across all agents per round; the interaction network shows influence connections at a single moment. Individual agent trajectories — who flipped stance, at what round, in which direction — are invisible unless you read through raw post logs. The Agent Belief Heatmap is a grid where each row is an agent and each column is a round, colored by stance (green = YES, red = NO, grey = NEUTRAL). At a glance, it makes visible the patterns that the aggregate chart erases: did consensus emerge gradually (all rows shifting green over time) or suddenly (half the agents flip in round 5)? Did a specific agent flip early and trigger a cascade? Is there a persistent holdout cluster that never joined consensus? For researchers, this is the individual-level view that separates "73% bearish consensus" from *how* that consensus actually formed. For the simulation design cycle, spotting a single persistent holdout agent who never moved makes stagnation checks more actionable.
**How:**
1. Add a `HeatmapView.vue` component alongside `BeliefDriftChart.vue` in the simulation results view. Fetch per-agent stance history from the existing trajectory data: for each simulation, the `<sim_dir>/posts/` directory contains all agent posts with round numbers; derive per-agent stance per round by reading each agent's final post stance per round from the leaderboard/trajectory data (already computed for the interaction network). Render as an HTML/CSS grid (no canvas required): `grid-template-columns: [agent_count] × [round_count]`, each cell a `<div>` with background color derived from stance (YES = `#22c55e` at full opacity, NO = `#ef4444`, NEUTRAL = `#6b7280`, missing = `#1f2937`). Cell size: 16×16px with 1px gap. On hover, show a tooltip: "Agent [name] · Round [N] · Stance: [YES/NO/NEUTRAL] · Belief: [yes_price]%." Add a "Heatmap" tab toggle in the visualization toolbar alongside "Belief Drift" and "Interaction Network."
2. Sort rows by the agents' final influence rank (leaderboard position) so the most influential agents appear at the top. Add a sort control in the toolbar: "Sort by: Influence (default) / First flip round / Name." Add a column header with round numbers (R1, R2 … RN). Add a legend row at the bottom: YES (green), NO (red), NEUTRAL (grey). For simulations with more than 20 agents, apply a compact display mode (12×12px cells, no labels). For simulations with more than 40 agents, allow vertical scroll with sticky column headers. The heatmap is fully rendered in the browser from cached trajectory data — no additional API calls after the initial page load.
3. In the simulation article generator, inject a one-line heatmap summary as structured context: "Heatmap pattern: [uniform early shift / late cascade / polarized holdout cluster / mixed]. N agents changed stance at least once (out of total). First mass flip occurred at round X (≥50% of agents showing same direction change in same round)." The article uses this to describe *how* consensus formed, not just what the final state was: "Consensus coalesced in a rapid cascade starting round 8, when 11 of 15 agents shifted simultaneously" is more informative than "73% bearish by round 12." Add `?view=heatmap` as a permalink variant on the share URL so researchers can share a direct link to the heatmap view of a specific simulation.

### 2. Per-Agent Trajectory CSV Export

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The belief trajectory export (PR #66) is aggregate-only: `[round, yes_price, no_price, neutral_price, top_agent]` — the summary statistics per round across all agents. Individual agent belief histories are not exported. For a researcher doing statistical analysis — regression models, survival analysis, influence attribution, agent-level clustering — the aggregate data is insufficient. They need the unit-record: `[agent_name, archetype, platform, round, yes_price, no_price, neutral_price, stance, post_count_this_round]` for every agent × round combination. This is the dataset that tools like R, pandas, and networkx need to run agent-level influence models. At 222 forks with a growing academic cohort, the gap between "we can export aggregate data" and "we can export the research dataset" is the gap between MiroShark as an illustration tool and MiroShark as a primary research instrument. Zero new data storage required — all of this information is already in `<sim_dir>/posts/` and `trajectory.json`.
**How:**
1. Add `GET /api/simulation/:id/export/agent-trajectory` returning a CSV with one row per agent × round. Column schema: `agent_id, agent_name, archetype, platform, round, yes_price, no_price, neutral_price, stance (YES/NO/NEUTRAL), post_count, was_top_influencer (bool — whether this agent held the top leaderboard position at end of this round), stance_changed_this_round (bool)`. Derive `yes_price`, `no_price`, `neutral_price` per agent per round from each agent's final post in that round (use the last belief_state entry for the agent in the round's post data). For rounds where an agent posted nothing, carry forward previous round's values. The endpoint streams the CSV directly (no intermediate file) using Python's `csv.writer` and Flask's `stream_with_context`. Set `Content-Disposition: attachment; filename="<sim_id>_agents.csv"`.
2. In the simulation results export toolbar (alongside "Download trajectory CSV" from PR #66), add a second export button: "Per-agent CSV." On click, trigger `GET /api/simulation/:id/export/agent-trajectory`. Add a brief tooltip on hover explaining the difference: "Aggregate trajectory: per-round totals. Per-agent trajectory: every agent × round combination — for individual-level analysis." In the Jupyter notebook export (built locally, not yet pushed), add a new pre-written analysis cell that loads the per-agent CSV: `agent_df = pd.read_csv('agents.csv'); stance_transitions = agent_df[agent_df['stance_changed_this_round'] == True].groupby(['archetype', 'round']).size()` — produces a stance-transition heatmap by archetype. In the OpenAPI spec, add the endpoint under an "Export" tag with the full CSV schema in the response description.
3. Add an `export_agent_trajectory: bool` field to the POST payload for `POST /api/simulation/:id/ensemble` (if the ensemble/multi-seed feature is built later): when true, aggregate the per-agent CSVs across all runs with a `run_id` column prepended, producing a single multi-run unit-record dataset. Document in `docs/API.md` with a note: "This endpoint is the recommended starting point for statistical modeling of MiroShark simulation data. It is the per-agent equivalent of the aggregate trajectory export introduced in PR #66." Add `docs/research-guide.md` covering the two export formats and suggesting appropriate analysis workflows for each.

### 3. Adversarial Robustness Mode

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** A MiroShark simulation that reaches 80% bullish consensus with no adversarial pressure has proved that agents with similar priors, talking among themselves, converge — not that the consensus is robust. A critic's obvious objection to any MiroShark finding: "of course they agreed, they weren't challenged." Adversarial Robustness Mode adds one configurable "devil's advocate" agent whose explicit goal is to challenge and disrupt emerging consensus — citing counter-evidence, seeding doubt in neutral agents, building coalitions against the dominant narrative. The simulation runs twice from the same seed: once baseline, once with the adversary injected. The results view shows a direct comparison: "Consensus held under adversarial pressure (80% vs. 74% bullish)" or "Adversarial pressure broke consensus at round 9 (consensus never stabilized above 55%)." The "Robust" or "Fragile" label is the research finding. For researchers using MiroShark to evaluate how resilient a narrative is to targeted skepticism — PR communications pre-testing, policy resilience analysis, protocol review — this is the feature that turns a directional signal into a stress-tested finding. The paper "Position: AI Agents Are Not (Yet) a Panacea for Social Simulation" identifies the absence of adversarial testing as a core validity gap in current LLM simulation tools. Adversarial Robustness Mode directly addresses this critique.
**How:**
1. Add an "Adversarial Mode" toggle in Step 2 (Configure), below the agent count selector. When enabled, show a "Devil's advocate config" expander with two fields: `adversary_intensity` (1–5 slider, default 3 — controls how aggressively the adversary challenges consensus: 1 = gentle skeptic, 5 = active disinformation spreader) and `adversary_strategy` (dropdown: "Evidence Challenger" / "Neutral Activator" / "Coalition Builder" / "Trust Eroder"). The adversary's persona is procedurally generated by the simulation setup step: given the scenario, generate a credible contrarian figure whose background gives them plausible standing to challenge the dominant view (e.g., for a crypto scenario, a former exchange compliance officer; for a corporate crisis, a short-selling analyst). The adversary is added to the agent roster as a regular agent but is initialized with a hardcoded `adversary: true` flag in their profile. When `adversary: true`, the agent's action-generation prompt includes an explicit adversarial directive prepended before the standard agent prompt: "Your role is to challenge the emerging consensus. Each round, identify the dominant narrative and compose a post that undermines it with credible counter-evidence or by surfacing overlooked risks. Amplify dissent from neutral agents. Do not simply disagree — provide specific, cited reasoning."
2. When Adversarial Mode is enabled at launch, run two sequential simulations: `baseline_id` (standard run) and `adversarial_id` (same config + adversary). Use identical random seeds for both. After both complete, store `{baseline_id, adversarial_id, seed}` in `<sim_dir>/adversarial_comparison.json`. Add `GET /api/simulation/:id/adversarial-comparison` returning: `{baseline_final_yes, adversarial_final_yes, consensus_delta (baseline_yes - adversarial_yes), consensus_sustained_baseline (bool: YES price > 0.65 for final 3 rounds), consensus_sustained_adversarial (bool), robustness_verdict: "Robust" (delta < 0.10) / "Pressured" (0.10–0.25) / "Fragile" (> 0.25 or consensus lost), adversary_influence_rank (adversary's leaderboard rank in the adversarial run), adversary_coalition_size (agents whose stance the adversary's posts can be attributed to shifting)}`.
3. In the simulation results view, when an adversarial comparison exists, add an "Adversarial Report" section in the sidebar. Show the robustness verdict as a colored badge ("Robust" = green, "Pressured" = orange, "Fragile" = red) with a one-line plain-English interpretation: "Consensus held within 8 percentage points under direct adversarial pressure — robust finding" or "Adversarial agent destabilized consensus in 3 rounds, reducing YES price from 78% to 51% — fragile result." Below the badge, show an overlaid belief drift chart (baseline line vs. adversarial line in distinct colors) and the adversary's influence rank ("Ranked #3 of 15 agents by post engagement"). In the article generator, inject the adversarial comparison as structured context when present: "Robustness: Adversarial robustness verdict is [Robust/Pressured/Fragile]. Baseline consensus: [X%]. Adversarial consensus: [Y%]. Delta: [Z pp]." Article language uses appropriate hedging based on verdict: Robust → confident language; Fragile → "results should be interpreted cautiously, as simulated consensus was sensitive to targeted skepticism."

### 4. Live Round Event Stream (Server-Sent Events)

**Type:** Integration
**Effort:** Small (hours)
**Impact:** The current monitoring options for a running simulation are: (a) poll the REST API every N seconds, (b) wait for the completion webhook to fire, (c) open the spectator watch page in a browser. None of these work for developers building CLI tools, custom dashboards, or automation pipelines who need a streaming event feed without a webhook server or browser. Server-Sent Events (SSE) is the HTTP-native solution: a persistent `GET /api/simulation/:id/events` connection that the server pushes events down as they complete. A developer can monitor a simulation with `curl --no-buffer https://miroshark.io/api/simulation/abc123/events` and see each round complete in real-time, with the `simulation_complete` event as the terminal signal. No webhook server, no polling loop, no browser dependency. SSE is unidirectional (server → client), built into every modern HTTP framework, and requires no additional infrastructure. For the automation audience who arrived via the OpenAPI spec and completion webhook, SSE is the streaming primitive that closes the last monitoring gap — and opens the door to external dashboards built on live simulation data.
**How:**
1. Add `GET /api/simulation/:id/events` as a Flask endpoint with `Content-Type: text/event-stream` and `Cache-Control: no-cache`. Use `flask.stream_with_context` and `flask.Response` with a generator. The generator polls `<sim_dir>/round_log.json` for new round entries (appended by the simulation runner after each round completes). When a new round entry appears, yield `data: {json}\n\n` where the JSON is `{event: "round_complete", round: N, yes_price: float, no_price: float, neutral_price: float, top_agent: string, post_count: int, timestamp: ISO}`. Polling interval: 500ms. When `<sim_dir>/complete.json` exists, yield a final `data: {event: "simulation_complete", rounds: N, final_yes: float, final_no: float, consensus: bool, article_url: string}\n\n` and close the generator. Add a `retry: 5000` comment line at stream start (tells clients to reconnect after 5s on connection loss). The round_log.json is written by the simulation runner's existing per-round completion hook — this is an append-only JSON array, one entry per round.
2. Document the SSE endpoint in the OpenAPI spec (use a `text/event-stream` response schema with a note: "SSE stream — use `EventSource` in browser or `curl --no-buffer` from CLI. The stream terminates with a `simulation_complete` event."). Add three usage examples in `docs/API.md`: (a) `curl --no-buffer` CLI example showing round-by-round output in a terminal; (b) JavaScript `EventSource` browser example feeding a live chart; (c) Python `sseclient` example for automation pipelines. In the spectator watch page (`/watch/<sim_id>`, PR #67), refactor the existing polling loop to use `EventSource` instead of `setInterval + fetch` — this eliminates unnecessary API calls during active simulations and simplifies the client code. The spectator page becomes an implicit integration test for the SSE endpoint.
3. Add a `belief_summary` field to each round event: `{yes_pct: float, no_pct: float, neutral_pct: float, swings: [{agent, from_stance, to_stance}] — agents who changed stance this round}`. This makes the event stream self-contained for clients building live belief visualizations — they don't need to make additional API calls to render a real-time belief drift line. Add a `?since=<round_number>` query parameter so clients can reconnect mid-simulation without replaying all past events: `GET /api/simulation/:id/events?since=7` starts the stream from round 8 onwards. Add the endpoint to `docs/API.md` under a new "Streaming" section. Note in the OpenAPI spec: "The SSE endpoint and the completion webhook cover the same events via different delivery mechanisms — SSE for connected clients, webhooks for server-side automation."

### 5. Academic Citation Block Generator

**Type:** DX
**Effort:** Small (hours)
**Impact:** PR #75 established the reproducibility config export as a "citation primitive" — a machine-readable record of exactly what was simulated. The missing piece is the human-facing output: the formatted citation block a researcher can paste into their paper's references. Generating BibTeX, APA, and IEEE citations from MiroShark simulation metadata is a single endpoint that closes the gap between "we have a reproducibility config" and "this simulation is actually citable in academic work." The researcher's workflow today: (1) run simulation, (2) get result, (3) manually write a citation (format varies, author is ambiguous — "MiroShark" or "aaronjmars"? or "MiroShark AI"?, date is just the run date?, URL is the share link?). With the Citation Block Generator, step 3 is "click Cite → copy BibTeX." For a project where 222 forks include an academic/researcher cohort that is the highest-engagement segment, removing the friction of citation construction directly upgrades MiroShark's scientific credibility. The NeuroClaw framework and MCP-SIM both identify provenance logging and citation infrastructure as core reproducibility requirements — MiroShark is one endpoint away from meeting that standard.
**How:**
1. Add `GET /api/simulation/:id/citation` accepting an `?format=bibtex|apa|ieee` query parameter (default: all three). Return a JSON object with three keys: `bibtex`, `apa`, `ieee`, each containing the formatted citation string. Derive citation fields from simulation metadata: `author` = "MiroShark Simulation Engine" (consistent machine-readable author for all simulations), `title` = scenario excerpt truncated to 100 characters (quoted and cleaned of special characters), `year` + `month` + `day` = simulation `created_at` date, `url` = the simulation's `share_url`, `note` = "Reproducibility config: [config_hash from PR #75 export]", `simulator_version` = current MiroShark version from `package.json`, `agent_count` and `round_count` as custom BibTeX fields. BibTeX entry type: `@misc`. APA format: `MiroShark Simulation Engine. ([year], [Month Day]). [scenario_title] [Simulation]. MiroShark. [url]`. IEEE format: `"[Scenario title]," MiroShark Simulation Engine, [url], [Month Day, Year].`
2. In the simulation results view, add a "Cite" button in the share toolbar (alongside "Share," "Export," "Embed"). On click, open a modal with three tabs: BibTeX, APA, IEEE. Each tab shows the formatted citation in a monospace code block with a "Copy to clipboard" button. Below the citation, show the reproducibility config hash (from PR #75) as a `Simulation fingerprint:` field with a copy button — this is the stable identifier researchers use to verify they're citing the correct simulation config. Add a note: "To verify reproducibility, compare this fingerprint against the exported config at [link to config export]." In the gallery explore view, add a small "Cite" icon on each gallery card (visible on hover, alongside the existing share/embed icons) — lets researchers cite interesting gallery simulations without opening the full detail view.
3. Add a `CITATION.cff` generation endpoint: `GET /api/simulation/:id/citation?format=cff` returns a CITATION.cff-format YAML string (the GitHub-standard citation format used for software repositories). Include all required fields: `cff-version`, `title`, `authors`, `date-released`, `url`, `version` (derived from `simulator_version`), `message: "If you use this simulation in your research, please cite it as follows."` In `docs/API.md`, add a "Citing MiroShark Simulations" section that explains the citation endpoint, lists all supported formats, and provides an example workflow: run simulation → export config (PR #75) → generate citation → embed in paper. In the README, add a "Citing results" section linking to the docs. This positions MiroShark alongside tools like CrystalPlasticitySim and JutulGPT — platforms that treat citation infrastructure as a first-class feature, not an afterthought.

## Selection Rationale

Five angles distinct from all 25 ideas suggested across May 2, May 4, and May 6, and from the open issue #70. Today's thread runs through a single theme: making simulation results *scientifically defensible* — not just shareable or embeddable, but citable, robust, and individually analyzable.

- **Agent Belief Heatmap** (#1) — The visualization gap. The aggregate drift chart is a summary; the heatmap is the evidence. Researchers need to see *how* consensus formed (rapid cascade vs. slow drift vs. persistent holdout cluster) to assess whether the outcome is meaningful. Uses existing data, pure frontend addition, zero new API work.
- **Per-Agent Trajectory Export** (#2) — The unit-record data gap. Aggregate exports (PR #66) summarize; per-agent export gives researchers the rows their statistical models actually need. Individual-level analysis — survival models, influence attribution, archetype clustering — is impossible without the unit record. Small implementation; removes the ceiling on research use cases.
- **Adversarial Robustness Mode** (#3) — The validity gap. A simulation that only runs in a friendly environment isn't a test — it's a demonstration. The "Position" paper identifies the absence of adversarial testing as the core validity criticism of LLM social simulation. Adversarial Mode answers that criticism directly: show that the consensus held (or didn't) under active challenge.
- **Live Event Stream (SSE)** (#4) — The streaming gap. CLI tools, custom dashboards, and automation pipelines need streaming events without webhook servers or browsers. SSE is the HTTP-native solution. Refactoring the spectator watch page to use it makes the feature self-validating. Closes the last real-time monitoring gap for developer integrations.
- **Academic Citation Block Generator** (#5) — The citation gap. PR #75 established the reproducibility primitive; the citation generator is the human-facing output. BibTeX + APA + IEEE cover 95%+ of academic citation needs. For a project at 1,117 stars with a growing research cohort, "citable in a paper" is a qualitatively different positioning than "fun to share on Twitter."
