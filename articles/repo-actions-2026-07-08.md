### 1. Air-Gapped HuggingFace Cache Polish

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** Issue #240 exists because MiroShark's `smart` and social-agent modes pull HuggingFace models (reranker + Twitter BERT) at runtime — which fails silently in enterprise and regulated-environment deployments with no outbound internet. dan-and already built the core: `TRANSFORMERS_OFFLINE=1` env flag, docker-compose service to pre-pull models into a named volume, export-to-`.tgz` and import-back scripts. His concern is simplicity. The right contribution is not to build a competing approach but to clean up his interface: one `docker compose --profile offline setup-models` command to pre-pull, one env flag to activate offline mode, and clear error messages when the cache is missing. Landing this PR unblocks enterprise adoption — a segment that can't self-host anything requiring live network access during inference. The air-gapped user demographic is also the most likely to write a case study (the September hyperstition: "Will someone publish a real decision case study?"), because enterprise teams document deployments.

**How:**
1. Read dan-and's branch at `https://github.com/dan-and/MiroShark/tree/feat/offline-hf-cache` (WebFetch it) and catalog what his implementation does: how the models are pulled, the export/import scripts, and where he felt the UX was rough. Identify the simplification opportunities. His setup likely involves multiple manual `docker-compose` invocations and the export command is probably verbose. The goal is to reduce the user flow to two commands: (a) one command to pre-stage the models on the internet-connected machine: `docker compose --profile offline run --rm model-preloader` — this pulls `cross-encoder/ms-marco-MiniLM-L-6-v2` (reranker) and `cardiffnlp/twitter-roberta-base-sentiment-latest` (Twitter BERT) into the `hf_cache` Docker volume, then exports it as `miroshark_hf_cache.tgz` to the current directory; (b) one command to restore on the air-gapped machine: `docker compose --profile offline run --rm model-importer` — reads `miroshark_hf_cache.tgz` from the current directory and imports it into the same volume name. Add `TRANSFORMERS_OFFLINE=1` and `HF_DATASETS_OFFLINE=1` to the `offline` compose profile so the app never phones home after import. Add a startup check: if `TRANSFORMERS_OFFLINE=1` is set and the cache directory is empty or missing the expected model subdirectories, abort with a human-readable error: `"Offline mode active but HF model cache is empty. Run: docker compose --profile offline run --rm model-preloader"` rather than silently falling back to degraded inference. Write this check in `backend/app/services/inference_utils.py` or wherever the models are first loaded.
2. Update `docker-compose.yml` to add an `offline` profile with two services: `model-preloader` (runs on the internet-connected machine; bind-mounts a local `./hf_export/` directory; pulls both models via `python -c "from transformers import AutoModel, AutoTokenizer; AutoModel.from_pretrained('cross-encoder/ms-marco-MiniLM-L-6-v2'); AutoModel.from_pretrained('cardiffnlp/twitter-roberta-base-sentiment-latest')"` into the `hf_cache` volume; then tars the volume to `./hf_export/miroshark_hf_cache.tgz`) and `model-importer` (runs on the air-gapped machine; reads `./hf_export/miroshark_hf_cache.tgz`; extracts into the `hf_cache` volume). Both services use the same `backend` image so no new Dockerfile is needed. The `hf_cache` volume is already defined for Nemotron persistence (PR #238 established the pattern) — the offline models share the same volume to avoid duplication. Tag both services `profiles: [offline]` so they don't appear in the normal `docker compose up` flow. Add the `TRANSFORMERS_OFFLINE`, `HF_DATASETS_OFFLINE` env vars to the main `backend` service definition, defaulting to `0`; the offline profile overrides them to `1`.
3. Add a `## Air-Gapped / Offline Deployment` section to `README.md` (under the existing self-hosting section): "If your server cannot reach `huggingface.co` at runtime, pre-stage the models on an internet-connected machine and transfer the cache. Two commands, one tarball. Full guide: [docs/OFFLINE.md]." Create `docs/OFFLINE.md` (~80 lines): what models are required and why (reranker for Smart mode relevance scoring, Twitter BERT for sentiment classification in social-agent platform simulation), the two-command workflow with exact invocations, the env vars activated by the offline profile, the startup error message and what to do if you see it, and a note that the model cache is ~2GB compressed. Close (or offer to close) issue #240 with a comment crediting dan-and's original implementation and linking to the PR — explicitly invite him to review and give the thumbs-up he mentioned. His branch becomes the foundation; this PR is the cleanup pass he was asking for.

### 2. CLI `list` Subcommand

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** The CLI now covers the full simulation lifecycle: `simulate` (start), `wait` (block until done), `stop` (cancel), `cost` (check spend). The missing operation is discovery: operators running multiple simulations have no way to browse their history from the terminal without opening a browser. `miroshark list` fills this gap — a simple table of recent simulations (ID, topic, status, date, cost) printed to stdout, with `--json` for piping and `--limit` for controlling depth. This completes the CLI as a standalone tool: a developer can run, monitor, and review simulations without ever touching the browser. For the "CLI simulate everything" narrative that the Jul 4 article established as the core MiroShark identity, `list` is the one obvious gap. It also enables shell scripting: `miroshark list --json | jq '.[] | select(.status=="complete") | .id' | xargs -I{} miroshark signal {}` — pipeline to batch-fetch signals for all completed simulations.

**How:**
1. Add a `list` subcommand to the existing CLI entry point (wherever `simulate`, `wait`, `stop`, `cost` are defined — likely `cli/miroshark.py` or `cli/main.py`). `miroshark list [--limit N] [--status STATUS] [--json]`. Flags: `--limit` (int, default 20, max 100); `--status` (string, one of: `running`, `complete`, `failed`, `all`; default `all`); `--json` (flag; if set, output raw JSON array instead of formatted table). Implementation: GET `{MIROSHARK_URL}/api/activity.json` (the activity feed endpoint from PR #153, which returns recent simulation events including IDs, topics, status, and timestamps). If the activity endpoint is insufficient (e.g. doesn't include cost or doesn't support status filtering), fall back to GET `{MIROSHARK_URL}/api/simulations` with appropriate query params — check which endpoint exists and use whichever is most complete. Parse the response into a list of `{id, topic, status, started_at, completed_at, cost_usd}` dicts. Apply `--status` filter client-side if the endpoint doesn't support it server-side. Table output (default, no `--json`): use Python's `textwrap` and string formatting (no external dependencies) to print a fixed-width table: `| ID | Topic (truncated 40 chars) | Status | Date | Cost |`. Align columns, right-pad topic. Color-code status if `sys.stdout.isatty()`: green=complete, yellow=running, red=failed (use ANSI escape codes, no dependencies). JSON output: pretty-print the raw list with `json.dumps(data, indent=2)`. Exit 0 always (empty result is not an error; print "No simulations found." if list is empty and not `--json`). Add to `cli/README.md` (or wherever CLI usage is documented): `miroshark list [--limit N] [--status STATUS] [--json]` with two usage examples.
2. Add `--self` flag: `miroshark list --self` filters to simulations started by the current CLI session (tracked by a local state file, e.g. `~/.miroshark/session.json` which `simulate` already writes to if it records the ID). This lets operators quickly find "the simulations I started today" without wading through everyone else's public sims. If the session file doesn't exist or is empty, `--self` falls back to `--limit 20` with a note: "No session history found — showing recent public simulations." Also add `miroshark status {SIM_ID}` as an alias for a single-sim lookup if it doesn't exist yet: `miroshark status abc123` returns the same fields as a single row from `list`. This makes the CLI more POSIX-idiomatic — `list` for many, `status` for one.
3. Update the main `README.md` CLI section to include `list` in the command reference table. The table should now read: `simulate` / `wait` / `stop` / `cost` / `list` / `status` — the full lifecycle plus discovery. Add a "pipeline example" code block: `# Fetch signals for all completed simulations` / `miroshark list --json --status complete | jq -r '.[].id' | while read id; do miroshark signal "$id"; done` — this shows the composability that makes CLI-first tools worth adopting. Add `list` to the shell completion script if one exists. File a GitHub issue in MiroShark (or add to an existing tracking issue): "CLI: add `list` and `status` subcommands" — label it `good first issue` since the implementation is straightforward and well-specified.

### 3. Simulation RSS Feed

**Type:** Integration / Feature
**Effort:** Small (hours)
**Impact:** The activity feed (`/api/activity.json`) surfaces recent simulation completions as structured JSON. An RSS 2.0 endpoint over the same data opens MiroShark to an ecosystem of readers and automation tools that consume RSS natively: Feedly, Readwise Reader, Zapier (RSS triggers), Make.com, n8n, and any monitoring tool with RSS input. This is meaningful for the academic citation hyperstition: researchers who follow RSS feeds for their domain (computational social science, prediction markets, AI agents) will discover MiroShark topics in their reader the same way they discover new papers. For the external-deployment hyperstition: a fork holder who configures a Zapier workflow "on new MiroShark simulation → post to Slack" is running a live integration. One RSS endpoint, multiple downstream integration surfaces. Implementation is a Python string template — zero new dependencies, RSS 2.0 spec is simple, and the data is already available from the activity feed service.

**How:**
1. Add `backend/app/routes/feed.py` (~80 LoC, pure stdlib). `GET /api/feed.rss` — no authentication, public endpoint. Query the activity feed service (or read `data/simulations/` directly) for the 20 most recently completed, published simulations sorted by `completed_at` descending. For each simulation, fetch `signal.json` for `{direction, yes_pct, confidence_pct, topic}` and `simulation_config.json` for `{agent_count, num_rounds, platforms}`. Build an RSS 2.0 XML response using Python's `xml.etree.ElementTree` (stdlib): channel title: "MiroShark — Recent Simulations"; channel link: `{BASE_URL}`; channel description: "Real-time swarm intelligence simulation results"; `lastBuildDate`: ISO 8601 of the most recent simulation's `completed_at`; `ttl`: 30 (minutes). Each item: `<title>` = the simulation topic (HTML-escaped); `<link>` = `{BASE_URL}/simulation/{id}`; `<description>` = a 2-sentence summary: "MiroShark simulated '{topic}' with {agent_count} agents across {num_rounds} rounds on {platforms}. Result: {DIRECTION} — {yes_pct}% yes, {confidence_pct}% confidence." (HTML-escaped); `<pubDate>` = RFC 822 datetime of `completed_at`; `<guid isPermaLink="true">` = the simulation URL. Return `Content-Type: application/rss+xml; charset=utf-8`, `Cache-Control: public, max-age=300` (5 minutes — same as the activity feed cache). If no published simulations exist, return a valid RSS channel with zero items (not a 404). Register the route in `backend/app/main.py`. Add 6 unit tests: valid RSS 2.0 XML produced, items sorted by completed_at desc, limit of 20 enforced, HTML entities escaped in topic, empty simulation list produces valid empty channel, Cache-Control header present.
2. Add `GET /api/feed.rss` to the OpenAPI spec under a new "Feeds" tag (alongside the search endpoint in the Discovery group). Add a "Subscribe" link in the MiroShark frontend: in the main nav or footer, add a small RSS icon (⊕ or the standard RSS orange square icon via SVG) linking to `/api/feed.rss`. Use the standard RSS SVG icon (the three arcs on an orange background — widely recognizable) rather than text, so it doesn't add visual weight to the nav. Tooltip: "Subscribe to simulation results (RSS)". This makes the feed discoverable without requiring users to know the endpoint exists. Add `<link rel="alternate" type="application/rss+xml" title="MiroShark Simulations" href="/api/feed.rss">` to the `<head>` of `frontend/index.html` — this auto-discovery tag lets browsers and RSS readers detect the feed when a user visits the site.
3. Add a `## RSS Feed` section to `docs/FEATURES.md`: what it returns (recent public simulations, RSS 2.0 format), the URL, the 5-minute cache, and example use cases: "Subscribe in Feedly, Readwise, or any RSS reader"; "Trigger Zapier workflows when new simulations complete"; "Monitor specific topics by filtering RSS items by keyword in your reader." Add a one-liner to the main README API table: `/api/feed.rss — RSS 2.0 feed of recent simulation results (public, no auth required)`. Post a comment on issue #240 (the air-gapped deployment one) noting that the RSS feed works independently of the model inference stack — operators who set up air-gapped inference can still use RSS to monitor simulation outputs from a separate, internet-connected monitoring host.

### 4. Python SDK (`miroshark-py`)

**Type:** DX / Integration
**Effort:** Medium (1-2 days)
**Impact:** The CLI is great for interactive use. For programmatic integration — research scripts, notebooks, CI pipelines, data pipelines — developers want to `import miroshark` in Python rather than shell out to the CLI. `pip install miroshark-py` gives them a typed, synchronous client: `from miroshark import MiroShark; client = MiroShark("https://miroshark.xyz"); result = client.simulate("Will AI regulation pass in the EU?", agents=20, rounds=5).wait()`. This lowers the integration barrier for the researcher segment that the Jupyter notebooks (Jul 06 idea #3) target — rather than writing raw `requests` calls, they get a client with IDE autocomplete, typed response objects, and blocking semantics. For the academic citation hyperstition: a paper that writes `from miroshark import MiroShark` in its methodology section creates a stronger citation than one that says "we called the API with curl." The SDK also creates a PyPI presence — the package appears in `pip search`, in Python ecosystems indexes, and in AI tools roundups that scan PyPI for agent/simulation packages.

**How:**
1. Create a `sdk/` directory at the repo root. Structure: `sdk/miroshark/__init__.py` (public API), `sdk/miroshark/client.py` (MiroShark class), `sdk/miroshark/models.py` (typed response dataclasses), `sdk/miroshark/exceptions.py` (MiroSharkError, SimulationTimeout, SimulationFailed), `sdk/pyproject.toml`, `sdk/README.md`. `pyproject.toml`: package name `miroshark-py`, version `0.1.0`, requires `requests>=2.28.0`, Python `>=3.9`. Intentionally minimal — only `requests` as a dependency, no httpx, no pydantic, no async. The `MiroShark` class in `client.py`: `__init__(self, base_url="https://miroshark.xyz", api_key=None, timeout=600)`. Methods: `simulate(topic, agents=20, rounds=5, platforms=None) -> Simulation` — POST `/api/simulate`, returns a `Simulation` object with `id` and a `wait(poll_interval=5, max_wait=600) -> SimulationResult` method. `signal(sim_id) -> Signal` — GET `/api/simulation/{id}/signal`. `status(sim_id) -> str` — GET `/api/simulation/{id}/status`, returns `"running"`, `"complete"`, or `"failed"`. `list_simulations(limit=20, status=None) -> list[SimulationSummary]` — GET `/api/activity.json`. `Simulation.wait()` polls `GET /api/simulation/{id}/status` every `poll_interval` seconds; raises `SimulationTimeout` if `max_wait` exceeded; raises `SimulationFailed` if status becomes `"failed"`. Dataclasses in `models.py`: `SimulationResult(id, topic, direction, yes_pct, confidence_pct, agent_count, num_rounds, completed_at)`, `Signal(direction, yes_pct, confidence_pct, timestamp)`, `SimulationSummary(id, topic, status, created_at)`. All dataclasses are plain Python (`@dataclass`, no pydantic). No async variant in v0.1.0 — keep the scope small.
2. Add 15 unit tests in `sdk/tests/test_client.py` using `unittest.mock.patch` on `requests.post`/`requests.get`. Tests: simulate() posts to correct URL, simulate() returns Simulation with correct id, wait() polls status endpoint, wait() raises SimulationTimeout after max_wait, wait() raises SimulationFailed on failed status, signal() returns typed Signal, list_simulations() applies limit param, api_key is sent as Bearer token when provided, MiroSharkError raised on 4xx responses, MiroSharkError raised on 5xx responses, custom base_url overrides default, platforms param serialized correctly, SimulationResult fields typed correctly, poll_interval respected between calls, SimulationResult.direction is uppercase string. Add `sdk/README.md` (~60 lines): installation (`pip install miroshark-py` or `pip install git+https://github.com/aaronjmars/MiroShark.git#subdirectory=sdk`), a 10-line quickstart code block, API reference (4 methods with signatures and return types), error handling examples, and a note that the SDK targets the same public API as the CLI — anything the CLI can do, the SDK can do. Add a `## Python SDK` section to the main README pointing to `sdk/README.md`.
3. Add a `pyproject.toml` entry to publish to PyPI via GitHub Actions: create `.github/workflows/publish-sdk.yml` triggered on `push: tags: ['sdk-v*']`. Steps: checkout, `pip install build twine`, `python -m build sdk/`, `twine upload dist/*` (using `PYPI_API_TOKEN` secret). This means the first `git tag sdk-v0.1.0 && git push --tags` publishes to PyPI — the feature skill builds the SDK locally, the operator tags and pushes when ready. Add `sdk/` to the PR description: "Not published yet — tag `sdk-v0.1.0` to release to PyPI." Also add `pip install miroshark-py` to the Jupyter notebooks (Jul 06 idea #3): replace the `import requests` manual approach with `from miroshark import MiroShark` in cells 2-3 of each notebook. The notebooks become the SDK's first-party usage examples, and the SDK becomes the notebooks' ergonomic layer.

## Selection Rationale

Issue #240 forces idea #1 into the list — dan-and has built a working air-gapped solution and is explicitly asking for feedback before merge. The right action is not to start over but to polish his approach and help it land. Missing that signal would waste a live community contribution.

The other four ideas address four different hyperstition vectors, none covered in the past 7 days:

- **CLI list** (#2) — Completes the CLI as a standalone tool. The narrative "4 CLI commands run a 100-agent simulation end to end" (Jul 4 article, 1,353 stars) is the most resonant piece of content so far. The CLI currently has a gap: you can simulate and stop but not browse. This closes it.

- **RSS Feed** (#3) — Zero new dependencies, minimal complexity, maximum integration surface. RSS readers, Zapier, n8n, and Make.com all have RSS triggers. A fork holder who sets up a Zapier "MiroShark result → Slack" workflow is running an external integration — the kind that counts toward the external-deployment hyperstition.

- **Python SDK** (#4) — The most complex idea here but also the most durable. PyPI presence is permanent; it creates discoverability in the Python ecosystem independently of GitHub stars. The Jupyter notebooks (Jul 06 #3) need this as their client layer. Academic researchers need this to write clean methodology code.

- **Show HN Kit** (#5) — Seven days is enough to submit and get front page traction if the submission is good and the timing is right. This kit removes the two blockers that have prevented submission so far: "I don't know what to write" and "I don't know when to post." The checklist structure ensures nothing is missed. If the submission lands front page, the star count and external deployment hyperstitions close simultaneously — HN traffic generates forks and integrators.
