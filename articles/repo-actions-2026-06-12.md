### 1. Agent Stance Flip Report

**Type:** Feature
**Effort:** Small (hours)
**Impact:** `agents.json` records each agent's initial and final stance. The aggregate trajectory chart shows how the crowd moved. Neither surface answers: *which specific agents changed their mind, from what to what, and at what round?* A simulation where 60% of agents started BEARISH and ended BULLISH has very different dynamics depending on whether 40 agents flipped early (the crowd led itself) or 2 high-influence agents flipped late and dragged the rest. `GET /api/simulation/:id/stance-flips` reads `agents.json` for initial/final stance pairs, cross-references per-round `trajectory.json` or `actions.jsonl` to identify the first round where each agent's stance changed, and returns `{flip_count, flip_rate_pct, top_flippers: [{agent, initial_stance, final_stance, flip_round, influence_score}]}` stored as `stance_flips.json`. A compact "Stance Shifts" panel on the watch page renders the flip rate as a percentage and lists the top 5 flippers with their before/after stance and round. For scenario designers, this is the "persuasion map" — it tells you which agents were persuadable and when the persuasion happened. That's the signal for designing better adversarial stress tests and knowing which agent archetypes to seed differently next time.

**How:**
1. Add `backend/app/services/stance_flip_service.py` (~140 LoC, pure stdlib). `compute_stance_flips(sim_dir)` reads `agents.json` (array of `{agent_id, name, initial_stance, final_stance, influence_score}`). For each agent where `initial_stance != final_stance`, this agent is a flipper. To find `flip_round`: read `actions.jsonl` line by line (each line: `{round, agent, stance, ...}`). For each flipper, scan chronologically and record the first `round` where `stance != initial_stance`. If `actions.jsonl` is absent (older sims), set `flip_round: null`. Compute: `flip_count` (count of flippers), `flip_rate_pct` (flip_count / total_agents * 100). Sort `top_flippers` by `influence_score` desc (most influential flippers first), cap at 10. Write `{generated_at: ISO, flip_count: int, flip_rate_pct: float, total_agents: int, top_flippers: [...]}` to `<sim_dir>/stance_flips.json` with mtime-based cache (only recompute if `agents.json` is newer than `stance_flips.json`). `get_stance_flips(sim_dir)` reads the cache if fresh. Add `GET /api/simulation/:id/stance-flips` (public for published sims, admin-gated for private). Returns the payload; 404 if sim not found; `{flip_count: 0, flip_rate_pct: 0.0, top_flippers: []}` if `agents.json` is absent. Add `stance_flips` to `surfaces_catalog.py` (38th surface) and `surface_stats.py`.
2. In `Step3Simulation.vue`, add a "Stance Shifts" toolbar toggle button (alongside the existing "@ Mentions", "Platforms" buttons). On click, fetch `GET /api/simulation/:id/stance-flips` and show an overlay panel. Panel content: a header row — "N agents flipped (X%)" — with a progress bar showing `flip_rate_pct` in violet. Below: a ranked list of top 5 flippers: `{rank}. {agent_name} — {initial_stance} → {final_stance}{flip_round != null ? " (round " + flip_round + ")" : ""}`. Stance labels use color chips: BULLISH in green, BEARISH in red, NEUTRAL in gray. If `flip_count === 0`, show "No stance changes detected — all agents held their initial conviction." Add `getStanceFlips(simId)` to `frontend/src/api/simulation.js`. In `EmbedDialog.vue`, add a "Stance Flips" section with the endpoint URL and a curl example. Bilingual EN/ZH labels on all panel text.
3. Add `GET /api/simulation/:id/stance-flips` to the OpenAPI spec under the "Analytics" tag with `StanceFlipResponse` schema: `{flip_count: int, flip_rate_pct: float, total_agents: int, top_flippers: [{agent: str, initial_stance: str, final_stance: str, flip_round: int|null, influence_score: float}]}`. Add 14 unit tests in `backend/tests/test_unit_stance_flips.py`: no flippers returns flip_count=0, single flipper detected correctly, flip_round extracted from actions.jsonl chronologically (not just first line), flip_round is null when actions.jsonl absent, flip_rate_pct = flip_count / total_agents * 100 (float), top_flippers sorted by influence_score desc, top_flippers capped at 10, agents with same initial and final stance excluded, cache written on first call, cache read on second call when agents.json unchanged, stale cache triggers recompute when agents.json newer, 404 for unknown sim_id, admin token required for private sim, bilingual stance labels (BULLISH/Bullish, 看涨) consistent. Add a "Stance Flip Report" section to `docs/FEATURES.md`: explains the flip detection logic (initial vs. final stance from agents.json), notes the flip_round precision caveat (requires actions.jsonl; otherwise null), and explains the influence_score sort order (why high-influence flippers appear first — they drove the most downstream opinion change).

### 2. Simulation Full-Text Search

**Type:** Feature
**Effort:** Small (hours)
**Impact:** With a growing inventory of simulations, operators have no way to find a specific simulation by topic without scrolling the gallery. An ecosystem partner who wants to find "all simulations about interest rates" has to browse manually. `GET /api/simulations/search?q=...` implements pure-stdlib case-insensitive regex search across `simulation_config.json` `title` and `description` fields for all published simulations. Returns `{results: [{sim_id, title, description_snippet, direction, confidence_score, created_at}], total: int}` with a 5-minute mtime-aware in-memory index (rebuilt when any sim dir has a newer `simulation_config.json` than the index timestamp). For the gallery, a search bar at the top of `/explore` dispatches against this endpoint. For ecosystem partners, this is the discoverability primitive — instead of requiring them to cache the full gallery locally, they can search by topic. No external search library, no database, no new dependencies.

**How:**
1. Add `backend/app/services/search_service.py` (~150 LoC, pure stdlib). `build_index(data_dir)` scans all published sim dirs (same scandir pattern as `leaderboard_service`). For each, reads `simulation_config.json` (fields: `title`, `description`), `signal.json` (fields: `direction`, `yes_pct`), `confidence.json` (field: `confidence_score`). Builds `_index: list[dict]` where each entry is `{sim_id, title, description, direction, confidence_score, created_at}`. Stores `_index_built_at: datetime`. `search(query, data_dir, limit=20)` calls `build_index` only if the index is stale (any published sim's `simulation_config.json` mtime newer than `_index_built_at`, or index is None). Compiles `pattern = re.compile(re.escape(query), re.IGNORECASE)`. Filters index entries where `pattern.search(entry['title'])` or `pattern.search(entry['description'] or '')`. For each match, generates `description_snippet`: first 120 chars of description with the matched term bolded using `**term**` markdown syntax. Sorts results by `created_at` desc (newest first). Returns `{results: matches[:limit], total: len(matches)}`. Special cases: empty query returns `{results: [], total: 0}` (no wildcard dump). Query `len < 2` returns 400 "Query too short." Add `GET /api/simulations/search` to `simulation.py` blueprint (unauthenticated, published sims only). Returns JSON; 400 on missing/short query; 200 with empty results if no matches. Add `search` to `surfaces_catalog.py` as a `utility` type surface.
2. In `ExploreView.vue` (the gallery page), add a search input at the top of the page above the existing filter tabs. On input (debounced 300ms), call `GET /api/simulations/search?q={value}` if `value.length >= 2`. While searching, replace the gallery grid with a "Searching…" spinner. On results, render matching sims in the same card grid layout as the gallery — but with the matched term highlighted in the title and snippet. If `total === 0`, show "No simulations found for '{query}'" with a "Clear search" button. On clear, restore the normal gallery view. Search state is reflected in the URL query param `?q=...` for shareability and browser back/forward. Add `searchSimulations(query)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH placeholder text ("Search simulations… / 搜索模拟…") and empty state message.
3. Add `GET /api/simulations/search` to the OpenAPI spec under the "Discovery" tag (alongside the gallery and explore endpoints) with `SearchResponse` schema: `{results: [{sim_id: str, title: str, description_snippet: str, direction: str, confidence_score: float|null, created_at: str}], total: int}`. Add 12 unit tests in `backend/tests/test_unit_search.py`: basic title match, basic description match, case-insensitive match, special-regex-char in query escaped (e.g. "C++"), empty query returns 400, single-char query returns 400, query with no matches returns empty results, description_snippet truncated at 120 chars, description_snippet has **bolded** match term, results sorted by created_at desc, index rebuilt when new sim added (mtime newer), unpublished sim excluded from results. Add a "Simulation Search" section to `docs/FEATURES.md`: explains the mtime-based index invalidation strategy (no manual invalidation needed), notes the query minimum length (2 chars), and explains why wildcard/empty queries are rejected (would dump the full inventory — use the gallery endpoint for that).

### 3. Confidence Component Breakdown

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The confidence score is a single number (0–100) composed of four sub-scores: stability (trajectory variance), convergence (final-round spread), participation (agent count relative to a baseline), and adversarial robustness (stress-test pass rate). Today, an operator who sees "confidence: 71" has no idea whether that reflects high stability with low convergence, or full participation with a failed adversarial test. The composite obscures the diagnosis. `GET /api/simulation/:id/confidence/components` re-runs the four component calculations individually and returns each sub-score (0–100) alongside its weight and label. Rendered as a 4-bar breakdown alongside the existing composite score display on the watch page: stability 88 / convergence 45 / participation 100 / adversarial 60 → composite 71. Operators immediately see "convergence is the weak link — agents hadn't fully settled by the final round." This is the "explain my score" surface. No new computation: it reuses the existing `confidence_service` logic with minor refactoring to expose components rather than only the composite.

**How:**
1. Refactor `backend/app/services/confidence_service.py` to extract component functions. Currently `compute_confidence` computes all four components inline. Extract: `compute_stability_score(trajectory)` → float 0–100, `compute_convergence_score(trajectory)` → float 0–100, `compute_participation_score(agent_count)` → float 0–100, `compute_adversarial_score(adversarial_report)` → float 0–100 (0 if report absent). Add `get_confidence_components(sim_dir)` (~40 new LoC) that reads the same source files as `compute_confidence` but returns a structured breakdown: `{composite: float, components: [{key: str, label: str, score: float, weight: float, description: str}]}` where components is fixed-order `[stability, convergence, participation, adversarial]` with human-readable labels ("Trajectory Stability", "Final-Round Convergence", "Agent Participation", "Adversarial Robustness") and one-line descriptions. Cache to `confidence_components.json` with same mtime pattern as `confidence.json`. Add `GET /api/simulation/:id/confidence/components` (public for published, admin-gated for private). Returns the payload; 404 if sim not found; `{composite: null, components: [...with score: null...]}` if trajectory.json absent. The existing `compute_confidence` is refactored to call the four extracted functions rather than re-implementing them — no behavior change, only structural.
2. In `Step3Simulation.vue`, below the existing confidence score display (wherever the single number is shown), add a "Why this score" expandable section (collapsed by default, toggled by a small "↓ breakdown" link). On expand, fetch `GET /api/simulation/:id/confidence/components` (cached after first load). Render 4 rows, each: `{label} — {score.toFixed(0)}/100` with a thin horizontal bar (violet fill, proportional to score out of 100). Below the 4 rows: a one-line description of each component pulled from the API response. The composite score row at the top of the section shows "Composite: {composite.toFixed(0)}/100 (weighted average)". If adversarial score is 0 and the adversarial report is absent, add a gray note "Adversarial test not run — score defaulted to 0." In `EmbedDialog.vue`, add a "Confidence Components" section with the endpoint URL and a note that this endpoint returns the composite's derivation. Add `getConfidenceComponents(simId)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels.
3. Add `GET /api/simulation/:id/confidence/components` to the OpenAPI spec under the "Analytics" tag alongside the existing confidence endpoints. Add `ConfidenceComponentsResponse` schema: `{composite: float|null, components: [{key: str, label: str, score: float|null, weight: float, description: str}]}`. Add 12 unit tests in `backend/tests/test_unit_confidence_components.py`: four components always present in response, component keys are exactly [stability, convergence, participation, adversarial], weights sum to 1.0 (tolerance ±0.01), composite equals weighted average of component scores (tolerance ±0.5), adversarial score is 0 when adversarial_report.json absent, stability score increases for lower-variance trajectory, convergence score increases for tighter final-round spread, participation score is 100 for agent_count above baseline, cache written on first call, cache read on second call when source unchanged, compute_confidence still returns same value after refactor (regression test), 404 for unknown sim_id. Add a "Confidence Component Breakdown" section to `docs/FEATURES.md`: explains all four components, their weights, and the design intent (stability + convergence are trajectory-derived; participation is agent-count normalized; adversarial is a quality gate that can be 0 without the test).

### 4. Simulation Fork Lineage Graph

**Type:** Feature
**Effort:** Small (hours)
**Impact:** MiroShark already supports forking a simulation (the fork button creates a copy with a new `sim_id`). But the parent-child relationship between the original and the fork is currently invisible. When a researcher forks a simulation 4 times to test different parameter sets, there is no way to see which fork produced the most divergent result or to navigate the family tree. `GET /api/simulation/:id/lineage` returns `{sim_id, parent: {id, title, direction} | null, children: [{id, title, direction, created_at}], depth: int}` by reading a `fork_parent_id` field written into `simulation_config.json` at fork time. A "Lineage" chip on the watch page (visible only if `parent` or `children` exist) opens a compact sidebar showing the family tree — ancestors above, siblings/children below, current sim highlighted. For researchers running parameter sweep forks, this is the "what did we fork from, and what did we fork into?" surface. Ecosystem partners who track simulation series need this to associate derivative sims with originals.

**How:**
1. Modify `backend/app/api/simulation.py` fork endpoint (wherever a new sim is created from an existing one): when writing the new `simulation_config.json`, include `"fork_parent_id": parent_sim_id`. If no fork parent, this field is absent (not null). Add `backend/app/services/lineage_service.py` (~120 LoC, pure stdlib). `get_lineage(sim_dir, sim_id, data_dir)` reads the current sim's `simulation_config.json` for `fork_parent_id` (if present). For the parent: read `<data_dir>/<parent_id>/simulation_config.json` for title and `<data_dir>/<parent_id>/signal.json` for direction (gracefully handle missing parent dir — parent may have been deleted). For children: scan all sim dirs, read `simulation_config.json` of each, collect those whose `fork_parent_id == sim_id`. For each child, read its `signal.json` for direction. Compute `depth`: follow parent chain recursively (max 10 hops to prevent cycles) to find the root; depth = hop count from root to current sim. Returns `{sim_id, parent: {...} | null, children: [...], depth: int}`. No disk cache (lineage is cheap to compute and must be live — new forks should appear immediately). Add `GET /api/simulation/:id/lineage` (public for published sims). Returns the payload; 404 if sim not found. If sim has no parent and no children, returns `{parent: null, children: [], depth: 0}` (not an error).
2. In `Step3Simulation.vue`, in the sim header area (near the title), add a "Lineage" chip — visible only when `lineage.parent !== null || lineage.children.length > 0`. Fetch `GET /api/simulation/:id/lineage` on page load (parallel with other metadata fetches). On chip click, open a sidebar panel titled "Fork Tree". Render: if parent exists — a "Forked from" row with the parent's title (linked to `/simulation/{parent.id}`), direction chip, and a "↑ ancestor" label. Below: current sim row highlighted with a violet border, labeled "You are here". Below that: each child listed as "→ fork: {child.title}" with direction chip and relative creation date. If `depth > 0`, show "Generation {depth+1}" sub-label on the current sim row. If no parent and no children, the chip is hidden entirely (no empty panel). Add `getSimulationLineage(simId)` to `frontend/src/api/simulation.js`. In the gallery `SimulationCard.vue`, if the sim has `fork_parent_id` set, add a small "fork" badge icon to the card. Bilingual EN/ZH labels.
3. Add `GET /api/simulation/:id/lineage` to the OpenAPI spec under the "Simulation" tag with `LineageResponse` schema: `{sim_id: str, parent: {id: str, title: str, direction: str} | null, children: [{id: str, title: str, direction: str, created_at: str}], depth: int}`. Add 12 unit tests in `backend/tests/test_unit_lineage.py`: sim with no parent returns parent=null, sim with parent returns correct parent fields, sim with children returns correct children array, depth=0 for root sim (no parent), depth=1 for first-generation fork, depth follows parent chain not children chain, deleted parent handled gracefully (parent dir missing → parent: null), new child fork appears in lineage immediately (no cache), children sorted by created_at asc, fork endpoint writes fork_parent_id to new sim config, gallery card shows fork badge when fork_parent_id present, 404 for unknown sim_id. Add a "Fork Lineage Graph" section to `docs/FEATURES.md`: explains the fork_parent_id write-time recording, notes the depth=10 cycle-prevention cap, and describes the live (non-cached) design rationale.

### 5. Per-Round Agent Participation Heatmap

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The trajectory chart shows what the crowd believed each round. The activity feed shows recent events. Neither surface answers: *how many agents were actually active in each round, and was that consistent across platforms?* Agents can become inactive mid-simulation (no posts, no reactions). A simulation where 80 agents participated in rounds 1–4 but only 12 were active by round 10 is structurally different from one with steady 80-agent participation throughout. `GET /api/simulation/:id/participation-heatmap` reads `actions.jsonl` and returns `{rounds: [{round: int, total_active: int, platforms: [{name: str, count: int}]}]}` — the count of distinct agents who took any action in each round, broken out by platform. On the watch page, a compact participation bar chart (one bar per round) sits beneath the existing trajectory chart. Thin bars indicate low-activity rounds; full bars indicate full engagement. For simulation designers, "participation collapse" in the final rounds is a data quality signal — it means the outcome was driven by a shrinking minority. Distinct from trajectory (stance %), agent count (total headcount), and activity feed (recent-events log).

**How:**
1. Add `backend/app/services/participation_service.py` (~110 LoC, pure stdlib). `compute_participation_heatmap(sim_dir)` reads `actions.jsonl` line by line. For each line: extract `round` (int), `agent` (str), `platform` (str, default "unknown"). Build `round_data: dict[int, dict]` where `round_data[r] = {total: set(), platforms: {name: set()}}` — `total` holds distinct agent names, `platforms[name]` holds distinct agents for that platform in that round. After scanning: for each round, convert sets to counts. Sort rounds ascending. Returns `{rounds: [{round: int, total_active: int, platforms: [{name: str, count: int}]}], max_active: int}` where `max_active` is the max `total_active` across all rounds (used by the frontend for bar scaling). Write to `<sim_dir>/participation_heatmap.json` with mtime-based cache. `get_participation_heatmap(sim_dir)` reads cache if fresh. Add `GET /api/simulation/:id/participation-heatmap` (public for published sims). Returns the payload; 404 if sim not found; `{rounds: [], max_active: 0}` if `actions.jsonl` is absent. Add `participation_heatmap` to `surfaces_catalog.py` (38th or 39th surface entry — after stance_flips if both ship together) and `surface_stats.py`.
2. In `Step3Simulation.vue`, below the existing trajectory/BeliefDriftChart component, add a "Participation" row. Fetch `GET /api/simulation/:id/participation-heatmap` on page load. If `rounds` has ≥2 entries, render a compact bar chart: one thin vertical bar per round, height proportional to `total_active / max_active`, min-height 4px even for rounds with 0 activity (so the bar is always visible). Bars colored violet if `total_active / max_active >= 0.7` (full engagement), yellow if 0.3–0.7 (partial), red if < 0.3 (low participation). On hover: tooltip "Round {n} — {total_active} agents active{platforms.length > 1 ? ", " + platforms.map(p => p.name + ": " + p.count).join(", ") : ""}". A legend below the chart: "● High ● Medium ● Low participation". If `rounds` is empty, hide the section. When the user scrubs the replay timeline to round N, the corresponding bar in the participation chart is highlighted with a brighter border (same scrub-sync mechanism as the existing round-hover on the belief chart). Add `getParticipationHeatmap(simId)` to `frontend/src/api/simulation.js`. In `EmbedDialog.vue`, add a "Participation Heatmap" section with endpoint URL and curl. Bilingual EN/ZH labels.
3. Add `GET /api/simulation/:id/participation-heatmap` to the OpenAPI spec under the "Analytics" tag with `ParticipationHeatmapResponse` schema: `{rounds: [{round: int, total_active: int, platforms: [{name: str, count: int}]}], max_active: int}`. Add 12 unit tests in `backend/tests/test_unit_participation.py`: total_active counts distinct agents not total action lines, same agent active twice in a round counts as 1, platform counts are per-platform distinct agents in that round, platform counts sum to ≥ total_active if agents appear on multiple platforms in one round (multi-platform agent counted once in total, once per platform), max_active is correct max across all rounds, rounds sorted ascending, empty actions.jsonl returns empty rounds, missing actions.jsonl returns empty rounds (not 404), cache written on first call, cache recomputed when actions.jsonl newer, low-participation round (< 30% of max_active) correctly classified in docs, 404 for unknown sim_id. Add a "Per-Round Agent Participation Heatmap" section to `docs/FEATURES.md`: explains the distinct-agent-per-round counting methodology, notes the multi-platform counting (an agent on two platforms in one round: 1 in total_active, 1 in each platform's count), and describes the participation collapse pattern (declining bars in final rounds as a data quality signal for scenario designers).

## Selection Rationale

Five analytics and tooling ideas, all distinct from the 37 push-blocked builds and the ten suggestions from the past 7 days (Jun 6 + Jun 8):

- **Agent Stance Flip Report** (#1) — the "who changed their mind?" surface. agents.json records initial and final stances; no existing endpoint reads those fields to produce a per-sim flip map. The flip rate and the identities of high-influence flippers are the most actionable signal for scenario designers who want to know which archetypes are persuadable and when the persuasion happened in the run.

- **Simulation Full-Text Search** (#2) — pure-stdlib mtime-indexed text search across published sim titles and descriptions. The gallery has filter tabs and sort controls but no keyword search. With 37 surfaces and growing inventories, integrators need discoverability. A `?q=interest+rates` query is the missing primitive for ecosystem partners building topic-specific dashboards.

- **Confidence Component Breakdown** (#3) — exposes the four sub-scores (stability, convergence, participation, adversarial) that produce the composite confidence number. The composite score currently has no explanation surface. "Your simulation is 71% confident because stability=88, convergence=45, participation=100, adversarial=60" is immediately more actionable than "71." The refactor also improves internal code structure by extracting the four components from the inline calculation.

- **Simulation Fork Lineage Graph** (#4) — records fork_parent_id at fork time and exposes the parent/child graph per simulation. Fork is already a supported action; the relationship is just invisible. Researchers running parameter sweeps across 5 forks of the same scenario need to navigate the family tree. Ecosystem partners need to associate derivative sims with originals in their own systems.

- **Per-Round Agent Participation Heatmap** (#5) — counts distinct active agents per round from actions.jsonl, broken out by platform. "Participation collapse" in the final rounds — where only 12 of 80 agents are still acting — is a data quality signal that the aggregate trajectory doesn't reveal. The heatmap bar chart makes this visible at a glance, synced with the existing replay timeline scrubber.
