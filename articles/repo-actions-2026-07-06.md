### 1. French i18n Completion Sprint

**Type:** DX / Community
**Effort:** Small (hours)
**Impact:** PR #239 pushed French from 1,627 to 1,723 translated strings, but 261 of 1,984 tr() calls are still untranslated — visible to any French-locale user as raw key names rendered verbatim in the UI. Before the community starts work on language #4 (German, Spanish, Japanese, Portuguese, Korean — the June 26 i18n contribution guide targets these), French must be at 100%. A 261-string gap in the base case contaminates the template that prospective translators are asked to copy. The completion sprint is not a new architectural contribution — it is a targeted audit-and-fill of the gap. The outcome is concrete and binary: `npm run dev`, cycle to French locale, confirm zero raw key names visible across all pages. Once French is clean, the "5 languages by September 1" hyperstition has a stable 3/5 baseline.

**How:**
1. Identify the 261 untranslated strings. Run `grep -rn "tr('" frontend/src/` to collect all `tr('...')` calls across the SPA. Cross-reference against the key set in `frontend/src/locales/fr.js` — any key present in the component files but absent from `fr.js` is a gap. Alternatively, load the dev server in French locale and scan each page for keys rendering as their own names (the i18n fallback behavior surfaces untranslated keys verbatim). Catalog the missing keys into three buckets: (A) keys that appear in new components added after PR #222 (the original French baseline), (B) keys that were present in the English source but missed in the initial translation, (C) keys added in PR #239's pass but still incomplete. Bucket A is likely the largest given that PR #239 added 96 strings — the gap is probably spread across recently added UI sections.
2. Translate the missing keys in `frontend/src/locales/fr.js`. Translation style: formal register consistent with the existing French dictionary (uses "vous" throughout; "vous" stays — do not mix registers). For UI labels and actions: prefer short, direct French ("Simuler" not "Lancer une simulation"; "Résultats" not "Afficher les résultats"). For error messages: full sentences with periods. For tooltips: noun phrases. Check `fr.js` for existing style patterns before translating each batch — the existing 1,723 translations are the style guide. Do not use Google Translate verbatim; cross-check idiomatically. Flag any key whose English source is ambiguous (e.g. "Signal" could be technical or vernacular) with a `// TODO: verify` comment rather than guessing.
3. Verification pass: `npm run dev` in `frontend/`, switch locale to French (FR button in LocaleToggle). Navigate every route in the SPA: home, simulation setup (Step1, Step2), simulation run (Step3), results view, embed dialog, mentions panel, stance flips panel, archetype atlas, trending topics, search. Any page that renders a key name verbatim (e.g. `simulation.agents.count` appearing as literal text) is a miss — return to step 2. When the full navigation produces zero key-name leakage, commit `frontend/src/locales/fr.js` with message `feat(i18n): complete French translations to 100% (1984/1984 tr() calls)`. Update `docs/CONTRIBUTING-I18N.md` (if it exists from the Jun 26 contribution guide) to note that French is the reference implementation: any new tr() calls added to the codebase must be added to both `fr.js` and `zh-CN.js` simultaneously to prevent future gaps.

### 2. GitHub Actions Workflow Template for Scheduled Simulations

**Type:** DX / Community
**Effort:** Small (hours)
**Impact:** The "Will someone outside MiroShark core team deploy and host a public-facing MiroShark instance by July 15, 2026?" hyperstition has 9 days left. A full platform deployment (Docker, cloud infrastructure, domain) is a multi-hour setup. A GitHub Actions workflow template requires nothing — fork holders already have GitHub Actions. The template defines a cron-scheduled job that: (1) calls `POST /api/simulate` on a hosted MiroShark instance (miroshark.xyz or self-hosted), (2) polls `GET /api/simulation/:id/status` until complete, (3) fetches the signal and posts it to a webhook (Slack, Discord, or a custom endpoint). A fork holder with a MiroShark API key can be running scheduled simulations in under 10 minutes. This is the minimal viable "external public deployment" — not a hosted instance, but a live integration that generates and consumes real simulation data on a schedule. It also serves as a reference architecture for operators who want to embed MiroShark into their own automation pipelines.

**How:**
1. Create `.github/workflow-templates/miroshark-schedule.yml` (and if the template directory format doesn't exist, create `examples/github-actions/miroshark-schedule.yml` instead — the goal is a copy-paste-ready file in the repo). The template: trigger is `schedule: [{cron: '0 9 * * 1'}]` (Monday 09:00 UTC as a default) plus `workflow_dispatch` (for manual runs). Env vars via GitHub secrets: `MIROSHARK_URL` (the hosted endpoint, e.g. `https://miroshark.xyz`), `MIROSHARK_API_KEY` (optional — for instances that require authentication), `WEBHOOK_URL` (where results are posted). Steps: (A) `simulate`: `curl -s -X POST "$MIROSHARK_URL/api/simulate" -H "Content-Type: application/json" -d '{"topic": "${{ env.TOPIC }}", "agent_count": 20, "num_rounds": 5}'` — stores `sim_id`. (B) `wait`: loop `GET /api/simulation/$SIM_ID/status` every 30s until `status == "complete"` or timeout 15m. (C) `fetch`: `GET /api/simulation/$SIM_ID/signal` — captures direction, yes_pct, confidence_pct. (D) `notify`: if `WEBHOOK_URL` is set, POST the result JSON to it. Include a commented block showing how to format a Slack message from the result. Parameterize `TOPIC` as a workflow input (string, default: "Will AI regulation pass in the EU this year?") so fork holders can change it without editing the YAML.
2. Create `examples/github-actions/README.md` (~60 lines) — minimal setup guide. Sections: (1) "What this does" — runs a MiroShark simulation on a cron schedule and posts the result to a webhook. (2) "Setup" — 5 steps: fork the repo, copy `miroshark-schedule.yml` to `.github/workflows/`, add `MIROSHARK_URL` as a GitHub secret (value: `https://miroshark.xyz` for the hosted instance or your self-hosted URL), add `WEBHOOK_URL` as a secret (value: your Slack/Discord webhook or any HTTPS endpoint that accepts POST with JSON), enable GitHub Actions in your fork's settings. (3) "Customizing the topic" — change the `TOPIC` workflow input default. (4) "Running manually" — Actions → miroshark-schedule → Run workflow. (5) "Cost" — one simulation with 20 agents and 5 rounds costs approximately $0.50–$1.00 on the hosted instance; GitHub Actions minutes used: ~2 minutes per run (free tier: 2,000 min/month). No MiroShark-side setup needed — it's all client-side.
3. Add a `## Automated Simulations` section to the main `README.md` (between the CLI and Contributing sections): "Run MiroShark simulations on a schedule from GitHub Actions — no server required. Copy the [workflow template](examples/github-actions/miroshark-schedule.yml) to your repo's `.github/workflows/`, set two GitHub secrets (`MIROSHARK_URL`, `WEBHOOK_URL`), and get weekly simulation results delivered to Slack or Discord automatically. Full setup guide: [examples/github-actions/README.md]." File a GitHub issue in MiroShark titled "Show & Tell: share your scheduled simulation workflow" — a prompt for fork holders using the template to post their automation configs. This creates a collection surface for the "external public deployment" hyperstition evidence.

### 3. Jupyter Notebook Research Gallery

**Type:** Community / DX
**Effort:** Medium (1-2 days)
**Impact:** The "Will a MiroShark simulation be cited in a peer-reviewed or pre-print paper by September 2026?" hyperstition requires researchers to find MiroShark useful enough to cite. The barrier is not the API — it's the last mile: "I have the data, how do I analyze it in my notebook?" A gallery of example `.ipynb` notebooks in a `notebooks/` directory answers this in the researcher's native environment. Notebooks are the standard publication artifact in computational social science, economics, and AI safety research. An arXiv paper that cites MiroShark is far more likely if the authors used a notebook they found in the repo to produce their figures — the notebook becomes the reproducibility artifact that satisfies peer review's "code availability" requirement. Each notebook is self-contained: installs dependencies inline (stdlib + matplotlib), hits the public MiroShark API with a real simulation ID, and produces a publication-ready figure. The gallery targets three use cases: belief drift visualization (the core phenomenon MiroShark models), comparative analysis across runs, and topic-level consensus statistics.

**How:**
1. Create `notebooks/01_belief_drift.ipynb` — "Visualizing Belief Drift in a MiroShark Simulation." Cells: (1) install/import: `requests`, `matplotlib`, `json`, `datetime` — all stdlib except requests. (2) Config: `SIM_ID = "your-simulation-id"` and `BASE_URL = "https://miroshark.xyz"` — two lines the researcher changes. (3) Fetch: `GET /api/simulation/{SIM_ID}/confidence/trajectory` and `GET /api/simulation/{SIM_ID}/signal`. (4) Plot: `matplotlib` line chart of `confidence_pct` per round (x-axis: round number; y-axis: 0–100%; title: "Belief Drift — {topic}"; annotated with final direction chip color: green=BULLISH, red=BEARISH, gray=SPLIT). (5) Export: `plt.savefig("belief_drift_{sim_id}.png", dpi=150, bbox_inches='tight')`. (6) Stats cell: print final direction, yes_pct, confidence_pct, agent_count, num_rounds as a formatted summary block. Create `notebooks/02_compare_runs.ipynb` — "Comparing Simulations Across Dates." Uses `GET /api/simulations/compare?ids=a,b,c` (if the comparison endpoint from Jul 4 idea #3 is built) or manual fetching of multiple `signal.json` endpoints. Plots: side-by-side bar chart of `yes_pct` per simulation, color-coded by direction. Trend line if runs are sorted by `completed_at`. Create `notebooks/03_topic_stats.ipynb` — "Topic-Level Consensus Statistics." Uses `GET /api/simulation/search?q=...` to find all simulations on a topic, then fetches each signal, computes mean/std of `yes_pct`, `confidence_pct`, and direction distribution (% BULLISH / SPLIT / BEARISH). Plots: violin plot of yes_pct distribution by direction bucket.
2. Each notebook must: (a) run top-to-bottom without errors against the live miroshark.xyz instance, (b) include a "Reproducibility" cell at the top that prints the API version, the simulation IDs used, and the run timestamp, (c) include a "Citation" cell at the bottom with a BibTeX block: `@misc{miroshark2026, title={MiroShark: Universal Swarm Intelligence Engine}, author={Aaron J. Mars}, year={2026}, url={https://github.com/aaronjmars/MiroShark}}` — this is the copy-paste citation block for a researcher's paper. Add `notebooks/requirements.txt`: `requests>=2.28.0` and `matplotlib>=3.7.0`. Add `notebooks/README.md` (~40 lines): what each notebook does, how to run locally (`pip install -r requirements.txt; jupyter notebook`), how to use with Google Colab (File → Open notebook → GitHub → aaronjmars/MiroShark), and the available API surfaces (links to `docs/FEATURES.md`). Each notebook should use a real simulation ID from a published public simulation — not a placeholder — so researchers can run the notebook immediately without needing their own simulation.
3. Add a `## Research & Notebooks` section to the main `README.md`: "For academic use, analysis, and reproducible research — see [notebooks/](notebooks/) for Jupyter notebooks covering belief drift visualization, multi-run comparison, and topic-level consensus statistics. All notebooks hit the public MiroShark API and run against real simulation data. Copy-paste BibTeX citation: [notebooks/01_belief_drift.ipynb → Citation cell]." File a GitHub issue in MiroShark: "Contribution wanted: add a notebook for your use case" with label `good first issue`. Body: "The `notebooks/` directory has 3 starter notebooks. ### 4. Operator Usage Analytics API

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** Operators running MiroShark instances — self-hosted or via Docker — currently have no visibility into aggregate usage patterns: How many simulations have run? What's the total spend? What time of day do simulations cluster? What agent counts are most common? This data exists on disk across the `data/simulations/` directory but is not surfaced anywhere. `GET /api/analytics/overview` (API-key gated or localhost-only) aggregates it into a structured response covering: simulation counts by period (today, 7d, 30d, all-time), total USD cost, cost by period, agent count distribution (histogram buckets: 5–10, 11–20, 21–50, 50+), rounds distribution, platform usage, and top 5 topics by frequency. For self-hosted operators this is capacity-planning data: "Do I need to upgrade my instance?" For the core team: an API surface that the showcase page (Jul 4 idea #5) and the community analytics could pull from. For the paper citation hyperstition: a researcher operating a self-hosted instance can quote "we ran 47 simulations across 12 weeks" with a concrete data source.

**How:**
1. Add `backend/app/services/analytics_service.py` (~150 LoC, pure stdlib). `build_overview(data_dir, now_iso)` scans all `data/simulations/*/state.json` (no publish gate — this is an operator-facing endpoint, not a public one; unpublished sims are included in operator analytics). Buckets simulations by `completed_at` into: today (same calendar day as `now_iso`), last_7d, last_30d, all_time. For each bucket: `count` (number of completed sims), `total_cost_usd` (sum of `cost_usd` from `simulation_config.json` where present, else 0.0), `avg_cost_usd`, `avg_duration_minutes` (from `completed_at` − `started_at` in state.json). Global stats: `agent_count_histogram` (dict of bucket labels → count: `"1-10"`, `"11-20"`, `"21-50"`, `"51+"`) from `simulation_config.json.agent_count`; `rounds_histogram` (same pattern); `platform_distribution` (dict of platform_name → count from `simulation_config.json.platforms`); `top_topics` (list of `{topic, count}` sorted descending, top 5, from `simulation_config.json.topic`). Returns `{generated_at: ISO, periods: {today: {...}, last_7d: {...}, last_30d: {...}, all_time: {...}}, distributions: {agent_count: {...}, rounds: {...}, platforms: {...}}, top_topics: [...]}`. Caches to `data/.analytics_cache.json` (5-minute mtime cache). Route: `GET /api/analytics/overview` — guarded by either: (a) request from `127.0.0.1` / `::1` (localhost-only by default), or (b) `X-Admin-Key` header matching `ADMIN_API_KEY` env var if set. If neither condition met: 403 with `{"error": "operator-only endpoint — set ADMIN_API_KEY or access from localhost"}`. Add 10 unit tests: all-time count reflects completed sims, incomplete sims excluded, today bucket uses correct date boundary, agent_count_histogram buckets sum to total, top_topics sorted descending, 5-min cache hit skips rescan, 403 returned for non-localhost without key, 200 returned from localhost, platform_distribution keys match sim platforms, generated_at ISO string present.
2. Add a minimal admin dashboard page `frontend/src/views/AdminView.vue` at `/admin` route (not in the main nav — accessed directly at `yourinstance.com/admin`). Requires the same auth as the API endpoint (prompts for the ADMIN_API_KEY if `ADMIN_API_KEY` is set on the server; skips auth prompt if accessing from localhost). Displays: a 4-tile stat row (total simulations all-time, total cost all-time, today's count, today's cost); a bar chart of simulations per period (today / 7d / 30d); a horizontal bar for agent_count distribution; a top-5 topics list. Dark-themed, consistent with the rest of the UI. No bilingual toggle needed (admin-only surface). Add `getAnalyticsOverview()` to `frontend/src/api/simulation.js`. This page is the operator's first-stop for "how is my instance performing?" without requiring log-parsing or database queries.
3. Add `GET /api/analytics/overview` to the OpenAPI spec under a new "Admin" tag with `x-internal: true` (marks it as operator-facing, not part of the public API surface). Add a "Operator Analytics" section to `docs/FEATURES.md`: what the endpoint returns, the auth options (localhost vs ADMIN_API_KEY), the 5-minute cache, and a note that incomplete/private simulations are included (operator analytics, not public metrics). Add a brief `## Analytics` section to the deployment docs (or README's self-hosting section): "Access operator analytics at `GET /api/analytics/overview` from localhost, or set `ADMIN_API_KEY` in your environment for remote access. The `/admin` frontend page renders this as a dashboard."

### 5. Topic Autocomplete API

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** Operators setting up a simulation face a blank input field: "What do you want to simulate?" Without any guidance, they either type a generic topic ("bitcoin") or spend time crafting a specific question. Both outcomes produce topics that may have already been simulated — creating fragmentation in the trending feed and making the comparison API less useful. `GET /api/topics/suggest?q=bitcoin` returns a ranked list of historical topics that match the query (prefix-match + fuzzy substring), each annotated with the most recent simulation's direction and yes_pct. The operator types "bit" and sees "Bitcoin price direction Q3 2026 — BEARISH 62%" — they can either adopt the existing framing (boosting topic normalization) or deliberately differentiate ("Will Bitcoin break $200K before year-end" vs "Bitcoin price direction Q3 2026"). For the trending topics endpoint (Jul 2 idea #4): normalized topics cluster correctly, producing cleaner trend signals. For the comparison API (Jul 4 idea #3): operators who reuse normalized topic strings can compare their runs against the community baseline. Zero new data — just a query layer over existing `simulation_config.json` topic strings. The autocomplete also serves as a discovery surface: operators who never thought to simulate "EU AI regulation" see it in suggestions while typing "AI".

**How:**
1. Add `backend/app/services/autocomplete_service.py` (~100 LoC, pure stdlib). `suggest_topics(data_dir, query, limit=10)` scans all `data/simulations/*/simulation_config.json` for published, completed simulations (checks `state.json` for both `publish_date` and `completed_at`). Builds a deduplicated topic index: for each unique normalized topic (`.lower().strip()`), stores the most recent `completed_at`, the most recent `signal.json` direction and `yes_pct`. Caches the full index to `data/.topic_index.json` (15-minute mtime cache — same window as trending). On query: applies two-pass matching against the cache. Pass 1 (prefix): topics where `normalized_topic.startswith(query.lower().strip())` — exact prefix takes priority. Pass 2 (substring): topics where `query.lower().strip() in normalized_topic` and not already in pass 1. Merge pass 1 + pass 2, sort by `completed_at` descending (most recently simulated topics surface first — they're likely most relevant). Take top `limit`. Returns `{suggestions: [{topic: str, last_simulated: ISO, last_direction: str, last_yes_pct: float, sim_count: int}], query: str, total: int}`. Route: `GET /api/topics/suggest` — unauthenticated (public endpoint; query param `q=`, min 2 chars else returns `{suggestions: [], query: q, total: 0}`; param `limit=` int clamp [1, 20]; `Cache-Control: public, max-age=60` — 60s, not 15min, since user is typing and staleness is more noticeable). Returns 200 always (empty suggestions is not an error). Add 10 unit tests: empty query returns empty list, single-char query returns empty list, prefix match appears before substring match, results sorted by completed_at desc, limit param respected, unpublished sims excluded, incomplete sims excluded, sim_count counts all sims on topic (not just latest), Cache-Control is max-age=60, total reflects matched topic count before limit clamp.
2. In `Step2EnvSetup.vue` (or wherever the simulation topic input lives), add an autocomplete dropdown below the topic text field. On input (debounced 300ms, min 2 chars): fetch `GET /api/topics/suggest?q={encodeURIComponent(input)}`. Display a dropdown of up to 8 suggestions (more would overflow): each row shows the topic string (truncated at 60 chars with ellipsis if longer), a direction chip (colored BULLISH/BEARISH/SPLIT), the yes_pct%, and a subtle "N runs" count in muted text. On suggestion click: populate the topic field with the full suggestion string and dismiss the dropdown. On Escape: dismiss. On ArrowUp/Down: navigate suggestions (standard autocomplete UX). Emit no additional API calls until the user submits the form. If the API returns zero suggestions (new topic being typed): no dropdown appears — the field behaves as a plain text input. Add `suggestTopics(query)` to `frontend/src/api/simulation.js`. Bilingual labels on the dropdown header: "Recent topics" / "最近话题" (shown only when suggestions are non-empty). No modal, no animation — just a basic `<ul>` positioned below the input, z-indexed above other elements.
3. Add `GET /api/topics/suggest` to the OpenAPI spec under the "Discovery" tag (alongside the full-text search and trending endpoints). Schema: `TopicSuggestResponse` with `suggestions: TopicSuggestion[]`, `query: string`, `total: integer`. `TopicSuggestion`: `topic`, `last_simulated`, `last_direction`, `last_yes_pct`, `sim_count`. Add a "Topic Autocomplete" section to `docs/FEATURES.md`: what the endpoint returns, the two-pass matching (prefix before substring), the 60s cache, the 2-char minimum, and that only published completed simulations contribute to the index. Add a curl example to the README API table: `curl "https://miroshark.xyz/api/topics/suggest?q=bitcoin"`. Note in `docs/FEATURES.md` that this endpoint is the normalization complement to the trending endpoint — operators who reuse exact topic strings from autocomplete suggestions will produce cleaner trending signals.

## Selection Rationale

PR #239 changing the context window (French at 86.9%, not 100%) drives idea #1 — the i18n contribution kit from Jul 4 assumes a complete French baseline; it doesn't exist yet. The other four ideas address the same pressure points as previous rounds but from angles not covered in the past 7 days.

- **French i18n Completion Sprint** (#1) — Binary outcome: 261 strings become zero. French reaches 100%. The "5 languages by September 1" hyperstition has a clean 3/5 baseline. The contribution guide (Jul 4 #1) and the template it points to are only valid once French is the complete reference implementation. This is a prerequisite, not an enhancement.

- **GitHub Actions Workflow Template** (#2) — The "external public deployment" hyperstition has 9 days left. A full Docker deployment is a multi-hour commitment. This template is a 10-minute commitment: copy a file, set two secrets, enable Actions. It defines "public deployment" in the most accessible terms possible — a fork holder running scheduled simulations and posting results to Slack is a legitimate external MiroShark integration. The template is also documentation of the API's automation capabilities.

- **Jupyter Notebook Research Gallery** (#3) — Researchers who want to cite MiroShark need two things: a stable URL for a specific result, and code that produced the analysis. The notebooks provide the second. They are also the lowest-friction academic on-ramp: `pip install requests matplotlib; jupyter notebook` gets a researcher from zero to a publication-ready belief drift figure in under 5 minutes. The BibTeX cell in each notebook removes the last barrier to citation.

- **Operator Usage Analytics API** (#4) — Self-hosted operators are flying blind. No aggregate stats, no cost visibility, no usage trends. This is infrastructure MiroShark needs before the showcase page (Jul 4 #5) and the "public case study" hyperstition can close — operators writing case studies need data to quote. The admin API and minimal dashboard create that data surface without requiring any external analytics service.

- **Topic Autocomplete API** (#5) — The blank-canvas problem at simulation setup has no solution yet. The trending feed (Jul 2 #4) shows what's popular after the fact; autocomplete helps before the run. Topic normalization via autocomplete adoption also improves the trending feed's quality — fragmented topic strings ("bitcoin", "Bitcoin", "bitcoin price", "BTC price") collapse into normalized clusters when operators adopt suggested strings. One service, two benefiting surfaces.
