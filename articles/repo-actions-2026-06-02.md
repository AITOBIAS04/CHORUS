### 1. Ecosystem Registry API

**Type:** Integration
**Effort:** Small (hours)
**Impact:** Four ecosystem projects added themselves to ECOSYSTEM.md this week. Every one of them has a website, an X handle, and a brief description — but that data is locked in a Markdown table, readable by humans and invisible to machines. A `GET /api/ecosystem` endpoint parses ECOSYSTEM.md into structured JSON: `{updated_at: ISO, projects: [{name, url, x_handle, github, description, added_at}]}` using Python's regex-based Markdown table parser (pure stdlib). This turns ECOSYSTEM.md from a PR-based append log into a live API. Downstream uses: HivemindOS can render a "MiroShark ecosystem" widget in their dashboard pulling from this endpoint; SyntheticsAI can list compatible tools without hardcoding; any Chinese developer platform that writes up MiroShark can link to a machine-readable "built on this" registry that grows with the community. A `GET /api/ecosystem/count` returns just the project count — suitable for embedding as a badge ("N projects building on MiroShark"). For the Chinese dev platform hyperstition, a structured ecosystem registry is the asset worth linking to; for the "10 community PRs" hyperstition, it creates a public artifact that attributing contributors find worth linking back to.

**How:**
1. Add `backend/app/services/ecosystem_service.py` (~80 LoC, pure stdlib). `get_ecosystem(repo_root)` reads `ECOSYSTEM.md` from the repo root (path relative to the Flask app's working directory). Uses `re.findall` to parse Markdown table rows into dicts: each row maps to `{name: str, url: str, x_handle: str|null, github: str|null, description: str}`. Filters out header rows and separator rows (`|---|`). Returns `{updated_at: ISO (from file mtime), projects: [...], total: int}`. Caches the result in memory for 60 minutes (a module-level dict with a timestamp — invalidated if file mtime changes). Add `GET /api/ecosystem` (public, no auth): returns the parsed JSON with `Cache-Control: public, max-age=3600`. Add `GET /api/ecosystem/count` (public): returns `{total: int, updated_at: ISO}` — a minimal endpoint for badge rendering and status widgets.
2. Add `EcosystemView.vue` at `/ecosystem` — a public page showing the ecosystem grid. Layout: dark-theme page header "Ecosystem" + subtitle "Projects building on MiroShark" + a count chip "N projects". Content: a responsive grid of cards (2 columns on desktop, 1 on mobile). Each card: project name in `font-semibold`, description in `text-gray-300 text-sm`, icon links row (globe icon → url, X/bird icon → x_handle if present, GitHub icon → github if present). Cards sorted by order of appearance in ECOSYSTEM.md (preserves the organic addition order). Add `/ecosystem` to the Vue Router. In the main navigation or the `/explore` toolbar, add an "Ecosystem" chip alongside Tags, Collections, and Leaderboard. In `frontend/src/api/simulation.js`, add `getEcosystem()` and `getEcosystemCount()` helpers. Optionally in the sidebar or footer, show the ecosystem count as a small "N ecosystem projects →" link.
3. Add `GET /api/ecosystem` and `GET /api/ecosystem/count` to the OpenAPI spec under a new "Community" tag with `EcosystemResponse` and `EcosystemProject` schemas. Add `ecosystem_count: int` to `GET /api/stats` (the existing platform stats endpoint). Add 8 unit tests in `backend/tests/test_unit_ecosystem.py`: parses valid ECOSYSTEM.md table correctly, filters header and separator rows, x_handle null when absent, github null when absent, total count matches parsed rows, cache returns same result on second call (file not re-read), cache invalidates when file mtime changes, GET /api/ecosystem/count returns correct total. In `docs/FEATURES.md`, add an "Ecosystem Registry" section explaining the Markdown parsing contract, the 60-minute cache, and the badge endpoint.

### 2. French Locale (i18n FR)

**Type:** Community / DX
**Effort:** Small (hours)
**Impact:** Issue #95 was filed on May 22 by a contributor asking whether a French locale PR would be welcome. The repo already has Chinese (zh-CN) localization for most Vue components — the i18n infrastructure exists and the precedent is set. French is the 5th most-spoken language globally and the dominant language in academic discourse across West Africa, the Maghreb, and Belgium/Switzerland/Quebec. For a platform whose thesis is that AI crowd simulation has social science applications — polling, policy research, deliberation — the French-speaking academic community is a natural constituency. Implementing FR completes the "multilingual from day one" narrative that zh-CN started: the README now mentions a Chinese section diagram, zh-CN FEATURES.md exists, zh-CN docs exist for several surfaces. Adding French closes the loop for the other major language in MiroShark's natural research constituency. It also directly closes an open issue filed by a community contributor — the fastest possible community responsiveness signal.

**How:**
1. Read the existing i18n infrastructure: find the Vue i18n config file (likely `frontend/src/i18n/index.js` or `frontend/src/locales/zh-CN.json`) to understand the key structure and how locale switching is currently handled. Identify all i18n keys currently defined in `en.json` (or equivalent English source file). Create `frontend/src/locales/fr.json` with French translations for every key. Translation guidance: use formal "vous" register (appropriate for a research/professional tool), preserve technical terms in English where French usage would be awkward (e.g. "consensus" stays "consensus", "round" stays "round"), translate UI labels and action words ("Run Simulation" → "Lancer la simulation", "Publish" → "Publier", "Download" → "Télécharger", "Confidence" → "Fiabilité", "Embed" → "Intégrer"). Add the `fr` locale to the Vue i18n `messages` config alongside `en` and `zh-CN`. In the language selector component (wherever `zh-CN` / English switching is implemented), add French as a third option: flag emoji 🇫🇷 + "Français". Default locale detection: respect the existing browser-preference detection if present; fall back to `en` if `fr` is not available for a key (standard Vue i18n fallback pattern).
2. Add French translations for all user-facing strings in the backend (wherever Flask returns human-readable error messages or the API returns display strings like `consensus_direction` labels). Check if the locale negotiation protocol documented in `docs/API.md` (PR #123 — locale-negotiation) handles `Accept-Language: fr` — if it does, add French variants for any server-side display strings (status labels, direction labels: "Bullish" → "Haussier", "Bearish" → "Baissier", "Neutral" → "Neutre", "Mixed" → "Mitigé"). In the OpenAPI spec, add `fr` to the list of supported locales wherever `zh-CN` and `en` appear. In `docs/LOCALES.md` (create if absent), add a "French (fr)" row to the locale status table alongside the existing Chinese entry: status "complete", completion date, link to the fr.json file.
3. Add `frontend/src/locales/fr.json` to the project with all keys from `en.json` translated. Add a smoke test: a simple CI-runnable script `scripts/check-locale-completeness.sh` that confirms `fr.json` has the same key count as `en.json` (exits non-zero if FR is missing any key — prevents future English additions from silently leaving French untranslated). In `docs/CONTRIBUTING.md`, add a "Adding a new locale" section: explains the pattern (copy en.json → xx.json, translate all values, add to i18n config, add to language selector, add to CI completeness check). Optionally, reply to issue #95 with a brief note (outside the scope of this autonomous build — operator action) once the FR PR is ready. Add 4 unit tests in `backend/tests/test_unit_locales.py`: `fr.json` parses as valid JSON, key count matches `en.json`, all values are non-empty strings, no key contains an untranslated `__MISSING__` marker.

### 3. Simulation RSS Feed

**Type:** Integration
**Effort:** Small (hours)
**Impact:** Four ecosystem projects just added themselves to ECOSYSTEM.md. Every one of them now faces the same integration problem: how do I know when MiroShark publishes a new simulation worth surfacing to my users? Today the answer is polling `GET /api/gallery`. RSS solves this: `GET /api/feed.rss` returns a standards-compliant RSS 2.0 feed of recently published simulations — title (scenario excerpt), description (consensus direction + confidence score + agent count), link (watch URL), pubDate (ISO). Any project can subscribe via standard RSS infrastructure without writing custom polling code: HivemindOS can pipe new sims into their dashboard; Capacitr can trigger on new publications; a research group can subscribe in Feedly and get MiroShark updates in their morning feed. RSS is also the standard format for every "submit your feed" integration on developer directories (dev.to, Hacker News, newsletter tools) — it makes MiroShark listable anywhere that accepts RSS. For operators who run private MiroShark instances for research teams, `?token=<admin_token>` appended to the feed URL includes private simulations in the feed (for internal distribution). Pure stdlib: `xml.etree.ElementTree` for XML generation, no new dependencies.

**How:**
1. Add `backend/app/services/rss_service.py` (~100 LoC, pure stdlib). `generate_rss_feed(data_dir, base_url, include_private=False, limit=20)` scans all sim dirs for published simulations (same scan as the gallery endpoint), sorts by `published_at` descending, and builds an RSS 2.0 XML document using `xml.etree.ElementTree`. Feed metadata: `<channel>` with `<title>MiroShark Simulations</title>`, `<description>Latest published simulations from MiroShark — swarm intelligence engine</description>`, `<link>` (base_url), `<lastBuildDate>` (most recent published_at). Each item: `<title>` (first 80 chars of scenario text), `<description>` (HTML string: "Direction: {consensus_direction} · Consensus: {yes_pct}% YES · Confidence: {confidence_score} · Agents: {n_agents} · Rounds: {n_rounds}"), `<link>` (base_url + `/watch/` + sim_id), `<guid>` (base_url + `/watch/` + sim_id), `<pubDate>` (RFC 2822 date from published_at). Add `GET /api/feed.rss` (public): returns the XML with `Content-Type: application/rss+xml; charset=utf-8` and `Cache-Control: public, max-age=300`. If `?token=<admin_token>` is present and valid, calls `generate_rss_feed(..., include_private=True)` to include private sims. Limit: 20 most recent published sims by default; `?limit=N` (max 50).
2. In `frontend/src/views/ExploreView.vue` (or the app header/footer), add an RSS subscription icon (📡 or the standard RSS orange square icon) that links to `/api/feed.rss`. Position it in the gallery toolbar alongside the "Leaderboard", "Tags", and "Collections" chips — a subtle but discoverable link for technically-oriented users. In `EmbedDialog.vue`, add an "RSS Feed" chip in the integrations section: "Subscribe to new simulations via RSS — paste `/api/feed.rss` into your RSS reader or pipeline." In the app's `<head>` (in `index.html` or the HTML template), add `<link rel="alternate" type="application/rss+xml" title="MiroShark Simulations" href="/api/feed.rss">` — this makes the feed auto-discoverable by browsers and RSS reader extensions. In `frontend/src/api/simulation.js`, add a `getRssFeedUrl()` helper that returns the feed URL (for clipboard-copy patterns).
3. Add `GET /api/feed.rss` to the OpenAPI spec under the "Discovery" tag (alongside `/api/leaderboard` and `/api/ecosystem`). Note: OpenAPI doesn't natively describe RSS, so document it as `produces: application/rss+xml` with a description: "RSS 2.0 feed of recently published simulations, newest first. Add `?token=<admin_token>` to include private simulations." Add 10 unit tests in `backend/tests/test_unit_rss.py`: feed is valid XML, channel title is "MiroShark Simulations", each published sim appears as an `<item>`, unpublished sims excluded, items sorted by pubDate descending, limit param respected (max 50), `Content-Type` header is `application/rss+xml`, private sims excluded without token, private sims included with valid admin token, invalid admin token excludes private sims (no 403 — just falls back to public view). In `docs/INTEGRATIONS.md`, add a "RSS Feed" section: explains the feed URL, the optional token param for private sims, and provides a curl example: `curl https://your-instance/api/feed.rss | xmllint --format -`.

### 4. Embed Theme Parameter

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** The simulation embed iframe — used whenever a simulation is shared on an external site — currently renders in the dark-space-violet theme unconditionally. Four ecosystem projects (HivemindOS, SyntheticsAI, Capacitr, Echo Oracle) are building UIs with their own visual languages; several run light-mode dashboards or hybrid themes. When they embed a MiroShark simulation, the dark purple iframe looks foreign in their UI. A `?theme=dark|light|system` query param solves this: `?theme=dark` (default) renders the current dark-space-violet theme; `?theme=light` renders a clean light palette (white background, dark text, blue-tinted accents); `?theme=system` reads the iframe's `prefers-color-scheme` media query and renders the matching theme automatically. All three are handled entirely in the frontend via a CSS class applied to the embed container — no backend changes, no new API endpoints. For `EmbedDialog.vue`, a theme selector (Dark / Light / Auto) updates the iframe snippet in real time. This is the most direct DX improvement for the four ecosystem projects currently mid-PR, and the most likely to be cited when they write up their integrations.

**How:**
1. In the embed entry point (the Vue component or route that renders the simulation embed, likely `EmbedView.vue` or a `?embed=true` param path in `SimulationWatchPage.vue`), read `?theme` from `window.location.search` on mount. Apply a CSS class to the root embed container: `theme-dark` (default), `theme-light`, or `theme-system`. For `theme-system`, add a `<style>` block with `@media (prefers-color-scheme: light) { .theme-system { ... } }` overrides. Define three CSS variable sets in `embed-themes.css` (new file in `frontend/src/styles/`): the dark set reuses the existing dark-space-violet variables; the light set uses `--bg: #ffffff; --bg-secondary: #f8f9fa; --text: #1a1a1a; --text-secondary: #6b7280; --accent: #7c3aed; --border: #e5e7eb; --chip-bg: #f3f4f6`; the system set uses CSS variables scoped to a media query. Validate the `?theme` param: only `dark`, `light`, `system` are accepted; any other value defaults to `dark`. Pass the resolved theme as a prop or CSS class down to all sub-components rendered in embed mode (`ConsensusPanel`, `AgentList`, `ReplayBar`, etc.) so they pick up the theme variables.
2. In `EmbedDialog.vue`, add a "Theme" selector row below the current embed options. Three buttons (pill/chip style): "Dark" (default, selected state = purple ring), "Light" (selected state = blue ring), "Auto" (selected state = gray ring). When a theme is selected, append `?theme=<value>` to the iframe `src` in the live preview and the copy-paste snippet. Update `updateEmbedUrl()` (or equivalent reactive function) to include the theme param. Show a note below the selector: "Auto matches the visitor's OS theme preference." Update the embed snippet copy-button output to include the theme param when non-default (omit `?theme=dark` from the snippet to keep default URLs clean; only append `?theme=light` or `?theme=system` when selected). In `frontend/src/api/simulation.js`, update `getEmbedUrl(simId, options)` (or equivalent helper) to accept a `theme` option.
3. Add `theme` as a documented query parameter on the embed route in the OpenAPI spec: enum `["dark", "light", "system"]`, default `"dark"`. Add 8 unit tests in `backend/tests/test_unit_embed_theme.py` (these are integration-layer tests for the embed route): embed route with no theme param returns 200, embed route with `?theme=dark` returns 200, embed route with `?theme=light` returns 200, embed route with `?theme=system` returns 200, embed route with `?theme=invalid` returns 200 (defaulting to dark — no 400), embed route with `?theme=light&round=5` preserves both params, embed route requires published sim (returns 404 for unpublished), CORS headers present. In `docs/FEATURES.md`, update the "Simulation Embed" section to document the `?theme` parameter with the three options and a note about `system` mode. In `EmbedDialog.vue`'s inline help tooltip, add: "Use 'Auto' theme to match the visitor's system dark/light preference automatically."

### 5. Simulation Scheduler (Time-Triggered Runs)

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** Every simulation MiroShark runs today is triggered synchronously: a user clicks "Run" and waits. For operators running recurring research cycles — "simulate public reaction to the Fed announcement every Thursday morning," "run the weekly scenario digest at 6am before the newsletter goes out," "schedule a competitive analysis simulation for Monday's team meeting" — there's no way to queue a future run without being online at the exact moment. A simulation scheduler adds `POST /api/simulation/schedule` accepting `{scenario: str, n_agents: int, n_rounds: int, model: str, archetypes: [...], run_at: ISO_datetime}`. On receive: validates the `run_at` is in the future (no same-second scheduling), writes a `<data_dir>/scheduled_runs.json` entry, and returns a `{scheduled_id: str, run_at: ISO}`. A Flask background thread (started with `app.after_request` or in `create_app()` using a daemon thread) wakes every 60 seconds, checks `scheduled_runs.json` for entries where `run_at <= now`, and launches each due simulation through the same `SimulationRunner` path as an immediate run. `GET /api/simulation/scheduled` (admin-gated) lists pending and recently-completed scheduled runs with status. `DELETE /api/simulation/scheduled/:id` cancels a pending run. For operators who use MiroShark as part of a weekly research workflow, the scheduler turns a synchronous research tool into an autonomous one — "set it, sleep, wake up to results."

**How:**
1. Add `backend/app/services/scheduler_service.py` (~200 LoC, pure stdlib). `schedule_run(data_dir, run_config)` validates `run_at` (must be a valid ISO datetime, must be at least 30 seconds in the future — prevents accidental immediate-duplicate triggers), generates `scheduled_id = hashlib.sha256(f"{run_at}{scenario[:50]}".encode()).hexdigest()[:12]`, writes to `<data_dir>/scheduled_runs.json` (atomic replace via write-to-tmp + rename pattern): an array of `{scheduled_id, run_at, status: "pending", config: {scenario, n_agents, n_rounds, model, archetypes}, created_at: ISO, started_at: null, sim_id: null}`. `get_scheduled_runs(data_dir)` reads the file and returns all entries. `cancel_scheduled_run(data_dir, scheduled_id)` sets status to "cancelled" for matching entries. `tick(data_dir, simulation_runner)` is the background worker function: reads `scheduled_runs.json`, filters entries where `status == "pending"` and `run_at <= now`, and for each due entry calls `simulation_runner.start(config)` (the same function as `POST /api/simulation/start`), updates `status → "launched"`, records `started_at` and the resulting `sim_id`. The background thread calls `tick()` every 60 seconds. Add `GET /api/simulation/scheduled` (admin-gated), `POST /api/simulation/schedule` (admin-gated), and `DELETE /api/simulation/scheduled/:id` (admin-gated). Start the background thread in `create_app()` using `threading.Thread(target=scheduler_loop, daemon=True).start()` with a `threading.Event` for clean shutdown.
2. In the new-simulation form (`Step1GraphBuild.vue`), add an "Advanced" collapsible section. Within it, add a "Schedule for later" toggle. When toggled on: show a datetime-local `<input>` for `run_at` (HTML5 native, no library needed), with a minimum value set to `now + 2 minutes` (computed reactively). When the form is submitted with a `run_at` value set: call `POST /api/simulation/schedule` instead of `POST /api/simulation/start`. On success, navigate to a new `/scheduled` page instead of the watch page, showing a confirmation card: "Simulation scheduled for [formatted datetime]. We'll start it automatically. Scheduled ID: [id]." Add `ScheduledRunsView.vue` at `/scheduled` (admin-gated): a table of pending and launched scheduled runs — columns: scenario excerpt (60 chars), scheduled time, status chip (Pending/Launched/Cancelled), SIM ID (when launched, link to watch page), Created At. A "Cancel" button on pending rows calls `DELETE /api/simulation/scheduled/:id`. Admin navigation link to `/scheduled`.
3. Add `POST /api/simulation/schedule`, `GET /api/simulation/scheduled`, and `DELETE /api/simulation/scheduled/:id` to the OpenAPI spec under a new "Scheduler" tag with `ScheduledRun` and `ScheduledRunsResponse` schemas. Add 14 unit tests in `backend/tests/test_unit_scheduler.py`: schedule_run writes to scheduled_runs.json, run_at in the past returns 400, run_at fewer than 30s ahead returns 400, duplicate scheduled_id prevention (same hash → update not append), cancel sets status cancelled, cancel of non-existent id returns 404, tick launches due pending runs, tick skips future runs, tick skips cancelled runs, tick skips already-launched runs, sim_id written after launch, started_at written after launch, GET returns all entries sorted by run_at ascending, admin token required for all three endpoints. In `docs/FEATURES.md`, add a "Simulation Scheduler" section explaining the 60s tick interval, the 30s minimum lead time, the daemon thread lifecycle, and a note: "Scheduled runs survive Flask restarts (persisted in scheduled_runs.json) but require the server to be running at the scheduled time — this is a lightweight scheduler for operator convenience, not a distributed job queue."

## Selection Rationale

Five angles distinct from all 10 ideas suggested May 28 and May 30, the 29 push-blocked local builds, and the 4 open ecosystem PRs:

- **Ecosystem Registry API** (#1) — Four ecosystem projects are mid-PR right now. The Ecosystem Registry API is the infrastructure that serves them and every future project: a machine-readable endpoint that turns ECOSYSTEM.md into a live API. It positions MiroShark as a legible hub, not just a standalone tool. Distinct from every prior idea because it's about the ecosystem graph itself, not simulation features.

- **French Locale** (#2) — Issue #95 is an open community ask that has gone unanswered for 11 days. Chinese localization shipped weeks ago. French completes the multilingual narrative and is the fastest possible signal of community responsiveness — close an issue filed by a contributor by implementing the exact thing they asked for. Pure translation work: no new architecture, no new dependencies, deterministic scope.

- **Simulation RSS Feed** (#3) — The completion webhook (May 28, #2) pushes results to a configured URL. RSS is the pull-subscription complement: ecosystem projects subscribe once and receive all future publications without polling or webhook configuration. Pure stdlib XML generation. Every RSS reader, webhook bridge (Zapier, n8n, Make), and developer newsletter tool accepts RSS. For the four new ecosystem projects, this is zero-configuration integration.

- **Embed Theme Parameter** (#4) — The 4 open ecosystem PRs represent projects with their own visual identities building on MiroShark's embed surface. The dark-space-violet theme is stunning for MiroShark's own pages but jarring when dropped into HivemindOS's dashboard or SyntheticsAI's light-mode research UI. A `?theme=dark|light|system` param is the standard web component courtesy — CSS variables only, zero backend changes, immediate value for four projects in mid-integration.

- **Simulation Scheduler** (#5) — All prior trigger mechanisms are synchronous (user-initiated) or event-driven (webhook-triggered). The scheduler is the first time-driven mechanism: a queued run that fires autonomously at a specified datetime. For operators running recurring research cycles — weekly scenario tracking, pre-event simulation queuing, automated digest generation — this turns MiroShark from a synchronous tool into an autonomous one. Distinct from the Inbound Launch Webhook (event-triggered, not time-triggered) and from every prior feature.
