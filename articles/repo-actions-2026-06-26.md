### 1. Agent Communication Transcript Export

**Type:** Feature
**Effort:** Small (hours)
**Impact:** MiroShark generates rich agent conversation data — posts, @-mentions, stance declarations, round-by-round progression — but the only way to read it today is through the frontend UI or raw JSON files. `GET /api/simulation/:id/transcript.md` returns a formatted Markdown document: a round-by-round conversation thread with each agent's name, stance chip (`[BULLISH]`/`[BEARISH]`/`[NEUTRAL]`), platform label, and post body, with @-mention patterns highlighted. `GET /api/simulation/:id/transcript.txt` is the plain-text variant (no Markdown). A "Download transcript" button in the simulation detail view (next to the existing share/embed controls) triggers the download. For the researcher audience (the "peer-reviewed paper" hyperstition), being able to say "here's the full agent conversation as a readable document" is the qualitative complement to the quantitative signal.json. For operators sharing results with non-technical stakeholders, a transcript PDF is more legible than a UI screenshot. No external dependencies — reads existing actions.jsonl, simulation_config.json, and state.json.

**How:**
1. Add `backend/app/services/transcript_service.py` (~140 LoC, pure stdlib). `build_transcript(sim_dir, fmt="md")` reads `simulation_config.json` (title, topic, agent_count, num_rounds, platforms), then scans `data/simulations/{sim_id}/{platform}/actions.jsonl` for all platform action files. Each JSONL line is parsed for `{agent_name, stance, platform, content, round, timestamp}`. Groups actions by round (ascending), then by timestamp within each round. For Markdown format: emits a header block (`# {title}\n*{topic}*\n{N} agents · {N} rounds · completed {date}*`), then per round a `## Round {N}` header, then per action a formatted block: `` **{agent_name}** `[{STANCE}]` *{platform}* \n> {content}\n ``. @-mention patterns (`@{agent_name}`) are highlighted in Markdown as `**@{agent_name}**`. For plain text: same structure without Markdown syntax. Returns the transcript string. `get_transcript(sim_dir, sim_id, fmt)` calls `build_transcript`, returns `{content: str, format: str, sim_id: str, agent_count: int, action_count: int, word_count: int}`. Add `GET /api/simulation/:id/transcript.md` (unauthenticated for published sims, 403 for unpublished; `Content-Type: text/markdown; charset=utf-8`; `Content-Disposition: attachment; filename="{sim_id}-transcript.md"`; `Cache-Control: public, max-age=3600`). Add `GET /api/simulation/:id/transcript.txt` (same auth, `Content-Type: text/plain`). If `actions.jsonl` files are absent (simulation not complete or no actions logged): return 404 with `{error: "Transcript not available — simulation has no recorded actions"}`. Add 10 unit tests in `backend/tests/test_unit_transcript.py`: single-round transcript has correct header, multi-round transcript groups actions by round, @-mentions highlighted in MD but not TXT, BULLISH/BEARISH/NEUTRAL chips correct, unpublished sim returns 403, sim with no actions.jsonl returns 404, action_count matches JSONL line count, word_count is positive for non-empty sim, round ordering is ascending, agent_name appears before content in each entry.
2. In `Step3Simulation.vue` (simulation detail view), add a "Download transcript" button to the existing action bar (next to the "Share" / "Embed" controls). Button is enabled only when `state.status === "complete"`. On click: opens a small dropdown with two options — "Markdown (.md)" and "Plain text (.txt)". Selecting one triggers a browser download via `window.location.href = /api/simulation/{sim_id}/transcript.{ext}`. No new API polling needed — the endpoint returns the file directly. Add `getTranscriptUrl(simId, fmt)` to `frontend/src/api/simulation.js` (returns the URL string; actual download handled via direct href). Button is visible to all users (unauthenticated); 403 response for unpublished sims shows a toast: "Transcript not available — simulation is not published." Bilingual EN/ZH button label ("Download Transcript" / "下载对话记录").
3. Add `GET /api/simulation/:id/transcript.md` and `GET /api/simulation/:id/transcript.txt` to the OpenAPI spec under the "Simulation" tag. No new schema needed — response is `text/markdown` and `text/plain`. Add a "Transcript Export" section to `docs/FEATURES.md`: explains the round-by-round structure, the @-mention highlighting, the two format options, and notes the 3600-second cache (transcript regenerated at most once per hour per simulation). Include a short example transcript excerpt (5 lines) showing the format. Note: for very large simulations (>500 agents, >10 rounds), transcripts may be large (>1MB) — the endpoint streams the response so clients can download without buffering the full body.

### 2. Round-Level State Snapshot API

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The current API exposes final simulation state: `signal.json` (consensus at end), `trajectory.json` (per-agent stance across all rounds in one blob), `confidence/trajectory` (aggregate confidence line). But operators and researchers regularly ask "what did consensus look like at round 3?" — not just the start and end. `GET /api/simulation/:id/rounds` returns a list of completed rounds with metadata (round number, action count, consensus pct at that round's close, dominant stance, timestamp). `GET /api/simulation/:id/rounds/:n/state` returns the simulation state at the end of round N: each agent's stance (extracted from `trajectory.json` at round N), the round's aggregate yes_pct, and the top-3 most active agents. This is the data foundation that the Simulation Replay Stepper (proposed Jun 18, never built due to push block) would need as its backend. It also enables operators to pinpoint exactly which round a stance flip happened, rather than computing it themselves from the full trajectory blob. Pure stdlib — reads trajectory.json and actions.jsonl, no new file writes.

**How:**
1. Add `backend/app/services/round_state_service.py` (~160 LoC, pure stdlib). `get_rounds_index(sim_dir)` reads `trajectory.json` to extract the set of round numbers (keys in the per-agent trajectory dictionaries). For each round N: counts actions in `actions.jsonl` where `round == N` across all platform files, computes `yes_pct_at_round_N` (fraction of agents whose stance at round N is "bullish" or equivalent positive stance), determines `dominant_stance` ("BULLISH" if yes_pct > 55, "BEARISH" if < 45, "SPLIT" otherwise), reads `state.json` for `started_at` + duration estimation. Returns `{sim_id, total_rounds: int, rounds: [{round: int, action_count: int, yes_pct: float, dominant_stance: str}]}`. Caches to `data/simulations/{sim_id}/rounds_index.json` with 15-minute mtime invalidation (invalidated when simulation completes). `get_round_state(sim_dir, round_n)` reads `trajectory.json` for each agent's stance at round N (the trajectory stores stances per round per agent; read `trajectory[agent_id][str(round_n)].stance`). Returns `{round: int, yes_pct: float, dominant_stance: str, agent_count: int, agents: [{agent_id: str, name: str, profession: str, stance: str, confidence: float|null}], top_active_agents: [{agent_id: str, name: str, action_count: int}]}`. `top_active_agents`: count actions in `actions.jsonl` where `round == N` and `agent_id == X`, return top 3 by count. Returns 404 if round N > total_rounds or trajectory.json absent. Add `GET /api/simulation/:id/rounds` (unauthenticated for published sims; `Cache-Control: public, max-age=900`). Add `GET /api/simulation/:id/rounds/:n/state` (unauthenticated for published sims; `Cache-Control: public, max-age=900`). Add 10 unit tests in `backend/tests/test_unit_round_state.py`: get_rounds_index returns correct total_rounds, round 1 yes_pct computed from agent stances in trajectory.json, dominant_stance "BULLISH" when yes_pct > 55, dominant_stance "SPLIT" when yes_pct = 50, get_round_state returns all agents with stances, top_active_agents limited to 3, round > total_rounds returns 404, unpublished sim returns 403, action_count per round matches JSONL line count, rounds_index.json cache written after first call.
2. In the API explorer or embed dialog, surface `GET /api/simulation/:id/rounds` as a new data endpoint. In the EmbedDialog (or developer docs section of the simulation view), add a "Round Data" section showing a curl example: `curl /api/simulation/:id/rounds` with a sample response excerpt. No new Vue component needed — this is primarily a developer/researcher surface. However: in `Step3Simulation.vue` operator view, update the existing timeline chart (if one exists) or add a minimal round-selector in the trajectory section — a horizontal row of round badges (Round 1, Round 2, …, Round N) where clicking a badge fetches `GET /api/simulation/:id/rounds/:n/state` and shows a modal with the round's agent stance breakdown (same layout as the existing signal panel but filtered to that round). The round badges are only shown when `total_rounds > 1`. Add `getRoundsIndex(simId)` and `getRoundState(simId, n)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels on the round selector and modal header.
3. Add `GET /api/simulation/:id/rounds` and `GET /api/simulation/:id/rounds/:n/state` to the OpenAPI spec under the "Simulation" tag. Schemas: `RoundsIndex` (`{sim_id, total_rounds, rounds: [RoundMeta]}`), `RoundMeta` (`{round, action_count, yes_pct, dominant_stance}`), `RoundState` (`{round, yes_pct, dominant_stance, agent_count, agents: [AgentRoundState], top_active_agents: [ActiveAgent]}`). Add a "Round-Level State" section to `docs/API.md`: explains the two endpoints, the trajectory.json parsing approach, the 15-minute cache, and notes that `rounds/:n/state` is the correct data source for building a round-by-round replay UI (with a link to the frontend overlay implementation if it ships). Note: the `confidence` field in each agent's state is null if the simulation did not capture per-agent confidence scores (present only in simulations run with extended thinking models).

### 3. GitHub Actions Workflow Template

**Type:** DX / Integration
**Effort:** Small (hours)
**Impact:** CLAUDE.md landed this week — MiroShark is explicitly designed for AI coding agents. GitHub Actions is where those agents live. A `templates/github-actions/` directory with two reusable workflow files puts MiroShark in the GitHub Actions ecosystem discovery path and gives operators a zero-friction path to automated simulation. `miroshark-weekly.yml`: runs a configurable simulation on a schedule (default: every Monday at 09:00 UTC), posts results as a GitHub issue comment using `gh`. `miroshark-on-pr.yml`: triggered on pull_request, runs a sentiment simulation on the PR title/description, posts consensus result as a PR comment (useful for teams doing product/market validation on proposals). Both templates require only `MIROSHARK_URL` and `MIROSHARK_ADMIN_TOKEN` as GitHub secrets — no new MiroShark features needed. The template includes a "Deploy your own MiroShark" note pointing to `docs/INSTALL.md`, which directly serves the "public-facing MiroShark instance by July 15" hyperstition. For the HackerNews hyperstition: "MiroShark + GitHub Actions = automated public sentiment checks on every PR" is a shareable technical story.

**How:**
1. Create `templates/github-actions/miroshark-weekly.yml` — a complete, copy-paste-ready GitHub Actions workflow. Key sections: `on: schedule: - cron: "0 9 * * 1"` + `workflow_dispatch` (for manual triggers). `env` block: `MIROSHARK_URL: ${{ secrets.MIROSHARK_URL }}`, `MIROSHARK_ADMIN_TOKEN: ${{ secrets.MIROSHARK_ADMIN_TOKEN }}`, `SIMULATION_TOPIC: ${{ vars.SIMULATION_TOPIC || 'Will AI regulation pass in the EU this year?' }}`. Steps: (1) `actions/checkout@v4`. (2) `Set up Python + install miroshark CLI` via `pip install miroshark` (or `pip install -e .` if working from source). (3) `Run simulation` — uses `miroshark simulate "$SIMULATION_TOPIC" --base-url "$MIROSHARK_URL" --output json > result.json`. (4) `Extract result` — `jq` to extract `direction`, `yes_pct`, `sim_id` from `result.json`. (5) `Post to GitHub issue` — `gh issue create --title "Weekly Simulation: $SIMULATION_TOPIC" --body "..."` with the result formatted as Markdown (direction, consensus pct, sim_id with link to `$MIROSHARK_URL/simulation/$SIM_ID`). Includes a `|| true` fallback on the gh step so the workflow doesn't fail if issue creation fails. Add clear comment blocks explaining each step and what to customize. Create `templates/github-actions/miroshark-on-pr.yml` — triggered on `pull_request: types: [opened, edited]`. Runs a simulation where the topic is `"${{ github.event.pull_request.title }}"`. Posts result as PR comment via `gh pr comment ${{ github.event.pull_request.number }} --body "..."`. Includes guard: only runs if `${{ vars.ENABLE_MIROSHARK_CI }}` is `"true"` — opt-in so repos that copy the template don't accidentally run simulations on every PR. Add `templates/github-actions/README.md` — explains what each template does, the required secrets (`MIROSHARK_URL`, `MIROSHARK_ADMIN_TOKEN`), the optional vars (`SIMULATION_TOPIC`, `ENABLE_MIROSHARK_CI`), and links to `docs/INSTALL.md` for hosting your own MiroShark endpoint. Inline security note: never log `MIROSHARK_ADMIN_TOKEN` in workflow steps — the templates are written to only pass it in headers/env vars, never echo it.
2. Add a "GitHub Actions Templates" section to `docs/INSTALL.md` (or create `docs/GITHUB-ACTIONS.md`). Content: what the templates do, how to copy them into a repo, the secret setup walkthrough, and two worked examples — the weekly simulation output (formatted as a GitHub issue body) and the PR comment output. Add a "GitHub Actions" badge or section in the repo README (after the Quick Start section): `[![Automate with GitHub Actions](https://img.shields.io/badge/Automate-GitHub%20Actions-2088FF?logo=githubactions)](templates/github-actions/)` — a badge linking to the templates directory. This surfaces the templates in README search results. Also add the templates directory to the top-level `.github/` discussion or `CONTRIBUTING.md` so contributors know it exists.
3. No backend changes, no OpenAPI changes. Add templates to `.gitattributes` as `templates/github-actions/*.yml linguist-language=YAML linguist-generated=false` so GitHub renders them correctly in the repo language stats (YAML workflows sometimes get misclassified). Add a brief `templates/github-actions/SECURITY.md` file: "Never hardcode `MIROSHARK_ADMIN_TOKEN` in workflow files. Store it as a GitHub secret. The templates here use `${{ secrets.MIROSHARK_ADMIN_TOKEN }}` exclusively." Add the new template path to `CODEOWNERS` (if present) or a `templates/` entry in `.github/CODEOWNERS` so PRs that modify the templates get flagged for review.

### 4. Jupyter Notebook Research Gallery

### 5. Topic Autocomplete API

**Type:** DX / Feature
**Effort:** Small (hours)
**Impact:** Every new MiroShark operator stares at a blank topic field. What should they simulate? The full-text search endpoint (built but push-blocked) lets them search *within* simulations; this is different — autocomplete *before* a simulation runs, suggesting topics that have been simulated before. `GET /api/topics/suggest?q=bitcoin&limit=5` returns the top-5 matching topics from the historical simulation catalog, scored by recency + frequency. The frontend adds a debounced autocomplete dropdown to the Step2 topic input: as the operator types, suggestions appear. Clicking a suggestion auto-fills the topic field and shows a "3 prior simulations on this topic" chip — giving operators confidence they're framing a question that the simulation engine handles well. For the Campaign Series (proposed Jun 24), topic autocomplete is also the natural UX for linking a new simulation to an existing series: type the series topic, autocomplete suggests "yes, 3 sims on this exact topic." The suggest endpoint is a direct scan of existing simulation_config.json files — pure stdlib, no new indexed data structures needed.

**How:**
1. Add `backend/app/services/topic_suggest_service.py` (~80 LoC, pure stdlib). `suggest_topics(data_dir, query, limit=5)` scans all `data/simulations/*/simulation_config.json` files (published only — checks `state.json` for `publish_date`). Extracts `topic` and `title` fields. Deduplicates by normalized topic string (`topic.lower().strip()`). Scores each unique topic by: (a) `frequency` = count of simulations on that topic; (b) `recency_boost` = 1.0 if any simulation on this topic was completed within the past 30 days, else 0.5. Final score = `frequency * recency_boost`. Filters to topics where `query.lower()` appears anywhere in the topic string (case-insensitive substring match). Sorts by score descending. Returns `{query: str, suggestions: [{topic: str, simulation_count: int, last_simulated_at: ISO|null}], total_unique_topics: int}`. If `query` is fewer than 2 characters: return empty suggestions list (don't scan on single-character queries). Cache to `data/.topic_suggest_index.json` with 5-minute mtime invalidation — rebuild the index when cache is stale. Add `GET /api/topics/suggest?q=:query&limit=:limit` (unauthenticated; `q` required, min 2 chars; `limit` optional, clamp [1, 20], default 5; `Cache-Control: public, max-age=120`). Returns 400 if `q` is fewer than 2 characters. Add 10 unit tests in `backend/tests/test_unit_topic_suggest.py`: query shorter than 2 chars returns 400, case-insensitive match ("bitcoin" matches "Bitcoin Price"), frequency counts deduplicated simulations correctly, recency_boost applied to topics with recent sims, results sorted by score desc, limit parameter clamps to 20, unpublished sims excluded from results, empty data_dir returns empty suggestions, total_unique_topics count correct, exact match appears first when frequency is equal.
2. In `Step2EnvSetup.vue` (the topic input step), add autocomplete behavior to the topic text input. On each keystroke (debounced 300ms), if the input length ≥ 2: fetch `GET /api/topics/suggest?q={input}&limit=5`. Show a dropdown below the input with up to 5 suggestion rows. Each row: `{topic text}` · `{simulation_count} simulation{s}` · `{last_simulated_at date or "recently"}`. Clicking a row: fills the topic input with the selected topic, closes the dropdown. ESC key closes the dropdown without selecting. On blur (click outside), close the dropdown. If the API returns 0 suggestions: hide the dropdown (don't show "no results" — topic input is free-form, so operators should feel free to enter novel topics). Keyboard navigation: up/down arrows cycle suggestions, Enter selects. Add `suggestTopics(query)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH: dropdown row metadata labels ("simulation" / "模拟", "recently" / "最近"). Don't show the dropdown while the user has explicitly selected a suggestion and not changed the input — only show on active typing.
3. Add `GET /api/topics/suggest` to the OpenAPI spec under a new "Discovery" tag. Schema: `TopicSuggestResponse` (`{query: str, suggestions: [TopicSuggestion], total_unique_topics: int}`), `TopicSuggestion` (`{topic: str, simulation_count: int, last_simulated_at: str|null}`). Add a "Topic Autocomplete" section to `docs/FEATURES.md`: explains the scoring (frequency × recency boost), the 2-character minimum, the 120-second cache, and notes that suggestions are drawn from published simulations only — novel topics are always valid even if no suggestion appears. Add a one-liner to `docs/API.md` under the Search/Discovery section: "For topic suggestions while configuring a new simulation, use `GET /api/topics/suggest?q=<partial-topic>&limit=5`."

## Selection Rationale

Five ideas grounded in what changed this week and what the hyperstitions need:

- **Agent Communication Transcript Export** (#1) — MiroShark generates a full agent conversation that no one can currently read as prose. The transcript endpoint turns that data into a document: shareable, citable, legible. For the researcher audience (paper citation hyperstition) and for non-technical stakeholders, "download the full conversation" is the missing complement to the confidence chart and signal summary. Two hours to build; the data is already there.

- **Round-Level State Snapshot API** (#2) — The confidence trajectory (built, push-blocked) shows the aggregate confidence line. But operators keep asking "what did round 3 look like?" The round state API answers that at the per-agent level. It's also the backend foundation that the Simulation Replay Stepper (proposed Jun 18, never built due to push block) needs — build the data layer first, the UI can follow. The round index endpoint doubles as a navigation primitive for any future round-aware frontend feature.

- **GitHub Actions Workflow Template** (#3) — CLAUDE.md signals that MiroShark is ready for AI-driven automation workflows. GitHub Actions is where those workflows live. The templates put MiroShark in the Actions ecosystem discovery path (GitHub's marketplace search, "Used by" section on template repos) and give operators a copy-paste path to weekly automated simulation. Directly serves the "public-facing MiroShark instance by July 15" hyperstition: operators who follow the template will host their own endpoint.

- **Jupyter Notebook Research Gallery** (#4) — The research community cites tools they can reproduce. A Binder badge in the README means anyone can run MiroShark data analysis in the browser, zero install. The notebook is also a shareable artifact — "here's a Jupyter notebook that visualizes MiroShark agent stance evolution" is a post that targets the ML/data science community on HackerNews and Twitter simultaneously.
