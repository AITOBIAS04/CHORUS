### 1. Interactive Simulation Replay Player

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The GIF export (PR #50) produces a static 12-frame animation — useful for sharing, useless for analysis. The spectator watch page (PR #67) lets you watch a simulation live, but you cannot pause, rewind, or step frame-by-frame through a completed simulation. The Interactive Replay Player is a browser-based VCR for completed simulations: play, pause, step forward/back by one round, and jump to any round via a scrubber. At each round, the belief drift chart animates up to that point, the top influencer is highlighted in the interaction network, and the round's most-shared post appears in a side panel with the agent's name, archetype, platform, and current stance. For researchers who want to present a simulation result in a talk or paper — "let me walk you through what happened round by round" — this is the presentation layer that currently doesn't exist. For builders embedding MiroShark outputs in dashboards, the replay player becomes an embeddable iframe (the 8th share surface), replacing the static GIF with an interactive experience. The implementation is frontend-only against existing data: `trajectory.json` for per-round belief prices, the agent posts in `<sim_dir>/posts/` for round content, and the existing belief drift SVG component for the animated chart.
**How:**
1. Add a `ReplayPlayer.vue` component in the simulation results view. The player consists of four elements: (a) the belief drift chart in "animated" mode — pre-computed SVG paths for all rounds, rendered cumulatively up to `currentRound` via a CSS clip-path that expands as `currentRound` increases; (b) a scrubber bar below the chart (an HTML range input, `min=1`, `max=N`, `step=1`, `value=currentRound`) — dragging it jumps to any round instantly; (c) transport controls (◀ Prev / ▶▶ Play / ▐▐ Pause / ▶ Next) as a centered row of buttons; (d) a "Round card" below the controls: `Round N of M · YES: X% · NO: Y% · NEUTRAL: Z% · Top influencer: [name] ([archetype], [platform])`. Below the round card, the top post from that round: agent avatar initials, agent name, stance badge, and the first 280 characters of the post body. In play mode, advance one round every 1.5 seconds (configurable via a ×1 / ×2 / ×4 speed toggle). Add a "Replay" tab toggle in the visualization toolbar alongside "Belief Drift," "Interaction Network," and "Heatmap."
2. Add `GET /api/simulation/:id/replay-data` returning a compact payload: `{total_rounds: N, rounds: [{round, yes_price, no_price, neutral_price, top_agent: {name, archetype, platform, stance}, top_post: {text, agent_name, stance}}]}` — one entry per round. Derive `top_post` as the highest-upvoted or first post from the top influencer in that round. This endpoint is cacheable (simulation data is immutable after completion) — set `Cache-Control: public, max-age=86400`. The `ReplayPlayer.vue` fetches this once on mount and drives all subsequent rendering from the client-side cache; no additional requests are made as the user scrubs or plays. Add `GET /api/simulation/:id/replay-data` to the OpenAPI spec under an "Export" tag with the `ReplayDataResponse` schema.
3. Add `/replay/<sim_id>` as a standalone shareable route (the 8th share surface). The `/replay/<sim_id>` page renders the `ReplayPlayer` component full-screen (no sidebar) with the scenario title and a "View full results →" link. Support `?round=N` as a query parameter to deep-link to a specific round (useful for presentations: share a URL that opens at the exact round where something interesting happened). In the simulation results toolbar, add a "Replay" share button alongside "Share," "Embed," "Export," "Cite" — it copies the `/replay/<sim_id>` URL to clipboard. In the Embed dialog, add a "Replay Player" tab (alongside "Simulation card" and "GIF") with an iframe embed code: `<iframe src="/replay/<sim_id>?autoplay=true" width="800" height="480">`. Add `?autoplay=true` mode that begins playing on load (for embedding in articles). Document the replay route and `?round=N` permalink in `docs/FEATURES.md` under a new "Replay Player" section.

### 2. Inbound Webhook — Launch Simulation via External Trigger

**Type:** Integration
**Effort:** Small (hours)
**Impact:** The completion webhook (PR #46) lets external systems react when a simulation finishes. HMAC verification (PR #79) secures outbound delivery. But the automation loop only goes one direction: MiroShark calls out, but cannot be called in. A Zapier workflow that monitors a Reddit thread for spike activity has no way to automatically trigger a MiroShark simulation when a spike is detected. A GitHub Actions job that detects a new release has no way to auto-run "how will the community react to this release?" without a human clicking "New Simulation." The Inbound Webhook closes this loop: a single authenticated `POST /api/webhooks/launch-simulation` endpoint that accepts a simulation config payload and starts a new simulation, returning `{sim_id, watch_url, events_url}` immediately. For n8n, Zapier, and GitHub Actions users — the audience who already built outbound integrations via PR #46 — this is the inbound half they've been waiting for. The HMAC-based auth pattern from PR #79 applies directly: the operator pre-configures a shared secret, and callers sign their payload with it. Zero new infrastructure: it's a thin authenticated wrapper around the existing `POST /api/simulation/start` code path.
**How:**
1. Add `POST /api/webhooks/launch-simulation` with HMAC-SHA256 request verification (same pattern as PR #79's outbound verification, reversed): the caller includes an `X-MiroShark-Signature: sha256=<hmac_hex>` header over the raw request body. The endpoint verifies the signature against `LAUNCH_WEBHOOK_SECRET` from config (a new env var, distinct from the completion webhook secret). On valid signature, parse the body as a standard simulation launch config (same schema as `POST /api/simulation/start`): `{scenario, n_agents, model, n_rounds, article_language, director_script_id (optional), persona_ids (optional)}`. Launch the simulation via the existing runner code path and return immediately (do not wait for completion) with HTTP 202: `{sim_id, watch_url: "/watch/<sim_id>", events_url: "/api/simulation/<sim_id>/events", completion_webhook_will_fire: bool}`. If the caller has configured a completion webhook, set `completion_webhook_will_fire: true` so automation pipelines know they'll receive a callback when done. On invalid signature: HTTP 401. On malformed body: HTTP 400 with validation errors (same as the existing start endpoint).
2. Add a `Launch Webhook` section to the Settings page (alongside the existing "Completion Webhook URL" and "Completion Webhook Secret" fields). Two new fields: "Launch Webhook Secret" (a configurable shared secret, auto-generated on first view with a "Regenerate" button) and a read-only "Launch Endpoint" display showing the full URL of `POST /api/webhooks/launch-simulation`. Below the fields, a "How to use" expandable panel with three usage examples: (a) curl example with the HMAC signature construction; (b) n8n HTTP Request node config (method: POST, authentication: "Header Auth," header name: `X-MiroShark-Signature`, example value computation); (c) GitHub Actions step using `curl` and `openssl dgst -sha256 -hmac "${{ secrets.LAUNCH_SECRET }}"` to construct the signature. Each example is syntax-highlighted in a monospace code block with a "Copy" button.
3. Add `POST /api/webhooks/launch-simulation` to the OpenAPI spec under a new "Webhooks" tag (alongside the existing completion webhook documentation). Include the full request schema (aliased from the simulation start schema), the `X-MiroShark-Signature` header as a required security header, and the 202 response schema (`LaunchWebhookResponse`). Add an `x-webhook-direction: inbound` custom property to the spec to distinguish it from outbound webhooks in generated API client SDKs. In `docs/API.md`, add a "Webhooks" section that documents both directions: "Outbound — Completion Notification" (PR #46, #79) and "Inbound — Launch Trigger" (this PR). Include a worked end-to-end example: "Monitor a Reddit post for comment velocity; when velocity exceeds threshold, POST to `/api/webhooks/launch-simulation` with the post title as scenario; receive completion webhook with the simulation article." Add a note: "The `sim_id` returned in the 202 response can be used immediately with the SSE event stream endpoint to monitor the simulation in real time."

### 3. Agent Conversation Thread View

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Three views currently describe a completed simulation: the belief drift chart (aggregate per-round numbers), the interaction network (who influenced whom, rendered as a graph), and the transcript export (all posts, chronological, flat). None of them show how the conversation actually flowed: which agent replied to whom, which post triggered a stance change three posts later, how a single contrarian post cascaded into a coalition shift. The social structure of the simulation — the thread beneath the aggregate — is invisible unless you read hundreds of raw posts in order. The Agent Conversation Thread View renders the simulation as a nested social media thread: top-level posts appear on the left; reply posts are indented under the post they responded to. Each post card shows the agent's initials, archetype chip, platform chip, stance badge, and post text. Agents whose stance changed this round are highlighted (a yellow border: "flipped here"). The view is filterable by round, by agent, and by stance — making it easy to find the moment a specific agent changed their mind and trace what they were responding to. For researchers using MiroShark to understand persuasion dynamics and information cascade — not just final consensus numbers — this is the read layer that the existing exports don't provide. It transforms the simulation from a chart into a narrative you can actually follow.
**How:**
1. Build `ThreadView.vue` as a new visualization tab alongside "Belief Drift," "Replay," "Interaction Network," and "Heatmap." The component fetches per-agent posts from a new endpoint `GET /api/simulation/:id/thread` returning `{rounds: [{round, posts: [{post_id, agent_name, archetype, platform, stance, stance_changed_this_round (bool), text, reply_to_post_id (string|null), timestamp_offset (ms from round start)}]}]}`. The reply_to_post_id field is derived from the simulation runner's post-generation step: agents are prompted to reference specific prior posts by post_id when composing replies (this prompt context is already present in the existing agent action loop — the post_id just needs to be extracted and stored alongside the post body in `<sim_dir>/posts/<round>.json`). Render as a nested list: top-level posts (reply_to_post_id = null) render as full-width cards; reply posts render indented beneath their parent with a connecting vertical line (CSS left-border, 2px solid #374151). Max nesting depth: 3 levels (flatten deeper replies to level 3 with a "→ In reply to [agent]" prefix). Round dividers separate each round group.
2. Add filter controls in the ThreadView toolbar: (a) "Round" — a dropdown to jump to a specific round or show all rounds; (b) "Agent" — a multi-select chip picker to show only posts from selected agents (useful for tracking one agent's full arc through the simulation); (c) "Stance changed" — a toggle that highlights (yellow border, `border-color: #fbbf24`) cards where the posting agent's stance changed in that round. Add a "Top cascade" auto-detection: identify the post with the most total descendants in the reply tree (the post that generated the most subsequent conversation) and pin a "🔥 Most-discussed post" badge on its card. In the simulation article generator, inject thread-aware context: "Thread analysis: longest reply chain is N posts, initiated by [agent_name] in round R. Top cascade post: '[first 80 chars].' First reply-triggered stance change at round [R], agent [name] flipped [stance] in response to [agent_name]'s post." The generated article incorporates thread structure into its narrative rather than describing only aggregate belief movement.
3. Add `GET /api/simulation/:id/thread` to the OpenAPI spec with the `ThreadResponse` schema. In the transcript export (PR #57), add a `reply_to:` field to each post entry when `reply_to_post_id` is set — making the thread structure machine-readable in the Markdown export. Add a `?format=threaded` query parameter to `GET /api/simulation/:id/transcript` that outputs the transcript as a nested structure (replies indented with `>` Markdown blockquote syntax) rather than flat chronological order. In the spectator watch page (PR #67), when new posts arrive via polling, render them in thread position rather than appending them to a flat list — so live viewers see the conversation thread assembling in real time. Add `?view=thread` as a permalink variant on the share URL, so researchers can share a link directly to the thread view.

### 4. Multi-Model Race Mode

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** MiroShark's model presets (Smart, Balanced, Cheap from the settings) let users pick which LLM powers their agents. But two researchers who each run the same scenario with different model presets will reach different conclusions — and have no systematic way to compare them. Is the "74% bearish consensus" a property of the scenario, or of the model that generated the agents? Multi-Model Race Mode answers this directly: run the same scenario with the same seed but two different model presets simultaneously, and show the results side by side. "Claude Sonnet agents: 78% bearish consensus by round 9. GPT-4o agents: 44% bearish, no consensus reached." The delta between the two runs is the model-induced variance — everything else held constant. This is categorically different from the A/B Comparison View (built, two unrelated simulations) and from Ensemble Runs (suggested May 6, same model with multiple seeds). Race Mode is specifically about isolating the model variable to understand whether consensus is scenario-driven or model-driven. For researchers evaluating LLM reliability in social simulation — the exact critique raised in the "AI Agents Are Not a Panacea" paper — this is the experiment they need but currently have to run manually across two separate browser sessions.
**How:**
1. Add a "Race Mode" toggle in Step 2 (Configure), shown only when the user selects a model preset. When enabled, a second model preset picker appears labeled "vs." (aligned horizontally with the primary picker as "Model A vs. Model B"). The two pickers cannot have the same value — selecting the same model on both shows an inline error: "Select two different models to run a race." At launch, a single API call to `POST /api/simulation/start-race` accepts `{scenario, n_agents, n_rounds, seed, model_a: string, model_b: string, ...rest of config}`. The backend launches two simulations in parallel (using Python's `concurrent.futures.ThreadPoolExecutor(max_workers=2)`), both with identical configs except model. Both simulations run to completion under their respective model IDs. Store `{race_id, sim_id_a, sim_id_b, model_a, model_b, seed}` in `<race_dir>/race_meta.json`. Return HTTP 202 with `{race_id, sim_id_a, sim_id_b, watch_url_a, watch_url_b}`.
2. Add a `RaceResultsView.vue` page at `/race/<race_id>`. Layout mirrors the A/B Comparison View (built locally) but adapted for model comparison: the page header shows "Model Race: [Model A] vs. [Model B] — Same scenario, same seed." The dual belief drift chart uses Model A's color (blue, `#3b82f6`) and Model B's color (amber, `#f59e0b`), with side-by-side and overlay toggle modes. Below the chart, a "Model Comparison" table (4 rows × 3 columns: Metric / Model A / Model B): Participation Rate, Stance Entropy (lower = more consensus), Consensus Velocity (rounds to reach consensus, `null` if no consensus), Cross-Platform Rate. Add a `Δ` column (Model B − Model A) and a "Model-sensitive?" badge on the row with the largest absolute Δ — the metric most affected by model choice. Below the table, a "Verdict" section: if both models reached consensus in the same direction, "Scenario-robust finding: both models converged to [YES/NO] consensus." If they diverged (different directions or one reached consensus and the other didn't): "Model-sensitive finding: results differ by model — verify with Adversarial Mode or Ensemble Runs." Add `GET /api/simulation/race/:race_id` returning `{race_meta, scorecard_a, scorecard_b, scorecard_diff}`.
3. In the gallery explore view, add a "Race" badge on cards that are part of a race (`race_id` is set in gallery metadata). Clicking the badge navigates to `/race/<race_id>`. In the simulation results view, when the current simulation is part of a race, add a "View Race Results →" banner at the top of the results page, linking to the race view. Add `POST /api/simulation/start-race` and `GET /api/simulation/race/:race_id` to the OpenAPI spec under a new "Race Mode" tag. In `docs/FEATURES.md`, add a "Multi-Model Race Mode" section with a worked example: "Run the same crypto policy scenario with Smart (Claude Sonnet) vs. Cheap (Qwen-72B) — compare whether model-class affects consensus direction or speed." Add a note: "Race Mode uses the same seed for both simulations. For seed-level variance, use Ensemble Runs (n_runs > 1). For scenario-level comparison, use the A/B Comparison View." This cross-referencing positions the three comparison primitives clearly for researchers.

### 5. Research Export Bundle (ZIP)

**Type:** DX
**Effort:** Small (hours)
**Impact:** A researcher who wants to submit a MiroShark simulation as evidence in a paper currently needs six separate downloads: trajectory CSV (PR #66), per-agent trajectory CSV (built locally), annotation JSON (built locally), quality report JSON (built locally), citation block (suggested May 8), and reproduce.json (PR #75). The Jupyter notebook (PR #80, just merged) is a seventh. Individually, each endpoint exists or will exist. Together, they're 7 clicks and 7 files that a reviewer has to manually reassemble. The Research Export Bundle is a single `GET /api/simulation/:id/export/bundle` endpoint that packages all of them into a single ZIP file named `<sim_id>_research_bundle.zip`. The archive has a predictable directory structure documented in `docs/research-guide.md`. Unzip it, point Jupyter at `data/`, and everything is in place. For the academic and institutional researcher cohort — the most fork-engaged segment — this is the difference between "I can export data from MiroShark" and "I can hand this ZIP to a co-author and they can reproduce my work." The bundle is also the natural companion artifact to the citation block: cite the simulation, attach the bundle, done.
**How:**
1. Add `GET /api/simulation/:id/export/bundle` as a Flask endpoint that assembles a ZIP file in memory using Python's `zipfile.ZipFile` (stdlib, zero new dependencies) and streams it to the client with `Content-Disposition: attachment; filename="<sim_id>_research_bundle.zip"`. The bundle structure: `<sim_id>_bundle/README.md` (auto-generated, lists all files and their descriptions); `data/trajectory.csv` (aggregate per-round CSV, same as PR #66 endpoint); `data/agent_trajectory.csv` (per-agent × round CSV, same as built-locally endpoint); `data/annotations.json` (all annotations, or `{}` if none); `data/quality_report.json` (quality guard report); `data/reproduce.json` (reproducibility config from PR #75); `data/config.json` (full simulation config); `analysis/simulation.ipynb` (the Jupyter notebook from PR #80, pre-configured with file paths pointing to `../data/`); `citation/citation.bib` (BibTeX block, same as suggested citation endpoint); `citation/citation_apa.txt` (APA format); `citation/citation_ieee.txt` (IEEE format); `article.md` (the generated simulation article). Each file is generated by calling the same code path as its individual endpoint — the bundle is a zip aggregator, not a new data source. Total bundle size for a typical 15-agent, 12-round simulation: approximately 80–150 KB.
2. In the simulation results toolbar, add a "Download Bundle" button (a download icon labeled "Research Bundle") as the primary export CTA — positioned before the individual export buttons. On hover, show a tooltip: "Download all simulation artifacts as a single ZIP: trajectory data, annotations, quality report, citation block, Jupyter notebook, and article." In the EmbedDialog export section, add "Research Bundle (.zip)" as the first option in the export list (above individual format exports). On the simulation detail page, when the simulation has annotations or quality flags, show a contextual nudge below the chart: "This simulation has [N annotations / quality flags] — include them in your research export." Clicking navigates to the bundle download. In the Jupyter notebook (PR #80), update the first cell to add a note: "If you received this notebook as part of a research bundle, the data files referenced below are in `../data/` relative to this notebook."
3. Add `GET /api/simulation/:id/export/bundle` to the OpenAPI spec under the "Export" tag. Response: `application/zip`, documented as a binary download. Add a `bundle_size_estimate_kb: int` field to `GET /api/simulation/:id` metadata — computed once after simulation completion and cached in `<sim_dir>/bundle_meta.json`. This lets API clients display the bundle size before the user triggers the download. In `docs/research-guide.md` (new file), document the bundle format completely: directory structure, each file's schema, how to use the Jupyter notebook, and how to verify reproducibility using `reproduce.json`'s config hash. Cross-reference from `docs/API.md` under a new "Research Workflows" section. In `README.md`, add a "For Researchers" section (alongside existing "For Builders") linking to the research guide. This creates a clear researcher on-ramp: run simulation → download bundle → open notebook → cite.

## Selection Rationale

Five angles distinct from all 15 ideas suggested across May 6, May 8, and May 10, and from the one open issue (#70):

- **Interactive Simulation Replay Player** (#1) — The GIF (PR #50) is static and single-use; the spectator page (PR #67) is live-only. Completed simulations have no interactive playback. Replay is the presentation layer — for talks, demos, and embeds — that turns a static chart into a step-by-step narrative. The `/replay/<sim_id>?round=N` permalink closes the last deep-linking gap in the share surface stack.
- **Inbound Webhook — Launch Trigger** (#2) — The completion webhook (PR #46, #79) sends. Nothing receives. The automation loop is half-open: n8n/Zapier pipelines can react to finished simulations but cannot start them. The inbound webhook is the missing half. HMAC auth pattern (PR #79) applies directly; the endpoint wraps the existing start code path. Closes the loop for the full trigger→monitor→receive cycle.
- **Agent Conversation Thread View** (#3) — The interaction network shows structural influence; the transcript export is flat text. Neither shows thread shape — which agent replied to whom, where the stance-flip cascade started, how a single contrarian post propagated. The threaded view is the social read layer for simulation results, distinct from every existing visualization.
- **Multi-Model Race Mode** (#4) — Ensemble Runs (suggested May 6) isolate seed variance. A/B Comparison (built) compares two independent sims. Race Mode isolates the model variable: same scenario, same seed, two different LLMs. The "model-sensitive / scenario-robust" verdict is the specific finding that addresses the reproducibility critics' core objection — LLM simulation results may be artifacts of the model, not the scenario.
- **Research Export Bundle** (#5) — Individual export endpoints exist (or are built). Getting all of them requires 6+ downloads and manual file assembly. The bundle collapses them into one ZIP with a predictable structure that Jupyter, co-authors, and journal reviewers can unpack without instructions. Pairs directly with the Jupyter notebook (PR #80, just merged) and the citation block (suggested May 8) as the third piece of the complete researcher handoff.
