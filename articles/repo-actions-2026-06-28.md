### 1. Simulation Badge API

**Type:** Feature / Growth
**Effort:** Small (hours)
**Impact:** A `GET /api/simulation/:id/badge.svg` endpoint returns a shields.io-compatible SVG badge: `[MiroShark | BULLISH 67%]` in two color-coded segments (left dark, right green/red/gray by direction). Any website, GitHub README, or blog post embeds it with one Markdown line: `![MiroShark Result](https://miroshark.xyz/api/simulation/abc123/badge.svg)`. Every embed is a passive CTA — viewers see the result badge, click it, land on the simulation page, discover MiroShark. Badges spread virally across READMEs and articles without any operator sharing effort. A companion `GET /api/simulation/:id/badge-url` endpoint returns copy-paste Markdown and HTML snippets so operators know exactly what to paste. Pure SVG string generation — no external dependencies.

**How:**
1. Add `backend/app/services/badge_service.py` (~80 LoC, pure stdlib). `generate_badge(sim_dir, sim_id)` reads `signal.json` for `{direction, yes_pct, confidence_pct}`. Direction maps to hex colors: BULLISH → `#2ea44f` (green), BEARISH → `#d73a49` (red), NEUTRAL → `#6e7681` (gray). Label text: left side `MiroShark`, right side `{DIRECTION} {yes_pct:.0f}%`. Generates an SVG string via Python string formatting (no external SVG lib) — two-segment rectangle (200×20px total): left segment 90px (`#555` background), right segment 110px (direction-colored). Uses `<text>` elements with `font-family="Verdana,DejaVu Sans,sans-serif" font-size="110"` (shields.io convention). Returns `{svg: str, direction: str, yes_pct: float}`. Add `GET /api/simulation/:id/badge.svg` (unauthenticated for published sims; `Content-Type: image/svg+xml`; `Cache-Control: public, max-age=3600`; ETag from sim completion timestamp). Returns a styled "not-found" gray badge SVG for unknown sim IDs and a "private" gray badge for unpublished sims — broken embeds fail gracefully. Add `GET /api/simulation/:id/badge-url` (JSON: `{badge_url, markdown, html}` — copy-paste snippets). Add 8 unit tests: BULLISH maps to green, BEARISH to red, NEUTRAL to gray, yes_pct renders as integer percentage, SVG is valid XML (ET.fromstring doesn't throw), unpublished sim returns 403 badge SVG, unknown sim_id returns 404 badge SVG, Cache-Control header is `public, max-age=3600`.
2. In `Step3Simulation.vue`, add a "Badge" button to the existing share/embed action bar (after the "Embed" button). On click: modal with the badge preview (rendered inline via `<img>` tag), plus three copy-paste snippets — Markdown, HTML (linked badge), raw URL — each with a "Copy" button. Button only enabled when `state.status === "complete"`. Add `getBadgeUrl(simId)` and `getBadgeSnippets(simId)` to `frontend/src/api/simulation.js`. Bilingual EN/ZH label ("Share Badge" / "分享徽章").
3. Add both routes to the OpenAPI spec under the "Simulation" tag. Add a "Badge" section to `docs/FEATURES.md`: color coding, cache TTL, graceful 404/403 badge fallbacks, Markdown example snippet. Add a one-liner to the main `README.md` Quick Start section: "Share results anywhere with embeddable badges — `GET /api/simulation/:id/badge.svg`."

### 2. Python SDK Package (`miroshark-py`)

**Type:** Integration / DX
**Effort:** Medium (1-2 days)
**Impact:** The CLI is a terminal tool. A Python SDK is a programmatic tool — makes MiroShark callable from scripts, notebooks, and CI pipelines. `pip install miroshark` → `from miroshark import MiroSharkClient` → five lines to run a simulation and get a result. PyPI listing adds MiroShark to the Python ecosystem's discovery surface: PyPI search, pip tab-completion, awesome-python lists, AI agent framework integrations. The SDK also makes the Jupyter gallery notebooks (proposed Jun 26) much simpler — replace raw `requests.get()` calls with clean SDK calls. For the "peer-reviewed paper citation by Sep 2026" hyperstition: researchers can `pip install miroshark` and reproduce any public simulation in one function call. Zero external dependencies — pure stdlib (`urllib.request`, `json`, `time`, `dataclasses`).

**How:**
1. Create `sdk/` directory. `sdk/miroshark/__init__.py` exports `MiroSharkClient`, `SimulationResult`, `SimulationError`. `sdk/miroshark/client.py` (~200 LoC, pure stdlib): class `MiroSharkClient(base_url, admin_token=None, timeout=600)`. Methods: `simulate(topic, agents=20, rounds=5, platforms=None, locale="en") → SimulationResult` — POSTs to `/api/simulate`, polls `/api/simulation/{id}` every 5s until complete. `status(sim_id) → dict`. `result(sim_id) → SimulationResult`. `list_simulations(limit=10, status=None) → list[dict]`. `stop(sim_id) → bool`. `estimate(agents, rounds, platforms=None) → dict`. `SimulationResult` is a dataclass: fields `sim_id, topic, direction, yes_pct, confidence_pct, agent_count, num_rounds, completed_at, raw: dict`. Properties: `is_bullish → bool`, `is_bearish → bool`, `summary → str` (one-line text). `SimulationError(message, status_code)` for HTTP errors. `sdk/pyproject.toml`: `name="miroshark"`, `version="0.1.0"`, `python_requires=">=3.8"`, zero dependencies. `sdk/README.md` — installation, 3-line quickstart, API reference. Add 10 unit tests in `sdk/tests/test_client.py` using `unittest.mock.patch` over `urllib.request.urlopen`: `simulate()` polls until status==complete; `result()` raises `SimulationError` on non-200; `SimulationResult.is_bullish` True when direction=="BULLISH"; `summary` includes direction and yes_pct; `list_simulations()` passes limit/status as query params; `stop()` returns True on 200; `estimate()` passes agents/rounds correctly; `SimulationError` carries status_code; missing base_url raises `ValueError`.
2. Add `sdk/` to the main `README.md` under a new "SDK" section (between "CLI" and "API"): 3-line quickstart code block, `pip install miroshark`, link to `sdk/README.md`. Add a GitHub Actions CI step (or `sdk-ci.yml`) that runs `pip install sdk/` and `python -m pytest sdk/tests/` on Python 3.8, 3.11, 3.12. Add `sdk/` to `CONTRIBUTING.md` under "SDK Contributions": "The Python SDK is in `sdk/` — pure stdlib, zero external dependencies. Add a method, add a test, open a PR." Add a PyPI badge to the main README: `[![PyPI version](https://badge.fury.io/py/miroshark.svg)](https://badge.fury.io/py/miroshark)`.
3. Add `Makefile` target `make publish-sdk` → `cd sdk && python -m build && python -m twine upload dist/*`. Document in `sdk/README.md` that initial PyPI upload requires `PYPI_TOKEN` as a GitHub secret and note the SDK versioning convention (SDK 0.x.y tracks API minor versions). No backend changes, no OpenAPI changes — the SDK is a client-side wrapper over existing endpoints. The PyPI listing is the discovery surface: `pip search miroshark` returns the package, and PyPI project pages are indexed by Google.

### 3. CLI `list` Subcommand

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** The CLI has `simulate`, `wait`, `stop`, `cost` — four action verbs. `list` is the obvious fifth: show what simulations are running or have run. `miroshark list` prints a formatted table (sim ID, status, direction, topic, created_at). `miroshark list --status running` shows only active ones. `miroshark list --json` dumps raw JSON for piping into `jq`. With `list` in place, MiroShark becomes fully scriptable: `COUNT=$(miroshark list --status running --json | jq length)` — "how many are running right now?" It's the CRUD verb that closes the loop — every system that can `start` something needs a way to `list` what exists. Small, clean, zero backend changes if the simulations list endpoint already exists.

**How:**
1. Add `list` subcommand to the existing CLI entry point (wherever `simulate`, `wait`, `stop`, `cost` are defined). Hits `GET /api/simulations` (or equivalent list endpoint). Arguments: `--status` (filter: `pending/running/complete/failed`, optional), `--limit` (integer, default 20, max 100), `--json` (raw JSON output). Table output uses plain `str.ljust`/`str.rjust` — no external libs. Column format: `{SIM_ID:<22} {STATUS:<12} {DIRECTION:<10} {TOPIC:<38} {CREATED}`. Column headers on first row. `--json` mode: `print(json.dumps(simulations, indent=2))`. Invalid `--status` value: print error and exit 1. Empty result: print "No simulations found." and exit 0. Uses the same `MIROSHARK_URL`/`MIROSHARK_ADMIN_TOKEN` env vars (or `--base-url`/`--token` flags) as other subcommands. Add 6 unit tests in the CLI test suite: default call returns table output with headers; `--json` flag returns JSON; `--status running` passes status filter to the API; `--limit 5` caps results; invalid `--status` value exits with error; empty result list prints "No simulations found." and exits 0.
2. Update `docs/CLI.md` (or equivalent): add a `list` section with full syntax, all flags, and two worked examples — table output (`miroshark list --limit 3`) and `--json` piped to `jq` (`miroshark list --status complete --json | jq '.[].topic'`). Update the main `README.md` CLI usage block to include `miroshark list` so new users see the full verb set at a glance.
3. If `GET /api/simulations` doesn't support `?status=` filtering server-side: add a trivial filter in the route handler alongside this PR — `if status: results = [r for r in results if r["status"] == status]`. No separate backend PR needed; one line of Python. Add a note in `CONTRIBUTING.md` under "CLI": "The `list` subcommand is in `cli/` — adding new output columns or filter options follows the same pattern as the existing `--status` filter."

### 4. Webhook Event Delivery

**Type:** Integration
**Effort:** Medium (1-2 days)
**Impact:** Operators hosting their own MiroShark instance want push notifications when simulations complete — not polling. `POST /api/webhooks` registers an HTTPS endpoint URL. When a simulation completes, MiroShark POSTs a signed payload (`sim_id`, `topic`, `direction`, `yes_pct`, `confidence_pct`, `completed_at`) with HMAC-SHA256 verification header. The integration unlocks Zapier, Make, n8n automations without a line of polling code: register a webhook URL → every simulation completion fires a Zap or triggers an n8n workflow → results land in Slack, Notion, a spreadsheet. For the "public-facing MiroShark instance by July 15" hyperstition: webhook support turns a deployed instance from a toy into a workflow tool. For the community that runs regular simulations (solo founders, researchers, teams doing weekly market checks), receiving results as pushes rather than polling is a meaningful quality-of-life improvement. Last suggested Jun 14 — 14 days ago, outside the 7-day dedup window.

**How:**
1. Add `backend/app/services/webhook_service.py` (~180 LoC, pure stdlib). Config stored in `data/webhooks.json` (list of `{id, target_url, secret, events, created_at, last_delivery, last_status}`). `register_webhook(target_url, secret, events)` validates `target_url` starts with `https://` (blocks HTTP), validates events are in `{"simulation.complete", "simulation.failed"}`, generates a UUID4 ID (using `uuid.uuid4().hex`), appends to `data/webhooks.json`. `deliver_webhook(webhook_id, event_type, payload)` signs payload with `hmac.new(secret.encode(), json.dumps(payload).encode(), sha256).hexdigest()`, sends via `urllib.request.urlopen` with headers `X-MiroShark-Event`, `X-MiroShark-Signature: sha256={hex}`, `X-MiroShark-Delivery: {uuid}`. Records `last_delivery` and `last_status` (HTTP response code). On failure: records error code, does not retry (caller handles retry via Zapier/n8n). `fire_webhooks(event_type, payload)` calls deliver for every webhook subscribed to event_type. Called from simulation completion path. Routes: `POST /api/webhooks` (admin auth required), `GET /api/webhooks` (admin auth), `DELETE /api/webhooks/:id` (admin auth). Add 10 unit tests: rejects `http://` URL, accepts `https://`, rejects unknown event type, generates unique ID, HMAC header matches expected, `fire_webhooks` calls deliver for matching webhooks only, failure records status code, admin auth required on all routes, DELETE removes webhook from list, empty events list is rejected.
2. Integrate `fire_webhooks` into the simulation completion path — after `state.json` is written with `status: "complete"`, call `fire_webhooks("simulation.complete", {...})`. Wrap in try/except with 5-second timeout so the runner never hangs on a slow webhook target. Add `fire_webhooks("simulation.failed", {...})` in the failure path. Add a "Webhooks" section to `docs/WEBHOOKS.md` (new file): intro, setup guide, payload schemas for both events, HMAC verification code (Python + JavaScript), and three integration examples — Zapier, n8n, Discord channel webhook. Link to `docs/WEBHOOKS.md` from the main README features list.
3. Add all three webhook routes to the OpenAPI spec under a new "Webhooks" tag. Schemas: `WebhookRegistration`, `WebhookResponse`. Add a README badge/section: "Webhooks — register HTTPS endpoints to receive simulation results via signed push notifications." The `docs/WEBHOOKS.md` file is the artifact that makes the integration discoverable — a shareable link for Zapier/n8n community posts.

## Selection Rationale

Five ideas grounded in where the project sits today and what the active hyperstitions need to move:

- **Simulation Badge API** (#1) — Every operator who embeds a badge in a README or blog post is advertising MiroShark to their audience passively. Badges spread viral without effort — they travel wherever operators share their results. The badge API is the smallest possible viral mechanism: one SVG endpoint, infinite reach.

- **Python SDK Package** (#2) — The CLI covers the terminal. A pip-installable SDK covers everything else: Python scripts, Jupyter notebooks, CI pipelines, AI agent frameworks. PyPI listing is a new discovery surface that GitHub search doesn't cover — `pip search miroshark` returns it, awesome-python lists index it, researchers cite it with `pip install miroshark` in their notebooks.

- **CLI `list` Subcommand** (#3) — The CRUD verb that closes the loop. `simulate` starts, `wait` blocks, `stop` cancels, `cost` reports — `list` shows what exists. Without it, operators script MiroShark by storing sim IDs manually. With it, `miroshark list --json | jq` becomes a first-class automation primitive. Small, clean, autonomously buildable in one PR.

- **Webhook Event Delivery** (#4) — The integration that turns a deployed MiroShark instance from a demo into a workflow tool. Every operator who registers a Zapier webhook gets results delivered to Slack, Notion, or a spreadsheet automatically. The Zapier/n8n/Make community posts about new integrations — webhook support is the feature that makes those posts possible.

- **Tutorial Workshop Kit** (#5) — The structural intervention for the "5 creators publish tutorials" hyperstition. Make tutorial-writing mechanical (run five commands, copy the output, fill in your topic) and creators will write tutorials. 281 forks with zero external content is a conversion problem, not a motivation problem.
