### 3. Research Campaign Tracker API

**Type:** Feature / Integration
**Effort:** Medium (1-2 days)
**Impact:** The academic citation hyperstition requires a researcher to run MiroShark for published work. Researchers don't run one simulation — they run 10–30 simulations as part of a methodology section: "We ran 15 simulations varying agent count (25, 50, 100) and platform composition (Twitter-only, Reddit-only, mixed) to test whether platform mix affects consensus stability." Without a grouping mechanism, 15 simulations appear in the gallery as 15 unrelated entries. With a Campaign Tracker, those 15 simulations are a named research campaign with a methodology description, aggregate statistics (consensus variance across runs, cost breakdown, average confidence), and a single shareable URL that a researcher can drop in a paper citation. The Campaign Tracker is the missing primitive that converts MiroShark from a "fun demo tool" into a "research methodology tool" — the distinction that determines whether a simulation gets cited or just run once and forgotten. It also creates natural content for the tutorial hyperstition: a recorded Jupyter walkthrough of setting up a research campaign is the ideal format for a first community tutorial.

**How:**
1. Add `backend/app/services/campaign_service.py` (~220 LoC, pure stdlib). Data model: each campaign stored as a JSON file in `data/campaigns/{campaign_id}/campaign.json`. Schema: `{ "id": str, "title": str, "description": str, "methodology": str, "created_at": ISO8601, "updated_at": ISO8601, "published": bool, "sim_ids": [str], "tags": [str] }`. Functions: `create_campaign(title, description, methodology) -> dict` (generates UUID campaign_id, writes campaign.json, returns campaign object); `add_simulation(campaign_id, sim_id) -> dict` (appends sim_id to campaign.sim_ids, validates sim exists in `data/simulations/`, updates `updated_at`); `remove_simulation(campaign_id, sim_id) -> dict`; `get_campaign(campaign_id) -> dict` (reads campaign.json, enriches with aggregate stats); `list_campaigns(published_only=False) -> list` (scans `data/campaigns/`, returns sorted by `updated_at` descending); `publish_campaign(campaign_id) -> dict` (sets `published: true`); `delete_campaign(campaign_id) -> None`. Aggregate stats computed in `get_campaign()`: `sim_count` (len of sim_ids), `published_sim_count` (how many sims in the campaign are published), `total_cost_usd` (sum of `cost_usd` from each sim's signal.json if available), `consensus_variance` (std dev of `final_bullish_pct` across published sims — measures consistency), `avg_confidence_pct` (mean of `confidence_pct` across published sims), `platform_mix` (sorted dict of platform → count across all sims' config.json), `topic_summary` (first 120 chars of each sim's topic, joined with " | "). Mtime-cache the aggregate stats to `data/campaigns/{id}/stats.json` with 5-min TTL. Add 14 unit tests covering: create round-trips correctly, add/remove simulation updates sim_ids, aggregate stats compute correctly for 3 mock sims, publish gate enforced for unauthenticated GET of unpublished campaign, list returns sorted by updated_at, delete removes campaign directory.
2. Add campaign API routes to `backend/app/api/` in a new `campaign.py` file (~80 LoC). Endpoints: `POST /api/campaigns` (create; body: `{title, description, methodology}`; returns campaign object), `GET /api/campaigns` (list published campaigns; public endpoint; returns array), `GET /api/campaigns/{id}` (get campaign with aggregate stats; published campaigns are public; unpublished require operator auth), `POST /api/campaigns/{id}/simulations` (add simulation to campaign; body: `{sim_id}`; operator-auth), `DELETE /api/campaigns/{id}/simulations/{sim_id}` (remove simulation; operator-auth), `POST /api/campaigns/{id}/publish` (publish campaign; operator-auth), `DELETE /api/campaigns/{id}` (delete campaign; operator-auth). Register the router in `backend/app/main.py`. Add all 7 endpoints to OpenAPI spec under a "Research Campaigns" tag with schemas `CampaignCreate`, `CampaignResponse`, `CampaignListEntry`, `CampaignStats`. Add `GET /api/campaigns` and `GET /api/campaigns/{id}` to `docs/API.md` under a new "Research Campaigns" section.
3. Add a `CampaignView.vue` frontend page at `frontend/src/views/CampaignView.vue` (~200 LoC) reachable at `/campaigns`. Layout: dark-themed card grid (same design system as the explore gallery at `ExploreView.vue`). Each campaign card shows: title, description (first 120 chars), sim count, published sim count, avg confidence percentage, consensus variance chip (green for low variance = consistent results, yellow for medium, red for high variance = noisy methodology). Clicking a card opens a campaign detail page (`/campaigns/{id}`): full methodology text block, a table of all simulations (topic, consensus direction chip, confidence%, cost, published_at), aggregate stats row (total cost, avg confidence, consensus variance, platform mix breakdown), and a "Cite this campaign" copyable citation block in APA format: `[Author, Year]. [Title]. MiroShark Research Campaign. https://miroshark.io/campaigns/{id}. Retrieved [date].` The citation block is the single most important UI element: it gives researchers a ready-made citation that links directly to reproducible methodology. Add `/campaigns` and `/campaigns/:id` routes to `frontend/src/router/index.js`. Add a "Research" link to the navbar alongside "Explore" and "Verified". Update `docs/FEATURES.md` to add a "Research Campaigns" section with the campaign creation workflow and citation format.

### 4. Operator Usage Analytics API

**Type:** Feature / Performance
**Effort:** Medium (1-2 days)
**Impact:** Operators running MiroShark for a research group, enterprise team, or public service have no way to understand their deployment without grepping through log files. How many simulations have completed? What's the average cost? Which topics are most requested? Which rounds do most simulations fail on? Without answers, operators can't right-size their API spend, can't identify slow-running configurations, and can't make data-driven decisions about which features to enable. An analytics endpoint closes this gap and simultaneously makes MiroShark feel enterprise-ready — the kind of tool an IT department will approve for a department-wide deployment when there's a usage dashboard. The endpoint also creates a hook for the CHORUS aeon agent: a daily analytics report ("Yesterday: 3 simulations, $2.41 total cost, avg confidence 71%, top topic: policy analysis") makes the operator feel like the system is managed rather than running blind. It also enables the `skill-leaderboard` skill to report deployment-level stats across forks — if 6 fork operators expose their analytics endpoint publicly, the leaderboard has real data to rank. This appeared in the candidate list three times (Jun 16, Jun 18, Jul 6) without being built; it belongs in the queue.

**How:**
1. Add `backend/app/services/analytics_service.py` (~200 LoC, pure stdlib). Function `compute_analytics(window_days: int = 30) -> dict`. Scans `data/simulations/` for all simulations (published and unpublished). For each simulation, reads `config.json` (created_at, agents, rounds, platforms, topics), `signal.json` (final_bullish_pct, confidence_pct, published, cost_usd if tracked), and `trajectory.json` header if available (last_round_completed — for failure detection). Computes and returns: `{ "window_days": int, "generated_at": ISO8601, "totals": { "simulations_created": int, "simulations_published": int, "simulations_failed": int (created but never published, older than 24h), "total_cost_usd": float, "total_agents_spawned": int (sum of agents × rounds across all sims) }, "averages": { "cost_per_simulation_usd": float, "confidence_pct": float (published sims only), "agents_per_sim": float, "rounds_per_sim": float, "completion_rate_pct": float (published/created × 100) }, "distributions": { "by_platform": { "twitter": int, "reddit": int, ... }, "by_agent_count_bucket": { "1-25": int, "26-50": int, "51-100": int, "100+": int }, "by_round_count": { "3": int, "5": int, "10": int, ... }, "by_consensus": { "bullish": int, "bearish": int, "neutral": int } }, "top_topics": [ { "topic_excerpt": str (first 80 chars), "sim_count": int, "avg_confidence": float } ] (top 10 by sim_count), "daily_volume": [ { "date": YYYY-MM-DD, "simulations": int, "cost_usd": float } ] (last window_days days, sorted asc), "failure_details": { "failed_sim_ids": [str] (first 10), "most_common_failure_round": int | null } }`. Mtime-cache the full result to `.analytics-cache/analytics.json` with a 1-hour TTL (analytics is expensive to compute on large deployments). Add 10 unit tests: empty deployment returns valid schema with zeros, completed sims counted correctly, failure detection uses 24h threshold, cost aggregation handles missing cost_usd gracefully (treat as 0), top_topics sorted correctly, daily_volume sorted ascending, cache TTL honored, window_days parameter filters correctly, platform distribution counts correctly, completion_rate computed as float not int.
2. Add `GET /api/analytics` to `backend/app/api/` in a new `analytics.py` file (~40 LoC). Query params: `?window_days=30` (default 30, max 365, min 1). Operator-authenticated endpoint (same auth as other operator-restricted endpoints). Returns the full analytics dict from `analytics_service.compute_analytics(window_days)`. Also add `GET /api/analytics/summary` (public endpoint, no auth) returning only the `totals` and `averages` sub-dicts — gives external parties (ecosystem partners, press) a non-sensitive public stat page without exposing topic content or cost details. Cache-Control: `private, max-age=3600` for the full endpoint; `public, max-age=3600` for the summary. Add to OpenAPI spec under an "Analytics" tag with `AnalyticsResponse` and `AnalyticsSummaryResponse` schemas. Add to `docs/API.md` under "Analytics". Add `ANALYTICS_PUBLIC_SUMMARY` bool env var (default `true`) to `docs/CONFIGURATION.md` — operators can set this to `false` to hide the summary endpoint entirely.
3. Add `GET /api/analytics` display to the operator admin panel if one exists, or add a minimal `AnalyticsView.vue` page at `frontend/src/views/AnalyticsView.vue` (~150 LoC) reachable at `/analytics` (operator-gated). Layout: stat tiles row (total simulations, published, completion rate %, total cost, avg confidence), a sparkline of daily volume (last 30 days), a platform distribution bar chart (horizontal, single color, matches the existing dark design system — no external chart library, pure CSS/SVG bars), and a top-topics list with per-topic confidence badges. The page is the "dashboard" operators will bookmark — make it the first thing that loads after setup. Add a `scripts/analytics-report.sh` utility (~20 LoC): calls `GET /api/analytics/summary`, extracts totals and averages, formats a one-paragraph plaintext report, and pipes it to `./notify`. Designed to run as a daily CHORUS cron step: "MiroShark: 3 simulations yesterday, $2.41 total cost, 67% completion rate, avg confidence 71%." Add the script reference to `docs/INSTALL.md` under "Operator Tools".

### 5. GitHub Actions Marketplace Action

**Type:** Integration / Growth
**Effort:** Medium (1-2 days)
**Impact:** Every GitHub repository is a potential MiroShark user. A `miroshark/run-simulation` GitHub Actions action makes MiroShark accessible to the 100M+ developers who already run CI in GitHub — without requiring them to install anything, stand up Neo4j, or read documentation. Use cases: a policy team adds it to trigger on PR open ("simulate stakeholder reaction to this policy draft before it merges"); a startup adds it to their release workflow ("simulate market reaction to this changelog before tagging v1.2.0"); a researcher uses it in a nightly workflow ("run today's simulation batch"). The action's presence in the GitHub Marketplace creates a discovery channel that the README, HN, and social media cannot reach: developers searching the Marketplace for "AI simulation" or "sentiment analysis" CI tools will surface MiroShark. The action also creates a compounding effect: every repo that adds it becomes a referral — "powered by MiroShark" appears in every workflow file of every team that adopts it. With 295 forks, there are 295 teams already interested enough to fork. An action gives them a zero-friction adoption path that doesn't require deploying their own instance.

**How:**
1. Create `action/` directory in the MiroShark repo (or as a separate `miroshark/run-simulation` repo — prefer separate repo for Marketplace listing, noted in PR description). Core file: `action/action.yml`. Metadata: `name: Run MiroShark Simulation`, `description: Simulate public reaction to any scenario using MiroShark's swarm intelligence engine`, `branding: { icon: 'activity', color: 'purple' }` (Marketplace icon). Inputs: `topic` (required, string — the scenario to simulate), `agents` (optional, default `50`), `rounds` (optional, default `5`), `platforms` (optional, default `twitter,reddit`), `miroshark-url` (required — the URL of the operator's MiroShark instance), `api-key` (optional — if the instance requires auth), `timeout-minutes` (optional, default `15`). Outputs: `simulation-id` (the sim ID), `consensus` (BULLISH/BEARISH/NEUTRAL), `confidence-pct` (float), `signal-url` (direct link to signal.json), `simulation-url` (link to the published simulation page). Implementation: `action/entrypoint.sh` (~60 LoC). Calls `miroshark-url/api/simulate` (POST with topic/agents/rounds/platforms), polls `miroshark-url/api/simulation/{id}/signal` every 30s until `status == "published"` or timeout, writes outputs using `echo "consensus=$CONSENSUS" >> $GITHUB_OUTPUT`. Add `action/README.md` (~80 lines): installation snippet (`uses: miroshark/run-simulation@v1`), full example workflow (`.github/workflows/simulate-pr-reaction.yml`), input/output reference table, a note about hosting requirements (the action calls an operator-hosted instance — it does not spawn infrastructure). Add a `Dockerfile` for the action using a minimal Python 3.11 Alpine base with `curl` and `jq` available — the entrypoint is shell, not Python, so the image stays under 50MB.
2. Add `.github/workflows/simulate-pr-reaction.yml.example` to the MiroShark repo (~40 lines). A ready-to-copy example workflow that organizations can drop into their own repos. Trigger: `on: [pull_request]`. Step: "Simulate stakeholder reaction to this PR" using `miroshark/run-simulation@v1` with `topic: "${{ github.event.pull_request.title }}: ${{ github.event.pull_request.body }}"` (uses PR title + first 200 chars of body as the scenario). Post-step: a `create-comment` action that posts the simulation result as a PR comment using the standard `actions/github-script` action: a formatted block with the consensus direction, confidence percentage, and a link to the full simulation. The comment template: `> **MiroShark Simulation Result**\n> Scenario: [PR title]\n> Consensus: **[DIRECTION]** at **[confidence]%** confidence\n> [View full simulation →](simulation-url)`. Add a second example: `.github/workflows/simulate-release.yml.example` — trigger: `on: [release]`, topic is the release notes body, posts the result to the release description.
3. Add Marketplace publishing workflow: `action/.github/workflows/release.yml` that triggers on tag push (format `v*`), runs `actions/checkout`, and publishes to the GitHub Marketplace via the standard marketplace release flow. Add a `MARKETPLACE.md` to the action directory with: category (`Utilities`), keywords (`ai-simulation, sentiment-analysis, swarm-intelligence, public-reaction, simulation`), and a 250-word Marketplace description. In the MiroShark main `README.md`, add a "GitHub Actions" section under Features linking to the action: "Run a simulation from any CI pipeline with `miroshark/run-simulation` → [GitHub Marketplace](https://github.com/marketplace/actions/run-miroshark-simulation)". ## Selection Rationale

All five ideas address the live crises (14-day social silence, token at ATL) while advancing the September hyperstition cluster with 42 days remaining.

- **Show HN Launch Kit (#1)** — The Hacker News post is the highest-leverage single action available to break the 14-day silence. Front-page HN posts in this category generate 300–800 comments, 1,000–3,000 uniques, and 200–400 stars in 24 hours. The kit removes the activation barrier: a maintainer who has the post draft, pre-written responses, and a demo seed ready can post tonight. Without the kit, the post stays in the "I should do this someday" queue indefinitely.

- **Spanish (ES) UI Locale (#2)** — The direct path to 5/5 on the September language hyperstition. ES is the highest-reach language available (500M native speakers, #2 in LATAM GitHub activity), uses the Latin alphabet (no font engineering), and follows the proven dictionary-only pattern from FR. With 42 days to the Sep 1 deadline and the locale already committed as a local branch, this is a 4-hour task that closes a hyperstition.

- **Research Campaign Tracker API (#3)** — The academic citation hyperstition requires researchers to treat MiroShark as a methodology tool, not a demo. Researchers need to group related simulations into named campaigns with aggregate statistics. The campaign tracker is the primitive that converts "I ran 15 simulations" into "here is a reproducible research campaign with a citable URL." Without it, MiroShark is memorable but not citable.

- **Operator Usage Analytics API (#4)** — Enterprise and research deployments require dashboards. An operator who can see "3 simulations yesterday, $2.41 total cost, 71% avg confidence" is an operator who will approve the tool for department-wide use. The analytics endpoint also feeds CHORUS's daily notification loop — turning aeon from a passive maintainer into an informed observer.
