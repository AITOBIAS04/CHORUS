## Ecosystem Context

MiroShark is 48 hours past shipping its 32nd share surface and 33rd API endpoint, and the open PR queue is down to one: `feat/batch-status-lookup`, which gives integrators a single POST to poll up to 20 simulations in one round-trip. That PR closes the last visible gap in the integrator-facing surface area — batch querying is the missing primitive once you have six ecosystem partners running automated workflows against the API.

What hasn't been built yet is the analytics layer that sits one level above the raw surfaces. The platform now exports: per-agent identity and stance arcs (agents.json, #137), per-round consensus trajectories (trajectory.json), confidence scores (confidence.json), coalition maps (coalitions.json), and action logs (actions.jsonl). Each of those surfaces answers a different question about a single simulation. None of them answer questions about patterns *within* a simulation's own internal structure — how sentiment diverges across platforms mid-run, whether confidence built steadily or only crystallized at the end, which agents discourse actually orbits when you read the post text rather than the action metadata.

That's where today's five ideas sit: internal-structure analytics that read the data already being generated and surface it in ways that are immediately useful for researchers, integrators, and operators — without adding new external dependencies or new data collection.

None of the five overlap with the 31 push-blocked builds, the batch-status open PR, the ten ideas from May 28 / May 30, the five from June 2, or the five from June 4.

### 1. Cross-Platform Sentiment Divergence

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Multi-platform simulations (Twitter + Reddit + Polymarket) generate platform-tagged action logs, but the only aggregate output today is a single platform-wide yes_pct. If Twitter agents ended at 72% YES and Reddit agents at 39% YES, the aggregate consensus (~56% YES) obscures a 33-point platform divergence that is itself the most interesting signal in the run. A `GET /api/simulation/:id/platform-sentiment` endpoint scans actions.jsonl, groups stance snapshots by platform tag, and returns per-platform final distributions: `{platforms: [{name, yes_pct, no_pct, neutral_pct, post_count, agent_count}]}` stored as `platform_sentiment.json`. A "Platform Breakdown" panel on the watch page renders these as a horizontal bar per platform — one row per platform, colored by direction. For operators modeling crowd behavior on specific platforms (crypto Polymarket vs. general Twitter vs. niche Reddit), this is the first surface that answers "where did the simulated crowd disagree with itself?" The signal is there in every multi-platform sim; this surface just reads it.

**How:**
1. Add `backend/app/services/platform_sentiment_service.py` (~130 LoC, pure stdlib). `compute_platform_sentiment(sim_dir)` reads `actions.jsonl` (one JSON per line; each line has `round`, `agent`, `platform`, `action_type`, `post_text`, `stance` — take the last recorded `stance` value per agent per platform as their "final" stance on that platform). Group by `platform` field. For each platform group, count agents with `stance == "Bullish"` (yes), `"Bearish"` (no), and `"Neutral"` (neutral). Compute `yes_pct = bullish_count / total * 100` etc. Include `post_count` (total lines for that platform) and `agent_count` (distinct agents). If `stance` field is absent from actions.jsonl (older sim format), fall back to scanning `state.json` per-agent stance and cross-referencing the platform from the most frequent platform in that agent's action lines. Write result to `<sim_dir>/platform_sentiment.json` with a one-time mtime-based cache (only recompute if `actions.jsonl` is newer than `platform_sentiment.json`). `get_platform_sentiment(sim_dir)` reads the cache if fresh. Add `GET /api/simulation/:id/platform-sentiment` (public for published sims, admin-gated for private). Returns the payload; 404 if sim not found; empty `{platforms: []}` if actions.jsonl is absent (single-platform sim or sim with no action data). Add `platform_sentiment` surface entry to `surfaces_catalog.py` (file type `integration`).
2. In `Step3Simulation.vue`, when the simulation's `config.json` lists more than one platform, render a "Platform Breakdown" panel below the main signal summary. Fetch `GET /api/simulation/:id/platform-sentiment` on sim load (parallel with the existing confidence and highlights fetches). Display a compact table: one row per platform — `{platform_name_chip} | {yes_pct}% YES | bar | {no_pct}% NO`. The bar is a split CSS bar (violet for yes, red for no, gray for neutral), width proportional to pct, full-width across the row. Include an `agent_count` sub-label under each platform name ("N agents"). If only one platform is present, hide the panel entirely (single-platform sims have no divergence to show). Add `getPlatformSentiment(simId)` to `frontend/src/api/simulation.js`. In `EmbedDialog.vue`, add a "Platform Sentiment" line in the embed section if `platform_sentiment.json` exists. Add `has_platform_breakdown: bool` to the gallery card payload.
3. Add `GET /api/simulation/:id/platform-sentiment` to the OpenAPI spec under the "Analytics" tag with `PlatformSentimentResponse` and `PlatformSentimentEntry` schemas (`{name: str, yes_pct: float, no_pct: float, neutral_pct: float, post_count: int, agent_count: int}`). Add 10 unit tests in `backend/tests/test_unit_platform_sentiment.py`: two-platform sim produces two entries, single-platform sim returns one entry (not empty), yes_pct + no_pct + neutral_pct sum to ~100 (float rounding tolerance ±0.5), post_count matches total action lines for that platform, agent_count counts distinct agents not total lines, cache hit (platform_sentiment.json newer than actions.jsonl) skips recompute, cache miss (actions.jsonl newer) triggers recompute, missing actions.jsonl returns empty platforms list, 404 for unknown sim_id, admin token required for private sim. In `docs/FEATURES.md`, add a "Cross-Platform Sentiment Divergence" section: defines the divergence signal (max yes_pct delta across platforms), notes the single-platform sim behavior (panel hidden), and explains the cache invalidation rule (actions.jsonl mtime).

### 2. Per-Round Confidence Score Trajectory

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The confidence service computes a single end-of-simulation score (0-100) from four components: stability (trajectory variance), convergence (final round spread), participation (agent count), and adversarial robustness. But that number conceals whether confidence built steadily from round 1 or spiked only in the final two rounds. A simulation that hits confidence 88 by round 4 of 12 is structurally more predictable than one that reaches 88 only at round 11 — the first result is robust to early stopping; the second was a late-round swing that could have gone the other way. `GET /api/simulation/:id/confidence/trajectory` applies the existing `confidence_service.compute_confidence` formula to trajectory slices (rounds 1 through N, extending the trajectory window incrementally) and returns `[{round, confidence_score}]` stored as `confidence_trajectory.json`. On the watch page, a small confidence sparkline is added next to the existing yes_pct timeline — a second thin line showing how trust in the result built over time. No new external data or dependencies: this is the existing formula applied to subsets of already-computed data.

**How:**
1. Add `compute_confidence_trajectory(sim_dir)` to `backend/app/services/confidence_service.py` (~80 new LoC). Reads `trajectory.json` (already read by the existing `compute_confidence`). For each round index `r` from 1 to `len(trajectory)`, call `compute_confidence(sim_dir, trajectory_slice=trajectory[:r])` — add a `trajectory_slice` optional parameter to `compute_confidence` so it accepts a pre-loaded slice rather than re-reading the file. For each slice, compute stability (variance of yes_pct in the slice), convergence (spread in the last round of the slice), participation (unchanged — agent count doesn't change by round), and adversarial (if `adversarial_report.json` exists, treat robustness as full if `pass_rate ≥ 0.7`, else scale linearly — same logic as today, but constant across the trajectory since adversarial runs once at the end). This means adversarial may be unavailable for early trajectory points; use 0 for that component when the adversarial report doesn't exist. Write `{generated_at: ISO, trajectory: [{round: int, confidence_score: float}]}` to `<sim_dir>/confidence_trajectory.json` with a one-time cache (same mtime pattern as `confidence.json`). Add `GET /api/simulation/:id/confidence/trajectory` (public for published sims, admin-gated for private). Returns the payload; 404 if sim not found; `{trajectory: []}` if fewer than 2 trajectory rounds exist.
2. In `Step3Simulation.vue`, in the replay timeline area (wherever the yes_pct sparkline is rendered), fetch `GET /api/simulation/:id/confidence/trajectory` on page load. If the response contains ≥2 data points, render a second sparkline overlaid on the same timeline container — a thin dashed `border-violet-400` line (confidence) vs. the existing solid line (yes_pct). Use separate Y-axes: yes_pct on the left (0-100%, labeled), confidence on the right (0-100, labeled "Confidence"). A legend below the chart labels both lines. When the user scrubs to round N in the replay, show a vertical rule that intersects both lines at round N, with a tooltip: `"Round N — YES: X% · Confidence: Y"`. This tooltip-on-scrub pattern is the same as the existing round-hover behavior — no new mechanism. Add `getConfidenceTrajectory(simId)` to `frontend/src/api/simulation.js`. In `EmbedDialog.vue`, note the confidence trajectory endpoint as a new programmatic export. Add `has_confidence_trajectory: bool` to the gallery card payload (true once the file exists).
3. Add `GET /api/simulation/:id/confidence/trajectory` to the OpenAPI spec under the "Analytics" tag alongside the existing `GET /api/simulation/:id/confidence`. Add `ConfidenceTrajectoryResponse` schema: `{generated_at: str, trajectory: [{round: int, confidence_score: float}]}`. Add 10 unit tests in `backend/tests/test_unit_confidence_trajectory.py`: 5-round sim returns 5 trajectory entries, entry for round 1 has lower confidence than round 5 (for a well-formed converging sim), adversarial component is 0 for all rounds when adversarial_report.json is absent, adversarial component is constant across all rounds when report exists, stability component increases monotonically for a converging sim, cache written on first call, cache read (not recomputed) on second call when trajectory.json is unchanged, empty trajectory returned when fewer than 2 rounds, 404 for unknown sim_id, admin token required for private sim, compute_confidence called with trajectory_slice parameter not re-reading file on each call (mock the file read). In `docs/FEATURES.md`, add a "Per-Round Confidence Trajectory" section: defines what each component contributes at partial trajectory (especially the adversarial component caveat — constant, derived from final-sim adversarial report, unavailable as 0 before the report is generated).

### 3. Agent Mention Network

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The Agent Interaction Graph Export (#137) builds edges from action metadata — which agents replied to, quoted, or reacted to which other agents. But the post text itself contains a different signal: which agents get explicitly named (`@agent_name`) in other agents' messages? An agent can reply to a post without mentioning the author by name, and can mention an agent without directly interacting with them. Mining `@`-patterns from `post_text` fields in actions.jsonl builds a directed mention graph: who is being talked about vs. who is talking. The most-mentioned agent is the agent discourse orbits — often different from the most-active poster. `GET /api/simulation/:id/mentions` returns `{top_mentioned: [{agent, mention_count, mentioned_by_count}], mention_matrix: [{from, to, count}]}` stored as `mentions.json`. A "Mentions" panel on the watch page surfaces the top 5 most-cited agents and a compact matrix. For researchers studying information diffusion ("which agent set the agenda?"), this is a qualitatively different signal from the action graph.

**How:**
1. Add `backend/app/services/mentions_service.py` (~120 LoC, pure stdlib). `compute_mentions(sim_dir)` reads `actions.jsonl` line by line. For each line with a `post_text` field, extract all `@word` patterns using `re.findall(r'@(\w+)', post_text)`. Resolve each extracted name against the known agent roster (read from `agents.json` if present, or from the set of all `agent` field values in actions.jsonl). Only count mentions of known agents (ignore `@twitter`, `@reddit`, etc. — non-agent handles). Record `(source_agent, mentioned_agent)` tuples. After scanning all lines: build `mention_matrix = [{from: source, to: target, count: n}]` for all (source, target) pairs with count ≥ 1, sorted by count desc. Build `top_mentioned = [{agent, mention_count: total times this agent was mentioned, mentioned_by_count: distinct agents who mentioned them}]` sorted by `mention_count` desc, limit 10. Write `{generated_at: ISO, top_mentioned: [...], mention_matrix: [...]}` to `<sim_dir>/mentions.json`. Cache: re-generate only if `actions.jsonl` is newer than `mentions.json`. `get_mentions(sim_dir)` reads the cache. Add `GET /api/simulation/:id/mentions` (public for published sims). Returns the payload; 404 if sim not found; `{top_mentioned: [], mention_matrix: []}` if no `@agent_name` patterns found (sims where agents don't use @ syntax). Add `mention_count: int` to the gallery card payload (total cross-agent mentions in the sim, summed from `mention_matrix`). Add `agent_mentions` to `surfaces_catalog.py`.
2. In `Step3Simulation.vue`, in the sidebar or below the agent list, add a "Mentions" collapsible panel (collapsed by default). Fetch `GET /api/simulation/:id/mentions` on page load. If `top_mentioned` is non-empty, render: a header "Most-Cited Agents" with a count chip (`N cross-agent mentions`). Below: a ranked list (max 5) — `{rank}. @{agent_name} — cited {mention_count}× by {mentioned_by_count} agents`. Below the list: a compact mention matrix table if `mention_matrix` has ≤ 15 entries: columns "From · To · Times". Rows sorted by count desc. For matrices > 15 entries, show only the top 10 rows and a "Show all →" link that expands inline. In the Agent Interaction Graph overlay (if rendered), add a toggle "Show text mentions" that overlays dashed edges for mention pairs alongside the existing solid action edges — distinguishing the two edge types with a legend: "— Action edge · ╌ Mention edge". Add `getAgentMentions(simId)` to `frontend/src/api/simulation.js`.
3. Add `GET /api/simulation/:id/mentions` to the OpenAPI spec under the "Analytics" tag with `MentionsResponse` schema: `{top_mentioned: [{agent: str, mention_count: int, mentioned_by_count: int}], mention_matrix: [{from: str, to: str, count: int}]}`. Add `mention_count: int` to the GalleryCard schema. Add 10 unit tests in `backend/tests/test_unit_mentions.py`: single @mention in one post produces one matrix entry, multiple @mentions in one post each counted separately, self-mention (agent mentions themselves) counted in matrix but not in top_mentioned (or included — decide: exclude self-loops as they're noise; add a test that enforces the decision), unknown @handle (not in agent roster) excluded, mention_count in gallery card equals sum of all mention_matrix counts, top_mentioned sorted by mention_count desc, top_mentioned capped at 10 entries, mention_matrix entries with count ≥ 1 only, cache written on first call, cache read on second call when actions.jsonl is unchanged. In `docs/FEATURES.md`, add an "Agent Mention Network" section: distinguishes the mention graph (text `@name` patterns) from the interaction graph (action-level reply/quote edges), explains the agent-roster filter (only known agent names count), and notes the self-loop policy (excluded from top_mentioned, included in matrix for completeness).

### 4. Simulation Narrative Export (Markdown)

**Type:** DX / Feature
**Effort:** Small (hours)
**Impact:** Researchers who want to share a simulation's findings in Notion, Obsidian, GitHub wikis, Slack, or a research note today have to manually copy-paste stats from three or four separate endpoints. `GET /api/simulation/:id/export.md` generates a single Markdown document from existing cached data: scenario title, final signal (direction + confidence score), trajectory table (round | yes% | no% | neutral%), top 5 agents by influence, coalition summary, quality assessment, and a footer with the source URL and generated date. Pure string formatting — no external API call, no AI generation, no new dependencies. The endpoint is also added to the archive bundle. For ecosystem partners building dashboards or reports on top of MiroShark, this is the "human-readable summary" primitive that fills the gap between machine-readable JSON exports and a polished report. Distinct from Jupyter Notebook Export (data scientist workflow, .ipynb), Simulation Clone JSON (forking, raw config), and Per-Agent Trajectory Export (agent-level raw data, CSV/JSON).

**How:**
1. Add `backend/app/services/narrative_export_service.py` (~160 LoC, pure stdlib). `generate_markdown(sim_dir, sim_id, base_url)` reads (with graceful fallbacks for missing files): `simulation_config.json` → title/scenario text; `signal.json` → direction (YES/NO/NEUTRAL), yes_pct, no_pct, neutral_pct; `confidence.json` → confidence_score (or `"N/A"` if absent); `trajectory.json` → `[{round, yes_pct, no_pct, neutral_pct}]` for the table; `agents.json` → top 5 agents sorted by `influence_score` desc (or by `final_stance` if influence absent); `coalitions.json` → coalition count, largest coalition size and label; `quality.json` → quality_health field (`"excellent"/"good"/"fair"/"poor"` or `"N/A"`). Assembles a Markdown string using f-string templates:

   ```markdown
   # {title}

   **Scenario:** {scenario_description}
   **Run date:** {completed_at}
   **Source:** {base_url}/simulation/{sim_id}

   ---

   ## Signal

   | Outcome | Share |
   |---------|-------|
   | YES (Bullish) | {yes_pct:.1f}% |
   | NO (Bearish) | {no_pct:.1f}% |
   | NEUTRAL | {neutral_pct:.1f}% |

   **Direction:** {direction} · **Confidence:** {confidence_score}/100 · **Quality:** {quality_health}

   ---

   ## Round-by-Round Trajectory

   | Round | YES% | NO% | NEUTRAL% |
   |-------|------|-----|---------|
   {trajectory_rows}

   ---

   ## Top Agents by Influence

   {agent_list}

   ---

   ## Coalitions

   {coalition_summary}

   ---

   *Generated by MiroShark · {generated_at}*
   ```

   No caching — generate fresh on each request (the source files are already cached). Add `GET /api/simulation/:id/export.md` (public for published sims, admin-gated for private). Returns `Content-Type: text/markdown; charset=utf-8` with `Content-Disposition: attachment; filename="{sim_id}.md"`. 404 if sim not found. Also add this file to the archive bundle zip (already generated by another endpoint) as `narrative.md`. Add `export_markdown` to `surfaces_catalog.py`.
2. In `EmbedDialog.vue`, in the "Export" section (alongside the existing JSON/GraphML/Jupyter download links), add a "Narrative (Markdown)" row: a download link for `GET /api/simulation/:id/export.md` with a tooltip "Human-readable summary — paste into Notion, Obsidian, or any Markdown editor." In `Step3Simulation.vue`, in the sim action toolbar (wherever the share/download buttons live), add a "Copy Markdown" button (clipboard icon) that fetches `GET /api/simulation/:id/export.md` and writes the response text to `navigator.clipboard.writeText()`. A brief "Copied!" toast confirms success. In `frontend/src/api/simulation.js`, add `getMarkdownExport(simId)` that fetches the endpoint and returns the response text (not JSON). In the archive bundle generation (wherever the zip is assembled), include `narrative.md`.
3. Add `GET /api/simulation/:id/export.md` to the OpenAPI spec under the "Export" tag (alongside archive and network endpoints) with a `text/markdown` response type and a note that the response includes all available data fields with graceful `N/A` fallbacks for missing files. Add 10 unit tests in `backend/tests/test_unit_narrative_export.py`: full data produces all sections non-empty, missing coalitions.json renders coalition section as "No coalition data available", missing agents.json renders top agents as "No agent data available", missing confidence.json renders confidence as "N/A", trajectory table has correct number of rows matching trajectory.json length, title comes from simulation_config.json title field, scenario_description truncated at 500 chars with "…" if longer, response Content-Type is text/markdown, Content-Disposition has sim_id in filename, admin token required for private sim. In `docs/FEATURES.md`, add a "Simulation Narrative Export" section: lists all source files read and their fallback behavior, notes the "no caching" design (fresh on each call since sources are cached), distinguishes from other export formats, and includes a one-liner curl example.

### 5. Operator Usage Analytics

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The Operator Dashboard shows a sim inventory. The Platform Stats endpoint (`/api/stats`) shows global platform-wide aggregates. Neither tells an operator "across everything you've run, you've simulated 14,200 agents across 892 rounds in 47 simulations — and your most-used platform is Twitter." `GET /api/operator/analytics` (admin-gated) aggregates across all sims owned by the deployment: `{total_sims, total_published, total_agents_simulated, total_rounds_run, most_used_platform, most_used_archetype, avg_confidence_score, busiest_day: {date, count}}`. Scans `state.json` (total_rounds), `agents.json` (agent count, platform and archetype frequency), and `confidence.json` (score) per sim dir. Cached to `operator_analytics.json` with 30-minute TTL (same pattern as activity_service). A new "Usage" stat block in the Operator Dashboard renders this as a single compact fact row at the top — "47 sims · 14,200 agents · 892 rounds · avg confidence 71 · peak: Mon Jun 2". Operators sharing their MiroShark deployment with a research team can screenshot this row as a usage summary. It's the "how much have we actually used this?" signal that makes simulation programs legible.

**How:**
1. Add `backend/app/services/operator_analytics_service.py` (~140 LoC, pure stdlib). `compute_operator_analytics(data_dir)` scans all sim dirs (same scandir pattern as `leaderboard_service` and `health_service`). For each sim dir: read `state.json` for `current_round` (total rounds run); read `agents.json` for agent count (len of agent array) and for each agent's `platform` (accumulate platform frequency) and `archetype` (accumulate archetype frequency); read `confidence.json` for `confidence_score` (accumulate for avg); read `config.json` for `published_at` (count published sims), `created_at` (bucket by date for busiest_day). After scanning all sims, compute: `total_sims` (all dirs), `total_published` (dirs with `published_at`), `total_agents_simulated` (sum of agent counts), `total_rounds_run` (sum of `current_round`), `most_used_platform` (mode of all platform values), `most_used_archetype` (mode of all archetype values), `avg_confidence_score` (mean of all non-null confidence scores, null if none), `busiest_day` ({date: ISO date string, count: int} for the date with most sim `created_at` values). Write result to `<data_dir>/operator_analytics.json` with 30-minute TTL. `get_operator_analytics(data_dir)` reads cache if fresh, else calls `compute_operator_analytics`. Add `GET /api/operator/analytics` to `operator.py` blueprint (admin-gated via `require_admin_token`, same as existing operator routes). Returns the payload; 200 always (empty/zeroed values if no sims exist). Add `last_updated: ISO` to the response for cache transparency.
2. In `OperatorDashboardView.vue`, at the top of the dashboard (above the 4 existing stat cards), add a "Usage Overview" section. On mount, fetch `GET /api/operator/analytics` alongside the existing sim list fetch. Render a single-line "fact strip" component: `{total_sims} sims · {total_agents_simulated.toLocaleString()} agents · {total_rounds_run.toLocaleString()} rounds · avg confidence {avg_confidence_score?.toFixed(0) ?? 'N/A'} · peak: {busiest_day?.date ? formatDate(busiest_day.date) : 'N/A'}`. Below the strip, two small chips: `Most-used platform: {most_used_platform ?? 'N/A'}` and `Favorite archetype: {most_used_archetype ?? 'N/A'}`. The section is hidden if `total_sims === 0`. If the analytics fetch fails (e.g., admin token expired), the section shows a single `text-gray-500` line "Usage analytics unavailable" — no layout disruption. Add `getOperatorAnalytics()` to `frontend/src/api/simulation.js` (sends the admin token from localStorage, same pattern as other admin calls). In the Operator Dashboard's mobile layout (single-column), the fact strip wraps to two lines with the same content — no truncation.
3. Add `GET /api/operator/analytics` to the OpenAPI spec under the "Operator" tag (alongside existing operator routes) with `OperatorAnalyticsResponse` schema: `{total_sims: int, total_published: int, total_agents_simulated: int, total_rounds_run: int, most_used_platform: str|null, most_used_archetype: str|null, avg_confidence_score: float|null, busiest_day: {date: str, count: int}|null, last_updated: str}`. Add 10 unit tests in `backend/tests/test_unit_operator_analytics.py`: empty data_dir returns all zeros, single sim counted correctly, total_agents_simulated is sum across all sims not max, most_used_platform is mode (not max), avg_confidence_score is mean (not latest), avg_confidence_score is null when no confidence.json files exist, busiest_day identifies correct date, cache written on first call, stale cache (>30min) triggers recompute, admin token required (401 without token). In `docs/FEATURES.md`, add an "Operator Usage Analytics" section: distinguishes this from Platform Stats (global, unauthenticated) and the Operator Dashboard sim list (per-sim inventory), notes the 30-minute cache TTL and the `last_updated` field for cache transparency.

## Selection Rationale

Five angles distinct from all 32 push-blocked local builds, the batch-status open PR, the five ideas from June 2, the five from June 4 (including Agent Archetype Atlas which landed June 6), and the ten from May 28 / May 30:

- **Cross-Platform Sentiment Divergence** (#1) — platform-specific sentiment breakdown within a single simulation. The data is already there (actions.jsonl has platform tags on every action); no surface currently reads it at the cross-platform aggregate level. The signal — "Twitter went 72% YES, Reddit went 39% NO" — is qualitatively different from the overall consensus and is exactly what operators designing multi-platform simulations need to validate platform-specific crowd behavior.

- **Per-Round Confidence Trajectory** (#2) — applies the existing confidence formula to incremental trajectory slices. The confidence service is already written; this is the temporal extension of it. A single confidence score at the end is like a single grade on a paper — it tells you the outcome but not how the result evolved. The trajectory answers whether the simulation was decisive from round 3 or only converged in the final two rounds, which is critical for operators who want to know if early stopping would have produced the same answer.

- **Agent Mention Network** (#3) — mines `@agent_name` patterns from post text in actions.jsonl. The interaction graph (already built and blocked) captures action-level edges (replies, quotes, reactions). Text mentions are a different signal: which agents get named in discourse even when they're not being directly interacted with? The most-cited agent is often not the most-active poster. This surface answers "who set the agenda?" — distinct from "who was most active?"

- **Simulation Narrative Export (Markdown)** (#4) — synthesizes signal.json, trajectory.json, agents.json, coalitions.json, and confidence.json into a single human-readable Markdown document. Fills the gap between machine-readable exports (JSON, GraphML, Jupyter) and a polished report. Specifically designed for paste-into-Notion / paste-into-Obsidian use cases that researchers have — not for data pipelines. Zero new external dependencies.

- **Operator Usage Analytics** (#5) — aggregates total sims, agents, rounds, platforms, and archetypes across the entire deployment for the admin. The Operator Dashboard has per-sim detail; Platform Stats has global aggregates (unauthenticated); neither gives an operator a personal usage summary. "You've simulated 14,200 agents across 47 sims" is the kind of number that makes a tool feel powerful and worth recommending. Same scan pattern as leaderboard and health services — 30-minute cached, pure stdlib.
