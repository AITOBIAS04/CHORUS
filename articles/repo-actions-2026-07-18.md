### 1. Simulation OG Image API

**Type:** Feature / Growth
**Effort:** Small (hours)
**Impact:** Every shared `https://miroshark.io/simulation/{id}` link currently unfurls blank on Twitter, LinkedIn, Discord, and Slack — no preview, no visual hook, no reason to click. A `GET /api/simulation/{id}/og-image` endpoint returning a PNG card (topic title, consensus outcome with directional arrow, confidence percentage, agent count, round count, subtle MiroShark wordmark) changes that calculus. When a researcher pastes a simulation link into their Discord server or a founder drops one in a Slack thread, the unfurl becomes the content — the simulation result is visible before anyone clicks. With 291 forks and 1,375 stars of potential ambassadors, the share rate is gated entirely on friction: if the link looks like something worth sharing, it gets shared. An OG image removes that friction. This is the highest-leverage thing MiroShark can do for the 12-day social silence without needing a marketing budget or external platform.

**How:**
1. Add `backend/app/services/og_image_service.py` (~120 LoC, pure stdlib). Function `generate_og_image(sim_id) -> bytes`. Reads `data/simulations/{id}/signal.json` (consensus direction, final_bullish_pct, confidence_pct, topic) and `data/simulations/{id}/config.json` (agent_count, rounds, platforms). Generates an SVG string using Python's string interpolation (no third-party library — pure stdlib template). SVG layout: 1200×630px (canonical OG dimensions), dark background (`#0f0f13`), topic title in large font (truncate at 60 chars with ellipsis), consensus outcome in a colored chip (green for BULLISH, red for BEARISH, gray for NEUTRAL), confidence percentage in large bold numerals (e.g. "73%"), agent count and round count in a subtitle row, MiroShark wordmark in bottom-right corner. Convert SVG to PNG via `cairosvg` (already present in most Python ML environments) with a fallback: if `cairosvg` is not installed, return the SVG directly with `Content-Type: image/svg+xml` — SVG unfurls are supported by most platforms since 2024. Cache the output to `.og-cache/{sim_id}.png` with a 24h mtime TTL. Add 5 unit tests: correct dimensions in SVG output, BULLISH uses green fill, BEARISH uses red fill, topic truncated at 60 chars, 404 for unpublished or missing simulation.
2. Add `GET /api/simulation/{id}/og-image` to `backend/app/api/simulation.py` (~30 LoC). Publish-gated. Returns PNG (or SVG fallback) with `Cache-Control: public, max-age=86400`. Add to OpenAPI spec under a "Sharing" tag. In the existing simulation HTML template (`backend/app/templates/simulation.html` or the equivalent FastAPI/Jinja2 route that serves the SPA), inject into `<head>`: `<meta property="og:image" content="/api/simulation/{id}/og-image" />`, `<meta property="og:title" content="MiroShark Simulation: {topic}" />`, `<meta property="og:description" content="Consensus: {direction} at {confidence}% confidence. {agent_count} agents, {rounds} rounds." />`, `<meta name="twitter:card" content="summary_large_image" />`. These tags are served server-side so crawlers see them without executing JS. Add a `GET /api/simulation/{id}/og-image` entry to `docs/API.md` under Sharing.
3. Add a "Share" button to `frontend/src/views/Step3Simulation.vue` (or the equivalent results view) — next to the existing toolbar buttons. Clicking opens a small dropdown: "Copy link" (copies `window.location.href` to clipboard), "Copy for Twitter" (copies a pre-formatted string: `Just ran a MiroShark simulation on "{topic}" — consensus: {direction} at {confidence}% confidence. https://miroshark.io/simulation/{id}`), "Copy OG preview URL" (copies the direct `/api/simulation/{id}/og-image` URL for embedding in blog posts). Add a 1-line OG image preview thumbnail in this dropdown (fetches the `/og-image` URL as an `<img>` tag at 300×157px, the inline preview size). The share button is the user-facing completion of the loop: simulation runs → result is visible → share button appears → unfurl on social shows the result. Without this button, most users don't know the link is shareable.

### 2. Air-Gapped HuggingFace Cache

**Type:** DX / Feature
**Effort:** Small (hours)
**Impact:** Issue #240 has been open 12 days (Jul 6 – Jul 18). It requests support for offline/air-gapped environments — the single most common blocker for enterprise and research institution deployments where outbound network access from compute nodes is restricted. MiroShark uses HuggingFace models for NER via the Gemini 3 Flash configuration, and when the runner has no internet, model downloads fail silently mid-simulation. A `scripts/prefetch-models.sh` script — following the same pattern as the existing `scripts/prefetch-*.sh` conventions in CHORUS — pre-downloads all required models to a `.hf-cache/` directory before Claude starts. The backend reads `HUGGING_FACE_HUB_OFFLINE=1` (HuggingFace's native env var) and `HF_HOME=./.hf-cache` to load exclusively from disk. Closing this issue demonstrates that MiroShark responds to contributor requests, which is the most visible trust signal for new potential contributors scanning the issue tracker.

**How:**
1. Add `scripts/prefetch-models.sh` (~40 LoC). Reads the list of required HuggingFace model IDs from `backend/app/config.py` or a new `backend/app/models.txt` file (one model ID per line — e.g., `google/flan-t5-base` or whatever NER model is currently in use). For each model ID: runs `python -c "from transformers import AutoTokenizer, AutoModel; AutoTokenizer.from_pretrained('{model_id}', cache_dir='.hf-cache'); AutoModel.from_pretrained('{model_id}', cache_dir='.hf-cache')"` to pre-download. The script is idempotent: checks if the model exists in `.hf-cache/` before downloading (uses `transformers-cli` if available, otherwise checks for the `config.json` sentinel file in the HF cache directory structure). Outputs progress to stdout: `Prefetching {model_id}...` / `Already cached: {model_id}`. Add `.hf-cache/` to `.gitignore` (already likely there — verify). Document in `docs/INSTALL.md` under a new "Offline / Air-Gapped Setup" section: "Run `./scripts/prefetch-models.sh` once on a networked machine, copy `.hf-cache/` to the target host, set `HF_HOME=$(pwd)/.hf-cache HUGGING_FACE_HUB_OFFLINE=1` in your environment."
2. Update `backend/app/config.py` (or equivalent settings module) to read `HF_HOME` and `HUGGING_FACE_HUB_OFFLINE` env vars and pass them to the `transformers` library at import time. Add a startup check: if `HUGGING_FACE_HUB_OFFLINE=1` is set and the `HF_HOME` path doesn't exist or the expected model cache subdirectory is missing, log a clear warning at startup: `[WARNING] HUGGING_FACE_HUB_OFFLINE=1 but .hf-cache/ not found. Run scripts/prefetch-models.sh first.` — rather than failing silently mid-simulation. Add a `GET /api/health/models` endpoint (~20 LoC) that returns: `{ "offline_mode": bool, "cached_models": [{ "model_id": str, "cached": bool }] }` — gives operators a quick diagnostic without needing shell access. Add 3 unit tests: startup warning fires when offline mode set and cache missing, health endpoint returns correct cached status per model, offline mode skips network calls in the NER pipeline.
3. Update `docs/INSTALL.md` to add an "Offline / Air-Gapped Setup" section. Update `README.md` to mention offline support under the Features section (one bullet: "Offline mode: pre-cache HuggingFace models for air-gapped environments"). Close issue #240 in the PR description with "Closes #240". Add `HUGGING_FACE_HUB_OFFLINE` and `HF_HOME` to the environment variable table in `docs/INSTALL.md` alongside the existing `OPENROUTER_API_KEY`, `ANTHROPIC_API_KEY`, etc. entries. Reference the `GET /api/health/models` endpoint in `docs/API.md` under a "Health" section alongside any existing health endpoints.

### 3. Python SDK (miroshark-py)

**Type:** Integration / DX
**Effort:** Medium (1-2 days)
**Impact:** The academic citation hyperstition requires researchers to use MiroShark for published work. Researchers don't call REST APIs from R Markdown or Jupyter notebooks manually — they import libraries. A `pip install miroshark` SDK with typed Pydantic models and a clean async interface removes the last friction between "I want to run 20 simulations with varying agent counts" and actually doing it. The SDK makes MiroShark feel like a first-class research tool — on par with `scipy`, `arviz`, `pymc` — rather than a web app you click through. It's also the artifact that turns the 291 forks from passive stars into active builders: a Python SDK is something a data scientist forks to extend, contributing back with custom authentication schemes, batch utilities, or integration adapters. The SDK ships as a standalone package in a new `sdk/python/` directory in the MiroShark repo, with a PyPI publish action that triggers on releases.

**How:**
1. Create `sdk/python/` with `pyproject.toml` (package name `miroshark`, version `0.1.0`, Python ≥3.10, dependencies: `httpx>=0.27`, `pydantic>=2.0`). Core module `miroshark/__init__.py` exports `MiroSharkClient`. `MiroSharkClient.__init__(base_url: str = "https://miroshark.io", api_key: str | None = None, timeout: float = 120.0)`. Methods: `simulate(topic: str, agents: int, rounds: int, platforms: list[str]) -> SimulationResult` (polls `/api/simulation/{id}/signal` until status == "published" with exponential backoff, max 5min), `get_signal(sim_id: str) -> SignalResult`, `get_confidence_trajectory(sim_id: str) -> ConfidenceTrajectory`, `compare(id_a: str, id_b: str) -> ComparisonResult`, `search(query: str, limit: int = 10) -> list[SearchResult]`, `list_published(limit: int = 20) -> list[SimulationSummary]`. All methods are sync by default (using `httpx` sync client) with async equivalents via `AsyncMiroSharkClient`. All response types are Pydantic v2 models matching the existing OpenAPI schema — zero parsing logic in the SDK, just model validation. Add `miroshark/exceptions.py` with `MiroSharkError`, `SimulationNotFoundError`, `UnauthorizedError`, `TimeoutError`. Add 12 unit tests using `httpx.MockTransport`: simulate returns `SimulationResult` on success, raises `TimeoutError` after max retries, `get_signal` parses signal.json fields correctly, `compare` maps delta fields correctly, `search` returns typed list, auth header sent when api_key provided, base_url trailing slash handled, all exception types raised for correct HTTP status codes.
2. Add `sdk/python/README.md` (~60 lines): quickstart with `pip install miroshark`, a 10-line example script (create client, run a simulation, print consensus), a reference table of all client methods with their return types. Add `sdk/python/examples/batch_compare.py` (~40 lines): a fully working example that runs 3 simulations on the same topic with different agent counts (10, 25, 50) and compares consensus pct across configurations — the exact workflow a researcher would use for a methods section. Add `sdk/python/examples/jupyter_demo.ipynb`: a minimal notebook (5 cells) that imports the SDK, runs one simulation, prints the result, and plots the confidence trajectory using matplotlib (pure stdlib-compatible plot, no seaborn). The notebook is what GitHub previews — it appears in the repo browser as a ready-to-run demonstration.
3. Add `.github/workflows/publish-sdk.yml` (requires `workflows` scope — note in PR description that this file needs manual merge after GH_GLOBAL unblocks, since PAT lacks `workflows` scope per CHORUS memory). The workflow: trigger on `push` to tags matching `sdk/python/v*`, runs `pip install build twine`, `python -m build`, `twine upload dist/*` with `TWINE_USERNAME=__token__ TWINE_PASSWORD=${{ secrets.PYPI_TOKEN }}`. Also add a `test-sdk` step in `tests/` or `sdk/python/tests/` that runs the mock transport tests above on every PR targeting `main`. Add `PYPI_TOKEN` to the secret list in `docs/INSTALL.md` under "Maintainer Setup". Mention the SDK in `README.md` under a new "SDKs & Integrations" section: "Python SDK: `pip install miroshark` ([PyPI](https://pypi.org/project/miroshark)) · [Quickstart →](sdk/python/README.md)".

### 4. i18n Contribution Kit

**Type:** Community / DX
**Effort:** Small (hours)
**Impact:** The "5 languages by September 1" hyperstition is at 4/5. Four languages (KO, JA, PT, ES) are built locally but blocked by GH_GLOBAL. The September deadline is 44 days away. The kit is a hedge: if GH_GLOBAL unblocks but only one slot merges in time, the kit activates community contributors who speak those languages to carry the rest. The core problem is that adding a language feels opaque — a Korean developer who wants to contribute needs to understand the full `locales/` architecture, find all 1,984 `tr()` call strings, and figure out which ones are missing in an existing locale file. An audit script reduces that to: clone, run one command, open one file, fill in the blanks. The CONTRIBUTING-i18n guide makes the process explicit. Together these two artifacts drop the contribution barrier from "requires full codebase understanding" to "requires 30 minutes and fluency in the target language." That's the unlock that gets the 5th language from a community member rather than only from the agent.

**How:**
1. Add `scripts/i18n-audit.py` (~80 LoC, pure stdlib). CLI usage: `python scripts/i18n-audit.py --target ko` (or `--target fr`, `--target de`, etc.). Reads `frontend/src/locales/en.js` as the authoritative source of truth using a simple regex to extract all `key: "value"` pairs into a dict. Reads `frontend/src/locales/{target}.js` (same extraction). Computes: `missing_keys` (keys in EN not in target), `extra_keys` (keys in target not in EN — stale translations), `empty_values` (keys present in target with empty string value), `coverage_pct` (len(target_keys) / len(en_keys) * 100). Outputs a Markdown report to stdout: coverage percentage, three sections (Missing, Extra/Stale, Empty), each as a bulleted list of key names with the EN value shown in brackets for context. Also writes a `frontend/src/locales/{target}.missing.js` file containing only the missing keys with empty string values — a contributor can open this file, fill in the translations, then copy the filled keys into the main locale file. The script is safe to run repeatedly (idempotent output). Add a shebang line and `chmod +x` instruction in the docs.
2. Add `docs/CONTRIBUTING-i18n.md` (~80 lines). Sections: **Overview** (dictionary-based approach, zero component changes, link to `locales/en.js`), **Prerequisites** (Node.js, Python 3.10+ for the audit script — both already required for dev setup), **Adding a new language** (step-by-step: 1. copy `locales/en.js` to `locales/{code}.js`, 2. run `python scripts/i18n-audit.py --target {code}`, 3. fill in `locales/{code}.missing.js`, 4. copy filled keys into main locale file, 5. update `i18n.js` to import and register the new locale, 6. update `LocaleToggle.vue` to add the locale to the cycle, 7. add a `README.{code}.md` stub), **Improving an existing language** (run the audit, fill in missing keys, open a PR), **Register and tone guidance** (formal register for developer tool context — reference DE as the formal register model, FR as the `vous` model; note that Hangul technical vocabulary conventions are documented in the KO locale file header). Link to `CONTRIBUTING.md` for general PR flow. This doc is the missing bridge between "I want to help with Korean" and "here is exactly what to do."
3. Update `CONTRIBUTING.md` to add an "Internationalization" section that links to `docs/CONTRIBUTING-i18n.md` with one sentence: "To add or improve a language, see the i18n contribution guide — it's the fastest way to make your first PR." Update `README.md` to mention the audit script under the Internationalization feature bullet: "Run `python scripts/i18n-audit.py --target {code}` to check coverage for any locale." File a new GitHub issue (in the PR description, suggest the maintainer open it manually since bot can't open issues without GH_GLOBAL): "Good first issue: Complete Korean (KO) translations — run `python scripts/i18n-audit.py --target ko` to see what's missing. ~200 strings remaining." Label it `good first issue` + `i18n`. A pinned good-first-issue for each language in the queue (KO, JA, PT, ES) gives the 291 forkers a concrete, scoped, low-code way to contribute — which is the fastest path to 10 community PRs.

### 5. Simulation RSS Feed

**Type:** Feature / Integration
**Effort:** Small (hours)
**Impact:** Researchers who want to track MiroShark's published simulation stream have no passive discovery mechanism — they must manually visit the site or poll the API. RSS is the canonical subscription protocol for academic and technical content: arXiv uses it, conference proceeding aggregators use it, and Slack/Discord integrations like RSSBot and MonitoRSS accept RSS URLs to post new items to channels. A `GET /api/simulations/rss` endpoint serving a valid Atom feed (one entry per published simulation: title = topic, updated = published_at, id = canonical URL, summary = consensus outcome + confidence + agent count) enables three distribution channels simultaneously: (1) researchers subscribe in Feedly or Inoreader and see new simulations alongside their paper feeds; (2) a MiroShark Discord server (or any partner's server) subscribes via RSSBot and auto-posts new simulation announcements; (3) a GitHub Actions job in the CHORUS aeon repo can poll the feed and fire a Telegram notification when a high-confidence simulation publishes. The feed is the passive presence MiroShark lacks in the 12-day social silence — it turns the simulation stream into a subscribable signal.

**How:**
1. Add `backend/app/services/rss_service.py` (~90 LoC, pure stdlib). Function `generate_rss_feed(limit: int = 20) -> str`. Scans `data/simulations/` for published simulations (same publish-gate check as all other surfaces). For each, reads `signal.json` (topic, final_bullish_pct, confidence_pct, direction, published_at) and `config.json` (agent_count, rounds). Sorts by `published_at` descending. Generates a valid Atom 1.0 XML string using Python string templates (no `feedgen` dependency — pure stdlib `xml.etree.ElementTree`). Feed metadata: `<title>MiroShark — Published Simulations</title>`, `<link rel="self" href="{base_url}/api/simulations/rss" />`, `<link href="{base_url}" />`, `<id>urn:miroshark:simulations</id>`, `<updated>` = most recent simulation's `published_at`. Each entry: `<title>{topic}</title>`, `<id>urn:miroshark:simulation:{sim_id}</id>`, `<link href="{base_url}/simulation/{sim_id}" />`, `<updated>{published_at}</updated>`, `<summary>Consensus: {DIRECTION} at {confidence}% confidence. {agent_count} agents, {rounds} rounds across {platform_count} platforms.</summary>`, `<category term="{direction}" />`. The feed URL uses `CANONICAL_URL` env var (defaulting to `https://miroshark.io`) so operators don't expose localhost URLs. Cache to `.rss-cache/feed.xml` with a 15-min mtime TTL. Add 6 unit tests: feed validates as parseable XML, entries sorted by published_at descending, unpublished simulations excluded, CANONICAL_URL used in entry links, entry summary includes correct consensus direction, feed updates TTL honored.
2. Add `GET /api/simulations/rss` to `backend/app/api/simulation.py` (~15 LoC). Returns the Atom XML string with `Content-Type: application/atom+xml; charset=utf-8` and `Cache-Control: public, max-age=900`. Also add `GET /api/simulations/feed.json` as a JSON Feed v1.1 endpoint (same data, `Content-Type: application/feed+json`) — JSON Feed is the modern alternative accepted by Micro.blog, NetNewsWire, and some Discord bots. Add both to the OpenAPI spec under a "Discovery" tag alongside the existing search endpoint. Add a `<link rel="alternate" type="application/atom+xml" title="MiroShark Simulations" href="/api/simulations/rss" />` tag to the SPA's `index.html` — browsers that detect this tag show a subscribe icon in the address bar (Firefox, Safari) and feed readers auto-discover the feed when given the root URL.
3. Update `docs/API.md` to add an RSS/Atom Feed entry under the Discovery section. Add a "Subscribe" link to the MiroShark homepage (`HomeView.vue`) — an RSS icon in the footer or header that links to `/api/simulations/rss`. In `README.md`, add one bullet under Features: "Simulation RSS feed: `GET /api/simulations/rss` — subscribe in any feed reader or pipe into Discord via RSSBot." Add a `scripts/poll-rss.sh` utility (~20 LoC) that fetches the feed, extracts the most recent simulation's URL and summary, and pipes it to `./notify` — designed to be added to the CHORUS `messages.yml` scheduler as a daily notification: "New MiroShark simulation published: {topic} — Consensus: {direction} at {confidence}%." This closes the loop: a new simulation is published → CHORUS detects it via RSS → aeon sends a Telegram notification → the operator is aware of real-time MiroShark activity without manually monitoring the site.

## Selection Rationale

All five ideas address the two live crises: 12 days of social silence and token at all-time low FDV, while moving the September hyperstition cluster forward.

- **OG Image API (#1)** — Every shared simulation link is currently a blank unfurl. With 291 forks and 1,375 stars of potential ambassadors, the share rate is entirely gated on friction. A visual preview card converts passive link-sharers into active distributors. This is the single highest-leverage anti-silence move that requires no external coordination and no social capital — it works every time someone pastes a link.

- **Air-Gapped HuggingFace Cache (#2)** — Issue #240 is 12 days old. Closing it demonstrates responsiveness and unblocks enterprise and research institution deployments — the exact audience most likely to produce an academic citation. A `GET /api/health/models` diagnostic endpoint also improves operator confidence in production deployments generally.

- **Python SDK miroshark-py (#3)** — Academic adoption requires a library, not a web app. The SDK is what converts a 291-fork base of Python developers from passive onlookers into active users who can run batch simulations in Jupyter notebooks. One batch study in a notebook → one paper citation → academic citation hyperstition closed.

- **i18n Contribution Kit (#4)** — The 5th language for the September hyperstition can come from the community if the audit script and guide exist. Four language commits are already blocked in local queue (KO, JA, PT, ES); the kit activates contributors for whichever language isn't built yet. It also produces 1–4 "good first issue" tickets that directly address the 10 community PRs hyperstition at 5+/10.

- **Simulation RSS Feed (#5)** — Passive discovery infrastructure that works without any active effort. Once live, the feed propagates across feed readers, Discord RSS bots, and aeon's own notification loop. A daily "new simulation published" notification from aeon to Telegram (via `scripts/poll-rss.sh`) turns the operator into an informed distributor rather than a passive maintainer.
