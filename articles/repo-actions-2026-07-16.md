### 2. Simulation Narrative Export

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** The academic citation and public case study hyperstitions both have the same blocker: researchers have no citation-ready output from a MiroShark simulation. The raw API surfaces (signed-result.json, consensus trajectory, stance-flips) are structured for developers, not for researchers who need methodology sections and BibTeX entries. A `GET /api/simulation/{id}/export/narrative` endpoint produces a structured Markdown report — title, date, topic, configuration table (agent count, rounds, platforms, model lineup), executive summary (consensus outcome, confidence level), a top-5 agent quote appendix (post text, agent demographics, platform, stance at time of post), a methodology paragraph (written in academic passive voice, suitable for a paper's footnote), and a BibTeX block pre-filled with the simulation ID, date, and MiroShark citation. A researcher who runs a simulation on "Will the EU AI Act reduce foundation model investment?" can export this file, paste the BibTeX into Zotero, and reference MiroShark in a methods section without writing a single line. This is the lowest-effort, highest-leverage path to the academic citation hyperstition — it takes an existing capability and adds the packaging researchers actually need.

**How:**
1. Add `backend/app/services/narrative_service.py` (~180 LoC, pure stdlib). Key function: `generate_narrative(sim_id) -> str`. Reads: `data/simulations/{id}/signal.json` (consensus outcome, confidence, direction), `data/simulations/{id}/trajectory.json` (round-by-round belief evolution — top 5 most influential agents by influence_score), `data/simulations/{id}/stance_flips.json` (if present — most dramatic stance shifts), per-platform JSONL action logs (sample 5–10 posts from high-influence agents). Outputs a Markdown string structured as: `# MiroShark Simulation Report` header, metadata table (sim ID, topic, date, config), `## Executive Summary` (2-3 sentences: what the consensus was, confidence level, how it evolved), `## Key Agent Perspectives` (5 quotes from top agents, each with: agent demographics in brackets, platform tag, round number, actual post text verbatim), `## Methodology` (3-4 sentences academic passive: "A swarm of N autonomous agents was instantiated…", model lineup, round structure, publish-gated access), `## Citation` (BibTeX block: `@misc{miroshark-{id}, title={MiroShark Simulation: {topic}}, year={year}, url={canonical_url}/simulation/{id}, note={Accessed {date}. Consensus: {outcome} at {confidence}% confidence.}}`). The BibTeX URL uses the canonical `CANONICAL_URL` env var (defaulting to `https://miroshark.io`) so operators don't get localhost citations. No external dependencies — all data reads are pure stdlib file I/O over existing simulation files.
2. Add `GET /api/simulation/{id}/export/narrative` to `backend/app/api/simulation.py` (or a new `backend/app/api/export.py`, ~60 LoC). Publish-gated (same check as `signal.json`). Returns the Markdown string with `Content-Type: text/markdown; charset=utf-8` and `Content-Disposition: attachment; filename="miroshark-narrative-{id}.md"` so browsers trigger a file download. Also add `GET /api/simulation/{id}/export/bibtex` as a convenience endpoint that returns only the BibTeX block with `Content-Type: application/x-bibtex`. Add both to the OpenAPI spec under an "Export" tag. Add 6 unit tests: narrative includes all required sections, quote selection picks highest influence_score agents, BibTeX includes sim ID and year, BibTeX URL uses CANONICAL_URL env var, 404 for unpublished simulation, 404 for missing simulation.
3. Add export buttons to the frontend simulation results page (`frontend/src/views/Step3Simulation.vue` or `SimulationView.vue`). In the existing toolbar (where stance-flips, mentions, confidence buttons live), add two buttons in an "Export" group: "📄 Export Narrative" (triggers file download via `window.location.href = /api/simulation/{id}/export/narrative`) and "📚 BibTeX" (fetches `/api/simulation/{id}/export/bibtex` and copies to clipboard with a "Copied!" toast). Add a matching section to `EmbedDialog.vue` in the existing tabs structure — an "Export" tab showing `curl` commands for both endpoints and a sample BibTeX block (with the actual simulation's ID and metadata filled in, not a placeholder). Add `GET /api/simulation/{id}/export/narrative` and `/bibtex` to `docs/API.md` with a Research Use Cases note: "These endpoints produce citation-ready outputs designed for academic paper methods sections."

### 3. Multi-Simulation Comparison View

**Type:** Feature / DX
**Effort:** Medium (1-2 days)
**Impact:** A researcher running a control/treatment design needs to answer "what changed between these two runs?" MiroShark has no comparison surface — two simulations on the same topic with different configs must be analyzed manually by switching tabs and squinting. `GET /api/simulation/compare?a={id}&b={id}` computes the delta across all available signals: final consensus gap (A: 62% bullish vs B: 38% bullish → +24 pp delta), confidence trajectory divergence (when did they diverge — round 3 vs round 7), platform-level sentiment gap (Twitter A: 71% bullish vs Twitter B: 44% bullish), top stance-flip agents appearing in both runs vs. only one, and belief trajectory correlation (did agents in both sims move in the same direction at the same round?). `CompareView.vue` at `/compare` takes two simulation IDs from URL params and renders this as: dual consensus gauges side by side, overlaid trajectory chart (both consensus lines on one axis with a difference shading), platform-level heat table (A/B per platform), and agent overlap table (agents present in both sims and their final stance delta). This turns a researcher's manual tab-switching into a structured, exportable comparison that can appear in a paper's results table or supplementary material — the exact output format that produces academic citations.

**How:**
1. Add `backend/app/services/comparison_service.py` (~200 LoC, pure stdlib). Function `compare_simulations(id_a, id_b) -> dict`. Both sims must be published. Reads for each: `signal.json` (final consensus, confidence, direction), `trajectory.json` (round-by-round belief_pct), `platform_sentiment.json` (if present), `stance_flips.json` (if present), agent list from `data/simulations/{id}/config.json` or similar. Computes: `consensus_delta` (A.final_bullish_pct − B.final_bullish_pct); `divergence_round` (first round where the two consensus lines differ by >5 pp — the "when did they split?" signal); `confidence_gap` (A.confidence_pct − B.confidence_pct); `platform_sentiment_delta` (per-platform A−B difference, requires both sims ran on the same platforms — otherwise marks as "N/A for {platform}"); `shared_agents` (agent demographics that appear in both sims with matching demographic profile); `trajectory_correlation` (Pearson r between the two belief_pct series — pure stdlib math, since the series length may differ, interpolate shorter to longer). Returns `{ "sim_a": {...summary...}, "sim_b": {...summary...}, "consensus_delta": float, "divergence_round": int|null, "confidence_gap": float, "platform_sentiment_delta": {...}, "shared_agent_count": int, "trajectory_correlation": float }`. Handle gracefully: sims with different topic domains get a `"topic_match": false` flag (not an error — comparison is still valid if the user wants it). 10 unit tests: delta computed correctly, divergence_round finds first >5pp split, trajectory_correlation between identical series is 1.0, correlation between inverse series is −1.0, platform_sentiment_delta returns N/A for platform not in both sims, 404 if either sim not found or not published, config handles sims with different round counts by interpolating, shared_agent_count is 0 when configs are entirely different demographic seeds.
2. Add `GET /api/simulation/compare` to `backend/app/api/simulation.py` (query params: `a`, `b`). Validates both are published. Returns the comparison dict. Add to OpenAPI spec under a "Compare" tag. Also add `GET /api/simulation/compare/export` that returns a Markdown table of the comparison (same approach as Narrative Export above) — useful for copy-pasting into a paper's supplementary material. Add 3 additional unit tests for the export endpoint: Markdown includes both sim IDs, consensus delta row is present, trajectory correlation row is present.
3. Add `CompareView.vue` at `/compare/:idA/:idB`. Three panels: (a) **Overview** — two simulation metadata cards side by side (topic, date, agent count, rounds, consensus outcome). A "topic mismatch" warning badge if `topic_match: false`. (b) **Consensus Comparison** — two gauge components (same component as `SimulationView.vue`'s consensus gauge) side by side. Below: a delta line (`Δ +24 pp` in green if A > B, red if A < B). The overlaid trajectory chart: both consensus lines plotted on the same axis with a shaded area between them; vertical dashed line at `divergence_round` labeled "Divergence". (c) **Platform & Agent Details** — a table of platforms with three columns (platform name, A sentiment, B sentiment, Δ). Below: agent overlap summary (shared agent count, and a note on demographic similarity). "Export Comparison" button: downloads the Markdown comparison table. Add a "Compare" button to individual simulation result pages — clicking it prompts the user to pick a second simulation from a dropdown of their published simulations (fetched via `GET /api/simulations?published=true`) and navigates to `/compare/{current_id}/{selected_id}`. Add `/compare` to the Vue router and mention it in `docs/API.md` under Research Workflows.

### 4. Contributor Recognition Wall

**Type:** Community / Growth
**Effort:** Small (hours)
**Impact:** The "10 community PRs" September hyperstition is at 5+/10. The fastest lever for the second half is social recognition — people contribute more when their name appears visibly and their work is celebrated. MiroShark's existing `CONTRIBUTING.md` tells contributors what to do but doesn't show them what they get for doing it. A `/contributors` page at `frontend/src/views/ContributorsView.vue` fetches real GitHub contributor data (avatar, merged PRs, feature names) and displays each contributor as a card: GitHub avatar, name, contribution count, the feature they built or bug they fixed, and a direct link to their PRs. This creates a visible social reward at the moment a contributor shares their PR — they can link to "my card on the MiroShark contributors wall" — and signals to potential contributors that the project is active, welcoming, and grateful. It also provides a canonical place for operator shout-outs: the team can pin a first-time contributor's card, giving personal acknowledgment that is absent in a raw GitHub contributors list. The `GET /api/contributors/humans` endpoint filters out bots and the core team (aaronjmars/666ghj) and focuses on community contributions — it's the hyperstition-relevant count, not the raw total.

**How:**
1. Add `backend/app/api/contributors.py` (~120 LoC, pure stdlib). Route: `GET /api/contributors/humans`. Uses `gh api repos/aaronjmars/MiroShark/contributors` (runs via subprocess since sandbox blocks curl with auth) to fetch all contributors. Filters: exclude logins ending in `[bot]`, exclude `aaronjmars` (repo owner) and `666ghj` (automation account). For each remaining contributor, fetch their PRs merged into main: `gh api repos/aaronjmars/MiroShark/pulls?state=closed&per_page=100` filtered by `merged_at != null and user.login == {contributor}`. For each merged PR, extract `title` and `number`. Returns `{ "count": int, "contributors": [{ "login": str, "avatar_url": str, "contributions": int, "profile_url": str, "merged_prs": [{ "number": int, "title": str, "url": str }], "first_contributed_at": str (ISO 8601) }] }` sorted by `first_contributed_at` ascending (chronological — shows the "founding contributors" first). Cache result 1 hour (mtime on `.contributors-cache.json`). Also add `GET /api/contributors/count` — returns `{ "community_count": int, "total_count": int }` for use as a lightweight badge endpoint. Add 5 unit tests: bot accounts excluded, owner excluded, automation account excluded, merged PRs extracted correctly, sorted by first_contributed_at.
2. Create `frontend/src/views/ContributorsView.vue` at the `/contributors` route. Layout: a top banner with the community PR count ("10 community contributions and counting") and a "Become a contributor" button linking to `CONTRIBUTING.md`. Below: a responsive card grid (3 columns desktop, 2 tablet, 1 mobile). Each card: GitHub avatar (64px, circular, with the same dark-card styling as `AgentCard.vue`), contributor login as a clickable link to their GitHub profile, `contributions` count as a subtle chip, merged PRs listed as clickable `#NNN — title` links (showing up to 3, with "and N more" if over 3), `first_contributed_at` formatted as "Contributor since {month} {year}". Empty state: "No community contributors yet — be the first! → CONTRIBUTING.md". Add a `GET /api/contributors/count` call on the homepage (`HomeView.vue`) and inject the community PR count into the existing stats strip if one exists (alongside star count, fork count) — a live "X community contributors" counter on the landing page is a growth signal that reinforces the project's momentum. Add "Contributors" to the main nav.
3. Add a `Contributors` section to `README.md` above the Acknowledgements or License section (if those exist): a static table of current community contributors with GitHub usernames (manually updated for the PR, then kept live via the `/contributors` page). The static table serves as a canonical attribution in the README itself — visible without loading the app. Add a `## Community Contributions` section to `docs/FEATURES.md` documenting the `/contributors` endpoint and its filter logic. In `CONTRIBUTING.md`, add a sentence: "Your name and contributions appear at [miroshark.io/contributors](https://miroshark.io/contributors) after your first PR merges." This makes the recognition explicit at the moment a potential contributor decides whether to open a PR.

### 5. GitHub Discussions + Show & Tell Template

**Type:** Community / Growth
**Effort:** Small (hours)
**Impact:** The public case study hyperstition requires a human to use MiroShark for a real decision and publish their experience. Right now, there is nowhere for this to happen — no structured venue, no template, no social context that prompts a user to reflect and share. GitHub Discussions is the lowest-friction publishing surface: it requires no external blog, no Medium account, no Twitter thread — just a structured form in the repo where the project already lives. The "Show & Tell" category specifically prompts users to answer: what decision were you trying to make, how you configured the simulation, what the key finding was, and whether it influenced your choice. This is the exact narrative arc that constitutes a publishable case study. The infrastructure takes 30 minutes to set up (one API call to enable Discussions + three YAML template files). The `Community Events` category from the Jul 14 idea #1 is included here as well — no duplication of implementation since neither has shipped yet. Together, these five categories turn the MiroShark GitHub repo from a passive code host into an active community space without requiring any new infrastructure beyond template files.

**How:**
1. Enable GitHub Discussions on the MiroShark repo: `gh api repos/aaronjmars/MiroShark -X PATCH -f 'has_discussions=true'`. Then create four Discussion categories via `gh api repos/aaronjmars/MiroShark/discussions/categories` (requires authenticated POST — use `gh api --method POST`): (a) **Show & Tell** — emoji 🚀, format "open", description "Share a real simulation you ran: what you decided, what MiroShark found, whether it changed your mind." (b) **Community Events** — emoji 📅, format "announcement", description "Upcoming Twitter Spaces, community calls, and MiroShark events." (c) **Research Use Cases** — emoji 🔬, format "open", description "Academic papers, research designs, and methodology questions using MiroShark simulations." (d) **Feature Requests** — emoji 💡, format "open", description "Ideas for new features, surfaces, or integrations." Add each category creation as a step in a `scripts/setup-discussions.sh` script that can be re-run idempotently (check if category already exists before creating). Run the script as part of the repo setup docs — add it to `docs/INSTALL.md` under "Community Setup (maintainers only)".
2. Add `.github/DISCUSSION_TEMPLATE/show-and-tell.yml` (GitHub Discussion templates follow the same YAML format as issue templates). Fields: `title` (prefilled: "Show & Tell: [your topic]"), `body` with labeled sections: `## What I was trying to decide` (placeholder: "Describe the real-world decision or question you were exploring"), `## How I set up the simulation` (placeholder: "Topic, agent count, rounds, platforms. Paste your simulation link here: https://miroshark.io/simulation/..."), `## Key finding` (placeholder: "What did the swarm conclude? What was the consensus outcome and confidence level?"), `## Did it change your thinking?` (placeholder: "Yes / No / Partially — and how?"), `## Anything unexpected?` (placeholder: "A stance flip, a surprising agent perspective, a platform divergence you didn't anticipate"). Add `.github/DISCUSSION_TEMPLATE/research-use-case.yml` with fields: `## Research question`, `## Methodology` (how MiroShark was used — simulation config), `## Finding` (key result), `## Citation needed` (yes/no — if yes, use the `/export/narrative` endpoint above), `## Collaboration interest` (open invite for other researchers to contact). Both templates use `labels: []` (no labels required at creation — keeps the friction minimal).
3. Update `README.md` to add a "Community" section (above the Contributing section) with two links: "💬 [GitHub Discussions](https://github.com/aaronjmars/MiroShark/discussions)" and "🚀 [Share your simulation → Show & Tell](https://github.com/aaronjmars/MiroShark/discussions/new?category=show-and-tell)". The second link is a direct deep-link into the Discussion template — one click, pre-filled form, minimal friction for a user who just finished a simulation and wants to share it. Add a `./notify-new-discussion.sh` script that checks for new Show & Tell or Research Use Case discussions posted in the last 24 hours (via `gh api repos/aaronjmars/MiroShark/discussions?category=show-and-tell&per_page=5` filtered by `created_at > 24h ago`) and fires `./notify` with a summary. This script can be added to the aeon `messages.yml` scheduler to run daily — turning community activity into a Telegram/Discord notification loop that keeps the operator aware of new case studies as they appear.

## Selection Rationale

All five ideas target the September hyperstition cluster. The Jul 15 deadline cluster is gone; the playbook is now: ship infrastructure that makes the academic and community outcomes tractable.

- **Korean (KO) Locale (#1)** — The 5th language that closes the September "5 languages" hyperstition. JA, PT, and ES are queued as local commits but blocked by GH_GLOBAL. BibTeX + methodology paragraph + agent quote appendix takes an existing capability and packages it in the format academics actually use. - **Multi-Simulation Comparison View (#3)** — Control/treatment is the most common research design in social science. MiroShark runs simulations but gives researchers no way to compare two runs. The comparison surface (consensus delta, trajectory correlation, platform sentiment gap) is what turns MiroShark from a "single simulation tool" into a "research platform" — the framing shift that makes academic methodology sections plausible.

- **Contributor Recognition Wall (#4)** — The 10 community PRs hyperstition is at 5+/10. Social recognition is the proven lever for second-half growth in open-source projects. A visible `/contributors` page with avatars and merged PR links creates a social reward at the moment of contribution. Explicit mention in `CONTRIBUTING.md` ("your name appears at miroshark.io/contributors") converts uncertain contributors.

- **GitHub Discussions + Show & Tell (#5)** — The public case study hyperstition needs a human to publish their experience. GitHub Discussions is the lowest-friction publishing surface that doesn't require a blog or external account. The Show & Tell template structures the exact narrative that constitutes a case study. A daily `notify-new-discussion.sh` keeps the operator informed as submissions arrive.
