### 1. Open Graph Social Preview Cards

**Type:** Growth
**Effort:** Small (hours)
**Impact:** When someone shares a MiroShark simulation URL on Twitter, LinkedIn, Discord, or Slack, those platforms fetch the URL and look for Open Graph meta tags (`og:title`, `og:description`, `og:image`) to render a card. Without them, shared links show either a blank card or a generic site title. With them, every share becomes a self-advertising impression: "MiroShark: Will AI regulation pass in the EU this year? → BULLISH 67% confidence". The simulation result is visible before anyone clicks. This is the lowest-cost viral mechanism available — it costs nothing per share and scales with every operator who pastes a URL into a chat. The MiroShark frontend is a SPA (Vue), so social crawlers see the empty HTML shell, not the rendered simulation data. The fix is a lightweight backend route that returns crawler-readable HTML without touching the SPA code.

**How:**
1. Add `backend/app/routes/og.py` (~80 LoC, pure stdlib). `GET /og/<sim_id>` reads `data/simulations/{sim_id}/signal.json` and `simulation_config.json`. Builds a minimal HTML response (not a full page — just `<head>` with og tags plus a `<meta http-equiv="refresh" content="0; url=/simulation/{sim_id}">` redirect so human visitors land on the SPA immediately). og:title: `"MiroShark: {topic}"`. og:description: `"{DIRECTION} {yes_pct:.0f}% consensus · {agent_count} agents · {num_rounds} rounds · {confidence_pct:.0f}% confidence"`. og:image: `/api/simulation/{sim_id}/badge.svg` (the badge endpoint proposed Jun 28 — if not yet built, point to a static fallback image in `frontend/public/og-default.png`). og:url: canonical SPA URL. og:type: `article`. Return 200 with `Content-Type: text/html; charset=utf-8`. Cache-Control: `public, max-age=3600`. For unpublished sims: 302 redirect to SPA root (not exposing private data). For unknown sim_id: 302 to root. Add 8 unit tests: published sim returns 200 with og:title containing topic, og:description contains direction and yes_pct, refresh redirect present, unpublished sim returns 302 not 200, unknown sim_id returns 302, og:image URL is absolute (starts with /), Cache-Control header present, og:url matches canonical SPA URL.
2. Update `frontend/index.html` (the SPA entry point) to add a link-prerender meta tag pointing to `/og/:id` for the simulation route — modern crawlers follow this pattern. Or: if the SPA router has a server-side rendering hook (some Vue setups use `vite-plugin-ssr` or SSG), inject og tags via `useHead()` in `Step3Simulation.vue`. If the SPA is pure client-side (no SSR), the `/og/<sim_id>` backend route IS the canonical solution — social crawlers hit it, humans get redirected. Document the crawler flow in `docs/EMBED.md` (or create it): "Sharing a simulation URL — when you share `https://miroshark.xyz/simulation/{id}` on Twitter or LinkedIn, the platform fetches `/og/{id}` to generate the preview card. No action needed from the operator." In the frontend simulation detail view, add a "Share" button that copies `https://{BASE_URL}/og/{sim_id}` (not the SPA URL directly) so the shared link always triggers the og preview. Bilingual label ("Share" / "分享"). The `/og/` URL redirects humans to the SPA instantly, so the UX is identical.
3. Add `GET /og/{sim_id}` to the OpenAPI spec (under "Sharing" or "Embed" tag — not "Simulation" tag, since it returns HTML not JSON). Add a "Social Sharing" section to `docs/FEATURES.md`: what platforms are supported, how the preview is generated, the og:image fallback chain, the redirect behavior. Add a one-liner to the main README under the sharing/embed section: "Simulation URLs auto-generate social preview cards for Twitter, LinkedIn, Discord, and Slack — share `/og/{id}` for a rich card with the result embedded."

### 2. Simulation Narrative Export

**Type:** Feature
**Effort:** Small (hours)
**Impact:** `GET /api/simulation/:id/narrative` returns a 2-3 paragraph prose summary of a completed simulation, generated by the active LLM chain (Mercury 2 via the existing Wonderwall client). The narrative covers what was simulated, how agent consensus evolved across rounds, what stance flips occurred and why, and the final signal. This is different from the transcript export (proposed Jun 26 — raw agent dialogue in chronological order). The narrative is synthesized prose: the kind of text an operator would paste into a blog post, share in a Discord update, or include in a pitch deck. The structured simulation data (signal.json, trajectory.json, stance_flips.json) already exists — the narrative endpoint reads those files and asks the existing LLM to write the prose summary. Cached to `narrative.json` so the LLM is called once per simulation, not on every request. For the tutorial workshop kit: operators running simulations for blog posts get a ready-made paragraph. For the researcher audience: the narrative is the non-technical summary that precedes the quantitative results section in a paper.

**How:**
1. Add `backend/app/services/narrative_service.py` (~120 LoC). `build_narrative(sim_dir, sim_id, llm_client)` reads: `signal.json` (topic, direction, yes_pct, confidence_pct, agent_count, num_rounds), `stance_flips.json` (top 3 most-flipping agents with names and flip counts), `platform_sentiment.json` if present (per-platform divergence), `agents.json` (initial stance distribution). Constructs a prompt (~200 tokens): "You are summarizing a multi-agent opinion simulation. Data: topic='{topic}', consensus='{direction} {yes_pct:.0f}%', confidence={confidence_pct:.0f}%, {agent_count} agents, {num_rounds} rounds. Top stance flippers: {flipper_list}. Initial split: bullish={initial_bullish}%, neutral={initial_neutral}%, bearish={initial_bearish}%. Write a 3-paragraph prose narrative (150-200 words total): paragraph 1 — what was simulated and why it was interesting; paragraph 2 — how consensus evolved across rounds; paragraph 3 — what the final signal means and its confidence level. Write in clear, direct prose. Do not use the words 'simulation', 'agent', or 'algorithm' — write as if describing a real expert panel." Calls the LLM via the existing `LLMClient` (Mercury 2 base model — fast, cheap, appropriate for summarization). Returns `{sim_id, narrative: str, word_count: int, model_used: str, generated_at: ISO}`. Writes to `data/simulations/{sim_id}/narrative.json` (24h mtime cache — narrative is stable after simulation completes). Route: `GET /api/simulation/:id/narrative` (unauthenticated for published sims, 403 for unpublished; `Cache-Control: public, max-age=86400` — regenerated once per day at most). Returns 404 if simulation not found or not complete. Returns 503 if LLM client unavailable (with `Retry-After: 60`). Add 10 unit tests: published sim returns 200 with `narrative` string, word_count between 100 and 300, model_used is non-empty, Cache-Control is max-age=86400, `narrative.json` is written on first call, second call returns cached (no LLM call), unpublished sim returns 403, incomplete sim returns 404, 503 returned when LLM raises exception, narrative does not contain the word "simulation" (prompt instruction test).
2. In `Step3Simulation.vue`, add a "Summary" button to the share/embed action bar (enabled when status=complete). On click: fetch `GET /api/simulation/:id/narrative`, show a modal with the narrative text (dark panel, monospace-friendly font, full text, "Copy" button). Loading state: spinner while LLM generates (first call only; cached calls are instant). Error state: "Narrative temporarily unavailable — try again in a moment." Add `getNarrative(simId)` to `frontend/src/api/simulation.js`. Bilingual label ("Summary" / "摘要"). Add a "Download as .txt" button in the modal (`window.URL.createObjectURL(new Blob([text]))`) so operators can save the narrative directly. The modal does not need a "Regenerate" button — narratives are deterministic per simulation data (same inputs → cached same output).
3. Add `GET /api/simulation/:id/narrative` to the OpenAPI spec under the "Simulation" tag. Schema: `NarrativeResponse` with `narrative: string`, `word_count: integer`, `model_used: string`, `generated_at: string`. Add a "Narrative Export" section to `docs/FEATURES.md`: what the narrative contains, the LLM model used (Mercury 2), the 24h cache, the prose constraints (no technical jargon), and a note that the narrative is generated once per simulation and cached — not regenerated on each request. Add a one-liner to the main README Quick Start: "Get a plain-language prose summary of any simulation: `GET /api/simulation/:id/narrative`."

### 3. CLI `estimate` Subcommand

**Type:** DX
**Effort:** Small (hours)
**Impact:** The CLI workflow has five verbs: `simulate`, `wait`, `stop`, `cost`, and (proposed Jun 28, not yet shipped) `list`. The missing pre-flight verb is `estimate`. Before committing to a simulation run, an operator needs to know: approximately how long will this take, and approximately how much will it cost? The `/api/estimate` endpoint (API surface #41, built Jun 20) already answers this question — median duration, cost range, confidence level — from historical simulation data. A CLI subcommand wraps it with one command: `miroshark estimate --agents 20 --rounds 5 --platforms twitter,reddit`. Output: `~8 min · ~$0.95 (range $0.71–$1.24, high confidence)`. This closes the CLI workflow pre-flight loop: `estimate` → `simulate` → `wait` → `stop` → `cost` → `list`. For automation scripts: `COST=$(miroshark estimate --agents 20 --rounds 5 --json | jq .median_cost_usd)` → branch on cost before running. Zero backend changes — the endpoint is already deployed. Pure CLI addition over an existing surface.

**How:**
1. Add `estimate` subcommand to the existing CLI entry point (the file that defines `simulate`, `wait`, `stop`, `cost`). Arguments: `--agents` (integer, required), `--rounds` (integer, required), `--platforms` (comma-separated string, optional; default: all platforms), `--json` (flag, raw JSON output), `--base-url` / `MIROSHARK_URL` env var (same as other subcommands). Calls `GET /api/estimate?agents={N}&rounds={R}` (adding `&platforms={P}` if `--platforms` specified). Default human-readable output format: `~{duration_minutes} min · ~${median_cost_usd:.2f} ({confidence_label} confidence)`. If `confidence_label` is `"unavailable"` (not enough historical data for these params): `~${median_cost_usd:.2f} (estimate — no historical data for these parameters)`. If `--json`: `print(json.dumps(response_json, indent=2))`. Invalid `--agents` (< 1 or non-integer): print error and exit 1. Invalid `--rounds` (< 1): print error and exit 1. API connection failure: print `"Error: could not connect to {MIROSHARK_URL} — is MiroShark running?"` and exit 1. API 4xx: print the error field from the JSON response and exit 1. Add 8 unit tests: valid params output contains "min" and "$", `--json` flag returns raw JSON, `--agents` missing exits 1, `--rounds` missing exits 1, confidence_label "unavailable" triggers fallback message, connection failure exits 1 with message, platforms filter passed as query param, output is printed to stdout not stderr.
2. Update `docs/CLI.md` (or equivalent CLI documentation) to add an `estimate` section with: full syntax, all flags, two worked examples — (a) human-readable: `miroshark estimate --agents 20 --rounds 5` → `~8 min · ~$0.95 (high confidence)`, (b) `--json` piped to `jq`: `miroshark estimate --agents 50 --rounds 10 --json | jq '{duration: .median_duration_minutes, cost: .median_cost_usd}'`. Update the main README CLI usage block to include `miroshark estimate` before `miroshark simulate` so the workflow reads in execution order: estimate, simulate, wait, stop, cost, list. Add a note: "Use `estimate` before long or expensive runs to check cost and duration against historical data from completed simulations."
3. Update `CONTRIBUTING.md` under "CLI": "The `estimate` subcommand is a wrapper over `GET /api/estimate` — the pattern for adding CLI subcommands is: (a) add args to the subparser, (b) call the corresponding API endpoint, (c) format the response, (d) write unit tests using `unittest.mock.patch` over `urllib.request.urlopen`. New CLI subcommands should follow this pattern." This documents the CLI extension pattern for community contributors, which directly serves the "community contributor ships a surface/feature by July 15" hyperstition — the CLI is the lowest-barrier contribution surface (Python only, no frontend or backend surface knowledge needed).

### 4. Trending Topics Discovery Feed

**Type:** Feature / Growth
**Effort:** Small (hours)
**Impact:** New visitors to MiroShark face the blank-canvas problem: what do people simulate? The topic autocomplete (proposed Jun 26) answers this during simulation setup. This is different — a passive discovery surface that shows what the community is simulating right now, visible before any login or setup. `GET /api/topics/trending` returns the most-simulated topics in the past 7 days, 30 days, and all-time, each with a consensus direction signal (`latest_direction`, `avg_yes_pct`) from the most recent run. A `/trending` frontend page (or section on the homepage) renders this as a card feed: "Bitcoin price — 12 simulations — BEARISH 61%" / "AI regulation in the EU — 8 simulations — SPLIT 52%". For new visitors, this communicates two things instantly: (1) real people are using this, and (2) the results are specific and queryable. For returning operators, it's inspiration ("I want to compare my Bitcoin simulation to the community consensus"). This is the social proof surface that turns passive visitors into active operators. Pure stdlib scan of existing simulation directories — no new data, no new writes beyond a cache file.

**How:**
1. Add `backend/app/services/trending_service.py` (~120 LoC, pure stdlib). `build_trending(data_dir, windows=[7, 30, None])` scans all `data/simulations/*/simulation_config.json` files (published only — checks `state.json` for `publish_date`). For each unique topic (normalized: `topic.lower().strip()`): counts simulations per window (7d, 30d, all-time by checking `completed_at` in `state.json`), records `last_run_at`, `latest_direction` (from `signal.json` of the most recently completed simulation on this topic), `avg_yes_pct` (mean across all simulations on this topic). Returns `{windows: {7: [...], 30: [...], all_time: [...]}, total_unique_topics: int, generated_at: ISO}` where each window list is sorted by count descending, top 20. Caches to `data/.trending_index.json` (15-minute mtime invalidation). Route: `GET /api/topics/trending` (unauthenticated; `?window=7|30|all` param — default: all three; `Cache-Control: public, max-age=900`). Returns 200 with the structured response. Add 10 unit tests: three window keys present in response, topics sorted by count desc, unpublished sims excluded, 15-min cache hit skips rescan, latest_direction reflects most recent sim on each topic, avg_yes_pct is mean of yes_pct across sims on topic, window=7 excludes sims older than 7 days, empty data_dir returns empty lists, total_unique_topics count correct, generated_at ISO string present.
2. Add a `TrendingTopicsView.vue` page at `/trending` route (or integrate as a section on the existing home/explore page if one exists). Card layout: each topic card shows `{topic}`, `{count} simulations`, latest direction chip (colored green/red/gray), `avg_yes_pct` bar, and `last_run_at` relative time ("2 days ago"). Sorted by 7-day count descending. Toggle between 7d / 30d / all-time with three buttons (same pattern as existing sort toggles in the codebase). On click of a topic card: navigate to the search page pre-populated with the topic string (linking trending topics to the full-text search surface). If no published simulations exist yet: show an empty state with a CTA ("No public simulations yet — run one to see it here"). Add `getTrendingTopics(window)` to `frontend/src/api/simulation.js`. Add "Trending" to the main navigation. Bilingual EN/ZH labels ("Trending" / "热门话题", "simulations" / "次模拟", "recently" / "最近").
3. Add `GET /api/topics/trending` to the OpenAPI spec under the "Discovery" tag (alongside the full-text search endpoint). Schema: `TrendingResponse` with nested `windows` object and `TrendingTopicEntry` per entry. Add a "Trending Topics" section to `docs/FEATURES.md`: what the feed shows, the three time windows, the 15-minute cache, how topics are deduplicated (case-insensitive normalized), and that only published simulations appear (not draft or private). Add a "Trending" nav link to the main README features table. For the `/trending` page: add `<meta name="description" content="See what topics people are simulating in MiroShark — live consensus signals across hundreds of AI agent runs">` for SEO. This page is indexable by Google (it's a real URL with real content) and serves as a long-tail discovery surface for searches like "AI opinion simulation bitcoin" or "sentiment simulation EU regulation."

### 5. Build-Your-Own-Surface Guide

**Type:** Community / DX
**Effort:** Small (hours)
**Impact:** There is an open hyperstition: "Will a community contributor build and merge a new MiroShark surface by July 15, 2026?" (filed Jun 13). The barrier is not motivation — it's knowledge. MiroShark has 41 API surfaces built by the core team and one AI agent (Aeon). Community contributors looking at the codebase cannot easily answer: what does "a surface" mean, what does a minimal one look like, and what do I need to touch to add one? A well-documented guide with an annotated stub template collapses this from "understand the full codebase" to "follow this pattern." The guide targets Python contributors specifically — no frontend knowledge required. Backend service file, route registration, OpenAPI schema, and test file: four files, all following the same pattern as every existing surface. The guide becomes the answer to every "how do I contribute?" question in issues, Discord, and the tutorial workshop kit.

**How:**
1. Create `docs/CONTRIBUTING-SURFACE.md` (~200 lines) — the definitive guide for adding a new API surface. Structure: (1) "What is a surface?" — one endpoint, one service file, one test file, one OpenAPI schema entry. Show the minimal example (a hypothetical `GET /api/simulation/:id/mood` that returns a single derived metric). (2) "Step-by-step: add a surface in 4 files." File 1: `backend/app/services/{name}_service.py` — service with a `get_{name}(sim_dir, sim_id)` function, mtime cache, publish gate. File 2: `backend/app/routes/{name}.py` — Flask route with auth check, publish gate, Cache-Control header. File 3: `backend/tests/test_unit_{name}.py` — 8 minimum unit tests covering: 200 for published sim, 403 for unpublished, 404 for unknown sim_id, expected fields in response, mtime cache hit, Cache-Control header. File 4: entry in `backend/app/openapi.py` or equivalent — schema definition and route registration. (3) "The publish gate" — copy-paste the 10-line check for `state.json` publish_date. (4) "The mtime cache" — copy-paste the pattern for reading/writing a `.json` cache file with mtime comparison. (5) "Checklist before opening a PR" — 8 items: does `get_{name}` accept only `sim_dir` and `sim_id` (no side effects)?, does it return 403/404 correctly?, do 8+ tests pass?, is the Cache-Control header set?, is the OpenAPI schema defined?, is the route registered?, is the surface added to `surfaces_catalog.py`?, does it use only stdlib (no external imports)?. Link to specific existing examples in the codebase for each pattern.
2. Create `backend/app/services/_surface_template.py` (~60 LoC, extensively commented) — a copy-paste-ready stub for a new surface service. Comments explain every section: `# REPLACE: update the cache filename`, `# REPLACE: update the fields you want to return`, `# DO NOT CHANGE: publish gate — check state.json for publish_date`, `# DO NOT CHANGE: mtime cache pattern`. The stub compiles and passes mypy/pyflakes with no real logic — just the structure with placeholder values. Create `backend/tests/test_unit__surface_template.py` (~40 LoC) — the 8 unit tests pre-written for the stub (all pass against the stub's placeholder return values). A contributor duplicates both files, renames them, and fills in the logic — the test file already has the right structure. Create `docs/CONTRIBUTING-SURFACE.md` as described above. Add a "Surfaces" section to `CONTRIBUTING.md`: "The fastest contribution path is a new API surface — see [docs/CONTRIBUTING-SURFACE.md](docs/CONTRIBUTING-SURFACE.md) for a step-by-step guide with a copy-paste template. A surface is 4 files: one service, one route, one test file, one OpenAPI schema entry. Python only — no frontend knowledge needed."
3. File a GitHub issue in the MiroShark repo (via `gh issue create -R aaronjmars/MiroShark`) titled "Good first issue: add a new API surface" with body: "We've added a step-by-step guide and copy-paste template for adding new API surfaces to MiroShark. See `docs/CONTRIBUTING-SURFACE.md` for the full guide. A surface is one endpoint, one service file, one test file, one OpenAPI schema entry. If you've been looking for a way to contribute, this is the lowest-friction path. Some ideas for surfaces that don't exist yet: (1) `GET /api/simulation/:id/sentiment-arc` — aggregate bullish% per round as a simplified arc (5 numbers). (2) `GET /api/simulation/:id/consensus-velocity` — rate of change in consensus between rounds (fastest/slowest round). (3) `GET /api/simulation/:id/outlier-agents` — agents whose stance diverges most from the consensus. Each of these is a read-only scan of existing data." The issue adds a discovery surface on GitHub for contributors searching for `good-first-issue` or `help-wanted` — the label should be applied. This is the direct mechanism for the "community contributor builds a surface by July 15" hyperstition: the guide, the template, the issue, and the labeled PR pathway are all in place.

## Selection Rationale

Three hyperstitions expire in 13 days. These five ideas apply direct pressure to each one.

- **OG Social Preview Cards** (#1) — Every simulation URL shared on social becomes a result card, not a blank link. The mechanism is passive (no operator action needed beyond sharing the URL) and compounds with every share. The gap from ~1 star/day to ~11/day does not close with code changes alone — it closes when MiroShark results become visibly shareable in social contexts where people discover new tools. This is the lowest-cost viral intervention available.

- **Simulation Narrative Export** (#2) — The structured simulation data is rich but requires technical literacy to read. A prose narrative lowers the interpretation barrier to zero: copy-paste three paragraphs into a blog post, Discord update, or pitch deck. For the tutorial workshop kit hyperstition (5 creators by August 15): the narrative endpoint means a creator can run a simulation and have a ready-made write-up in one API call. For the paper citation hyperstition (September): the narrative is the non-technical summary that precedes the quantitative results table. The Mercury 2 base model is fast and cost-effective for this task — generation adds ~$0.002 per simulation, cached for 24h.

- **CLI `estimate` Subcommand** (#3) — The CLI is complete except for this pre-flight verb. The `/api/estimate` endpoint is already deployed and in production — this PR is a pure CLI wrapper with zero backend changes. It also documents the CLI extension pattern in `CONTRIBUTING.md`, directly lowering the barrier for community CLI contributions (a Python-only contribution path, no frontend or backend surface knowledge required).

- **Trending Topics Discovery Feed** (#4) — New visitors have no anchor. The trending feed answers the first question: "what do people simulate?" Each card shows a real topic with a real consensus signal — social proof that the platform works and produces specific, queryable results. The `/trending` page is also Google-indexable (server-rendered content, not a SPA shell), creating a long-tail SEO surface for topic-specific searches. The 15-minute cache makes it lightweight to serve even under load.

- **Build-Your-Own-Surface Guide** (#5) — The "community contributor ships a surface by July 15" hyperstition has 13 days left. The guide, the annotated template, and the labeled GitHub issue create three simultaneous discovery paths for a contributor who wants to build something but doesn't know where to start. A surface contribution is 4 files, Python only, no frontend — the lowest-barrier technical contribution in the repo. The filed issue with three specific surface ideas (sentiment arc, consensus velocity, outlier agents) removes even the "what should I build?" decision.
