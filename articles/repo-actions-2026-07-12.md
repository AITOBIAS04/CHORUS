### 2. Simulation Diff API

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** Researchers who use MiroShark for legitimate experimental work run the same topic multiple times — with different agent counts, different rounds, different platform configurations — and compare results to draw conclusions. Currently there is no structured way to do this comparison; they must call `/api/simulation/{id}/signal` twice and diff the JSON manually. `GET /api/simulation/diff?a={id}&b={id}` returns a structured comparison object: which direction changed (BULLISH → BEARISH), the delta in `yes_pct` and `confidence_pct`, the difference in agent count and round count between the two runs, a per-platform breakdown of how sentiment diverged, and a stance flip delta (did simulation B produce more flip events?). This is the minimum viable "experiment" surface — the thing that lets a researcher write "we ran identical topics under two configurations and MiroShark produced consistent/divergent results." For the academic citation hyperstition: a paper's methodology section that shows a diff between two runs with quantified deltas is more rigorous than one that shows a single run. Implementation is pure comparison arithmetic over existing surface data — zero new data collection, zero new dependencies.

**How:**
1. Add `backend/app/routes/diff.py` (~120 LoC, pure stdlib). `GET /api/simulation/diff?a={sim_id_a}&b={sim_id_b}`. Validation: both IDs required; both must be published; both must have `complete` status; return 422 with clear message if either fails validation. Response JSON: `{ "simulation_a": { "id", "topic", "completed_at", "agent_count", "num_rounds", "platforms" }, "simulation_b": { "id", "topic", "completed_at", "agent_count", "num_rounds", "platforms" }, "signal_diff": { "direction_a", "direction_b", "direction_changed": bool, "yes_pct_a", "yes_pct_b", "yes_pct_delta", "confidence_pct_a", "confidence_pct_b", "confidence_pct_delta" }, "config_diff": { "agent_count_delta", "round_count_delta", "platform_overlap": ["twitter"], "platforms_only_in_a": [], "platforms_only_in_b": [] }, "stance_diff": { "flip_count_a", "flip_count_b", "flip_rate_a", "flip_rate_b" }, "platform_sentiment_diff": { "twitter": { "bullish_delta", "bearish_delta" }, ... } }`. For each numeric delta: `b_value - a_value` (positive = B is higher). Include `"same_topic": bool` (true if `topic_a.lower() == topic_b.lower()` — helpful for researchers who always run same-topic comparisons). Return `Cache-Control: public, max-age=300`. Add to `backend/app/main.py`. Add 8 unit tests: valid diff for two published sims, 422 when sim A not found, 422 when sim B not found, 422 when sim A not published, direction_changed=true when directions differ, direction_changed=false when same, yes_pct_delta arithmetic correct, platform_overlap correctly computed.
2. Add the diff endpoint to the OpenAPI spec under a new "Comparison" tag. Add it to `surfaces_catalog.py` as the 42nd surface with `type: "comparison"`. Update `surface_stats.py`. Add a "Compare" action to the simulation detail page in `Step3Simulation.vue`: a text input or dropdown where the user pastes or selects a second simulation ID, and a "Compare" button that opens a diff panel showing the structured comparison. Keep the UI minimal — the panel shows the `signal_diff` block (direction, yes_pct_delta, confidence_pct_delta as colored ±number badges) and the `platform_sentiment_diff` as a two-column bar chart per platform. No external chart libraries needed — use CSS flex bars with inline width percentages, matching the existing `StanceFlipsPanel.vue` pattern.
3. Add a `## Simulation Diff` section to `docs/FEATURES.md` with the endpoint signature, response field descriptions, a worked example (two runs of the same topic with a director-mode event injected in one), and an explicit note that the diff works across different topics — comparing two unrelated simulations is valid (no topic-match requirement). Add to `docs/API.md` under a "Comparison" group. Add one line to the README features table: "Simulation Diff | Compare two simulations — signal delta, stance shift, per-platform sentiment divergence". Add a `--diff {id_b}` flag to the CLI: `miroshark signal {id_a} --diff {id_b}` prints the diff as a formatted text table. This completes the CLI's analytical surface: simulate, wait, stop, cost, list, signal, signal --diff.

### 3. Per-Agent Belief Timeline

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** `trajectory.json` captures belief drift at the population level — aggregate `bullish_pct`, `bearish_pct`, `neutral_pct` per round. What it cannot show is what happened to Agent #42 specifically: a retail investor persona who started NEUTRAL, went BULLISH in round 3 after a news injection, then flipped BEARISH in round 7 after reading another agent's argument. That per-agent belief arc is the most granular observability surface MiroShark can offer, and it's the kind of data a researcher cites when analyzing individual-level opinion dynamics vs. population-level trends. `GET /api/simulation/{id}/agent/{agent_id}/timeline` returns the per-round belief record for a single agent: their declared direction and confidence at each round they participated, plus the actions they took (post, reply, like, trade) that correlate with belief shifts. This completes the agent-level observability stack: `agents.json` gives static profiles, `mentions.json` gives interaction network, `timeline` gives temporal belief evolution. Three surfaces, one agent story. Zero new data collection — `trajectory.json` and `actions.jsonl` already contain this information at the per-agent level; the endpoint just aggregates it per agent ID.

**How:**
1. Add `backend/app/services/agent_timeline_service.py` (~200 LoC, pure stdlib). The service scans the simulation's per-platform `actions.jsonl` files and the existing `trajectory.json` for records keyed by `agent_id`. For each round, extract: `{ "round": int, "direction": "BULLISH"|"BEARISH"|"NEUTRAL"|null, "confidence_pct": float|null, "actions": [{ "type": "post"|"reply"|"like"|"trade", "platform": str, "timestamp": str }] }`. The `direction` and `confidence_pct` for each round come from the agent's belief state as stored in the per-round trajectory snapshot; if a round has no belief record for that agent (they didn't participate), include the round with `"participated": false` and `null` for direction/confidence. Sort rounds ascending. Cache the result to `agent_timeline_{agent_id}.json` in the simulation's data directory (mtime-based, same pattern as all other surface services). Add `GET /api/simulation/{id}/agent/{agent_id}/timeline` to `backend/app/routes/agent_timeline.py` (~50 LoC): validate `sim_id` exists and is published, validate `agent_id` exists in `agents.json`; return the timeline with `Cache-Control: public, max-age=300`. 404 if either not found; 403 if not published. Add 10 unit tests: correct round count, direction per round, participated=false for skipped rounds, actions listed per round, caching (second call uses cached file), 404 for unknown agent, 404 for unknown sim, 403 for unpublished sim, rounds sorted ascending, action types correctly classified.
2. Add the endpoint to the OpenAPI spec under the existing "Agents" tag (alongside `agents.json` and `archetypes`). Add to `surfaces_catalog.py` as the 43rd surface (`type: "agent-detail"`). Update `surface_stats.py`. In `EmbedDialog.vue`, add an "Agent Timeline" section that shows the endpoint URL pattern for any specific agent (since the embed dialog already lists all other per-simulation surfaces, per-agent timeline fits naturally in the same list). In the main simulation UI (`Step3Simulation.vue`), when the user clicks on an agent in any agent list (e.g., the top-agents panel), show a timeline sparkline in the agent detail view: a horizontal bar per round showing direction (color-coded: green/red/gray) and confidence (opacity or bar height). Keep it minimal — 5-10 lines of inline CSS, no external chart library.
3. Add `## Per-Agent Belief Timeline` to `docs/FEATURES.md` under the Agents section: endpoint signature, response structure, a note that `participated: false` entries are included (for sparse participation), and an example use case: "Track how a specific persona's opinion evolved in response to news injections." Add to `docs/API.md`. Add a CLI shortcut: `miroshark agent-timeline {sim_id} {agent_id}` — prints each round as a row: `Round 3 | BULLISH (72%) | 2 actions`. This mirrors the `miroshark signal` and `miroshark cost` commands as single-artifact CLI lookups.

### 4. Verified Predictions JSON API

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** The `/verified` UI page exists — it's where MiroShark tracks prediction calls from past simulations that were validated against real-world outcomes. What doesn't exist is a machine-readable version: `GET /api/verified` returning the same data as structured JSON. The academic citation hyperstition requires a researcher to publish work referencing MiroShark; the strongest evidence they can include is not "we ran a simulation" but "MiroShark's simulation predicted X and X happened." A JSON API for verified predictions makes that evidence queryable: filter by direction, confidence tier, topic keyword, date range. It also enables autonomous agents (including this one) to include verified prediction statistics in articles ("MiroShark's BULLISH calls on X were validated Y% of the time over Z simulations"). The `/verified` page is already the most credibility-building surface in the entire product — giving it an API endpoint is the minimum viable act of making that credibility machine-readable. Implementation reads from whatever backing store the `/verified` UI page reads from — no new data collection.

**How:**
1. Add `backend/app/routes/verified.py` (~100 LoC, pure stdlib). `GET /api/verified` — no authentication, public endpoint. Query params: `limit` (int, default 20, max 100); `direction` (string, optional: BULLISH/BEARISH/NEUTRAL); `outcome` (string, optional: `correct`/`incorrect`/`partial`); `since` (ISO 8601 date, optional); `q` (keyword search on topic, optional). Read from the same backing store as the /verified UI page (whether that's a `verified.json` file in `data/`, a database table, or derived from `signal.json` + a manual verification log — check the existing implementation). Response: `{ "count": int, "verified": [{ "simulation_id", "topic", "direction", "yes_pct", "confidence_pct", "predicted_at", "outcome_determined_at", "outcome": "correct"|"incorrect"|"partial", "outcome_notes": str, "verification_url": str|null }] }`. Include a `"meta"` block: `{ "total_verified": int, "correct_count": int, "incorrect_count": int, "partial_count": int, "accuracy_pct": float }` — this is the headline number that appears in articles. Return `Cache-Control: public, max-age=3600`. Add 7 unit tests: returns array for public endpoint, limit param applied, direction filter applied, meta accuracy_pct computed correctly, empty result returns valid JSON, since filter applied, keyword search on topic applied.
2. Add to OpenAPI spec under a new "Verification" tag. Add to `surfaces_catalog.py` as the 44th surface (`type: "verification"`). Update `surface_stats.py`. In the /verified UI page, add a "Copy API URL" button next to the existing page controls — one click copies `https://miroshark.xyz/api/verified` to clipboard. Add a "JSON" toggle that switches the page to render the raw API response in a `<pre>` block (useful for developers who find the page while browsing and want to quickly copy the endpoint). Add the verified predictions API to `EmbedDialog.vue` under a "Verification" tab: show the endpoint URL, a curl example with direction filter, and the meta block preview (total verified, accuracy %).
3. Add `## Verified Predictions API` to `docs/FEATURES.md` under a "Verification" section: endpoint signature, all query params with types and defaults, the meta block fields, and an explicit note that `accuracy_pct` is computed as `correct_count / (correct_count + incorrect_count)` (partials excluded from the denominator). Add to `docs/API.md` under "Verification". Write a CLI shortcut: `miroshark verified [--direction BULLISH] [--limit 10] [--json]` — prints a formatted table of verified predictions and the accuracy summary line. Add a one-liner to README features table: "Verified Predictions API | Queryable record of past predictions validated against real outcomes — with accuracy statistics."

### 5. Research Series / Campaign Tracker

**Type:** Feature / DX
**Effort:** Medium (1-2 days)
**Impact:** A researcher studying how public opinion on a topic evolves over weeks or months runs MiroShark repeatedly — same topic, different dates — and wants to analyze the trend. Currently there is no way to group simulations together: each run is an isolated entity with no relationship to others. A "research series" — a named collection of simulations — enables longitudinal research workflows. `POST /api/series` creates a series with a slug and optional description. `POST /api/series/{slug}/add?simulation_id={id}` adds a simulation to the series. `GET /api/series/{slug}` returns all simulations in the series sorted by `completed_at`, plus an aggregated trend: the `direction` per run (as an ordered array), the `yes_pct` time series, the `confidence_pct` time series, and a computed `trend_direction` (RISING/FALLING/STABLE based on `yes_pct` slope over the series). This is the minimum viable longitudinal study surface. For the academic citation hyperstition: a paper that writes "we ran 6 simulations over 6 weeks and observed a consistent BEARISH shift (yes_pct: 61% → 52% → 48% → 44% → 41% → 39%)" has a publishable methodology with a single API endpoint as its data source. For the public case study hyperstition: a blogger who writes "I tracked AI regulation sentiment every week for 2 months using MiroShark series" is exactly the external content that closes that target.

**How:**
1. Add `backend/app/models/series.py` (~60 LoC, pure stdlib) — a `Series` dataclass: `slug` (str, URL-safe), `title` (str), `description` (str, optional), `created_at` (ISO 8601), `simulation_ids` (list[str]). Store in `data/series/{slug}.json` — one file per series, same pattern as `data/simulations/{id}/`. Add `backend/app/routes/series.py` (~180 LoC, pure stdlib). `POST /api/series` — body: `{ "slug": str, "title": str, "description": str }`. Validates slug: URL-safe pattern `^[a-z0-9-]{3,64}$`; 409 if already exists. Creates `data/series/{slug}.json`. Returns `201` with the series object. `POST /api/series/{slug}/add` — body: `{ "simulation_id": str }`. Validates sim exists and is published. Appends ID to `simulation_ids` list (dedup — don't add the same ID twice). Returns `200` with updated simulation count. `GET /api/series/{slug}` — reads all simulations in the series, fetches their `signal.json`, sorts by `completed_at`, computes aggregated trend: `{ "slug", "title", "description", "created_at", "simulation_count", "simulations": [{ "id", "topic", "direction", "yes_pct", "confidence_pct", "completed_at" }], "trend": { "yes_pct_series": [61, 52, 48, 44, 41, 39], "direction_series": ["BULLISH", "BULLISH", "BEARISH", "BEARISH", "BEARISH", "BEARISH"], "confidence_pct_series": [78, 74, 71, 69, 67, 65], "trend_direction": "FALLING", "first_run": "2026-06-01", "last_run": "2026-07-12", "span_days": 41 } }`. `trend_direction`: compute linear regression slope of `yes_pct_series`; if slope > +1.0/run → RISING; if slope < −1.0/run → FALLING; else STABLE. `GET /api/series` — returns list of all series slugs with title, created_at, simulation_count (no full simulation data). Add 10 unit tests: create series, 409 on duplicate slug, add sim to series, dedup (add same sim twice → count stays 1), get series returns sorted simulations, trend FALLING for declining yes_pct, trend RISING for increasing yes_pct, trend STABLE for flat, 404 for unknown slug, get all series returns list.
2. Add all series endpoints to OpenAPI spec under a new "Series" tag. Add `GET /api/series/{slug}` to `surfaces_catalog.py` as the 45th surface (`type: "series"`). Add a "Series" section to the MiroShark frontend: a dedicated `/series/{slug}` route in the Vue router showing the series title, description, simulation list as a timeline (cards sorted by date), and a trend chart (line chart of yes_pct over time using the same CSS-bar technique used elsewhere — no external chart library). Add a "Add to Series" button in `Step3Simulation.vue` for published simulations: a text input for the series slug (or a dropdown of existing series if the user has created any in this browser session), plus a "Create new series" modal. Keep the UI minimal — this is primarily an API feature; the UI just needs to make it discoverable.
3. Add `## Research Series` to `docs/FEATURES.md` under a new "Research Tools" section alongside the Verified Predictions API: what a series is (named collection of simulations with longitudinal trend tracking), all endpoint signatures, the `trend_direction` computation (linear regression slope), and a worked example: "Running 'Will AI regulation pass in the EU?' monthly and observing the `yes_pct` decline over 6 runs." Add to `docs/API.md`. Add a CLI shortcut: `miroshark series create {slug} "{title}"`, `miroshark series add {slug} {sim_id}`, `miroshark series show {slug}` (prints the trend as ASCII sparkline: `▂▂▃▄▅▆` for RISING, `▆▅▄▃▂▁` for FALLING). Add one line to README: "Research Series | Group simulations into named collections, track yes_pct trend over time — longitudinal research in one endpoint."

## Selection Rationale

Three days to July 15; those hyperstitions are not the target. All five ideas serve the September cluster and the durable research use case.

- **Japanese (#1)** — The most direct path to closing the September "5 languages" hyperstition. `README.ja.md` already exists in the repo, which is as close to a community signal as the project will get without a contributor PR. The dictionary pattern is proven (French, German). This is a build-it-now, close-it-permanently contribution.

- **Simulation Diff (#2)** — The academic citation hyperstition requires researchers to document methodology. "We compared two configurations" is standard experimental practice; `GET /api/simulation/diff` makes that comparison a first-class API endpoint rather than a manual JSON subtraction exercise. Low complexity, high research credibility.

- **Per-Agent Belief Timeline (#3)** — Completes the agent observability stack. `agents.json` is static profile; `mentions.json` is interaction graph; `timeline` is temporal belief evolution. A researcher who can trace individual agent opinion arcs over 10 rounds has a publishable story about individual-level vs. population-level opinion dynamics. Zero new data collection — the data already exists in `actions.jsonl` and `trajectory.json`.

- **Verified Predictions API (#4)** — The /verified UI page is already the highest-credibility surface in the product. Making it machine-readable with an accuracy meta block gives any article author (including autonomous agents) a single API call to cite: "MiroShark's verified prediction accuracy across N runs is X%." This is the strongest evidence for the academic citation hyperstition.

- **Research Series (#5)** — Longitudinal research is impossible without a way to group simulations. A named series with trend computation is the minimum viable research instrument for the "public case study" hyperstition — a blogger who tracks the same question monthly and publishes the trend is exactly the case study that closes that target. The CLI sparkline makes the trend visible without opening a browser.
