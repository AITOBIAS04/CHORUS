The existing `telegram_notify.py`, `discord_notify.py`, and `slack_notify.py` services mean the announcement broadcast requires zero new notification infrastructure — it's a script that calls what's already there.

**How:**

Add a `.github/DISCUSSION_TEMPLATE/community-event.yml` template with frontmatter fields: `event_type` (enum: Twitter Space, Discord Voice, Community Call), `date` (ISO 8601), `topic`, `speakers` (list), `registration_link` (optional), `description`. When someone creates a new Discussion in this category using the template, it auto-populates a structured post. The template file follows the same GitHub Discussion template format as `.github/ISSUE_TEMPLATE/` — this is a static YAML file, zero backend required.
2. Add a `/community` route to the Vue router and create `frontend/src/views/CommunityView.vue`. The page fetches open GitHub Discussions in the "Community Events" category via `GET /api/community/events` (a new backend endpoint, ~60 LoC, that calls the GitHub REST API at `/repos/aaronjmars/MiroShark/discussions?category=community-events` and returns upcoming events sorted by date). Each event renders as a card: event type badge, date + countdown, speaker list, topic, registration link button. Below the upcoming events list: a "Past Events" accordion showing closed Discussions in the same category. A "Schedule an Event" button links directly to the GitHub Discussion template — one click, structured form, no friction. The page should load the locale-toggle (same pattern as all other views) and handle the empty state gracefully: "No upcoming events — schedule the first one" with the GitHub link.
3. Add `backend/app/api/community.py` (~80 LoC, pure stdlib). `GET /api/community/events` — fetches GitHub Discussions in the "community-events" category via `gh api` (since sandbox blocks curl with auth). Returns `{ "upcoming": [...], "past": [...] }` with each event as `{ "id", "title", "date", "event_type", "speakers": [], "url", "registration_link" }`. Cache result 15 minutes (mtime on a `.community-events-cache.json` scratch file). Add a `notify-community-event` script at `scripts/notify-community-event.sh` — takes a Discussion URL as argument, fetches the structured fields, formats a notification (`*Community Event* — {title} on {date}, speakers: {list}, join: {url}`), and calls `./notify`. Add `GET /api/community/events` to the OpenAPI spec under a "Community" tag. Add a "Community" link to the main nav in `App.vue` or `MainView.vue`.

### 2. Simulation Cron Scheduler API

**Type:** Feature
**Effort:** Medium (1-2 days)
**Impact:** Every research use case that might produce a case study or academic citation requires longitudinal data — same topic, different dates, trend over time. Research Series (Jul 12) gives researchers a container to organize runs; but they still have to manually trigger each one. A scheduler that fires a pre-configured simulation on a cron expression turns a weekly manual action into a standing study. `POST /api/simulation/schedule` with `{ "topic": str, "cron": "0 9 * * 1", "config": {...}, "series_slug": str (optional) }` — the scheduler stores the job in `data/schedules/{id}.json` and a background thread (or GitHub Actions cron, if operators deploy this way) fires it at the specified cadence. For self-hosted Docker deployments: a lightweight scheduler thread inside the backend process. For GitHub Actions deployments: the endpoint writes a `data/pending-schedules/` file that a cron workflow picks up. Either way, each triggered run optionally appends to a Research Series, making the longitudinal trend fully automatic. A researcher who writes "I scheduled weekly simulations of the EU AI Act vote for 8 weeks" has a methodology that takes 2 minutes to set up and produces a publishable data series. That is the exact scenario that closes the "public case study" hyperstition.

**How:**
1. Add `backend/app/models/schedule.py` (~50 LoC) — a `SimulationSchedule` dataclass: `id` (UUID), `topic` (str), `cron_expression` (str), `config` (dict — same shape as simulation POST body), `series_slug` (str, optional), `created_at` (ISO 8601), `last_run_at` (ISO 8601, nullable), `next_run_at` (ISO 8601, computed), `run_count` (int), `enabled` (bool). Store in `data/schedules/{id}.json` (one file per schedule). Add `backend/app/services/scheduler_service.py` (~180 LoC, pure stdlib). Key functions: `list_schedules()` → all schedules sorted by next_run_at; `create_schedule(topic, cron_expression, config, series_slug)` → validates cron syntax (5-field, whitelist of common patterns: `* * * * *`, `0 N * * *`, `0 N * * W`, `0 N */M * *`; reject exotic expressions); `compute_next_run(cron_expression, from_dt)` → pure stdlib cron evaluator for the whitelisted patterns; `run_due_schedules()` → called by the background worker or the cron check endpoint; finds schedules where `next_run_at <= now` and `enabled = true`; triggers each via `SimulationManager.start_simulation(topic, config)`; updates `last_run_at`, `run_count`, `next_run_at`; if `series_slug` is set, calls `SeriesService.add_simulation(series_slug, new_sim_id)` (from the Jul 12 Series idea, if that's been built) or writes to `data/series/{slug}.json` directly.
2. Add `backend/app/api/schedules.py` (~120 LoC). Routes: `POST /api/simulation/schedule` — create a new schedule (body: `{ "topic", "cron", "config", "series_slug" }`; validate cron; return 201 with schedule object including computed `next_run_at`). `GET /api/simulation/schedules` — list all schedules (`{ "count", "schedules": [...] }`). `GET /api/simulation/schedule/{id}` — get single schedule with run history from the last 10 `last_run_at` values (stored as array). `PATCH /api/simulation/schedule/{id}` — update `enabled` (pause/resume), `cron`, or `topic`. `DELETE /api/simulation/schedule/{id}` — removes the schedule file. `POST /api/simulation/schedule/tick` — trigger-safe check endpoint: finds and runs any due schedules; designed to be called by a cron job (`*/5 * * * *` in the deployment's cron) or by the frontend polling. Returns `{ "ran": [sim_id, ...], "count": int }`. Add a background ticker in the main app startup: a `threading.Timer` that calls `run_due_schedules()` every 5 minutes if `SCHEDULER_ENABLED=1` env var is set (default `1` for Docker; `0` for GitHub Actions where the external cron is the trigger). Add 10 unit tests: create schedule, cron next_run computed correctly for daily 09:00, run_due_schedules fires sim for due schedule, run_due_schedules skips disabled schedule, pause/resume via PATCH, delete removes file, list returns sorted by next_run_at, 422 on invalid cron pattern, run_count increments, series_slug appends sim_id.
3. Add `GET /api/simulation/schedules` and `POST /api/simulation/schedule` to the OpenAPI spec under a new "Scheduler" tag. Add a `ScheduleView.vue` in `frontend/src/views/` at `/schedules`: lists active schedules as a table (topic, cron expression, next run countdown, run count, status badge, pause/delete buttons). A "New Schedule" modal: topic input, cron expression input with common presets (daily, weekly, every Monday), simulation config (agent count, rounds, platforms with sensible defaults), optional series slug. The cron input shows a human-readable translation (`0 9 * * 1` → "Every Monday at 09:00 UTC") computed client-side. Add CLI shortcut: `miroshark schedule create "{topic}" "0 9 * * 1"` and `miroshark schedule list` and `miroshark schedule pause {id}`.

### 4. Webhook Delivery Dashboard

**Type:** DX
**Effort:** Small (hours)
**Impact:** `webhook_service.py` and `docs/WEBHOOKS.md` exist — MiroShark has a webhook system. The problem: there is no UI to register, inspect, or debug webhooks. A developer who finds `WEBHOOKS.md` has to construct raw `curl` commands to register an endpoint, has no visibility into whether their webhook fired, and has no way to replay a failed delivery. This is the pattern that kills integrations before they start: the feature is technically there, but the developer experience friction stops anyone from completing setup. A `WebhooksView.vue` at `/webhooks` provides: a registration form (URL, events to subscribe, secret token), a delivery history table (timestamp, event, HTTP status, latency, response body snippet), and a "Replay" button for failed deliveries (4xx/5xx). The backend already has `webhook_service.py` — the additions are a delivery log writer and a replay endpoint. The delivery dashboard converts a buried docs feature into a first-class UX that any developer can discover and use in under 5 minutes. More webhook integrations = more external touchpoints = more potential for case studies and community contributors.

**How:**
1. Extend `backend/app/services/webhook_service.py` to add delivery logging. Each webhook dispatch already sends an HTTP POST — after each attempt, write a delivery record to `data/webhooks/deliveries/{webhook_id}-{timestamp}.json`: `{ "webhook_id", "event_type", "simulation_id", "delivered_at", "status_code", "latency_ms", "response_body_snippet": first 500 chars, "success": bool }`. Cap the delivery log at 100 records per webhook (delete oldest on write). Add a `replay_delivery(delivery_id)` function that re-POSTs the original payload to the registered endpoint and records a new delivery entry with `{ "replayed_from": original_delivery_id }`. Add `list_deliveries(webhook_id)` that reads and returns all delivery records sorted by `delivered_at` desc. Add `list_webhooks()` that reads all `data/webhooks/{id}.json` files. These are pure file-system operations — no new dependencies.
2. Add `backend/app/api/webhooks.py` (new file, ~120 LoC, pure stdlib) with routes: `POST /api/webhooks` — register a new webhook (`{ "url", "events": ["simulation.complete", "simulation.start", "simulation.error"], "secret": str (optional, HMAC signing key) }`; validate URL scheme is HTTPS; return 201 with webhook `id` and `created_at`). `GET /api/webhooks` — list all registered webhooks (no secret exposure — return `{ "id", "url", "events", "created_at", "delivery_count", "last_delivery_at", "last_delivery_status" }`). `DELETE /api/webhooks/{id}` — unregister. `GET /api/webhooks/{id}/deliveries` — list last 100 deliveries sorted newest-first with `status_code`, `latency_ms`, `success`, `delivered_at`, `event_type`. `POST /api/webhooks/{id}/deliveries/{delivery_id}/replay` — replay the specific delivery; returns the new delivery record. Add these to `backend/app/main.py`. Add 8 unit tests: register webhook, list returns all registered, delete removes, list_deliveries returns sorted, replay sends to endpoint (mock HTTP call), replay creates new delivery with replayed_from, 404 for unknown webhook, 400 for non-HTTPS url.
3. Create `frontend/src/views/WebhooksView.vue` at `/webhooks` route. Three sections: (a) **Registered Webhooks** — a table listing all webhooks with columns: URL (truncated with tooltip), events (badge list), created at, last delivery status (green tick / red X), delivery count. Row actions: Delete button, "View deliveries" link. (b) **Register Webhook** form (collapsible or modal): URL input, checkboxes for events to subscribe (`simulation.complete`, `simulation.start`, `simulation.error`), optional HMAC secret input with a "Generate" button (crypto.randomUUID() on client). On submit: POST to `/api/webhooks`, show the generated webhook ID and a `curl` test command pre-filled. (c) **Delivery History** (shown when "View deliveries" is clicked for a specific webhook): table with `delivered_at`, event type, HTTP status code badge (green 200, red 4xx/5xx), latency, response snippet. Per-row "Replay" button. Replay result shows inline (status code + latency). All interactions follow the existing Vue patterns (dark theme, same button/badge classes as `StanceFlipsPanel.vue`). Add "Webhooks" link to the main nav.

Lower barrier to first run means more runs, more case studies, more data for the verified predictions surface. The template gallery is also the natural entry point for the academic research use case: a researcher who wants to replicate a standard methodology picks a template, runs it, and has a cite-ready configuration in a structured format.

**How:**
1. Audit `backend/app/preset_templates/` contents (list all files via GitHub API). Each template is likely a JSON or Python dict with fields like `name`, `description`, `agent_count`, `num_rounds`, `platforms`, `demographics`. If the format is not already standardized, define a `TemplateSpec` dataclass in `backend/app/models/template.py`: `{ "id": str (slug), "name": str, "description": str, "category": "crypto" | "policy" | "product" | "social" | "research" | "custom", "agent_count": int, "num_rounds": int, "platforms": list[str], "demographics": dict, "tags": list[str], "example_topics": list[str] (2-3 sample prompts the template works well with), "estimated_cost_range": str ("$0.50–$1.20"), "estimated_duration_range": str ("4–8 min") }`. Ensure all existing preset templates conform to this spec (add missing fields with sensible defaults). Add 2-3 new domain templates if existing set is sparse: `ai-regulation.json` (15 agents, news + reddit + twitter, policy-expert + researcher demographics, topic seed: "Will the EU AI Act's Article 6 requirements reduce foundation model investment?"), `crypto-market.json` (25 agents, twitter + reddit, retail-investor + defi-native demographics, topic seed: "Will [TOKEN] reach new ATH within 30 days?"), `product-launch.json` (20 agents, twitter + hackernews, early-adopter + developer demographics, topic seed: "Will [PRODUCT] achieve product-market fit in its target segment?"). Primary CTA: "Use Template" — navigates to the simulation creation flow with all fields pre-populated from the template config. Secondary CTA: "Copy API" — copies the `curl -X POST /api/templates/{id}/simulate -d '{"topic":"..."}'` command to clipboard. Add `/templates` to the Vue router and to the main nav as "Templates". Add CLI shortcut: `miroshark templates list` (tabular output of all templates) and `miroshark templates use {id} "{topic}"` (triggers `POST /api/templates/{id}/simulate` and prints the simulation ID + polling command).
