### 1. Coalition Detection in Interaction Network

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The interaction network graph (already shipped) renders directed edges showing who influenced whom. What it doesn't show is the meso-level structure that matters most in social simulation: which agents clustered into self-reinforcing groups? A simulation showing "74% YES consensus" tells you the outcome. Coalition detection tells you *how it formed* — did a dominant 7-agent coalition absorb neutral agents over time, or did two opposing coalitions of 5 and 4 agents compete for the 6 undecided agents in the middle? These are categorically different consensus formation stories, and the current product can't distinguish them visually. Coalition Detection applies Louvain-style community detection (pure Python stdlib, no new dependencies — a greedy Union-Find variant on weighted edge adjacency is sufficient for graphs under 50 nodes) to the interaction graph at simulation completion. Each detected coalition is auto-labeled by its dominant stance and primary archetype ("7-agent YES coalition — Financial Analyst led"), assigned a distinct palette color (4 colors, cycling), and rendered as a colored subgraph background region in the network view. The sidebar gains a "Coalitions" panel: each cluster listed with agent count, dominant stance, most influential member, internal cohesion score (ratio of intra-cluster edges to total edges), and a one-sentence auto-generated description. The Quality Guard gains a new check: "Coalition dominance" — when one cluster accounts for >80% of all influence edges, flag it as a potential echo chamber (severity: warning). For researchers studying polarization dynamics and narrative coalition formation, this transforms the network view from a wiring diagram into an interpretable social structure analysis. It also makes the "Interaction Network" tab genuinely distinct from the belief drift chart, giving it standalone analytical value rather than serving as an aesthetic complement.
**How:**
1. After simulation completion, compute the coalition graph: build an adjacency matrix from `<sim_dir>/interaction_graph.json` (the existing per-round influence records), weight edges by influence score sum across all rounds, and run a greedy community detection pass — merge the two nodes with the highest combined edge weight, repeat until no merge improves modularity Q (≥ 0.1 improvement threshold). Store result in `<sim_dir>/coalitions.json`: `{coalitions: [{id, label, color, agents: [{name, archetype, platform, stance_at_end}], dominant_stance, cohesion_score, primary_influencer}], modularity_score, echo_chamber_flag (bool)}`. Add `GET /api/simulation/:id/coalitions` returning this structure. The computation is deterministic (no randomness) and cached on first request.
2. In `InteractionNetwork.vue`, fetch coalitions from the new endpoint on mount alongside the existing graph data. Render each coalition's agent nodes with a shared background ellipse (SVG `<ellipse>` with 15% opacity fill in the coalition color), positioned using the convex hull of the coalition's node coordinates. Add a legend chip row below the network: one chip per coalition (colored dot + label). On click, highlight only that coalition's nodes and edges (dim others to 20% opacity). In the results sidebar, add a "Coalitions" section below the existing leaderboard: each coalition as a card with stance badge, agent count, primary influencer, and cohesion score rendered as a thin horizontal progress bar. When `echo_chamber_flag` is true, show an orange "Echo chamber risk" banner above the coalition list with the Quality Guard message.
3. In the article generator, inject coalition context as structured input before the narrative prompt: "Coalition structure: N distinct coalitions detected (modularity Q=[score]). Largest: [label], [M] agents, cohesion [score]. Echo chamber: [yes/no]." The generated article describes the social architecture — "Two distinct coalitions formed by round 4: a 7-agent YES bloc anchored by Financial Analyst archetypes, and a 5-agent NO bloc led by a Risk Manager. The remaining 3 agents oscillated between coalitions before joining the majority by round 9." Add `GET /api/simulation/:id/coalitions` to the OpenAPI spec under the "Analysis" tag with the full `CoalitionResponse` schema. Document a "Reading coalition scores" section in `docs/FEATURES.md`: what modularity Q means (0 = no structure, 1 = perfect modules), what cohesion score measures, and how to interpret echo chamber flags.

### 2. Simulation Lifecycle Webhooks (State Machine Events)

**Type:** Integration
**Effort:** Small (hours)
**Impact:** The current completion webhook (PRs #46, #79) fires exactly once — when a simulation finishes. For server-side automation pipelines, this single event is the entire window into simulation progress. A Zapier workflow that wants to post "Simulation reached early consensus at round 5!" to Slack has no way to detect mid-simulation events without either polling the REST API continuously or standing up an SSE client (a separate delivery mechanism suggested May 8 for connected clients). State machine webhooks extend the existing delivery infrastructure (HMAC-SHA256 signing from PR #79, delivery log from PR #73) to fire on every meaningful lifecycle transition: `simulation.created` (immediately after launch, carries `{sim_id, config_summary, watch_url}`), `simulation.round_complete` (after each round, carries `{round, yes_price, no_price, neutral_price, top_agent, consensus_forming: bool}`), `simulation.consensus_reached` (fires the moment YES/NO/NEUTRAL first crosses the configured consensus threshold, mid-simulation, carries `{round, outcome, yes_price, confidence}`), `simulation.completed` (carries the full existing completion payload), `simulation.failed` (carries `{error_type, round_failed_at, message}`). The operator configures per-event subscriptions: a single webhook URL can subscribe to `[consensus_reached, completed]` without receiving `round_complete` spam. The "consensus reached mid-simulation" event is the missing primitive for time-sensitive automation — a trading research pipeline that wants to react when consensus forms doesn't need to wait for the final round to finish. The full state machine also unlocks "simulation watchdog" patterns: subscribe to `round_complete` events and detect stagnation externally by comparing consecutive round payloads, without any changes to MiroShark's internal logic.
**How:**
1. Extend `<sim_dir>/config.json` to include a `webhook_subscriptions: {url: string, events: string[], secret: string}` field at launch. The existing `completion_webhook_url` and `completion_webhook_secret` remain as aliases for `{url, events: ["simulation.completed"], secret}` — full backwards compatibility. Add event-specific subscription configuration to the Settings page: a checkbox matrix (rows = events, columns = configured webhook URLs) letting operators choose which events trigger which endpoints. In the simulation runner, after each meaningful state transition, call a new `deliver_lifecycle_event(sim_id, event_name, payload)` function that (a) constructs the event envelope `{event, sim_id, timestamp, data: {...}}`, (b) signs with HMAC-SHA256 (same as PR #79), (c) fans out to all subscribed webhook URLs for that event, (d) logs delivery result to `<sim_dir>/delivery_log.json` (same format as PR #73, with `event` field added). The `consensus_reached` event is emitted in the round loop when YES or NO first crosses the `consensus_threshold` setting (default 0.65).
2. In the delivery log view (Settings → Webhook Delivery Log from PR #73), add an `Event` column showing the event name for each delivery entry. Add a filter dropdown: "All events / round_complete / consensus_reached / completed / failed." In the Settings page under "Completion Webhook," rename the section to "Lifecycle Webhooks" and add an event subscription matrix below the existing URL/secret fields. Provide defaults: new webhook URLs default to subscribing to `[consensus_reached, completed, failed]` with `round_complete` opt-in (to avoid log spam on long simulations). Each event type in the matrix has a tooltip explaining its trigger and payload.
3. In the OpenAPI spec, document each event type under a new "Lifecycle Webhooks" section: event name, trigger condition, full payload schema, and example JSON. Add a `?subscribe_to` query parameter to `POST /api/simulation/start`: a comma-separated list of event types to subscribe the configured webhook to for this specific simulation (overriding the global Settings default). This lets callers programmatically vary event subscriptions per run — e.g., only subscribe to `round_complete` for a short 5-round diagnostic run, but not for production 15-round runs. Document the complete state machine as a Mermaid diagram in `docs/WEBHOOKS.md`: `created → (round_complete)* → [consensus_reached] → completed` and the `* → failed` error path. Add a note in `docs/WEBHOOKS.md`: "The `consensus_reached` event fires mid-simulation and is distinct from `completed` — it indicates the statistical threshold was crossed but the simulation continues running remaining rounds. If you need only the final outcome, use `completed`."

### 3. Scenario Starter Kit — Pre-Built Prompt Library + "Inspire Me" Button

**Type:** DX
**Effort:** Small (hours)
**Impact:** The hardest moment in the MiroShark user journey is the blank scenario text field. Every new visitor who understands the concept ("I can simulate how people will react to this") still faces the question "what exactly do I type?" The field offers no examples, no format guidance, and no signal about what kinds of scenarios produce interesting simulations vs. degenerate ones. The Scenario Starter Kit eliminates cold-start abandonment with two components: (a) a curated `scenarios/library.json` file — 80 pre-written, categorized, field-tested scenario prompts across domains: crypto/market (25), geopolitics/policy (20), tech/product (20), social/culture (15) — and (b) an "Inspire Me →" button next to the scenario textarea in Step 1 that opens a browsable picker. The picker shows 5 randomly sampled scenarios from the library (filtered to match any text the user has already typed), each displayed as a one-line card with a domain chip and a "Use this" button that populates the textarea. Users can shuffle for 5 new suggestions. A "Browse all" link opens a modal showing all 80 scenarios grouped by category. The scenarios are carefully formatted — active voice, specific enough to generate genuine disagreement, with a clear temporal horizon ("Will X happen by Y?" or "How will [community] react to Z?") — modeling the scenario format implicitly for users who read a few examples before writing their own. For repeat users who run MiroShark programmatically, `GET /api/scenarios/library` exposes the full library as JSON — enabling automated scenario testing workflows. For the feature skill, this is one of the highest-leverage DX improvements available: it directly addresses the conversion gap between "landing on the app" and "completing a first simulation run."
**How:**
1. Add `scenarios/library.json` to the repo root: a JSON array of `{id, text, domain, tags: string[], difficulty: "easy"|"medium"|"hard", predicted_controversy: float}`. Populate with 80 entries across four domains. Example entries: `{"id": "crypto-001", "text": "Will Ethereum's transition to proof-of-stake improve its long-term price outlook?", "domain": "crypto", "tags": ["ethereum", "consensus-mechanism", "price"], "difficulty": "easy", "predicted_controversy": 0.72}`, `{"id": "policy-014", "text": "Will the EU's AI Act create a competitive disadvantage for European AI startups by 2027?", "domain": "geopolitics", "tags": ["EU", "AI regulation", "startups"], "difficulty": "medium", "predicted_controversy": 0.81}`. Add `GET /api/scenarios/library` as a Flask endpoint that reads and returns this file. Support `?domain=crypto` and `?limit=5&shuffle=true` query parameters.
2. In `Step1Scenario.vue` (the scenario input step), add an "Inspire Me →" button to the right of the scenario textarea label. On click, call `GET /api/scenarios/library?limit=5&shuffle=true` (optionally filtered by any text already in the textarea using a simple keyword match on the `tags` array). Render results as a compact vertical list of 5 cards: each card shows the scenario text, a colored domain chip (crypto = amber, geopolitics = blue, tech = purple, social = green), and a "Use this" button. On "Use this," populate the textarea with the scenario text and close the picker. Add a "Shuffle" link below the list (refetches 5 new random scenarios) and a "Browse all 80 →" link that opens a full modal with category tabs. The picker is rendered as a floating panel anchored below the textarea (CSS `position: absolute`, `z-index: 50`, `box-shadow: lg`) — no route change, no modal overlay.
3. In `GET /api/scenarios/library`, add a `?similar_to=<text>` parameter that returns the 5 library scenarios with the highest keyword overlap with the provided text (simple TF-IDF cosine similarity over the concatenated `text + tags` fields, computed in pure Python). This powers a "refine my idea" flow: user types a partial scenario, clicks "Inspire Me →," and sees the 5 most similar fully-formed scenarios from the library. In `docs/FEATURES.md`, add a "Scenario Library" section documenting the `GET /api/scenarios/library` endpoint and the available query parameters. Note that the library is a static JSON file and can be extended by community PRs — "To contribute scenarios, open a PR editing `scenarios/library.json`. Include at least one scenario per domain and set `predicted_controversy` based on how much you'd expect a typical 10-person group to disagree on the question." This makes the library a community contribution surface.

### 4. miroshark-sdk — Python Client Package

**Type:** Integration
**Effort:** Medium (1–2 days)
**Impact:** MiroShark's API is fully documented in OpenAPI spec, but every developer who uses it — Jupyter notebook users, automation pipeline builders, researchers scripting batch runs — must hand-roll HTTP requests. `requests.post()`, polling loops, CSV parsing, HMAC signature construction: these are 40 lines of boilerplate that live nowhere in the project and are reimplemented by every user independently. The `miroshark-sdk` Python package eliminates all of it. Installed with `pip install miroshark-sdk`, it exposes a clean client: `from miroshark import MiroSharkClient; client = MiroSharkClient("http://localhost:5000"); sim = client.run("Will BTC hit $200K by 2027?", n_agents=10, n_rounds=8)`. The `run()` method is blocking: it launches the simulation, polls the status endpoint at 10-second intervals, and returns a `Simulation` object when complete. `sim.trajectory_df` returns the aggregate trajectory as a `pandas.DataFrame` (if pandas is installed) or a list of dicts (if not — zero hard dependencies). `sim.agent_trajectory_df` returns the per-agent trajectory. `sim.quality_report` returns the quality guard dict. `sim.citation("bibtex")` returns the BibTeX string. `sim.download_bundle("/path/to/dir")` downloads and extracts the research bundle ZIP. The SDK also exposes an async variant (`AsyncMiroSharkClient`) using `asyncio` + `urllib` (no `httpx`/`aiohttp` required — pure stdlib) for users running batch simulations in parallel. Ships in `sdk/python/` with a `pyproject.toml` and a 20-line `README.md` example. For the researcher cohort that just got the Jupyter export (PR #80), a first-class Python client is the next logical capability unlock — the difference between "I can analyze MiroShark results in Jupyter" and "I can run MiroShark from Jupyter."
**How:**
1. Create `sdk/python/miroshark/` with `__init__.py`, `client.py`, `models.py`, `exceptions.py`, and `pyproject.toml`. `client.py` contains `MiroSharkClient` with methods: `run(scenario, n_agents, n_rounds, model, **kwargs) → Simulation` (blocking — polls every 10s until `status == "complete"`), `start(scenario, ...) → str` (returns `sim_id`, non-blocking), `get(sim_id) → Simulation`, `poll(sim_id, timeout_seconds=600) → Simulation` (blocks until done or timeout), `list_gallery(limit=20, **filters) → List[SimulationSummary]`, `get_library_scenarios(domain=None, limit=5) → List[ScenarioTemplate]`. All HTTP via `urllib.request` (stdlib only — zero hard dependencies). `models.py` defines `Simulation(sim_id, status, config, trajectory, agent_trajectory, quality_report, coalitions)` as a dataclass with properties: `trajectory_df` (returns `pandas.DataFrame` if pandas available, else list), `agent_trajectory_df`, `citation(format="bibtex") → str`, `download_bundle(dest_dir: str)`. `exceptions.py` defines `MiroSharkError`, `SimulationTimeoutError`, `SimulationFailedError`.
2. Add `AsyncMiroSharkClient` in `client.py` using `asyncio` + `urllib` (pure stdlib): `async def run(...)`, `async def poll(...)`. Add a `batch_run(scenarios: List[str], **shared_config) → List[Simulation]` method on both sync and async clients: launches all simulations in parallel, returns when all complete. For the async client, uses `asyncio.gather`. For the sync client, uses `concurrent.futures.ThreadPoolExecutor`. Add `sdk/python/README.md` with a complete 20-line example: import, instantiate, `client.run()`, access `sim.trajectory_df`, `sim.citation("bibtex")`, `client.batch_run(["scenario A", "scenario B"])`. Add `sdk/python/examples/` with 3 Jupyter notebooks: `01_basic_run.ipynb`, `02_batch_comparison.ipynb`, `03_export_and_cite.ipynb` — each a self-contained demonstration.
3. In `pyproject.toml`, set `requires-python = ">=3.9"`, `dependencies = []` (zero hard deps), `[project.optional-dependencies] pandas = ["pandas>=1.3"]`, `[project.urls] Homepage = "https://github.com/aaronjmars/MiroShark"`. Add a GitHub Actions workflow `sdk/python/.github/workflows/test-sdk.yml` that runs on PRs: installs the package, runs `sdk/python/tests/` (a unittest suite covering `start()`, `poll()`, `batch_run()` against a mock HTTP server using `http.server`). In `README.md` (main repo), add a "Python SDK" section below the "For Researchers" section: one-liner install command, two-line usage example, link to `sdk/python/README.md`. In `docs/API.md`, add a "Python SDK" section at the top of the file: "If you're using Python, the `miroshark-sdk` package wraps all API endpoints. Manual HTTP requests are only needed for languages other than Python." Cross-reference the SDK in `docs/research-guide.md` as the recommended interface for programmatic research workflows.

### 5. OpenMetrics / Prometheus Export Endpoint

**Type:** Integration / Performance
**Effort:** Small (hours)
**Impact:** A MiroShark instance running in production has no structured observability surface. Operators using Grafana, Prometheus, Datadog Agent, or any scrape-based monitoring stack must treat MiroShark as a black box — they can check if the process is alive, but they can't see simulation throughput, model usage distribution, or consensus rates without writing custom log parsers. The `GET /metrics` endpoint emits Prometheus-format text metrics (the de facto standard for operator observability, supported by every monitoring stack): `miroshark_simulations_total{status, model}` (counter), `miroshark_simulations_active` (gauge), `miroshark_simulation_duration_seconds` (histogram: 0.5, 2, 5, 10, 30 minute buckets), `miroshark_consensus_reached_total{outcome}` (counter, labeled yes/no/neutral/none), `miroshark_agents_per_simulation` (histogram), `miroshark_api_requests_total{endpoint, method, status_code}` (counter), `miroshark_webhook_deliveries_total{status}` (counter). Output is `text/plain; version=0.0.4` — the Prometheus text exposition format is 4-field plain text, implementable in 60 lines of Python with zero dependencies (no `prometheus_client` library needed). The endpoint is authentication-optional: protected by `METRICS_TOKEN` env var (passed as `Bearer` token) if set, open if unset. Ships with `monitoring/grafana-dashboard.json` — a pre-built 6-panel Grafana dashboard (simulation throughput, model usage donut, consensus rate gauge, p50/p95 duration histogram, active count, error rate) as a portable JSON file operators can import in one click. For the DevOps/infrastructure-focused fork segment — operators running MiroShark in production for their team, startup, or research lab — this is the observability primitive they'd add on day one of deployment. For MiroShark itself, it opens the door to community-contributed monitoring configs, which is a discovery surface for operator-facing tooling.
**How:**
1. Add `GET /metrics` as a Flask endpoint with `Content-Type: text/plain; version=0.0.4; charset=utf-8`. Derive all metric values from existing data sources: simulation counts from `<data_dir>/*/config.json` and `<data_dir>/*/complete.json` files (scan once at startup, then maintain in-memory counters updated by the existing simulation runner hooks — no new database table needed). Implement `MetricsRegistry` in `backend/app/services/metrics.py` as a simple in-memory dict of `{metric_name: value}` updated by events. The simulation runner calls `metrics.increment("simulations_total", labels={"status": "completed", "model": config.model})` at each lifecycle transition. The histogram metrics (`duration_seconds`, `agents_per_simulation`) are maintained as running sum/count/bucket arrays — no external library. Format as Prometheus text: `miroshark_simulations_total{status="completed",model="gpt-4o"} 42`. Add `METRICS_TOKEN` to `.env.example` with a comment: "If set, `GET /metrics` requires `Authorization: Bearer <token>`. Leave unset for open metrics (suitable for internal networks)."
2. Add a `monitoring/` directory at the repo root. In `monitoring/grafana-dashboard.json`, define a 6-panel Grafana dashboard: (a) "Simulations / hour" — time series on `rate(miroshark_simulations_total[1h])`; (b) "Model usage" — pie chart on `miroshark_simulations_total` grouped by `model` label; (c) "Consensus rate" — gauge on `miroshark_consensus_reached_total{outcome=~"yes|no"} / miroshark_simulations_total{status="completed"}`; (d) "Duration p50/p95" — stat panels on `histogram_quantile(0.5, ...)` and `histogram_quantile(0.95, ...)`; (e) "Active simulations" — stat panel on `miroshark_simulations_active`; (f) "Webhook error rate" — stat panel on `miroshark_webhook_deliveries_total{status="failed"} / miroshark_webhook_deliveries_total`. Include `monitoring/prometheus.yml` — a minimal Prometheus scrape config: `scrape_configs: [{job_name: miroshark, static_configs: [{targets: ["localhost:5000"]}], metrics_path: /metrics}]`.
3. Add `GET /metrics` to the OpenAPI spec under a new "Observability" tag with a note: "Returns Prometheus text exposition format, not JSON. Content-Type: text/plain; version=0.0.4." In `docs/API.md`, add an "Observability" section documenting all exposed metric names, their labels, and how to configure Prometheus and Grafana. Include a minimal `docker-compose.monitoring.yml` snippet: `prometheus` and `grafana` services preconfigured to scrape MiroShark and import the dashboard. In `README.md`, add a "Monitoring" section (alongside "For Researchers" and "For Builders"): a one-paragraph description of the metrics endpoint with a link to `monitoring/grafana-dashboard.json` and the docker-compose snippet. Note explicitly: "The metrics endpoint exposes only aggregate counters and histograms — no simulation content, no user data, no scenario text." This is the operator reassurance that monitoring adoption doesn't require any privacy trade-off.

## Selection Rationale

Five angles distinct from all 20 ideas suggested across May 8, 10, and 12, and from the 5 on May 6, and from all 14 features currently built and awaiting push:

- **Coalition Detection** (#1) — The interaction network view currently shows wiring, not social structure. Community detection transforms it into an echo-chamber analyzer: "how many coalitions formed, how cohesive were they, was there an echo chamber?" This is the analytical layer that makes the network tab a standalone research tool rather than a decorative companion to the belief drift chart. Pure Python stdlib graph algorithm — zero new dependencies.
- **Lifecycle Webhooks** (#2) — The completion webhook fires once. Everything between simulation.created and simulation.completed is invisible to server-side automation. Mid-simulation events — especially `consensus_reached` — unlock time-sensitive automation patterns (react when consensus forms, not when the simulation finishes) that the current webhook architecture can't express. Extends PR #79's HMAC delivery infrastructure with zero new primitives.
- **Scenario Starter Kit** (#3) — The blank scenario field is the conversion killer. Every session starts with a blank text box and no guidance. An 80-scenario curated library with a shuffle button addresses cold-start abandonment for new users and provides a contribution surface for the community. The `GET /api/scenarios/library` endpoint makes it programmable for batch-testing workflows. The most direct DX improvement available that hasn't been suggested yet.
- **Python SDK** (#4) — Zero-dependency pip-installable client that wraps all API endpoints. The Jupyter export (PR #80, just merged) proves researchers are using MiroShark programmatically. The SDK is the next step: from "I can analyze results in Jupyter" to "I can run and retrieve simulations from Jupyter in 5 lines." `sim.trajectory_df`, `sim.citation("bibtex")`, `client.batch_run(scenarios)` — the three operations researchers actually need, without any HTTP boilerplate.
- **Prometheus Export** (#5) — No existing MiroShark endpoint tells operators what the platform is doing at runtime. Prometheus text format is 60 lines of pure Python and compatible with every monitoring stack (Grafana, Datadog, Uptime Robot, Fly.io metrics). The pre-built Grafana dashboard lowers the operator setup cost to one import click. For forks running in production, this is the observability primitive that signals "this project takes production deployments seriously."
