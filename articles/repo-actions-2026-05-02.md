## Ecosystem Context

MiroShark has crossed 970 stars with a feature set that now covers almost every surface a solo researcher or analyst could want: seven share surfaces (share card, replay GIF, transcript, trajectory CSV, public gallery, RSS feeds, and the incoming live watch page), three export formats (Markdown, JSON, CSV/JSONL), a completion webhook for Slack/Discord/Zapier, a Langfuse observability integration, and full zh-CN internationalization across 90 files. The platform layer is effectively complete.

The fork rate tells a clear story: 197 forks at 976 stars is a 20% fork-to-star ratio — nearly double the typical open-source project (10–12%). These aren't passive forks. The push-recap from May 1 showed 10 new forks in 24 hours, and Aaron's tweet "just build on aeon" signals the intent: convert passive forkers into active deployers. The activation gap is still wide — less than 1% of forks have modified `aeon.yml` — but the surface area is now large enough that the right set of features could close it.

What the repo needs in this phase isn't more share surfaces or integration breadth. It needs depth in three specific areas:

**1. Organizational primitives for repeat users.** Researchers running 10+ simulations have no container for their work. They can't group simulations into a named study, track belief trend across runs of the same topic, or export a bundled research dataset. The history page is a flat list. For anyone doing longitudinal work, MiroShark currently forces them to manage their own spreadsheet.

**2. API completeness for the automation audience.** The completion webhook and OpenAPI spec opened MiroShark to programmatic use. But two of the most distinctive features — the Trace Interview (agent interrogation) and mid-run belief state — have no API surface at all. The automation audience (n8n, GitHub Actions, research pipelines) can trigger simulations and receive final results, but can't query agents or monitor belief evolution mid-run.

**3. Discovery acceleration at the 1K threshold.** MiroShark is 24 stars from 1,000. The community template gallery is the growth lever that turns the existing "Preset Templates" feature into a viral discovery channel — each shared template is a domain-specific entry point that organic search can index.

These five ideas are distinct from all 10 ideas suggested on April 30 (batch runs, cross-narrative, checkpoint/resume, persona library, GH Actions action, Jupyter export, cost estimator, cross-simulation analytics, scheduling, n8n/Zapier templates) and do not overlap with the open PR.

### 1. Community Scenario Template Gallery

**Type:** Growth / DX
**Effort:** Small (hours)
**Impact:** MiroShark ships with 6 hardcoded preset templates (crypto launch, corporate crisis, political debate, product announcement, campus controversy, historical what-if). These are the fastest path to a first simulation, but they're a ceiling, not a ramp. A community template gallery lets any user save their simulation config as a named, tagged, shareable template — and browse templates published by others. At 197 forks and a growing academic user base, there are already users with domain-specific simulation designs worth sharing: DeFi governance vote, EU AI Act compliance debate, earnings call, token launch sentiment, central bank policy decision. Each shared template is a domain-specific entry point for organic search (e.g., "MiroShark template: EU AI Act debate") and a usage guide for new users who don't know what to simulate. The template gallery is the viral flywheel that turns the existing preset feature into a community-driven discovery channel — and the 24 stars to the 1K milestone make this timing matter.
**How:**
1. Add `POST /api/templates` to save any simulation config (scenario text, agent count, rounds, model preset) as a named template with tags and a short description. Store in a `templates` SQLite table: `id, label, description, tags (JSON), config_json, fork_count, published (bool), created_at`. Add `GET /api/templates` (list with tag filter), `GET /api/templates/:id` (detail), `POST /api/templates/:id/fork` (copies config into a new simulation launch), and `DELETE /api/templates/:id`. Add a "Save as Template" button in the simulation results toolbar — opens a modal with label, description, and tag multi-select (DeFi, Policy, Marketing, Crisis, Historical, Custom).
2. Add `/templates` page as a community gallery page mirroring the `/explore` gallery layout. Template cards show: label, description excerpt, tags as chips, agent count, round count, fork count, and a "Use template" CTA. "Use template" pre-fills the Step 1 launch form with the template's config — the user adds any overrides they want before launching. "Fork template" copies the template to the user's library for private modification. Include a tag filter chip row at the top and a search bar for keyword lookup across labels and descriptions.
3. Seed the gallery with 10 community templates beyond the existing 6 presets: US election cycle polling, EU AI Act compliance debate, token launch sentiment, corporate whistleblowing crisis, climate policy vote, sports team scandal, startup acquisition rumor, central bank rate decision, product recall crisis, scientific preprint controversy. Add a "Submit a template" link (GitHub issue template for community contributions). Add a "Templates" nav entry in the sidebar and a "Community Templates" row in the README features table. In the article generator, include a one-line footnote: "Template: [label]" when a simulation was launched from a saved template.

### 2. Fork / Counterfactual Diff Comparison View

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** The fork feature (PR #17) is the most intellectually powerful primitive in MiroShark — "what would happen if I injected this event at round 5?" — but there's no analytical surface for comparing fork vs. original. A user who forks a simulation currently has to open two tabs and visually compare charts. The diff view answers the research question directly: how much did belief distribution change? At what round did the fork diverge? Which agents shifted? This is the controlled-experiment interface that turns forking from an exploratory tool into a publishable research method. At 197 forks of the repo, a growing academic cohort, and the Director Mode (PR #31) and Counterfactual Branching already live, this is the feature that closes the loop on counterfactual research design. "We forked the simulation at round 5 with an event injection; belief divergence emerged at round 7, reaching a 23% delta by round 12" is a publishable finding. Currently MiroShark can produce that result but can't present it.
**How:**
1. Add `GET /api/simulation/:id/diff/:base_id` accepting two simulation IDs (fork and parent). Return: `{belief_drift_overlay: [{round, fork_yes, base_yes, delta}], final_price_delta, consensus_round_delta, divergence_round: first round where |fork_yes - base_yes| > 0.15, agent_overlap: [{agent_name, fork_stance, base_stance, agreement_score}] for top 5 agents present in both runs}`. Derive everything from existing `trajectory.json` and leaderboard data — no new storage. Add `GET /api/simulation/:id/diff/:base_id/export` returning the diff as a flat CSV: `[round, base_yes, fork_yes, base_no, fork_no, delta]` for import into Excel, R, or the Jupyter notebook export.
2. In the simulation history view, when a simulation has `forked_from` metadata, show a "Compare to original" button alongside "Fork" and "Share." On click, open a `DiffView.vue` overlay with a synchronized two-panel layout: overlaid belief drift chart (original lines vs. fork lines in distinct colors, with a vertical "Divergence marker" at the computed `divergence_round`), and a KPI delta row below: final price delta (green/red arrow + %), consensus round delta (how many rounds earlier/later fork reached consensus), and an agent overlap score (0–100% of top agents agreeing). Include a "Download diff CSV" button.
3. In the article generator, detect when the simulation is a fork (has `forked_from`) and inject the counterfactual framing into the prompt: "This simulation is a fork from [parent scenario excerpt]. Belief diverged from the baseline at round N, reaching a delta of X% by the final round." This makes the generated article automatically frame the fork as a controlled experiment result. Add a "Diff" tab alongside "Belief Drift" and "Interaction Network" in the results view — only visible when the simulation has a parent fork.

### 3. Programmatic Agent Interrogation API

**Type:** Integration / DX
**Effort:** Small (hours)
**Impact:** The Trace Interview (PR #26) is MiroShark's most differentiated capability: cross-examining an agent on its actual posts, trades, and stance changes rather than reconstructing intent post-hoc. It's also the only major feature with no API surface. Adding `POST /api/simulation/:id/agents/:agent_id/query` exposes this to automated pipelines. The GitHub Actions composite action (suggested Apr 30) can embed "Why did the top agent flip bearish at round 7?" in a PR comment. n8n and Zapier workflows can route agent interrogations to Slack. Researchers running batch studies can interrogate 50 agents with a fixed question set programmatically. This is the single API addition that unlocks the most differentiated MiroShark capability for the entire automation audience — and it reuses the exact context-building logic the UI modal already uses, so the implementation surface is small.
**How:**
1. Add `POST /api/simulation/:id/agents/:agent_id/query` accepting `{question: string, session_id?: string}`. Load the agent's trace from `<sim_dir>/posts/` filtered to this agent's posts, plus their belief state from `trajectory.json`. Construct the same grounded context the UI modal uses: agent biography, platform, archetype, per-round post excerpts, and stance-change events. Call the Smart LLM slot with the question + context. Return `{answer: string, agent_name, agent_archetype, citations: [{round, post_excerpt}]}`. If `session_id` is provided, cache the last 4 Q&A pairs in `<sim_dir>/interview_sessions/<session_id>.json` for context injection on subsequent calls (enabling multi-turn dialogue). Add a rate limit: max 20 queries per simulation per 5 minutes, returning `429 Too Many Requests` with `Retry-After` to prevent cost explosions from automation loops.
2. Add `GET /api/simulation/:id/agents` returning the full agent list for a completed simulation: `[{agent_id, name, archetype, platform, final_stance, influence_rank}]`. This is the prerequisite endpoint — a caller needs agent IDs before querying. Add `GET /api/simulation/:id/agents/:agent_id` for the full agent profile including biography and per-round belief history. Document both endpoints in the OpenAPI spec under a new "Agent Interrogation" tag, with 8 example questions in the spec description: "Why did you change your stance in round X?", "Which agent influenced you most?", "What evidence would have changed your mind?", "Summarize your reasoning arc across all rounds."
3. Add `GET /api/simulation/:id/interview-sessions` listing all session IDs with their Q&A history — useful for debugging automation pipelines and reviewing past interrogations. In the README integrations table, add a "Agent Interrogation API" entry pointing to the new OpenAPI spec tag. In the article generator, optionally inject a "Key agent quote" from a pre-set interrogation question ("What was the most important factor in your final stance?") fired against the top-ranked agent — adds a direct agent voice to the generated article.

### 4. Mid-Run Belief Threshold Alert Webhooks

**Type:** Integration
**Effort:** Small (hours)
**Impact:** The completion webhook (PR #46) fires once when a simulation ends. For monitoring-oriented use cases — traders watching simulations as early-signal tools, researchers detecting critical dynamics mid-run — waiting for completion is too late. A mid-run belief alert fires the moment agent consensus crosses a user-defined threshold: "notify me when bearish agents exceed 60%", "alert when YES consensus reaches 0.80", "fire when belief split flips from bullish-majority to bearish-majority." For a 15-round simulation where 80% bearish consensus appears at round 3, the signal arrives 12 rounds before the completion webhook would fire. Combined with Director Mode (PR #31), belief alerts enable adaptive simulation design: inject a breaking news event only when consensus reaches a critical threshold. MiroShark already has per-round belief data and the webhook dispatch infrastructure from PR #46 — this feature wires them together.
**How:**
1. Add a `belief_alerts` field to the simulation launch config: `[{metric: "yes_price" | "bullish_pct" | "bearish_pct", operator: ">" | "<" | ">=", threshold: float, webhook_url: string, label: string, fire_once: bool}]`. Store alert configs in `<sim_dir>/alert_config.json`. In the simulation runner, after each round completes, evaluate all alert conditions against the round's belief summary. When a condition fires, dispatch a webhook payload (reusing `_dispatch_webhook` from PR #46) with `{event: "belief_threshold_reached", alert_label, metric, value, threshold, round, simulation_id, scenario, share_url}`. Track trigger state in `<sim_dir>/alert_state.json` to prevent duplicate fires when `fire_once: true`. Allow up to 5 alerts per simulation.
2. In the simulation launch UI (Step 2 – Configure), add a collapsible "Mid-run alerts" section below the existing completion webhook URL field. "+ Add alert" opens an inline row: metric dropdown (YES price / Bullish % / Bearish % / Neutral %), operator dropdown (above / below), threshold input (0–100 for percentages, 0.00–1.00 for price), webhook URL field, and a label input. Show configured alerts as chips in the pre-run summary step. Add a "Test webhook" button that fires a dummy payload immediately so users can verify their endpoint before launching.
3. Add `GET /api/simulation/:id/alerts` returning all configured alerts with trigger status: `pending` (not yet fired), `fired` (with `{round, value, fired_at}`), or `never_triggered` (simulation completed without the threshold being crossed). In the simulation detail sidebar, show an "Alerts" section with each alert's current status and the live metric value (updated each round for running simulations). Add a "Clear alerts" button. Document the alert payload schema in the OpenAPI spec alongside the completion webhook — same consumer, different trigger event.

### 5. Simulation Series / Longitudinal Study Tracker

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** Individual simulations and the flat history page are at opposite ends of the organizational spectrum. Researchers running multi-simulation studies — "how has DeFi governance sentiment evolved across 8 simulations over 6 weeks?", "compare audience reaction to 4 product launch variants" — have no container for their work. They manage their own spreadsheets. A Study groups simulations into a named, curated series with a time-series belief dashboard and a bundled export. At 976 stars with a growing academic cohort, this is the organizational primitive that turns MiroShark from a one-off analysis tool into a longitudinal research platform. The Study is meaningfully different from the Cross-Simulation Analytics dashboard (suggested Apr 30) — that covers all-runs aggregate stats; a Study covers a curated, named subset of runs as a coherent research project. The difference is the same as a journal article's methodology section vs. a Google Analytics dashboard.
**How:**
1. Add `POST /api/studies` accepting `{label: string, description?: string, simulation_ids: []}`. Store studies in a `studies` SQLite table: `id, label, description, simulation_ids (JSON array), created_at`. Add `GET /api/studies` (list), `GET /api/studies/:id` (detail with full member simulation metadata), `PATCH /api/studies/:id/simulations` (add or remove simulation IDs by passing `{add: [], remove: []}`), and `DELETE /api/studies/:id`. In the simulation history view, add "Add to Study" to each completed simulation card — opens a dropdown of existing studies or a "Create new study" inline form.
2. Add `/studies/:id` detail page with a study-level dashboard: (a) a timeline line chart — `final_yes_price` (y-axis) vs. simulation date (x-axis) across all member simulations in chronological order, revealing trend direction over time; (b) a stacked area chart of belief distribution (bullish/neutral/bearish %) per simulation in chronological order — visualizes distribution evolution across the series; (c) a "Recurring agents" table showing agents who appear across multiple member simulations with their average final stance and consistency score. Below the charts: a sortable member-simulation table (date, scenario excerpt, final YES price, consensus round, quality score, top agent) with links to individual simulation detail pages.
3. Add `GET /api/studies/:id/export` returning a ZIP file containing: (a) `study_summary.csv` — one row per simulation (id, date, scenario excerpt, final_yes_price, consensus_round, top_agent, quality_score); (b) individual transcript JSON files for each member simulation; (c) `study_notebook.ipynb` — a Jupyter notebook pre-populated with study-level analysis cells (time-series price chart, belief evolution heatmap per simulation, recurring-agent consistency matrix). Add "Export study" button on the study page. In the article generator, allow the user to select a study as the context for a comparative article: "Across [N] simulations in this series, bullish consensus declined from X% to Y% over [timeframe]."

## Selection Rationale

Five distinct angles, none overlapping with the ten ideas from April 30 or the open spectator watch PR:

- **Community Template Gallery** (#1) — The 1K-star growth lever. Turns 6 hardcoded presets into a community-driven discovery channel. Each domain-specific template is an organic SEO entry point and a usage guide. At 197 forks with builders looking for a "just build on aeon" moment, templates lower the floor to under 30 seconds for a first simulation.
- **Fork Diff View** (#2) — Closes the loop on counterfactual research. Fork exists since PR #17; Director Mode since PR #31. The diff view is the missing analytical surface that turns forking from exploration into a publishable controlled experiment. "Belief diverged at round N with X% delta" is a method-section finding.
- **Agent Interrogation API** (#3) — Unlocks the most differentiated MiroShark capability for automation. Trace Interview (PR #26) is UI-only. One API endpoint turns it into a research pipeline primitive — automation can query agents, embed answers in PR comments, route to Slack, or run batch interrogations. Small lift; uniquely high leverage.
- **Mid-Run Belief Alerts** (#4) — The real-time monitoring primitive. Completion webhooks arrive too late for signal-oriented use cases. Belief threshold alerts fire during the simulation — 12 rounds earlier for a threshold reached at round 3 of 15. Wires existing per-round belief data to existing webhook dispatch. Small implementation; changes the use-case category for monitoring-oriented users.
- **Study Tracker** (#5) — The longitudinal research container. Cross-Simulation Analytics (suggested Apr 30) covers all-runs aggregate stats; Study covers a curated, named series as a research project. The difference matters for researchers who need to scope their analysis to a defined set of simulations and export a bundled dataset. The study-level Jupyter notebook export extends the per-simulation notebook (already built) into a multi-run comparative research artifact.
