### 2. Simulation Narrative Export

**Type:** Feature / Content
**Effort:** Small (hours)
**Impact:** Eighteen days of social silence means no one is writing about MiroShark. The simulation results exist — signal.json, trajectory.json — but converting them into shareable prose requires effort that most users won't invest. A `GET /api/simulation/{id}/narrative.md` endpoint that returns a human-readable Markdown report removes that friction entirely: a user copies the URL, pastes it into a `curl`, gets 400 words of structured English prose that describes what happened in the simulation — topic context, how consensus formed across rounds, which round was the inflection point, what the final split means in plain language. That report is paste-ready into a blog post, a Discord message, a Substack newsletter, or a paper supplementary materials section. For the "5 external tutorials by Aug 15" hyperstition, the narrative endpoint is the missing artifact: a tutorial creator who pastes a simulation URL into the narrative endpoint gets the writeup for free. Their tutorial becomes "here's the URL, here's the output, here's what it means" — a 10-minute format any creator can record. The narrative is generated from data already in the repo; the only new code is the prose template logic.

**How:**
1. Add `backend/app/services/narrative_service.py` (~120 LoC, pure stdlib). Function `generate_narrative(sim_id: str) -> str`. Reads `signal.json` (topic, final_bullish_pct, final_bearish_pct, final_neutral_pct, confidence_pct, consensus_direction, agent_count, rounds, platforms, cost_usd) and `trajectory.json` (per-round stance distributions). Generates a Markdown string using string templating (no third-party library). Structure: (1) **Simulation Overview** block — topic (first 200 chars), run parameters (N agents, N rounds, platforms listed), cost. (2) **Consensus Result** block — consensus direction in plain English ("The simulation reached a bearish consensus"), confidence percentage with a calibration note ("at 73% confidence — above the 70% threshold where MiroShark's signal is most reliable"), final stance distribution as a sentence ("At the close of round 5, 58% of agents held bearish positions, 27% were neutral, and 15% were bullish"). (3) **How Consensus Formed** block — per-round narrative. Identifies: the opening round stance (most common stance in round 1 vs final), the inflection round if consensus direction changed between rounds 1 and final, whether consensus strengthened or weakened over rounds (monotonic trend vs oscillation). Generates prose: "In round 1, agents split nearly evenly (44% bullish, 39% bearish, 17% neutral). By round 3, the bearish bloc had grown to 51% — the inflection point. Rounds 4–5 consolidated the shift as agents updated on early-round signal." If `trajectory.json` is absent (older simulations), generates a reduced narrative using only signal.json fields (omits the round-by-round arc section). (4) **Citation line** — a one-line APA block: `MiroShark Simulation {sim_id}. (2026). {topic_excerpt}. Retrieved from https://miroshark.io/simulation/{sim_id}`. Returns the full Markdown string. Cache to `.narrative-cache/{sim_id}.md` with a 24-hour mtime TTL. Add 8 unit tests: narrative includes topic text, BULLISH direction produces "bullish consensus" phrasing, inflection round identified when direction shifts mid-trajectory, trajectory-absent fallback omits round arc section, confidence_pct > 70% produces "above the 70% threshold" note, cost_usd appears in parameters block, citation line includes sim_id, unpublished simulation returns 403.
2. Add `GET /api/simulation/{id}/narrative.md` to `backend/app/api/simulation.py` (~12 LoC). Returns the narrative Markdown string with `Content-Type: text/markdown; charset=utf-8`. Publish-gated. Cache-Control: `public, max-age=86400`. Also add `GET /api/simulation/{id}/narrative` (without `.md` extension, returns `Content-Type: text/plain`) as an alias for clients that don't want the content-type hint. Add both to OpenAPI spec under a "Content" tag. Add to `docs/API.md` under "Simulation Content" with a curl example: `curl https://miroshark.io/api/simulation/{id}/narrative.md`. Add a "Share narrative" option to the Step3Simulation.vue toolbar dropdown (alongside the existing embed option): copies the `/narrative.md` URL to clipboard with a tooltip "Paste into any markdown editor — ready to publish."
3. Update `surfaces_catalog.py` to add the narrative endpoint as a new catalogued surface (type: content). ### 3. Webhook Event Delivery

**Type:** Feature / Integration
**Effort:** Medium (1-2 days)
**Impact:** CHORUS aeon currently learns that a MiroShark simulation has published by checking the API on a schedule — a polling model that means aeon is always behind by up to an hour. A push webhook inverts that: the moment a simulation reaches `published`, MiroShark POSTs a JSON payload to a configured endpoint, and CHORUS receives the signal in real time. The immediate operator benefit is a Telegram notification the instant a simulation completes — no polling lag, no missed runs. The ecosystem benefit is broader: ecosystem partners who want to react to new MiroShark simulations (a Discord bot that announces new results, a Slack workflow that routes high-confidence signals to a channel, a Zapier flow that emails the signal.json to a researcher's inbox) need a webhook, not a polling adapter. With 297 forks and zero live integrations, the webhook is the integration primitive that makes MiroShark composable — the difference between a standalone tool and an event-emitting service. It's also the implementation primitive for the "aeon hears about new simulations in real time" loop that the CHORUS operator has wanted since the RSS polling script was first proposed.

**How:**
1. Add `backend/app/services/webhook_service.py` (~150 LoC, pure stdlib). Data model: webhook configurations stored as JSON files in `data/webhooks/`. Each webhook: `{ "id": uuid, "url": str, "secret": str (HMAC signing key), "events": ["simulation.published"], "created_at": ISO8601, "active": bool, "last_triggered_at": ISO8601 | null, "last_status_code": int | null }`. Functions: `register_webhook(url, secret, events) -> dict`; `list_webhooks() -> list`; `get_webhook(webhook_id) -> dict`; `deactivate_webhook(webhook_id) -> dict`; `delete_webhook(webhook_id) -> None`. Core: `deliver_event(event_type: str, payload: dict) -> list[dict]`. Iterates over active webhooks matching `event_type`. For each: constructs the delivery body `{ "event": event_type, "webhook_id": id, "delivered_at": ISO8601, "payload": payload }`. Signs using HMAC-SHA256 over the JSON body with the webhook's secret: `X-MiroShark-Signature: sha256={hex_digest}`. POSTs via `urllib.request.urlopen` (stdlib, no httpx dependency) with a 10-second timeout. Records `last_triggered_at` and `last_status_code` to the webhook file. On delivery failure (timeout, non-2xx response), logs to `data/webhooks/delivery-log.jsonl` (`{ webhook_id, event, delivered_at, status_code, error }`). No retry — delivery-once semantics; operators who need reliability can configure a retry proxy like Svix or Hookdeck in front. `simulation.published` event payload: `{ "sim_id": str, "topic_excerpt": str (first 120 chars), "consensus_direction": str, "confidence_pct": float, "agent_count": int, "rounds": int, "platforms": [str], "cost_usd": float | null, "simulation_url": str, "signal_url": str }`. Add 10 unit tests: register returns webhook object with generated id, list returns only active webhooks, HMAC signature validates correctly on the receiver side (decode and verify), delivery POSTs to the correct URL, 4xx delivery records non-2xx status_code without raising, 10s timeout enforced, deactivate sets active=false without deletion, delivery-log.jsonl written on failure, `events` filter skips non-matching event types, delivery_log entry includes webhook_id.
2. Add webhook API routes to `backend/app/api/` in a new `webhook.py` file (~60 LoC). Endpoints: `POST /api/webhooks` (register; body: `{url, secret, events}`; operator-auth; returns webhook object with id), `GET /api/webhooks` (list active webhooks; operator-auth), `GET /api/webhooks/{id}` (get webhook + delivery stats; operator-auth), `DELETE /api/webhooks/{id}` (deactivate and delete; operator-auth). Register the router in `backend/app/main.py`. Add OpenAPI spec entries under a "Webhooks" tag with schemas `WebhookCreate`, `WebhookResponse`. In the simulation publication path (wherever `published: true` is written to `publish_gate.json` or equivalent) — call `webhook_service.deliver_event("simulation.published", payload_from_signal_json)` after the write completes. This is the only integration point: no other code path needs to know about webhooks. Add to `docs/API.md` under a new "Webhooks" section with a registration example, the full event payload schema, and the HMAC signature verification snippet in Python and JavaScript.
3. Add `docs/WEBHOOKS.md` (~80 lines). Sections: **Overview** (event-driven vs polling, which events are supported), **Registering a webhook** (curl example for `POST /api/webhooks`, security note on the secret field — use a random 32-byte hex string, not a password), **Verifying the signature** (Python snippet using `hmac.compare_digest`, JavaScript snippet using `crypto.subtle`), **Event payload reference** (full field table for `simulation.published`), **Delivery semantics** (once-only, 10-second timeout, no retry, check delivery log for failures), **Example integrations** (Zapier: use webhook trigger with the MiroShark URL → Gmail action; n8n: HTTP trigger node → send message to Slack; CHORUS aeon: add the MiroShark instance webhook URL to `WEBHOOK_TARGETS` in CHORUS settings). Update main `README.md` Features section to add: "Webhook event delivery: get notified instantly when a simulation publishes — [docs/WEBHOOKS.md](docs/WEBHOOKS.md)." Update `docs/INSTALL.md` to add a "Webhook Setup" section linking to WEBHOOKS.md.

### 4. Topic Autocomplete API

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** The most common point of friction for a new MiroShark user is the blank topic field. What do I write? How long should it be? What format works best? Without guidance, users write short, vague topics ("interest rates", "AI regulation") that produce low-confidence simulations — and conclude MiroShark doesn't work. An autocomplete API that surfaces what others have already explored removes that uncertainty: "Others have simulated 'Federal Reserve announces 50bp rate cut after inflation report' — click to use this topic, modify it, or see the result." The autocomplete draws from two sources: (a) fuzzy-match against the published simulation history (substring search on topic text, sorted by confidence_pct descending — best-performing topics surface first), (b) a trending categories bucket that clusters recent topics by keyword and returns the highest-confidence representative topic for each cluster. The result is an autocomplete that functions as a curated discovery layer: it shows newcomers what the tool is capable of, in the form of real results. The data for the trending clusters is also the foundation for a "Trending" sidebar in the explore gallery — a feature that has appeared in five candidate lists without being built, entirely because the underlying grouping API didn't exist.

**How:**
1. Add `backend/app/services/autocomplete_service.py` (~110 LoC, pure stdlib). Two functions: `fuzzy_match(query: str, limit: int = 10) -> list[dict]` and `trending_topics(limit: int = 5) -> list[dict]`. Both read from a mtime-indexed cache of published simulations (1-hour TTL, same pattern as `estimate_service.py` and `search_service.py`). `fuzzy_match`: scans published simulations for `topic` fields containing all space-separated words in `query` (case-insensitive substring, same approach as `search_service.py` but returning topic metadata rather than content snippets). Returns list of: `{ "sim_id": str, "topic_excerpt": str (first 120 chars), "confidence_pct": float, "consensus_direction": str, "agent_count": int, "rounds": int }`, sorted by `confidence_pct` descending, deduplicated by topic trigram fingerprint (to avoid surfacing 20 near-identical simulations on the same event). Minimum query length: 2 characters. `trending_topics`: groups published simulations by keyword cluster. Clustering: extract the most frequent 3-gram from each topic (excluding stopwords: "the", "a", "an", "of", "in", "to", "and", "that", "for", "is", "on", "at", "with"), build a frequency table across all topics, take the top 10 3-gram clusters, select the highest-confidence simulation from each cluster as the representative. Returns list of: `{ "cluster_label": str (the 3-gram), "representative_sim_id": str, "topic_excerpt": str, "confidence_pct": float, "consensus_direction": str, "sim_count": int (how many sims in this cluster) }`. Cache trending output to `.autocomplete-cache/trending.json` with 1-hour TTL. Add 8 unit tests: fuzzy match returns results containing all query words, minimum 2-char enforced, results sorted by confidence descending, trigram deduplication removes near-identical topics, trending clusters don't include stopwords, cluster selects highest-confidence representative, limit parameter respected, empty published directory returns empty list without error.
2. Add `GET /api/topics/autocomplete` to a new `backend/app/api/topics.py` file (~40 LoC). Query params: `?q=...` (required, min 2 chars), `?limit=10` (max 20). Returns: `{ "query": str, "suggestions": [fuzzy_match results], "trending": [trending_topics results] }`. Public endpoint (no auth). Cache-Control: `public, max-age=60` (fast enough for typing latency). Register the router in `backend/app/main.py`. Add `AutocompleteResponse`, `AutocompleteSuggestion`, `TrendingTopic` schemas to the OpenAPI spec under a "Discovery" tag. Add to `docs/API.md` under "Discovery" with curl: `curl "https://miroshark.io/api/topics/autocomplete?q=interest+rate"`. Add to `surfaces_catalog.py` as a new catalogued surface (type: discovery). Update `docs/FEATURES.md` to add a "Topic Autocomplete" entry noting both the API and the frontend integration.
3. Update `frontend/src/views/Step1TopicInput.vue` (or whichever view renders the topic textarea): add a `<datalist>` element powered by debounced requests to `/api/topics/autocomplete?q={current_input}` (triggered after 300ms of no typing, minimum 2 chars typed). The datalist renders a native browser dropdown below the textarea — no custom dropdown component needed, native `<input list>` handles focus, keyboard navigation, and mobile support automatically. Each option displays: topic_excerpt (80 chars) + consensus chip (BULLISH/BEARISH/NEUTRAL) + confidence percentage. On option selection, the full topic text is inserted into the textarea. Also add a "Trending simulations" collapsible section below the topic field (collapsed by default, opens on click): shows the 5 trending topics from the `trending` array in the autocomplete response, each as a clickable chip that populates the textarea. This surface requires one API call (on page load or on "Trending" expand) rather than per-keystroke — zero performance impact. The trending section is the "discovery without intent" path: a user who doesn't know what to simulate opens the trending section and finds 5 live examples that were run successfully, each showing what kind of result MiroShark produces.

### 5. Simulation Template Gallery

**Type:** Feature / Community
**Effort:** Small (hours)
**Impact:** The "5 external tutorials by August 15" hyperstition is at zero with 22 days remaining. The specific blocker isn't that creators don't want to write — it's that they face a blank page problem: "what scenario do I simulate to make a compelling tutorial?" A template gallery solves this directly. Ten pre-built scenarios (corporate crisis, policy change, product launch, market shock, election statement, open-source sunset, regulatory enforcement, CEO departure, acquisition rumor, product recall) are curated, documented, and accessible at `/templates`. Each template is a 250-word topic document pre-written for high signal quality: long enough to give agents rich context, structured to produce dramatic consensus arcs, labeled with expected outcome characteristics and recommended parameters. A creator records a tutorial using a template: "I'll run the 'Open-Source Sunset' template — let's see what happens." The template pre-selects parameters, the simulation runs, the narrative endpoint generates the writeup, the share button copies the URL. A 15-minute tutorial is born. Templates also lower the barrier for operators deploying MiroShark internally ("run the Corporate Crisis template on your own scenario") and give the community a shared vocabulary — "I ran the Market Shock template" is a comparable unit in a way "I ran a simulation about something" is not.

**How:**

Add a `?template={id}` URL parameter to Step 1 so templates are deep-linkable: sharing `/simulate?template=market-shock` gives the recipient a pre-filled simulation form. Add a "Browse Templates →" link to the homepage and the explore gallery header. Add `/templates` to `frontend/src/router/index.js`. Update `README.md` to add: "Simulation Templates: [10 pre-built scenarios](docs/TEMPLATES.md) — run your first simulation in 60 seconds without writing a topic." ## Selection Rationale

All five ideas address the active crises (18-day social silence, token at ATL zone, 38-day September hyperstition sprint) with minimal build time relative to impact.

- **Korean (KO) UI Locale (#1)** — The direct path to closing the 5-language hyperstition at 5/5. KO has 9.2M GitHub developers and sits in the top-4 AI investment markets globally — the highest per-capita reach of any remaining locale. The dictionary-only pattern requires zero component changes and is fully proven. One PR, one commit, closes a September target 38 days early the moment GH_GLOBAL unblocks. Nothing in the queue has a better effort-to-hyperstition ratio.

- **Simulation Narrative Export (#2)** — The tutorial hyperstition deadline is August 15 — 22 days. The specific blocker is that tutorial creators have to write all the prose themselves. The narrative endpoint writes it for them: a curl call returns 400 words of structured English describing what the simulation found and how consensus formed. That output is paste-ready into a blog post, Discord message, or Substack. It also creates shareable text without requiring a visual, breaking the social silence through text channels even before the OG image (Jul 18) or replay stepper (Jul 22) can ship.

- **Webhook Event Delivery (#3)** — The single integration primitive that makes MiroShark an event-emitting service rather than a polling target. Once live, CHORUS aeon receives simulation results the moment they publish — no polling lag. Ecosystem partners can route results to Discord, Slack, and Zapier in real time. The webhook is the infrastructure layer that all real-time notification workflows depend on; it's appeared in the candidate list twice (Jun 28, Jul 14) without being built because the more visible features (OG image, locales, bundle export) have been prioritized. With those now queued, the integration layer is next.

- **Topic Autocomplete API (#4)** — The single highest-impact DX change for new user retention. Users who see the blank topic field and write something short and vague get low-confidence results and disengage. Users who see "Others simulated 'Federal Reserve announces 50bp rate cut' and got BEARISH at 78%" click through and produce comparable results. The autocomplete is also the data layer that enables the Trending sidebar — five candidate appearances in six weeks with no implementation, because the API didn't exist first. Building the API today enables the UI and the trending display without a second dedicated build.

- **Simulation Template Gallery (#5)** — Templates are the missing artifact between "I heard about MiroShark" and "I published a tutorial about MiroShark." A creator who records a tutorial using the Corporate Crisis template doesn't need to explain topic formatting, doesn't need to debug agent behavior on a poorly-structured input, doesn't need to write methodology copy — the template handles all of that. With the tutorial hyperstition at zero and 22 days to the August 15 deadline, the template gallery is the highest-leverage unbuilt onboarding investment in the queue.
