### 1. i18n Language Contribution Kit

**Type:** Community / DX
**Effort:** Small (hours)
**Impact:** The "5 languages by September 1" hyperstition is at 3/5. French proved the pattern: one dictionary file, `i18n.js` update, `LocaleToggle.vue` cycling, zero changes to the 34+ component files. The barrier to language #4 is not technical complexity — it is the absence of a step-by-step guide. A contributor who speaks German, Spanish, Japanese, or Portuguese cannot easily answer: what files do I touch, what format do the keys need, how do I test locally? A language contribution guide with an annotated starter template reduces this from "read the French PR diff" to "follow these 4 steps." Unlike the surface guide (Jul 02), this targets non-developer contributors: translators, bilingual users, and fork holders who are not Python engineers. The i18n contribution path requires only text editing — the lowest technical barrier in the entire codebase.

**How:**
1. Create `docs/CONTRIBUTING-I18N.md` (~150 lines). Structure: (1) "How i18n works in MiroShark" — the dictionary-only architecture: `frontend/src/locales/{lang}.js` exports a flat object of `{key: translation}` pairs; `i18n.js` maps locale keys to their dictionaries; `LocaleToggle.vue` cycles through locales; no changes to component files. (2) "Step-by-step: add a new language in 3 files." File 1: `frontend/src/locales/{lang}.js` — duplicate `fr.js` as the starting point (it has all 1,627 keys); translate each value; file should export a default object. File 2: `frontend/src/i18n.js` — add the new locale key (`'de'`, `'es'`, `'ja'`) to the `locales` array and import the new dictionary. File 3: `frontend/src/components/LocaleToggle.vue` — add the new locale button to the cycle. (3) "How to test locally" — `npm run dev` in `/frontend`; cycle through locales with the toggle; check that every page renders without the key name showing (key names appearing verbatim mean a missing translation). (4) "Checklist before opening a PR" — 6 items: all 1,627 keys translated (no untranslated English values left), locale key added to `i18n.js`, toggle updated, no component files changed, `npm run dev` shows translated UI without visible key names, PR title format: `feat(i18n): [Language Name] ([locale code]) frontend translations`. (5) "Language candidates" — list top 5 most-starred languages in MiroShark's fork geography: Spanish, German, Japanese, Portuguese, Korean. Note that formal register (vous/Sie/usted) vs informal is a translation decision to make consistently.
2. Create `frontend/src/locales/_template.js` (~30 LoC) — a copy of `fr.js` with all values replaced by their English originals. A contributor opens this file, translates each value, renames it to their language code, and submits. The template has a comment at the top: `// Translate each value below. Do not change the keys. See docs/CONTRIBUTING-I18N.md.` and a footer comment: `// Total keys: 1627. If your translation is missing keys, check fr.js for the full list.` This eliminates the need to find the French file, understand its structure, and figure out what the English originals are.
3. File a GitHub issue in MiroShark titled "Call for translators: help bring MiroShark to 5 languages" with labels `good first issue` and `i18n`. Body: "We're at 3 languages (EN, ZH-CN, FR). We'd like to reach 5 by September 2026. The contribution is text-only — no Python, no backend knowledge needed. See [docs/CONTRIBUTING-I18N.md] for a step-by-step guide and a starter template in [frontend/src/locales/_template.js]. Language candidates: Spanish (es), German (de), Japanese (ja), Portuguese (pt), Korean (ko). One PR per language. If you want to claim a language, comment below — we'll coordinate to avoid duplicates." Add a "Community" section to the main `CONTRIBUTING.md` pointing to this guide: "Translation contributions require no coding background — see [docs/CONTRIBUTING-I18N.md]."

### 2. Simulation RSS / Atom Feed

**Type:** Feature / Growth
**Effort:** Small (hours)
**Impact:** `GET /api/simulations/feed.rss` returns the 20 most recently completed and published simulations as a standards-compliant RSS 2.0 feed. Each item: `<title>` = topic, `<description>` = direction + yes_pct + confidence_pct + agent_count + round count, `<link>` = canonical simulation URL, `<pubDate>` = completed_at, `<guid>` = simulation ID (permanent). This is a passive distribution surface that operates without any operator action after initial setup. RSS feeds are indexed by Feedly (4M+ users), Inoreader, and similar aggregators — a MiroShark feed on Feedly means every published simulation reaches subscribers automatically. It also makes MiroShark submittable to AI newsletter aggregators, Planet Python, and dev.to RSS-based discovery feeds. For researchers: an RSS feed is citable as a live data source in a paper's methodology section. Pure stdlib XML generation — no external dependencies, no new data model, no write paths. Scans existing published simulation directories. The endpoint also addresses "show proof the platform is active" for new visitors: the feed's last-build date is a live signal that the platform is producing results.

**How:**
1. Add `backend/app/services/rss_service.py` (~100 LoC, pure stdlib). `build_feed(data_dir, limit=20)` scans `data/simulations/*/state.json` for published, completed simulations (publish_date and completed_at both set), sorts by completed_at descending, takes top `limit`. For each simulation: reads `signal.json` (topic, direction, yes_pct, confidence_pct, agent_count, num_rounds) and `simulation_config.json` (sim_id). Generates RSS 2.0 XML with `xml.etree.ElementTree`: channel title = "MiroShark Simulations", channel description = "Latest AI agent opinion simulations", channel link = `{BASE_URL}`, channel `lastBuildDate` = RFC 822 datetime of most recent sim, `ttl` = 15 (15 minute cache). Each item: `title` = topic, `description` = `"{direction} · {yes_pct:.0f}% consensus · {agent_count} agents · {num_rounds} rounds · {confidence_pct:.0f}% confidence"`, `link` = `{BASE_URL}/simulation/{sim_id}`, `guid isPermaLink="false"` = sim_id, `pubDate` = RFC 822 completed_at. Returns the XML string. Writes to `data/.rss_cache.xml` with mtime check (15-minute cache — same window as trending). Route: `GET /api/simulations/feed.rss` (unauthenticated; `Content-Type: application/rss+xml; charset=utf-8`; `Cache-Control: public, max-age=900`). Also serve `GET /api/simulations/feed.atom` as Atom 1.0 (same data, different XML schema). Add 10 unit tests: feed contains correct number of items (≤ limit), item title matches topic, item description contains direction and yes_pct, guid matches sim_id, unpublished sims excluded, incomplete sims excluded, lastBuildDate is most recent completed_at, Cache-Control header present, Content-Type is rss+xml, 15-minute mtime cache hit returns cached feed.
2. Add an `<link rel="alternate" type="application/rss+xml" title="MiroShark Simulations" href="/api/simulations/feed.rss">` tag to `frontend/index.html` — browser RSS autodiscovery. Modern browsers and RSS readers scan HTML for this tag; when a user visits the MiroShark homepage with an RSS reader extension installed, it detects the feed automatically. Add a small RSS icon (🗞 or SVG) to the frontend footer/nav linking to `/api/simulations/feed.rss` — low-profile but visible to the users who care (developers, researchers, newsletter writers). No modal, no announcement — just the icon.
3. Add `GET /api/simulations/feed.rss` and `GET /api/simulations/feed.atom` to the OpenAPI spec under a "Discovery" tag. Add a "RSS / Atom Feed" section to `docs/FEATURES.md`: what the feed contains, the 15-minute cache, the 20-item limit, how to subscribe in Feedly/Inoreader (URL: `https://miroshark.xyz/api/simulations/feed.rss`). Add a one-liner to the README features table: "Live RSS/Atom feed of latest public simulations — subscribe in any RSS reader." Submit the feed URL to [Feedly's self-subscription page](https://feedly.com/i/subscription/feed%2Fhttps%3A%2F%2Fmiroshark.xyz%2Fapi%2Fsimulations%2Ffeed.rss) if Feedly is publicly accessible.

### 3. Simulation Comparison API

**Type:** Feature
**Effort:** Small (hours)
**Impact:** `GET /api/simulations/compare?ids=a,b,c` returns a structured JSON side-by-side comparison of signal data from multiple simulations. For operators who run the same topic across multiple dates or agent configurations, this answers: did consensus change? Which run had higher confidence? Did the direction flip? This is the "before/after" view that researchers need to build a case study — the concrete quantitative evidence that "I ran this simulation in March and again in July and the consensus shifted from SPLIT to BEARISH." The "public case study by September 1" hyperstition requires a mechanism for operators to produce specific, comparable results. The comparison endpoint is that mechanism. It also serves the "paper citation" hyperstition: a paper's methodology section can cite `GET /api/simulations/compare?ids=...` as a reproducible comparison protocol. The data model already exists in `signal.json` — this is a read-only aggregation of existing surfaces. Supports up to 10 simulation IDs per request (pagination is not needed; if comparing more than 10, the operator should filter first).

**How:**
1. Add `backend/app/services/comparison_service.py` (~120 LoC, pure stdlib). `build_comparison(data_dir, sim_ids)` takes a list of sim IDs (max 10; raises ValueError if more). For each ID: reads `data/simulations/{sim_id}/signal.json` (topic, direction, yes_pct, confidence_pct, agent_count, num_rounds, completed_at) and `data/simulations/{sim_id}/state.json` (publish_date — skips unpublished silently). Computes cross-simulation stats: `topic_match` (True if all topics are case-insensitively identical), `direction_consensus` (direction string if all agree, else "MIXED"), `yes_pct_range` (min, max, delta), `confidence_range` (min, max), `trend` ("improving" if yes_pct increases monotonically by completed_at order, "declining", "fluctuating", or "stable" if max delta < 5%). Returns `{simulations: [{sim_id, topic, direction, yes_pct, confidence_pct, agent_count, num_rounds, completed_at}], comparison: {topic_match, direction_consensus, yes_pct_range, confidence_range, trend}, total: N, generated_at: ISO}`. Route: `GET /api/simulations/compare` (unauthenticated for published sims; query param `ids=` comma-separated, max 10; `Cache-Control: no-store` — comparisons are per-request, no caching). Returns 400 if `ids` missing or empty, 400 if more than 10 IDs supplied, 200 with the structured response (silently excludes unknown or unpublished IDs; if all IDs are invalid, returns an empty simulations array with total=0). Add 10 unit tests: valid request returns 200, simulations array length matches valid IDs, unknown sim_ids silently excluded, unpublished sims excluded, topic_match True when all topics match, direction_consensus "MIXED" when directions differ, yes_pct_range delta correct, trend "improving" when yes_pct increases monotonically, >10 IDs returns 400, missing ids param returns 400.
2. In `Step3Simulation.vue`, add a "Compare" button to the simulation action bar (visible for published, completed sims). On click: open a modal that lets the operator paste up to 9 additional simulation IDs, then `GET /api/simulations/compare?ids={current},{...others}`. Display: a table with one row per simulation (topic, direction chip, yes_pct%, confidence%, completed_at), a summary row showing direction_consensus and yes_pct_range, and a trend label ("Consensus improving", "Consensus declining", "Fluctuating"). If `topic_match` is False: show a yellow warning banner "These simulations cover different topics — comparison may not be meaningful." Add `compareSimulations(ids)` to `frontend/src/api/simulation.js`. Bilingual labels ("Compare" / "对比", "Trend" / "趋势").
3. Add `GET /api/simulations/compare` to the OpenAPI spec under the "Simulation" tag. Schema: `ComparisonResponse` with nested `ComparisonMeta` (topic_match, direction_consensus, yes_pct_range, confidence_range, trend) and `ComparisonEntry` per simulation. Add a "Simulation Comparison" section to `docs/FEATURES.md`: what the endpoint computes, the 10-ID limit, the silent exclusion of unknown/unpublished IDs, and a worked example showing a 3-sim comparison that reveals a trend. Add a curl example to the README API table: `curl "https://miroshark.xyz/api/simulations/compare?ids=id1,id2,id3"`.

### 5. Community Showcase Page

**Type:** Community / Growth
**Effort:** Medium (1-2 days)
**Impact:** The "public case study by September 1" hyperstition requires two things: (1) a mechanism for operators to submit their story, and (2) a public surface that makes submitted stories visible. GitHub Discussions (idea #4) handles (1). A `/showcase` page handles (2). The showcase is a curated gallery of real-world decisions that were informed by MiroShark simulations — each entry: operator name or handle, the decision they were trying to make, the simulation they ran (topic, direction, confidence), and what they did with the result. Unlike the trending topics feed (Jul 02 — automated, based on frequency), the showcase is manually curated, operator-submitted, and story-driven. For new visitors: "281 forks" is a number. A showcase entry that says "I used MiroShark to decide whether to pivot my startup's go-to-market strategy before a $40K marketing commitment" is a story. Stories convert visitors into users at a rate numbers cannot match. The showcase also creates an SEO surface for searches like "AI simulation real world use case" or "opinion simulation business decision" — long-tail queries that the technical documentation cannot rank for.

**How:**
1. Define the showcase data format: `showcase/entries/{slug}.md` — a Markdown file with YAML frontmatter: `title`, `author` (GitHub handle or anonymized), `date`, `topic` (the simulation topic), `sim_id` (optional — links to the live simulation if public), `direction`, `yes_pct`, `confidence_pct`, `decision` (1-2 sentences: what decision were they making?), `outcome` (1-2 sentences: what did they do with the result?), `tags` (list). Add a JSON Schema file at `showcase/schema.json` for validation. Create two seed entries (from the operator team's own simulations — using real historical sim data from the repo's public simulations) to populate the page before any community submissions arrive. Seed entry 1: a startup go-to-market decision. Seed entry 2: an investment thesis check. Add a GitHub Issue template at `.github/ISSUE_TEMPLATE/showcase-submission.yml` (YAML-format issue template) with fields for: the decision, the simulation topic and result, a brief outcome description, public or anonymous. Add a note at the top of the template: "Submissions are reviewed and added to showcase/ by the maintainers. Anonymous entries are fine."
2. Add `backend/app/services/showcase_service.py` (~80 LoC, pure stdlib). `get_showcase(showcase_dir)` scans `showcase/entries/*.md`, parses YAML frontmatter (using `re` — no external YAML library), returns a list sorted by `date` descending. Each entry: `{slug, title, author, date, topic, sim_id, direction, yes_pct, confidence_pct, decision, outcome, tags}`. Caches to `showcase/.index.json` (1h mtime cache — showcase updates infrequently). Route: `GET /api/showcase` (unauthenticated; `Cache-Control: public, max-age=3600`). Returns `{entries: [...], total: N}`. Add 8 unit tests: entries sorted by date desc, YAML frontmatter parsed correctly, missing optional fields (sim_id) handled gracefully, 1h cache hit skips rescan, entries without `published: true` excluded (add a `published` flag to the frontmatter schema), total count correct, route returns 200, Cache-Control header present.
3. Add `ShowcaseView.vue` at `/showcase` route. Card layout: each entry shows `title`, `author` chip, `date`, a "Simulated" label with `topic` and direction chip + `yes_pct%`, and a two-panel "Decision" / "Outcome" description below. Link to the live simulation if `sim_id` is set (badge: "View simulation →"). CTA at top: "Have a decision you made with MiroShark? [Submit your story →](link to GitHub Issue template)". Empty state: "No showcase entries yet — [be the first](link to submission template)." Bilingual ("Showcase" / "案例展示"). Add "Showcase" to the main navigation. Add a one-liner to the README: "Real-world decisions informed by MiroShark simulations — [browse the showcase →](link)." The showcase link in the README is the anchor text that drives long-tail SEO for "MiroShark use cases."

## Selection Rationale

Two new hyperstitions filed today, both expiring September 1: 5 languages (at 3/5) and a public case study (at 0/1). These five ideas are almost entirely focused on creating the mechanisms that make those hyperstitions closeable.

- **i18n Language Contribution Kit** (#1) — French proved the pattern; this formalizes it. The guide + template + labeled issue creates three discovery paths for a translator who doesn't know where to start. Language #4 is one contributor away — the kit removes the "I don't know what to touch" barrier. Spanish, German, Japanese, and Portuguese all have fork holders; the question is whether they find the on-ramp.

- **Simulation RSS / Atom Feed** (#2) — Passive distribution at zero marginal cost. Every completed simulation hits Feedly and any subscribed newsletter automatically. RSS is also a research protocol: a paper can cite the live feed as a data source. The 15-minute cache is lightweight. No new data, no new write paths — pure aggregation of what already exists.

- **Simulation Comparison API** (#3) — The case study hyperstition needs operators to produce "before and after" evidence. A comparison API is the structured tool for that: "I ran this simulation three months apart and here's what shifted." Researchers building pre-prints need a citable comparison method. The endpoint is read-only, pure stdlib, and the data is already on disk in `signal.json`.

- **GitHub Discussions Activation & Seeding** (#4) — There is no community channel. A "Show & Tell" pinned thread is the lowest-friction mechanism for collecting public usage stories — operators can share in a comment without writing a blog post or filing an issue. The "Research & Papers" thread surfaces academic work before it's published. Three structured posts seed the conversation so the space does not open empty.

- **Community Showcase Page** (#5) — A curated gallery of real decisions informed by MiroShark simulations. Stories convert visitors at a rate numbers cannot. The submission path (GitHub Issue template) is intentional — it creates a paper trail and makes showcase curation a maintainer action, not an automated process. The showcase page is also the long-tail SEO surface that the technical docs cannot cover: "AI simulation real-world use case", "opinion simulation business decision."
