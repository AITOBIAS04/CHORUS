### 2. Simulation Embed Widget

**Type:** Feature / Integration
**Effort:** Small (hours)
**Impact:** `EmbedDialog.vue` already exists — it's MiroShark's 41-surface showcase, generating curl examples and API links for every data endpoint. What it doesn't provide is a self-contained embeddable artifact: a single URL that renders a styled result card a developer can drop into any webpage with one `<iframe>` tag or one `<script>` include. The distinction matters for the external deployment hyperstition: an operator who copies a curl URL is using the API; an operator who embeds a `<miroshark-card>` on their personal site is running a public integration. This is the difference between a developer tool and a distribution mechanism. The embed widget makes MiroShark's output portable — a simulation result can appear in a blog post, a Discord server (link preview), a Product Hunt launch, or a company dashboard without the reader needing to click through to `miroshark.xyz`. For the academic citation hyperstition: a paper can include a live embed of the simulation it cites rather than a static screenshot. Implementation is straightforward: a minimal server-side route returns an HTML page sized for embedding, with the simulation's signal pre-rendered and zero external dependencies (no CDN calls, works offline once loaded).

**How:**
1. Add `backend/app/routes/embed.py` (~120 LoC, pure stdlib). `GET /embed/{simulation_id}` — a lightweight HTML page, no auth required (publish-gated: only published simulations are embeddable). The route reads `signal.json` and `simulation_config.json` for the given simulation ID and returns a complete, self-contained HTML document with inline CSS and inline SVG — no external stylesheets, no JavaScript dependencies, no CDN calls. Design: dark background (`#0f1117`), a centered card (max-width 480px), the simulation topic in bold white at the top, the direction badge (BULLISH / BEARISH / NEUTRAL) in the appropriate color (green / red / gray using the existing MiroShark color palette from the Vue frontend: `--color-bullish: #22c55e`, `--color-bearish: #ef4444`, `--color-neutral: #6b7280`), `yes_pct` and `confidence_pct` as two horizontal stat bars, agent count and round count as small metadata chips at the bottom, and a "Powered by MiroShark" wordmark linking to `miroshark.xyz`. Dimensions: designed for `600×200px` (standard iframe embed size). Include `<meta property="og:*">` tags so link previews in Discord/Slack/Twitter render the signal data. Return `Cache-Control: public, max-age=300`. Add the route to `backend/app/main.py`. Add 6 unit tests: 200 response for published sim, 404 for unknown ID, 403 for unpublished sim, direction badge color matches signal.direction, og:title includes topic, Cache-Control header present.
2. Add `GET /embed/{simulation_id}.js` — a web component definition (~100 LoC, vanilla JS, no framework). When this script is loaded, it registers a `<miroshark-card>` custom element. The element accepts a `sim-id` attribute, fetches `/api/simulation/{sim-id}/signal` on mount, and renders the same card design as the iframe version using Shadow DOM (isolated styles, no CSS bleed). Usage on any external page: `<script src="https://miroshark.xyz/embed/abc123.js"></script>` followed by `<miroshark-card sim-id="abc123"></miroshark-card>`. The web component auto-updates if the simulation is still running (polls `/api/simulation/{id}/status` every 10s and refreshes the card when status changes to `complete`). Include a `theme` attribute: `theme="light"` for a white card variant (for embedding in light-mode sites). Add CORS headers to the `/embed/*.js` route so external origins can load the script. No npm install needed — this is a native browser API (Custom Elements v1, supported in all modern browsers since 2020).
3. Update `EmbedDialog.vue` to add two new embed options alongside the existing API tab structure. New tab: "Embed". Show two embed codes side by side: (a) `<iframe src="https://miroshark.xyz/embed/{id}" width="600" height="200" frameborder="0"></iframe>` with a copy button; (b) the web component snippet (2-line version: script tag + custom element) with a copy button. Below the two snippets, show a live preview of the embed iframe (using the `/embed/{id}` URL in a sandboxed iframe within the dialog). Add a "Share" section below the embed codes: direct link to `https://miroshark.xyz/embed/{id}` with an "Open" button (opens in new tab) and a "Copy link" button. Add `## Embed Widget` to `docs/FEATURES.md` with the two embed methods, sizing recommendations, and the live-update behavior note. Add one line to the main README's API table: `/embed/{id} — embeddable result card (iframe + web component)`.

### 3. GitHub Actions Marketplace Action

**Type:** Integration / Growth
**Effort:** Medium (1-2 days)
**Impact:** A reusable GitHub Action — `aaronjmars/miroshark-simulate-action` — published to the GitHub Actions Marketplace creates a discoverability channel that operates independently of GitHub stars, HN, or npm. Any developer browsing the Actions Marketplace for "AI simulation" or "opinion research" tools finds MiroShark there. More concretely: it enables CI-driven simulations. A product team can add a `miroshark-simulate-action` step to their release workflow that runs a market sentiment simulation before deploying a pricing change — or a research team can run simulations as part of a data pipeline without writing custom API calls. This is the cleanest integration vector for the "external public deployment" hyperstition: a public GitHub repository that includes `uses: aaronjmars/miroshark-simulate-action@v1` in its `.github/workflows/` is a visible, searchable external deployment. The action wraps the existing API — no backend changes required — and the entire implementation is a ~150-line Node.js script plus `action.yml`.

**How:**
1. Create a new directory structure in the MiroShark repo: `actions/simulate/` (alternative: create a separate repo `aaronjmars/miroshark-simulate-action` — prefer the separate repo for cleaner Marketplace listing and independent versioning). `action.yml`: name "MiroShark Simulate", description "Run a MiroShark swarm intelligence simulation and get structured results", `runs: using: 'node20'`, `main: dist/index.js`. Inputs: `topic` (required, string); `miroshark_url` (optional, default `https://miroshark.xyz`); `agents` (optional, default `20`); `rounds` (optional, default `5`); `platforms` (optional, default `twitter,reddit,linkedin`); `wait_timeout` (optional, default `600`, seconds). Outputs: `simulation_id` (string); `direction` (BULLISH/BEARISH/NEUTRAL); `yes_pct` (0-100, string); `confidence_pct` (0-100, string); `signal_json` (full JSON string of signal.json for downstream `fromJSON()` parsing). `index.js` (~150 LoC, Node.js, no external npm dependencies — use only `https` stdlib): POST to `{miroshark_url}/api/simulate` with the topic and config; poll `GET /api/simulation/{id}/status` every 10s until `complete` or `failed` or timeout; on complete, GET `/api/simulation/{id}/signal`; set all outputs via `@actions/core` (the one allowed dependency, bundled via `@vercel/ncc` into `dist/index.js`). On `failed` status: call `core.setFailed('Simulation failed')`. On timeout: call `core.setFailed('Simulation timed out after ${wait_timeout}s')`. Bundle with `ncc build index.js -o dist` so the action runs with zero network installs in CI.
2. Add a `README.md` to the action repo (~80 lines): what the action does (1 paragraph), a minimal usage example (10 lines of YAML — start a simulation, use the outputs), a full example with conditional steps based on `direction` output (e.g., "if direction is BEARISH, open a GitHub issue"), all inputs and outputs in a reference table, a note that `miroshark_url` can point to a self-hosted instance, and a link to `miroshark.xyz` for the public instance. Add the Marketplace `action.yml` metadata: `branding: { icon: 'activity', color: 'green' }`. Write a `test.yml` GitHub Actions workflow in `.github/workflows/test.yml` within the action repo that runs the action against `miroshark.xyz` with a test topic (`"Is automated testing worth the investment?"`) — this serves as both CI and as a live demo that's visible in the action repo's Actions tab. Create a GitHub release `v1.0.0` with the compiled `dist/` committed — the release is what gets indexed by the Marketplace.
3. Add a `## GitHub Actions` section to the main MiroShark `README.md` and `docs/FEATURES.md` with the minimal 10-line usage example and a "Marketplace" badge linking to the action's Marketplace page. Add `miroshark-simulate-action` to the MiroShark `ECOSYSTEM.md` or equivalent listing file (the same file that tracks ecosystem integrations — PR #242 updated `ECOSYSTEM.md` for the RootAI CDN; this action gets the same treatment). File the action repo on GitHub with topics: `github-actions`, `ai-simulation`, `swarm-intelligence`, `miroShark` so it appears in topic searches. Submit for Marketplace listing via the GitHub UI (requires the `action.yml` with branding metadata to be on the default branch and a valid `README.md`). The Marketplace listing is free and goes live immediately after submission review (~24-48h). Once listed, link to it from the main repo README badge row.

### 4. JavaScript/TypeScript SDK (`miroshark-js`)

**Type:** DX / Integration
**Effort:** Medium (1-2 days)
**Impact:** The Python SDK idea (Jul 8 #4) targets data scientists and researchers. `miroshark-js` targets a different segment: frontend developers, Node.js automation scripts, browser-side integrations, and the JavaScript-first AI tooling ecosystem (LangChain.js, Vercel AI SDK, etc.). An npm package creates a parallel discoverability surface — `npm search miroshark` finds it; JavaScript AI tooling roundups index it; the TypeScript ecosystem gets typed response objects for free. More specifically: the GitHub Actions action (#3) can use `miroshark-js` as its underlying client (instead of raw `https` stdlib), making the client library and the action mutually reinforcing. For the academic citation hyperstition: Jupyter-like environments (Observable notebooks, Quarto documents) run JavaScript; a researcher can embed a live MiroShark query in an Observable notebook with `import { MiroShark } from 'miroshark-js'`. The scope is intentionally narrow — no async generators, no streaming, no WebSocket — to match the Python SDK's `v0.1.0` conservatism and keep the implementation verifiable.

**How:**
1. Create `sdk-js/` directory structure: `sdk-js/src/index.ts` (public API), `sdk-js/src/client.ts` (MiroShark class), `sdk-js/src/types.ts` (TypeScript interfaces), `sdk-js/src/errors.ts` (MiroSharkError, SimulationTimeout, SimulationFailed), `sdk-js/package.json`, `sdk-js/tsconfig.json`, `sdk-js/README.md`. `package.json`: name `miroshark-js`, version `0.1.0`, `main: dist/index.js`, `types: dist/index.d.ts`, `type: "module"`, engines `node: ">=18"` (for native `fetch`), zero runtime dependencies. The `MiroShark` class in `client.ts`: `constructor(options?: { baseUrl?: string; apiKey?: string; timeout?: number })`. Methods: `simulate(topic: string, options?: { agents?: number; rounds?: number; platforms?: string[] }): Promise<Simulation>` — POSTs to `/api/simulate`, returns a `Simulation` object. `Simulation` class: `.id: string`, `.wait(options?: { pollInterval?: number; maxWait?: number }): Promise<SimulationResult>` — polls `/api/simulation/{id}/status`. `signal(simId: string): Promise<Signal>`. `status(simId: string): Promise<'running' | 'complete' | 'failed'>`. `list(options?: { limit?: number; status?: string }): Promise<SimulationSummary[]>`. TypeScript interfaces in `types.ts`: `SimulationResult`, `Signal`, `SimulationSummary` — same fields as the Python SDK dataclasses, typed as TypeScript `interface`. Use native `fetch` throughout (available in Node ≥18, all modern browsers). No polyfills. Compile with `tsc` to `dist/` with `declaration: true` so `.d.ts` types are auto-generated. `tsconfig.json`: `target: "ES2020"`, `module: "ESNext"`, `moduleResolution: "bundler"`.
2. Add 15 unit tests in `sdk-js/src/client.test.ts` using Node.js built-in `test` and `mock` (Node ≥18, no Jest/Vitest required — this keeps the zero-external-dependency goal in testing too). Mock `globalThis.fetch` for all tests. Tests parallel the Python SDK tests: simulate() posts to correct URL, returns Simulation with id, wait() polls status, wait() throws SimulationTimeout after maxWait, wait() throws SimulationFailed on failed status, signal() returns typed Signal, list() applies limit param, apiKey sent as Bearer token, MiroSharkError thrown on 4xx, MiroSharkError thrown on 5xx, custom baseUrl overrides default, platforms serialized as comma-joined string, SimulationResult fields typed correctly, pollInterval respected, direction is uppercase string. Add `sdk-js/README.md` (~60 lines): installation (`npm install miroshark-js`), a 10-line quickstart (`import`/`await simulate/wait`), the same API reference table as the Python SDK with TypeScript type signatures, an Observable notebook usage note, and a browser usage note (works in any modern browser that supports native `fetch` — no bundler required for ES module import via CDN: `import { MiroShark } from 'https://esm.sh/miroshark-js'`).
3. Add a `package.json` script: `"publish": "tsc && npm publish --access public"`. Add GitHub Actions workflow `sdk-js/.github/workflows/publish.yml` (or `.github/workflows/publish-sdk-js.yml` in the main repo): triggered on `push: tags: ['sdk-js-v*']`; steps: checkout, `npm ci`, `npm test`, `npm run build`, `npm publish`. Use `NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}` secret. This makes release a single `git tag sdk-js-v0.1.0 && git push --tags`. Add `sdk-js/` to the main README's ecosystem section alongside the Python SDK: "**SDKs:** Python (`pip install miroshark-py`) · JavaScript (`npm install miroshark-js`)". Add to `docs/FEATURES.md` under a "Client Libraries" section. File the package on npm under the `miroshark` topic. Add the CDN URL (`esm.sh/miroshark-js`) to the GitHub Actions action's README as an alternative to bundling — frontend devs who don't want to `npm install` can use the CDN URL directly in browser scripts.

### 5. Simulation Data Bundle Export (ZIP)

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** MiroShark has 41 queryable surfaces. Researchers who want to analyze a simulation offline currently have to make 41 separate API calls. `GET /api/simulation/{id}/bundle.zip` packs all simulation artifacts into a single download — the signal, belief trajectory, agent profiles, platform sentiment, stance flips, mention network, confidence trajectory, signed result, and simulation config — organized in a directory structure with a human-readable `README.txt` summarizing what each file contains. This is the highest-leverage contribution for the academic citation hyperstition: a paper's methodology section currently has to document 5-10 API calls; with the bundle, it documents one. The bundle is also the right format for archival repositories (Zenodo, OSF, Harvard Dataverse) that the Zenodo DOI auto-deposit idea (suggested May 30, never built) targets — a single `.zip` is what data repositories expect as a submission. Implementation uses Python's `zipfile` stdlib — zero new dependencies. Side effect: makes it trivially easy for the MiroShark team to audit or share a complete simulation snapshot for debugging or demonstration purposes.

**How:**
1. Add `backend/app/routes/bundle.py` (~150 LoC, pure stdlib). `GET /api/simulation/{id}/bundle.zip` — publish-gated, no auth required for published simulations. The route builds a `zipfile.ZipFile` in memory (`io.BytesIO`) and adds the following files with their canonical paths inside the archive: `signal.json`, `trajectory.json`, `agents.json` (per-agent profiles), `simulation_config.json`, `platform_sentiment.json`, `stance_flips.json`, `mentions.json`, `confidence_trajectory.json`, `signed_result.json`. For each file: if the simulation's data directory contains the file, add it to the ZIP; if a surface hasn't been computed yet (e.g., the simulation was published before platform_sentiment was added), skip it gracefully (log a warning, don't 500). Add a synthetic `README.txt` as the first entry in the ZIP: a plain-text header ("MiroShark Simulation Bundle — {topic} — {completed_at}") followed by a table of contents with one line per included file, a 1-sentence description of what each file contains, and the canonical API URL for each surface (so a researcher can re-fetch any individual file). Add `bundle_meta.json` as the last entry: `{ "simulation_id", "topic", "exported_at", "miroshark_version", "surface_count", "files" }` — this gives downstream tooling a machine-readable manifest without parsing the ZIP contents. Return `Content-Type: application/zip`, `Content-Disposition: attachment; filename="miroshark-{id[:8]}.zip"`, `Cache-Control: public, max-age=3600` (bundles are immutable once a sim is complete). Add 8 unit tests: valid ZIP returned for published sim, 404 for unknown sim, 403 for unpublished sim, README.txt present as first entry, bundle_meta.json present, missing surface files are skipped without error, Content-Disposition header correct, cache-control header present.
2. Add `GET /api/simulation/{id}/bundle.zip` to the OpenAPI spec under a new "Export" tag (alongside `signed-result.json` in the Integrity group). Add it to `surfaces_catalog.py` as the 42nd surface with `type: "export"`. Update the surface count in `surface_stats.py`. Add a "Download bundle" button to the simulation detail page in the Vue frontend: in `Step3Simulation.vue` or wherever the existing "Embed" and share buttons live, add a `<a href="/api/simulation/{id}/bundle.zip" download>Download All Data (.zip)</a>` link. Style it as a secondary button (outline, not filled) to distinguish it from primary actions. Tooltip: "Download all simulation artifacts as a single ZIP file — includes signal, trajectory, agent profiles, and all 41 surfaces." Add a ZIP icon (can use an SVG or the 📦 emoji if the codebase uses emoji in UI — check existing conventions). Add the bundle link to `EmbedDialog.vue` as a new tab alongside the existing embed options: "Export" tab with the download link, `bundle_meta.json` preview (showing file list), and a note that the bundle is suitable for submission to Zenodo, OSF, and other data repositories.
3. Add a `## Data Bundle Export` section to `docs/FEATURES.md`: what's in the bundle (file list with descriptions), the URL pattern, the metadata manifest format, and example use cases ("Download for offline analysis", "Submit to Zenodo as a dataset", "Attach to a research paper as supplementary materials", "Debug a simulation by inspecting all intermediate states"). Add one line to the main README API table: `/api/simulation/{id}/bundle.zip — download all simulation artifacts as a ZIP archive`. Write a `scripts/fetch-bundle.sh` convenience script (~20 lines): takes a simulation ID and optional output path, calls `curl -L -o` or falls back to WebFetch, and prints a summary of what was downloaded. This gives CLI users a one-liner to fetch the full dataset without remembering the URL pattern: `./scripts/fetch-bundle.sh abc123 ~/research/sim-abc123.zip`.

## Selection Rationale

Five days to the July 15 deadline cluster. The Show HN Kit (Jul 8) gave the submission playbook; none of this week's ideas can force HN front page. What these five ideas do is build durable infrastructure that pays off whether or not July 15 closes.

- **Spanish (#1)** — The most direct unblocked path to the September "5 languages" hyperstition. French took two PRs (initial + completion sprint); Spanish can do it in one if the initial coverage is 100%. Takes the language count to 4/5. The remaining slot is open to whatever community contribution arrives first.

- **Embed Widget (#2)** — External deployment hyperstition needs someone to run MiroShark publicly. An iframe embed is the lowest-friction path: someone writes a blog post, drops in the embed code, and becomes an external deployer. The web component angle also gives JavaScript developers a native integration pattern that doesn't require any framework knowledge.

- **GitHub Actions Action (#3)** — Marketplace discoverability is independent of GitHub stars, HN, and npm. Any public repo that uses the action appears in GitHub's `used by` count for the action — a live counter of external integrations. The action also composes with the JS SDK (#4): the action can use `miroshark-js` under the hood, making both artifacts mutually reinforcing.

- **JS SDK (#4)** — Python SDK (Jul 8 #4) targets researchers; `miroshark-js` targets the far larger JavaScript ecosystem. Observable notebooks, Quarto documents, and browser-based demos all run JavaScript. The CDN usage path (`esm.sh/miroshark-js`) means zero install — any frontend dev can try it in a CodePen or JSFiddle without a build step.

- **Data Bundle (#5)** — The academic citation hyperstition requires a researcher to use MiroShark in published work. Right now that means documenting 5-10 API calls in a methodology section. The bundle reduces it to one. Zenodo compatibility (ZIP format + `bundle_meta.json` manifest) puts MiroShark one click away from a citable DOI-backed dataset — the exact artifact a researcher needs to attach to a paper submission.
