### 1. CLI `simulate` One-Command End-to-End

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The CLI has `cost` and `wait` but no unified `simulate` command. Operators running batch jobs or CI pipelines must: call the REST API to launch, run `wait` to block until done, call `cost` to fetch results. A `simulate <topic>` subcommand collapses this into one call. For CI integration — "run a consensus check on each deploy" — or for operators scripting weekly sentiment tracking, the single command is the primitive that makes automation practical. The CLI is already moving in this direction; this closes the loop.

**How:**
1. Add `cli/commands/simulate.py` (~120 LoC, pure stdlib + `httpx` already in the project). `simulate_cmd(topic, agents, rounds, platforms, model, wait, output_format)` — parses arguments, posts `POST /api/simulation/run` with a generated `simulation_config` (topic → title, sane defaults for unspecified fields). If `--wait` flag (default True): calls the same poll loop as `wait` subcommand (checks `GET /api/simulation/:id` until status is "complete" or "failed", 10s poll interval). On completion: prints a result summary — consensus direction, `yes_pct`, round count, total cost if cost.json is present. Supports `--output json` for machine-readable output (the full `GET /api/simulation/:id` response as JSON). Supports `--no-wait` to return immediately with the simulation ID. Registers the command as `miroshark simulate` in `cli/main.py` alongside `cost`, `wait`, and existing commands. Add `BASE_URL` config read from `~/.miroshark/config.toml` (same pattern as existing CLI config) with `--base-url` flag override. Add 8 unit tests in `cli/tests/test_simulate.py`: successful simulate returns sim_id, `--no-wait` exits immediately after POST, `--wait` polls until complete, `--output json` emits valid JSON, failed simulation prints error and exits non-zero, topic with spaces is URL-safe in the POST body, `--agents` overrides the default agent count, `--base-url` overrides config file.
2. In `cli/main.py`, register `simulate` under the main command group with `@app.command()` (Typer pattern consistent with existing commands). Help text: `"Run a simulation end-to-end: configure, launch, wait, and print results."`. Named params: `topic` (positional, required), `--agents` (int, default 20), `--rounds` (int, default 5), `--platforms` (str multiple, default ["twitter"]), `--model` (str, defaults to env `MIROSHARK_MODEL` or `"auto"`), `--wait/--no-wait` (default wait=True), `--output` (enum: "text"|"json", default "text"), `--base-url` (str, default from config). In text mode, output format: "Launching simulation: {topic}\n  → {sim_id}\nRunning [round 1/5] .\nRunning [round 2/5] ..\n...\n\nResult: BULLISH (73% yes) · 5 rounds · 37 agents · $0.42\n". The dot-per-poll-tick is the same visual feedback as the existing `wait` command. No color dependency — plain stdout so CI logs stay readable.
3. Update `docs/CLI.md` (or create it if absent) with a `simulate` section: the full parameter reference, three examples (quick one-liner, CI mode with `--output json`, async launch with `--no-wait`), and a note on the `~/.miroshark/config.toml` BASE_URL configuration. Add a one-line entry to `README.md` in the "Quick Start" or "CLI" section: `miroshark simulate "Will BTC reach $200k?" --agents 20 --rounds 5` as the zero-friction entry point for new users. Also update the OpenAPI spec's "CLI" or "Getting Started" section if one exists.

### 2. Open Graph / Social Preview Cards

**Type:** Growth
**Effort:** Small (hours)
**Impact:** miroshark.xyz is live. Sharing a simulation URL on X, Discord, Slack, or in a chat currently produces a blank link — no title, no image, no preview. Open Graph meta tags are the zero-config fix. `GET /api/simulation/:id/og-meta` returns the five OG fields needed for a rich preview card: `title`, `description` (consensus summary), `image` (a static template SVG with the consensus pct overlaid), `url`, `type`. The frontend injects these as `<meta>` tags in the `<head>` for simulation detail pages (SSR or via a lightweight pre-render middleware). Every shared result becomes a preview card: "Will ETH reach $5K? — 68% BULLISH · MiroShark · 40 agents". For the 1,500-star target and the Hacker News hyperstition, every organic share is a growth artifact. This is a two-hour build with compounding long-term effect.

**How:**
1. Add `backend/app/services/og_meta_service.py` (~80 LoC, pure stdlib). `get_og_meta(sim_dir, base_url)` reads `simulation_config.json` (title, agent_count, num_rounds, platforms), `signal.json` (yes_pct, direction, confidence_pct), and `state.json` (status, completed_at). Returns `{title: str, description: str, og_image_url: str, og_url: str, og_type: "website", twitter_card: "summary_large_image", twitter_title: str, twitter_description: str}`. Title format: `"{sim_title} — {direction} ({yes_pct:.0f}% consensus) · MiroShark"`. Description format: `"{agent_count} agents · {num_rounds} rounds · {platform_count} platform(s) · Completed {completed_at_date}"`. `og_image_url`: points to `GET /api/simulation/:id/og-image` (see step 2). If simulation is not yet complete (status != "complete") or signal.json is absent: title = `"{sim_title} — In Progress · MiroShark"`, description = `"{agent_count} agents simulating consensus..."`, `og_image_url` points to a static default card. Add `GET /api/simulation/:id/og-meta` (unauthenticated for published sims, returns 403 for unpublished). Response: `Cache-Control: public, max-age=3600`. Add 8 unit tests in `backend/tests/test_unit_og_meta.py`: complete sim returns direction + yes_pct in title, incomplete sim returns "In Progress" title, unpublished sim returns 403, missing signal.json falls back gracefully, description includes agent_count and num_rounds, og_url constructed correctly from base_url + sim_id, og_type is always "website", twitter_card is always "summary_large_image".
2. Add `GET /api/simulation/:id/og-image` that generates a minimal SVG card (inline, no PIL/Pillow required — pure stdlib SVG string generation). Card template: dark background (#0e0e1a), left half shows MiroShark wordmark (text element) + simulation title (wrapped text, max 60 chars, `<text>` with `textLength`), right half shows a large consensus number (e.g., "73%") in violet (#7c5cfc), with "BULLISH" or "BEARISH" label below in green/red. Bottom row: "{N} agents · {N} rounds · miroshark.xyz". SVG dimensions: 1200×630 (standard OG image size). `Content-Type: image/svg+xml`. `Cache-Control: public, max-age=86400`. If simulation is not complete, return the same template with "In Progress" replacing the consensus number. Note: SVG OG images work in Discord and Slack previews; Twitter/X requires a PNG, but Twitter crawlers will fall back to the `<title>` if the SVG isn't rendered — acceptable trade-off to avoid a Pillow dependency.
3. In `frontend/src/views/Step3Simulation.vue` (or the simulation detail view), add a `useHead()` call (or manual `document.head` manipulation if no head management library) that sets `<meta property="og:title">`, `<meta property="og:description">`, `<meta property="og:image">`, `<meta property="og:url">`, `<meta name="twitter:card">`, `<meta name="twitter:title">`, `<meta name="twitter:description">`, `<meta name="twitter:image">` using data from `GET /api/simulation/:id/og-meta`. Fetch the OG meta once when the simulation detail page mounts (one HTTP call; not polled). On unmount, restore the default page `<title>` and remove the OG tags. Add `getOgMeta(simId)` to `frontend/src/api/simulation.js`. Add `GET /api/simulation/:id/og-meta` and `GET /api/simulation/:id/og-image` to the OpenAPI spec under the "Simulation" tag. Add a "Social Preview" section to `docs/FEATURES.md`: explains the OG meta fields, the SVG card format, the Twitter/X limitation, and notes that meta tags are injected client-side (which means crawler visibility depends on pre-render — Cloudflare Workers or a simple pre-render script at miroshark.xyz would give full crawler coverage, but client-side injection alone covers Discord, Slack, and most chat apps).

### 3. Simulation Campaign Series Tracker

**Type:** Feature
**Effort:** Medium (1-2 days)
**Impact:** Operators tracking sentiment on a recurring topic — "ETH price consensus", "BTC quarterly outlook", "AI regulation" — run the same simulation weekly. Today those runs are unlinked. `POST /api/series` creates a named series with a slug and topic description. `POST /api/series/:slug/simulations/:sim_id` links a published simulation to the series. `GET /api/series/:slug/timeline` returns the linked simulations sorted chronologically with `yes_pct`, `direction`, and `completed_at` per entry — enough to plot a consensus trend line over time. A "Series" view on the site renders this as a timeline chart with the consensus direction annotated per data point. Operators presenting "six months of ETH sentiment" have a dedicated surface for that story, not a screenshot gallery.

**How:**
1. Add `backend/app/services/series_service.py` (~160 LoC, pure stdlib). Series metadata stored in `data/series/{slug}.json`: `{slug: str, name: str, description: str, topic: str, created_at: ISO, sim_ids: [str]}` (append-only list; sims linked in order of the POST call). `create_series(data_dir, slug, name, description, topic)` validates slug format (lowercase alphanumeric + hyphens, 3–50 chars; 409 if slug already exists), writes the metadata file. `link_simulation(data_dir, slug, sim_id)` reads `data/simulations/{sim_id}/state.json` (must exist; 404 if not), checks `publish_date` is set (403 if unpublished), appends `sim_id` to `series[slug].sim_ids` (idempotent — skip if already present). `get_timeline(data_dir, slug)` reads the series metadata, then for each `sim_id` reads `simulation_config.json` (title, agent_count, num_rounds), `signal.json` (yes_pct, direction, confidence_pct), `state.json` (completed_at); returns `{slug, name, description, topic, created_at, entry_count: int, entries: [{sim_id, title, completed_at, yes_pct, direction, confidence_pct, agent_count, num_rounds}]}` sorted by `completed_at` asc. Caches to `data/series/{slug}_timeline.json` with 5-minute mtime invalidation (invalidate cache when a new sim is linked). Add `POST /api/series` (admin-gated; body: `{slug, name, description, topic}`). Add `POST /api/series/:slug/simulations/:sim_id` (admin-gated). Add `GET /api/series/:slug` (unauthenticated — returns metadata only, no sim details). Add `GET /api/series/:slug/timeline` (unauthenticated — returns full timeline). Add `GET /api/series` (unauthenticated — lists all series with slug, name, entry_count). Add 12 unit tests in `backend/tests/test_unit_series.py`: slug validation rejects uppercase letters, slug validation rejects >50 chars, duplicate slug returns 409, link unpublished sim returns 403, link unknown sim_id returns 404, link is idempotent (same sim_id twice doesn't duplicate), timeline sorted by completed_at asc, timeline excludes sims with no signal.json (no error — just missing yes_pct/direction fields), empty series returns entry_count=0 and empty entries list, unknown slug returns 404, admin token required for POST endpoints, GET timeline is unauthenticated.
2. Add `frontend/src/views/SeriesView.vue` — a new page at `/series/:slug`. Fetches `GET /api/series/:slug/timeline`. Layout: series name + description header; a consensus trend chart (line chart, x-axis = completed_at dates, y-axis = yes_pct 0–100%, a horizontal midline at 50%, each data point labeled with its direction chip: green BULLISH above 50, red BEARISH below). Below the chart: a table listing each simulation entry — date, title (clickable link to `/simulation/:sim_id`), consensus, yes_pct, confidence chip. Add a `SeriesListView.vue` at `/series` that fetches `GET /api/series` and renders a grid of series cards (name, topic, entry count, last entry date, link to `/series/:slug`). Add route entries in `router/index.js`. Add `getSeries(slug)`, `getSeriesTimeline(slug)`, and `listSeries()` to `frontend/src/api/simulation.js`. Bilingual EN/ZH labels throughout. Add a "Series" nav link in the main navigation (next to Explore or Archive). No admin UI for creating/linking series — that's done via API; the frontend is read-only for the public view.
3. Add `POST /api/series`, `POST /api/series/:slug/simulations/:sim_id`, `GET /api/series`, `GET /api/series/:slug`, and `GET /api/series/:slug/timeline` to the OpenAPI spec under a new "Series" tag. Schemas: `SeriesMetadata`, `SeriesCreateRequest`, `SeriesTimelineResponse`, `SeriesTimelineEntry`. Add a "Simulation Campaign Series" section to `docs/FEATURES.md`: explains the slug format constraints, the link-then-timeline workflow (API-first), and notes that series are append-only (simulations cannot be unlinked via the API, only by editing the JSON file manually). Provide a three-step quick-start example: create a series → link two sims → fetch timeline. Note: the cache on `_timeline.json` is 5 minutes; immediately after linking a sim, the GET timeline response may lag by up to 5 minutes until the cache is invalidated.

### 4. Simulation Tags & Labels

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The full-text search endpoint (#14 — built as part of the push-blocked builds) searches simulation titles. Tags are the intentional complement: operators label their simulations with `["crypto", "eth", "bearish-test"]` at run time or post-hoc. `PUT /api/simulation/:id/tags` sets the tag list. `GET /api/simulations?tag=crypto` filters the gallery. The ExploreView adds a tag filter strip below the existing sort controls. Tags are operator-defined strings — no taxonomy enforced — so the feature requires no curation and no migrations. The design pattern (free-form tags on structured data) is how every simulation platform that's scaled past 1,000 operators has solved organization. With 279 forks and a growing operator base, the lack of simulation organization will become a real friction point at 300+ forks.

**How:**
1. Add `backend/app/services/tags_service.py` (~80 LoC, pure stdlib). Tags stored in `data/simulations/{sim_id}/tags.json`: `{sim_id: str, tags: [str], updated_at: ISO}`. `set_tags(sim_dir, sim_id, tags)` validates: max 20 tags, each tag is 1–30 chars lowercase alphanumeric + hyphens (strip leading/trailing whitespace, lowercase, reject invalid format with 400). Writes to `tags.json`. `get_tags(sim_dir, sim_id)` reads `tags.json` or returns `{sim_id, tags: [], updated_at: null}` if absent. `list_simulations_by_tag(data_dir, tag)` scans all sim dirs for `tags.json` files containing the tag string (case-insensitive match), returns `[sim_id, ...]`. Add `PUT /api/simulation/:id/tags` (admin-gated; body: `{tags: [str]}`; replaces the full tag list; returns the updated `{sim_id, tags, updated_at}`). Add `GET /api/simulation/:id/tags` (unauthenticated for published sims, 403 for unpublished). Add `GET /api/simulations?tag=:tag` (unauthenticated; returns list of sim_id strings for published sims matching the tag; limited to 100 results, sorted by most recently tagged). Include `tags` in the `GET /api/simulation/:id` response as an optional field (read from `tags.json` if present, empty array if absent — no schema change, additive). Add 10 unit tests in `backend/tests/test_unit_tags.py`: set_tags writes tags.json, get_tags returns empty array when file absent, tag validation rejects >30 char tags, tag validation rejects >20 tags, tag validation normalizes to lowercase, list_by_tag returns only published sims with that tag, list_by_tag is case-insensitive, PUT replaces full tag list (not appends), admin token required for PUT, GET /api/simulation/:id/tags returns 403 for unpublished sim.
2. In `Step3Simulation.vue` (operator view, admin-gated section), add a tag editor: a small input chip field below the simulation title. Shows existing tags as removable chips. A text input `+ Add tag` that adds a tag on Enter or comma. On change (debounced 1s), fires `PUT /api/simulation/:id/tags` with the full updated tag list. Shows a brief "Tags saved" confirmation (1.5s toast). In `ExploreView.vue`, fetch `GET /api/simulations` and collect the union of tags across all loaded simulations (from the `tags` field now included in list responses). Render a tag filter strip below the sort bar: chips for each unique tag found, with a count badge ("crypto · 12"). Clicking a tag chip: filters the gallery client-side (sims where `tags.includes(tag)`); clicking again deselects. Multiple tags: AND filter (sim must have all selected tags). The active filter chips appear in a "Filtered by:" row above the gallery grid. Add `setTags(simId, tags)` and `getTags(simId)` to `frontend/src/api/simulation.js`. Update `getSimulations()` to include `tags` in the response fields requested. Bilingual EN/ZH labels on the editor placeholder, filter strip header, and empty-state message.
3. Add `PUT /api/simulation/:id/tags` and `GET /api/simulation/:id/tags` to the OpenAPI spec under the "Simulation" tag. Add `GET /api/simulations?tag` filter to the list endpoint spec. Add `TagsResponse` schema: `{sim_id: str, tags: [str], updated_at: str|null}`. Add a "Simulation Tags & Labels" section to `docs/FEATURES.md`: explains the tag format constraints (lowercase, alphanumeric + hyphens, 1–30 chars, max 20 per simulation), the AND-filter behavior when multiple tags are selected in ExploreView, and notes that tags are operator-set (no public tag submission). Note: tags are not indexed — `GET /api/simulations?tag=:tag` does a linear scan of all sim dirs; for installations with >10,000 simulations, this should be replaced with an indexed lookup, but the linear scan is correct and fast for typical operator deployments.

### 5. Contributor Leaderboard

On hover: expand to show `most_recent_pr_title` in a tooltip (truncated to 40 chars + "…"). Below the grid: a "Contribute" link pointing to `CONTRIBUTING.md` in the repo. Cap the display at 10 contributors (top 10 by rank) with a "View all contributors on GitHub →" link for the rest. Add `getContributors()` to `frontend/src/api/simulation.js`. Bilingual EN/ZH section header and "Contribute" CTA.
3. Add `GET /api/contributors` to the OpenAPI spec under a new "Community" tag with `ContributorLeaderboardResponse` schema: `{generated_at: str, repo: str, contributors: [{rank: int, login: str, avatar_url: str, contributions: int, html_url: str, most_recent_pr_title: str|null}]}`. Add a "Contributor Leaderboard" section to `docs/FEATURES.md`: explains the GitHub API source, the 1-hour cache, the `GH_TOKEN` env var (optional but recommended to avoid the 60 req/hr unauthenticated limit), and notes that "contributions" from GitHub's API counts all commit activity (not just merged PRs) — a separate note clarifies that the `most_recent_pr_title` is the more meaningful signal for "who has merged PRs." Note: bots (login ending in `[bot]`) are filtered from the leaderboard — check `login.endswith("[bot]")` before including in the result.

## Selection Rationale

Five ideas grounded in what's actually happening this week:

- **CLI `simulate` command** (#1) — `cost` and `wait` landed in the same week. The CLI has two-thirds of a complete simulation loop. `simulate` closes it. Operators running batch jobs or CI checks need a one-liner, not a three-step API dance. This is the natural next commit in the series.

- **OG / Social Preview Cards** (#2) — miroshark.xyz is live. Branding is updated. The domain is pointed. The one thing missing: shared links don't preview. Every MiroShark result shared on X, Discord, or in a Slack message is a blank URL. OG tags take two hours and produce compounding growth. With the 1,500-star hyperstition target 167 stars away and 21 days left, this is the highest-leverage growth lever that hasn't been pulled yet.

- **Simulation Campaign Series** (#3) — Operators who run MiroShark at scale run the same topic weekly. There's no infrastructure for that. Without a series primitive, they're manually tracking simulation IDs in a spreadsheet. The timeline chart is the artifact that makes longitudinal analysis presentable — "here's 8 weeks of ETH consensus" as a clean chart rather than eight separate URLs.

- **Tags & Labels** (#4) — 279 forks. Growing operator base. No simulation organization beyond the existing full-text search (which is already built but push-blocked). Tags are the lightweight answer: free-form, no taxonomy, operator-defined. The AND-filter in ExploreView turns tags into a real discovery tool as the gallery grows past 20 simulations per operator.

- **Contributor Leaderboard** (#5) — The 10-merged-PR hyperstition needs 5 more. The leaderboard is a social proof artifact: it shows that specific named humans are building MiroShark. For potential contributor #6–10, seeing that the community is active and recognized is a different signal than "there's a CONTRIBUTING.md." The avatar grid costs two hours and targets the retention of the current growth trajectory.
