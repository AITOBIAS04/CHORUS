### 1. Simulation Fork from Round (Counterfactual Branching)

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** PR #76's Lineage Navigator added the read layer for tracking parent-child simulation relationships: every forked simulation knows its parent. The write layer is still incomplete. Today, forking copies the entire simulation config and re-runs from round 1. What doesn't exist is forking from a *specific round* — "at round 6 the agents reached consensus; what if I injected a market-shock event at round 4 instead and watched from there?" Round-level forking is the most natural counterfactual research operation: you want to branch from a known shared history and diverge it with a single intervention, not run two independent simulations from scratch. The round state is fully available in `trajectory.json` and the per-round post archive — the simulation runner just needs a `start_from_round: N` option that seeds belief states and agent histories from the prior run rather than generating fresh. The lineage graph in PR #76 automatically records the branch point: `{parent_sim_id, branched_at_round: N, injected_event: string}` — same data model, one new field. For researchers designing controlled experiments, this is the operation that makes MiroShark a proper counterfactual lab rather than a sequence of fresh starts.
**How:**
1. Add `start_from_round: int` and `parent_sim_id: string` as optional fields to the simulation launch API (`POST /api/simulation/start`). When `start_from_round` is set, the runner reads `<parent_sim_dir>/trajectory.json` up to the specified round, seeds each agent's belief state and post history from the parent's round-N snapshot, and begins generating from round N+1 forward with the new scenario text (or Director Mode event, if `director_event_at_start: string` is also provided). The new simulation inherits the parent's agent roster (same names, archetypes, platforms) but starts from mid-state. Write `{parent_sim_id, branched_at_round, director_event}` to `<sim_dir>/lineage.json` at launch — the Lineage Navigator picks it up automatically with no additional code changes.
2. In the simulation results view, add a "Fork from round" action in the timeline strip (the per-round hover row below the belief drift chart). On hover over any completed round dot, show a context menu with two options: "Fork from here" (opens a pre-filled launch modal: parent config loaded, `start_from_round` set to the hovered round, cursor placed in a "What changes at this point?" text field for the injected event) and "Compare rounds" (scrolls to the A/B comparison view and locks the selected round as the split point). In the launch modal, show a preview of the parent's belief state at the selected round: "At round N — YES: 58%, NO: 31%, NEUTRAL: 11% — 3 agents have flipped stance so far." This gives the user enough context to write a meaningful injected event without re-reading the full transcript.
3. In the Lineage Navigator view (PR #76), when a child simulation was branched at a specific round, render the branch point on the parent's belief drift chart as a vertical dashed line (same color as the child node in the lineage tree). Tooltip: "Fork point: [child_sim_id] branched here at round N — [injected event excerpt]." If multiple children branched from the same parent at different rounds, render one line per branch. Add `branched_at_round` to the `GET /api/simulation/:id/lineage` response schema. In the OpenAPI spec, document `start_from_round` and `parent_sim_id` as optional fields on the launch schema, with a note: "When provided, the simulation inherits agent state from the parent at the specified round — use for counterfactual experiments." Add a "Counterfactual Experiments" section to `docs/API.md` with a worked example: same scenario, forked at round 4 with two different Director Mode events, compared in the A/B view.

### 2. Agent Persona Library

**Type:** Feature / Community
**Effort:** Small (hours)
**Impact:** Every MiroShark simulation generates agent personas procedurally from the scenario text. Two runs of the same scenario with different seeds generate different agent populations — which is stochastic by design, but creates a reproducibility problem. A researcher who wants to test "how does this *specific* agent population respond to five different scenarios?" has no way to save and re-use an agent configuration today. The Persona Library is a named, persistent agent configuration: archetype, platform, initial belief bias (YES/NO/NEUTRAL weight), and a freeform backstory paragraph. When configuring a simulation, users can mix: 2 personas from the library + 3 auto-generated. Libraries start private; users can mark personas as public to contribute to a community catalog. For the academic and research cohort — the most active fork segment — reproducible agent populations are the difference between "I ran an interesting simulation" and "I ran the same agent population across 10 different scenarios and here's what I found." The community catalog compounds over time: as researchers publish niche personas ("ESG Investment Director", "Short-Selling Analyst", "First-Generation Crypto Holder"), the catalog becomes a discovery surface for simulation methodology.
**How:**
1. Add a `personas` SQLite table: `{id, label, archetype, platform, initial_stance (YES/NO/NEUTRAL/MIXED), stance_weight (float 0–1 toward initial_stance), backstory (text, max 500 chars), public (bool), created_at}`. Add REST endpoints: `POST /api/personas`, `GET /api/personas?public=true`, `GET /api/personas/:id`, `DELETE /api/personas/:id`. In Step 2 (Configure), add a "Persona Library" section below the agent count selector: a two-column layout — left panel lists saved personas (name, archetype, platform, stance badge), right panel is a form to create a new persona. Drag from library to an "Agent Roster" preview area that shows the mixed population: library personas (pinned, rendered as filled cards) + auto-generated slots (rendered as dotted empty cards). The total count stays locked to the user's agent count setting; each library persona occupies one slot, reducing the auto-generated count by 1.
2. At simulation launch, when a persona is included, write their full profile to the agent initialization data (same format as auto-generated profiles, but using the saved backstory and stance weights rather than LLM generation). The simulation runner treats saved personas identically to auto-generated agents — same action vocab, same round loop. In `<sim_dir>/config.json`, add a `persona_library_ids: [id, ...]` field listing which saved personas were used. The Reproducibility Config Export (PR #75) automatically includes this field — so a shared simulation's config export also captures the persona references. In the article generator, inject the persona-sourcing note as structured context when any library personas are present: "Agent population: 2 library personas (ESG Investment Director, Short-Selling Analyst) + 3 auto-generated from scenario text."
3. In `GET /api/personas?public=true`, add `sort` (by: `date`, `usage_count` — how many simulations used this persona) and `filter` (by archetype, platform, initial_stance). Add a "Community Personas" tab in the Configure step that queries public personas and presents them as a browsable catalog with a search field. Each card: persona name, archetype chip, platform chip, stance badge, usage count ("Used in 14 simulations"), and a preview of the backstory's first 100 chars. Add `POST /api/personas/:id/fork` to copy a public persona into the user's private library for customization (same pattern as simulation forking). Document all endpoints under a new "Persona Library" tag in the OpenAPI spec. Add `docs/personas.md` with a guide covering: creating personas, mixing library + auto-generated, forking community personas, and how persona IDs are captured in reproducibility configs.

### 3. Simulation A/B Comparison View

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The Lineage Navigator (PR #76) handles vertical relationships — parent simulation and its forks. Horizontal comparison — two simulations with no ancestry relationship — has no dedicated view. A researcher running MiroShark to compare "how does the same topic play out with 10 agents vs. 20 agents?" or "model A vs. model B on the same scenario" currently opens two browser tabs and reads charts separately. The A/B Comparison View puts two published simulations side by side on a single page: synchronized dual belief drift charts (both locked to the same round axis so trajectories are directly comparable), a scorecard diff table (Polarization Index Δ, Influence Concentration Δ, Consensus Velocity Δ, Narrative Volatility Δ), and annotation timelines for each. It is the first cross-simulation analysis surface that is not ancestry-constrained. For operators comparing model presets, for researchers comparing parameter sensitivity, and for anyone building reports that argue "scenario A produced more robust consensus than scenario B" — this closes the last gap in MiroShark's analysis layer. The implementation draws entirely from existing data: trajectory.json, scorecard.json, and annotations.json for each simulation, plus the dual chart rendering that the ensemble confidence bands (suggested May 6) will eventually reuse.
**How:**
1. Add a `/compare` route: `/compare?a=<sim_id_a>&b=<sim_id_b>`. The page fetches trajectory, scorecard, and annotations for both simulations in parallel and renders a two-column layout. Left column: Simulation A with its full scenario title, model, agent count, round count, and belief drift chart. Right column: Simulation B — same layout. Both charts render in the same `<svg>` coordinate space (shared Y axis, same round scale): YES price as a solid line, NO price as a dashed line, per simulation. The two simulations' YES lines in different colors (Sim A = `#3b82f6` blue, Sim B = `#f59e0b` amber) so they can be overlaid on the same chart when the user clicks "Overlay" (a toggle in the toolbar that switches between side-by-side and overlaid views). Overlay mode draws both YES lines on a single chart canvas — useful for direct trajectory comparison at the same scale.
2. Below the dual charts, render a "Scorecard Comparison" table: 4 rows (one per metric), 3 columns: Metric / Sim A score / Sim B score. Each score cell is colored by tier (green = low, yellow = moderate, red = high/critical) using the existing Quality Guard thresholds. Add a "Δ" column showing the absolute difference. Row with the largest Δ gets a "Most different" badge — the metric where the two simulations diverged most. For each metric, a one-line interpretation under the diff: "Sim B reached consensus 4 rounds faster (Consensus Velocity 23 vs. 64)" — derived from the diff direction and the metric's semantics (lower is faster for Velocity, lower is less polarized for Polarization). Below the scorecard, render annotation timelines for both simulations as a two-row Gantt-style bar: each row is a simulation, each annotation is a colored tick mark at its round position. Hovering a tick shows the annotation text.
3. In the gallery explore view, add a "Compare" mode: a checkbox on each gallery card (visible on hover). When 2 cards are checked, a persistent "Compare selected →" banner appears at the bottom of the screen. Click it to open `/compare?a=<id_a>&b=<id_b>`. Maximum 2 selections — selecting a third card auto-unchecks the previous second. In the simulation results toolbar, add a "Compare with..." button that opens a modal: a gallery search picker (reuses the gallery full-text search from PR #69) to select any published simulation. On selection, navigates to `/compare?a=<current_id>&b=<selected_id>`. Add a share button on the compare page that generates a shareable URL (`/compare?a=X&b=Y`) — making the comparison itself a shareable artifact. Document the compare route in `docs/FEATURES.md` and add a note in the OpenAPI spec cross-referencing the two sim IDs.

### 4. Narrative Evolution Tracker (Scheduled Reruns)

**Type:** Integration / Feature
**Effort:** Medium (1–2 days)
**Impact:** Every MiroShark simulation is a point-in-time snapshot: "given this scenario, what does the swarm believe on May 10?" But the scenarios that matter most are ongoing — a token's price narrative, a policy debate, a product reputation crisis. The researcher who wants to track how consensus on "will Ethereum hit $10K in 2026?" evolves from May to August has to manually re-run the same config every week and mentally compare results. The Narrative Evolution Tracker makes this automatic: save a scenario config as a "Tracked Narrative" and specify a rerun interval (daily, weekly, monthly). Each run appends a new data point to a time-series stored alongside the simulation series. The gallery shows a "⟳ Tracking" badge on tracked scenarios with a mini sparkline showing consensus trend over the last 4 runs. The tracker endpoint (a lightweight cron-compatible endpoint) can be called by n8n, Zapier, or GitHub Actions — the same automation audience that built integrations around the completion webhook. For market researchers, PR teams, and academics studying narrative drift, this is the longitudinal primitive that turns MiroShark from a one-shot tool into a monitoring platform.
**How:**
1. Add a `tracked_narratives` SQLite table: `{id, label, base_config_json (full simulation launch config as JSON), interval (enum: daily/weekly/monthly), next_run_at (ISO timestamp), webhook_url (optional), created_at, last_run_at, run_count}`. Add REST endpoints: `POST /api/tracked-narratives` (save a config as a tracked narrative — accepts the same config schema as `POST /api/simulation/start` plus `interval`), `GET /api/tracked-narratives` (list all), `GET /api/tracked-narratives/:id` (detail + run history), `POST /api/tracked-narratives/:id/trigger` (manually trigger a rerun immediately). Each triggered run launches a new simulation (same code path as `/api/simulation/start`) and writes `{tracked_narrative_id, run_index: int}` to the new sim's `config.json`. The tracker links each run to the narrative series: `GET /api/tracked-narratives/:id/runs` returns all run sim IDs in chronological order.
2. Add a time-series endpoint: `GET /api/tracked-narratives/:id/series` returning `[{run_index, sim_id, final_yes, final_no, final_neutral, consensus_velocity, polarization_index, ran_at}]` — one entry per completed run. Use this to render the trend sparkline: a 4-point mini line chart (last 4 runs only) showing the YES price trend, embedded as an SVG inline on gallery cards when `tracked_narrative_id` is set. In the gallery explore view, add a "⟳ Tracking" filter chip (alongside "Clean", "Annotated") that shows only tracked narratives. On the narrative detail page (`/tracked/<id>`), render a full time-series chart: all runs on the same axis, YES/NO/NEUTRAL lines, round axis normalized to the longest run. Below the chart, a table of all runs: run date, final consensus, consensus velocity, top agent. Clicking any row navigates to that run's full simulation detail page.
3. Add `POST /api/tracked-narratives/:id/trigger` to the GitHub Actions webhook path — a route that queues an immediate rerun when called with a valid admin token. Document this as the integration point for external schedulers: "Call this endpoint from a cron job, n8n workflow, or GitHub Actions schedule to trigger programmatic reruns." In the completion webhook payload, add `tracked_narrative_id` and `run_index` fields when the simulation belongs to a series — so webhook consumers can detect tracked-series completions and route them separately. In the results toolbar, add a "Track this scenario" button that opens a panel: the current simulation's config is pre-loaded, the user selects an interval, clicks "Start tracking." This saves the config as a new tracked narrative and immediately queues run 2 (since the current sim is run 1). Document the full tracked-narratives API under a "Tracking" tag in the OpenAPI spec and add a "Longitudinal Analysis" section to `docs/API.md`.

### 5. Gallery Creator Profiles

**Type:** Community / Growth
**Effort:** Medium (1–2 days)
**Impact:** The MiroShark gallery at `/explore` is a chronological stream of published simulations with no authorship signal. There is no way to discover "all simulations by researcher X," follow a specific creator, or understand which humans are behind the most interesting published work. With 224 forks and a growing academic and practitioner community, the gallery is starting to have enough content volume that creator-centric discovery matters — the same shift from chronological to creator-organized that every platform goes through at scale. Creator Profiles adds a lightweight authorship layer with no authentication requirement: when publishing a simulation, users optionally supply a creator handle (free text, no email, no OAuth). `/creators/<handle>` shows all their published simulations, aggregate stats (total runs, avg consensus velocity, most-used model, most-explored topics derived from scenario clustering), and an RSS subscription button (one creator feed per handle, powered by the existing RSS infrastructure from PR #60). A "By [handle]" chip appears on gallery cards from creators who use handles. For the project's growth, creator profiles create pull-through from individual researchers' audiences — when a professor publishes 10 simulations under their handle and links to their profile from a paper, their readers arrive at MiroShark with a ready-made tour of relevant content.
**How:**
1. Add `creator_handle: string` (optional, max 40 chars, alphanumeric + hyphens) to the gallery publish payload (`POST /api/simulation/:id/publish`). Store in `<sim_dir>/gallery_meta.json` alongside existing metadata. Add a `GET /api/creators/<handle>` endpoint returning `{handle, sim_count, published_sims: [{sim_id, scenario_excerpt, final_yes, consensus_velocity, model, published_at}], stats: {avg_consensus_velocity, avg_polarization_index, top_model, total_runs}}`. Compute `top_model` from the `model` field across all published sims (mode). Derive `topic_clusters` from the first 5 words of each scenario excerpt — group by common leading terms (stopword-filtered) and return the top 3 clusters as `{label, count}`. Add `GET /api/creators` returning the top 20 most active creators (by sim_count desc) for a "Top Creators" discovery surface.
2. Add a `/creators/<handle>` page to the frontend. Layout: a profile header (handle displayed as `@handle`, total sim count, top model chip, top topic clusters as tag chips); below, a grid of simulation cards (same card component as `/explore` — no new component needed); a sidebar with aggregate stats displayed as metric cards (same 2×2 layout as the Impact Scorecard panel); at the bottom, an RSS subscribe button: "Subscribe to @handle's simulations" linking to `/creators/<handle>/feed.xml`. Implement `/creators/<handle>/feed.xml` as a Flask route that renders a filtered Atom feed (same `atom_feed.py` template from PR #60, filtered to `creator_handle == handle`). The feed title: `"MiroShark — @handle's simulations"`. Add a "Top Creators" sidebar widget on `/explore`: a small ranked list of the top 5 most active handles with their sim counts, linking to their profiles.
3. In the gallery explore view, render a "By @handle" chip below the scenario title on gallery cards where `creator_handle` is set. Clicking the chip navigates to `/creators/<handle>`. Add a `creator` filter to the gallery search: `GET /api/simulation/public?creator=<handle>` returns only that creator's published simulations. In the gallery search bar (from PR #69), support `creator:<handle>` as a search operator: typing `creator:alice` filters to Alice's simulations while still supporting full-text search on the remaining query terms. Add `GET /api/creators/<handle>` to the OpenAPI spec under a new "Creator Profiles" tag. In `docs/FEATURES.md`, add a "Creator Profiles" section explaining the handle system, the RSS subscription, and the `creator:<handle>` search operator. Note explicitly: "No account creation or email required — handles are first-come, first-claimed per instance."

## Selection Rationale

Five angles distinct from all 15 ideas suggested across May 4, May 6, and May 8, and from the two open PRs:

- **Simulation Fork from Round** (#1) — PR #76 built the lineage read layer. Fork-from-round is the write-side complement: counterfactual branching from a specific round, not from config scratch. The most natural research operation — shared history, one divergence point — that MiroShark currently forces researchers to approximate by running two independent sims from scratch. The lineage graph picks it up automatically.
- **Agent Persona Library** (#2) — Closes the reproducibility gap at the agent-population level. Per-Agent Trajectory Export (built) gives individual-level data. Reproducibility Config Export (PR #75) captures what was run. The Persona Library captures *who* was run — the agent configurations that produced those trajectories. Researchers need all three to make a fully citable, reproducible simulation claim.
- **Simulation A/B Comparison** (#3) — First cross-simulation analysis view that is not ancestry-constrained. Lineage handles vertical (parent-child); A/B handles horizontal (unrelated sims, same topic). The synced dual chart + scorecard diff table + overlay mode is the analytical layer that report-builders and model-comparers need. Pure frontend addition against existing data endpoints.
- **Narrative Evolution Tracker** (#4) — Turns MiroShark from a point-in-time tool to a monitoring platform. The scheduled-rerun primitive + time-series endpoint is the longitudinal layer that ongoing research use cases (narrative drift, policy sentiment tracking, market sentiment monitoring) require. Integrates naturally with the completion webhook audience already building automation pipelines.
- **Gallery Creator Profiles** (#5) — The authorship layer that the gallery is missing at 224 forks and growing content volume. No auth required — handles are lightweight enough to avoid friction while being meaningful enough to build creator-centric discovery. The creator RSS feed reuses PR #60's infrastructure directly, and the `creator:<handle>` search operator extends PR #69's search without new architecture.
