### 1. Simulation Replay Stepper

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** Sixteen days of social silence require content that shows what MiroShark actually *does* — not just what it produces. A round-by-round replay stepper is the missing demo primitive. `trajectory.json` already stores per-round agent stances; the stepper needs only a timeline scrubber UI to replay them. Screen-recorded, this becomes the kind of 30-second GIF that makes someone stop scrolling: you see a bearish consensus form, dissolve as counter-information propagates in round 3, and re-coalesce neutral by round 5. That arc — 200 agents updating beliefs in real time — is MiroShark's value proposition, currently invisible unless you run your own simulation. The stepper makes it watchable without running anything. It's also the natural demo artifact for the "5 external tutorials" hyperstition: a screen recording of the replay stepper playing through a dramatic simulation is the easiest tutorial format for a creator who doesn't want to write a 2,000-word post.

**How:**
1. Add `frontend/src/components/StepperControls.vue` (~120 LoC). Props: `trajectoryData` (parsed trajectory.json content) and `currentRound` (reactive integer). Renders: a prev/next round button pair, a round indicator label ("Round 3 / 5"), a play/pause toggle that auto-advances every 1.5s, and an `<input type="range">` scrubber from 0 to `max_round`. Emits `update:currentRound` on all changes (v-model compatible). Style: charcoal background, violet accent for the active round indicator, same border-radius and padding as the existing toolbar buttons. Keyboard shortcut: left/right arrow keys advance rounds when focus is inside the stepper container. Add a "Share this round" button that copies a URL with `?round=N` appended; when the published simulation page loads with `?round=N` in the URL, it pre-initializes the stepper at round N so shared links land at a specific moment.
2. Modify `BeliefDriftChart.vue` to accept a `replayRound` prop (optional integer). When set, filter trajectory entries to `round <= replayRound` before rendering — the chart shows only stances accumulated up to that round. The existing chart is cumulative by design; adding the filter is a single conditional in the data-prep function. In `Step3Simulation.vue`, add a "▶ Replay" button to the simulation toolbar (alongside "@ Mentions", "↻ Stance Shifts"). When clicked: fetch `/api/simulation/{id}/trajectory` if not already cached, initialize `StepperControls` at round 0, and mount it below the visualization area. Play mode auto-advances from round 0 to `max_round`, pausing 1.5s per round. On completion, show a "Share replay" prompt.
3. Add `GET /api/simulation/{id}/trajectory` to `backend/app/api/simulation.py` (~15 LoC) if not already exposed. Returns the trajectory.json content filtered to `{ sim_id, max_round, rounds: [{ round, agents: [{ id, stance, confidence }] }] }`. Publish-gated, Cache-Control: `public, max-age=60`. Add `TrajectoryResponse` to OpenAPI spec, add to `docs/API.md` under "Simulation Data", add to `surfaces_catalog.py` as a new catalogued surface (type: replay).

### 3. Simulation Data Bundle Export

**Type:** Feature / Integration
**Effort:** Small (hours)
**Impact:** The academic citation hyperstition requires a researcher to treat MiroShark as *methodology* — reproducible, citable, attachable to a paper. MiroShark already stores everything needed: `config.json` (parameters), `signal.json` (outcome), `trajectory.json` (per-round stances), per-platform action logs. But retrieving it requires 4–6 separate API calls that most researchers won't discover. A single `GET /api/simulation/{id}/bundle.zip` changes that: one download, one ZIP, everything needed to attach a complete reproducible simulation to a paper appendix. The bundle includes a pre-formatted `methodology.md` (cite-ready APA block + methods-section boilerplate) and a `reproduce.sh` that re-runs the same simulation from scratch — the two artifacts that convert "I used this tool" into "I used this methodology." Three hours of backend work removes the single biggest barrier between "interesting demo" and "tool I can cite in a published paper."

**How:**
1. Add `backend/app/services/bundle_service.py` (~120 LoC, pure stdlib). Function `create_bundle(sim_id: str) -> bytes`. Validates simulation exists and is published (raises 404/403 otherwise). Reads from `data/simulations/{sim_id}/`: `config.json`, `signal.json`, `trajectory.json`, `publish_gate.json`, and all `actions-{platform}.jsonl` files (glob `actions-*.jsonl`). Builds an in-memory ZIP via `zipfile.ZipFile(io.BytesIO(), mode='w', compression=zipfile.ZIP_DEFLATED)`. Bundle layout: `{sim_id}/config.json`, `{sim_id}/signal.json`, `{sim_id}/trajectory.json`, `{sim_id}/actions-{platform}.jsonl` (one per platform), `{sim_id}/methodology.md`, `{sim_id}/reproduce.sh`, `{sim_id}/README.md`. Generates `methodology.md` from config.json + signal.json: **Citation** section (APA: `[Operator]. (2026). [Topic, 80 chars]. MiroShark Simulation {sim_id}. Retrieved from https://miroshark.io/simulation/{sim_id}`), **Parameters** table (topic, agents, rounds, platforms, cost_usd), **Outcome** table (consensus, confidence_pct, final_bullish/bearish/neutral_pct), **Methodology Note** (2-paragraph boilerplate describing grounded agents, platform-specific behavior, Neo4j graph memory, OpenRouter LLM calls — suitable for a paper methods section verbatim). Generates `reproduce.sh` (~15 lines): `./miroshark simulate --topic "{topic}" --agents {agents} --rounds {rounds} --platforms {platforms}`. Returns the ZIP bytes. Add 8 unit tests: valid ZIP, all expected files present, methodology.md contains the sim_id, reproduce.sh contains the correct topic, bundle raises 403 for unpublished sim, 404 for missing sim, all platform jsonl files included, ZIP_DEFLATED compression applied.
2. Add `GET /api/simulation/{id}/bundle.zip` to `backend/app/api/simulation.py` (~15 LoC). Returns `StreamingResponse(bundle_service.create_bundle(sim_id), media_type='application/zip', headers={'Content-Disposition': f'attachment; filename="miroshark-sim-{sim_id}.zip"'})`. Publish-gated. Cache-Control: `public, max-age=3600`. Add `bundle.zip` to OpenAPI spec (binary response, application/zip). Add to `docs/API.md` under "Simulation Data" with curl: `curl -o sim.zip https://miroshark.io/api/simulation/{id}/bundle.zip`. Add to `surfaces_catalog.py` (type: export). Add a "↓ Research Bundle" link to `Step3Simulation.vue`'s toolbar: an `<a href="/api/simulation/{id}/bundle.zip" download>` anchor (native browser download, zero JS overhead), placed beside the embed button.
3. Add `docs/research.md` (~80 lines): a landing page for researchers explaining MiroShark's methodology in academic register, the bundle download, the APA citation format, and a sample `methodology.md` excerpt. This doc becomes the URL a researcher drops in their paper when citing MiroShark. Update main `README.md` to add a "For Researchers" section (4–5 lines) linking to `docs/research.md` and mentioning the bundle endpoint — placed between Features and Quick Start so it's findable by the exact audience the academic citation hyperstition targets.

### 4. Per-Round Cost Curve

**Type:** Performance / DX
**Effort:** Small (hours)
**Impact:** MiroShark's "$1 per simulation" claim is its most powerful marketing statement and its least auditable one. Operators approving enterprise deployments want to know: which round costs the most? Is cost front-loaded (early exploration rounds) or back-loaded (later rounds with richer graph memory)? Can we cut cost by stopping at round 3 when consensus stabilizes? Without a cost curve, these questions are unanswerable — the only cost signal is the total in `signal.json`. A `GET /api/simulation/{id}/cost-curve` endpoint makes the $1 claim a *breakdown*, not a ballpark: "Round 1 was 31% of total cost, rounds 4–5 were 18% combined." That breakdown is what an IT department needs to approve department-wide deployment ("we can cap cost at round 3 if budget is tight"). It also creates the observability hook for a future early-stop optimization, and produces the exact type of concrete technical detail that breaks social silence — "we ran cost accounting on 50 MiroShark simulations; here's the distribution" is a thread that costs nothing to write once the data exists.

**How:**
1. Add `backend/app/services/cost_curve_service.py` (~80 LoC, pure stdlib). Function `compute_cost_curve(sim_id: str) -> dict`. Reads `trajectory.json` and `signal.json`. If trajectory.json entries contain per-round cost fields (`cost_usd`, `token_count`, or `api_calls`), extract them directly and set `"cost_granularity": "per_round"`. If absent (older simulations), distribute `signal.json.cost_usd` evenly across rounds as best-effort and set `"cost_granularity": "estimated"`. Returns: `{ "sim_id", "total_cost_usd", "cost_granularity", "rounds": [{ "round": int, "cost_usd": float, "cumulative_cost_usd": float, "pct_of_total": float, "agent_count_active": int|null, "api_calls": int|null }] }`. The `pct_of_total` per round is the operationally useful field — "Round 1: 31%" is the shareable claim. Cache to `.cost-curve-cache/{sim_id}.json` with 1-hour TTL. Add 6 unit tests: per-round data extracted when present, fallback distributes evenly across rounds, cumulative sum is monotonically non-decreasing, pct_of_total sums to ~100% (±0.1% float), empty trajectory returns 404, publish-gate enforced.
2. Add `GET /api/simulation/{id}/cost-curve` to `backend/app/api/simulation.py` (~10 LoC). Publish-gated. Cache-Control: `public, max-age=3600`. Add `CostCurveResponse` to OpenAPI spec. Add to `docs/API.md`. Add to `surfaces_catalog.py` (type: analytics). Add `CostCurveChart.vue` (~80 LoC): a horizontal segmented bar showing per-round cost as a proportion of total, with each segment labeled on hover (round number + pct_of_total). Color: sequential violet from light (round 1) to dark (final round) — the gradient encodes accumulation direction visually. Mount below the signal.json summary card in `Step3Simulation.vue`, rendering only when `signal.json.cost_usd` is non-zero. This chart is the first data visualization on the simulation page that finance/operations stakeholders will export for budget justification slides.
3. Add a `COST_TRACKING_ENABLED` env var to `docs/CONFIGURATION.md`: when `true`, MiroShark logs per-round LLM costs to trajectory.json during simulation execution, enabling `"cost_granularity": "per_round"` precision. Default `false` (estimation fallback active). Document in `docs/INSTALL.md` under "Operator Tools" how to enable it. Update the main README Features bullet list to add: "Per-round cost curve — see exactly where your $1 went, round by round." This converts the cost claim from a single number into an auditable record, which is what converts an interested evaluator into a deploying operator.

### 5. Simulation Diff API

**Type:** Feature / Research
**Effort:** Medium (1-2 days)
**Impact:** A simulation is most valuable in comparison to another simulation. "Does changing the topic wording from 'announces layoffs' to 'announces restructuring' shift the consensus?" "Does Reddit consensus differ from Twitter consensus for the same event?" "Does 150 agents produce a different outcome than 50?" These questions require running two simulations and comparing them — and currently that comparison is manual: copy two `signal.json` files, open a spreadsheet, compute deltas by hand. A `GET /api/simulations/diff?a={id1}&b={id2}` makes comparison one API call: consensus direction change (BULLISH → NEUTRAL), confidence delta (−17pp), stance distribution shift, cost delta, platform mix diff. The diff JSON is itself a research artifact — a researcher can attach two simulation bundles plus a diff to a paper and have complete A/B methodology in three files. It also creates shareable content that requires zero creativity to write: "We ran the same MiroShark simulation with two different framings — here's how the consensus shifted" is a thread that the data writes for you.

**How:**
1. Add `backend/app/services/diff_service.py` (~150 LoC, pure stdlib). Function `compute_diff(sim_id_a: str, sim_id_b: str) -> dict`. Validates both simulations exist and are published (404/403 respectively). Reads `signal.json` and `config.json` for both. Returns: `{ "sim_a": { "id", "topic_excerpt" (80 chars), "published_at" }, "sim_b": { same }, "config_diff": { "agents_delta": int, "rounds_delta": int, "platforms_added": [str], "platforms_removed": [str], "topic_similarity_pct": float (Jaccard similarity of space-tokenized topics, capped 0–100) }, "outcome_diff": { "consensus_a": str, "consensus_b": str, "consensus_changed": bool, "confidence_delta_pp": float (b−a), "final_bullish_pct_delta_pp": float, "final_bearish_pct_delta_pp": float, "final_neutral_pct_delta_pp": float }, "cost_diff": { "cost_a_usd": float|null, "cost_b_usd": float|null, "cost_delta_usd": float|null }, "interpretation": str (auto-generated one-sentence summary: "Changing from '{topic_a_30chars}' to '{topic_b_30chars}' shifted consensus from {consensus_a} to {consensus_b} with a {confidence_delta:+.0f}pp confidence change.") }`. `topic_similarity_pct` is the most useful field for researchers: high similarity = controlled variation; low similarity = fundamentally different scenarios. Cache to `.diff-cache/{id_a}-{id_b}.json` with 24-hour TTL. Add 10 unit tests: same sim vs itself = all-zero deltas + 100% similarity, consensus_changed reflects correctly, missing sim = 404, unpublished sim = 403, platforms_added/removed computed correctly, Jaccard correct for empty topics, interpretation includes both consensus values, cost_delta null when either sim lacks tracking, negative delta formatted with sign, bidirectional call (a,b) = inverse deltas of (b,a).
2. Add `GET /api/simulations/diff` to `backend/app/api/simulation.py` (~15 LoC). Query params: `?a={sim_id}&b={sim_id}`. Both must be published. Cache-Control: `public, max-age=86400`. Add `SimulationDiffResponse` to OpenAPI spec under a new "Comparison" tag. Add to `docs/API.md` under "Comparison" with curl example: `curl "https://miroshark.io/api/simulations/diff?a=SIM_A&b=SIM_B"`. Add to `surfaces_catalog.py` (type: comparison). Add a "Compare" button to the `Step3Simulation.vue` toolbar: when clicked, shows a one-input modal ("Paste a simulation URL or ID to compare"), fetches the diff, and mounts `SimulationDiffPanel.vue` as a drawer overlay.
3. Add `frontend/src/components/SimulationDiffPanel.vue` (~130 LoC). Props: `diffData`. Layout: two-column header (Sim A topic_excerpt | Sim B topic_excerpt, truncated to 40 chars each), then three row groups — **Config** (agents/rounds deltas as ±N chips; platforms added/removed as colored tags; topic similarity as a horizontal bar with "X% similar" label), **Outcome** (`consensus_a → consensus_b` with directional arrow; confidence delta as ±Npp with green/red; stance distribution deltas as a before/after bar pair), **Cost** (cost_a, cost_b, delta with +/− prefix). Bottom: `interpretation` text in italic as a pull-quote. Add a "Copy diff JSON" button (copies raw API response) and a "Cite this comparison" button that generates APA: `[Operator]. (2026). A/B Simulation Comparison: [topic_a] vs. [topic_b]. MiroShark. Retrieved from https://miroshark.io/api/simulations/diff?a={id_a}&b={id_b}`. The compare-and-cite workflow is the research primitive that converts MiroShark from a demo into a controlled-experiment methodology tool — the exact requirement for the academic citation hyperstition.
