### 1. Agent Archetype Atlas

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The agents.json surface (27th, PR #137) exports per-agent identity data — archetype name, initial stance, final stance, platforms used — for every simulation. But that data exists only per-simulation; there's no view of how archetypes behave across the entire platform. If "Risk Manager" has appeared in 60 published simulations, what's the aggregate pattern? Does it typically start neutral and converge bearish? Does it flip stances more than "Market Analyst"? A `GET /api/agents/archetypes` endpoint aggregates agents.json data across all published sims into a platform-wide atlas. Each archetype entry: `{name, sim_count, avg_initial_bullish_pct, avg_final_bullish_pct, flip_rate, avg_influence_score, most_common_platform}`. A `GET /api/agents/archetypes/:name` returns detailed stats for a single archetype. A new `/archetypes` page shows the "personality atlas" — a grid of archetype cards with stance distribution bars, flip-rate badges, and sim counts. Practically: operators building new simulations can reference the atlas to predict how archetypes will behave ("this scenario needs a high-flip archetype to generate interesting dynamics"), and researchers studying agent behavior across scenarios have a quantitative baseline. This is the first cross-simulation analytics surface that uses the agents.json data that just landed.

**How:**
1. Add `backend/app/services/archetype_atlas_service.py` (~150 LoC, pure stdlib). `build_atlas(data_dir)` scans all published sim dirs for `agents.json` files (the per-sim agent export — same path as `GET /api/simulation/:id/agents.json`). For each agent entry, record: `archetype_name` (from the `archetype` field), `initial_stance`, `final_stance`, `platforms`, `influence_score` (if present). After scanning all sims, group by `archetype_name` and compute: `sim_count` (number of sims this archetype appeared in), `avg_initial_bullish_pct` (pct of appearances where initial_stance == "Bullish"), `avg_final_bullish_pct` (same for final), `flip_rate` (pct of appearances where initial_stance != final_stance), `avg_influence_score` (mean across all appearances where present, null if never present), `most_common_platform` (mode of platforms across all appearances). Writes result to `<data_dir>/archetype_atlas.json` with 1h mtime-based TTL (rebuild if file older than 3600s). `get_archetype(data_dir, name)` reads the atlas and returns the single entry matching `name` (case-insensitive). Add `GET /api/agents/archetypes` (public, returns full atlas array sorted by `sim_count` desc) and `GET /api/agents/archetypes/:name` (public, returns single entry or 404). Both endpoints serve from the cached atlas file — no per-request scan. Add `archetype_count: int` to `GET /api/stats` (count of distinct archetypes across all published sims from the atlas).
2. Add `ArchetypeAtlasView.vue` at `/archetypes`. Page header: "Agent Archetype Atlas" with a subtitle "How each archetype behaves across all published simulations" and a count chip "N archetypes". Sort controls: two sort buttons ("Most Used" and "Most Volatile" — volatile = highest flip_rate). Content: a responsive grid of archetype cards (2 columns desktop, 1 mobile). Each card: archetype name (`font-semibold text-lg`), sim count chip (`N sims`), then three horizontal stat bars (Bullish start, Bullish end, Flip rate) each rendered as a `w-full bg-gray-800 rounded` with a fill proportional to the percentage. At the bottom of each card: `Most common on: {platform}` and optionally `Avg influence: N.N` if present. Clicking a card expands it inline (or navigates to `/archetypes/:name`) showing the full detail view: a simple before/after stance comparison table (columns: Initial Stance distribution, Final Stance distribution, Δ), plus a "Find simulations with this archetype →" link that calls `GET /api/gallery?archetype=<name>` (add `?archetype=` filter support to the gallery endpoint while building this — one extra filter on the scan loop, same pattern as `?tags=`). In `Step1GraphBuild.vue`, add a small "Atlas →" link next to each archetype in the multi-select that opens a tooltip or side panel showing the archetype's atlas card stats. Add `/archetypes` to Vue Router. Add an "Archetypes" chip in the `/explore` toolbar alongside Tags, Collections, and Leaderboard.
3. Add `GET /api/agents/archetypes` and `GET /api/agents/archetypes/:name` to the OpenAPI spec under the "Analytics" tag with `ArchetypeAtlasResponse` (array of `ArchetypeEntry`) and `ArchetypeEntry` schemas: `{name: str, sim_count: int, avg_initial_bullish_pct: float, avg_final_bullish_pct: float, flip_rate: float, avg_influence_score: float|null, most_common_platform: str|null}`. Add the `?archetype=` query param to `GET /api/gallery` in the spec. Add 12 unit tests in `backend/tests/test_unit_archetype_atlas.py`: atlas builds from valid agents.json files, archetype with single appearance, flip_rate 0.0 when all stances unchanged, flip_rate 1.0 when all stances flip, avg_influence_score null when no influence data, most_common_platform correctly identifies mode, atlas cached on second call (no re-scan within TTL), cache rebuilds when file older than TTL, `GET /api/agents/archetypes` returns sorted by sim_count desc, `GET /api/agents/archetypes/:name` returns 404 for unknown name, case-insensitive name match, `archetype_count` in stats endpoint. In `docs/FEATURES.md`, add an "Agent Archetype Atlas" section: explains the scan source (agents.json per sim), the 1-hour cache, the `flip_rate` definition (pct of appearances where initial ≠ final stance), and the `?archetype=` gallery filter.

### 2. Platform Activity Timeline

**Type:** Feature
**Effort:** Small (hours)
**Impact:** There is no temporal view of MiroShark's activity. An operator who runs simulations weekly has no way to see "we did 12 simulations this month vs 4 last month." A researcher studying the platform's usage patterns can't see whether activity is accelerating, plateauing, or concentrated on certain days. A GitHub-style contribution heatmap — a 13-week × 7-day grid where each cell represents one day, colored by simulation run count — answers all of these at a glance. `GET /api/activity?days=90` returns `[{date, run_count, published_count, avg_yes_pct}]`, scanned from sim directory mtimes (same scan pattern as the health and leaderboard services). A new `/activity` page renders the grid with hover tooltips (date, N simulations, N published, avg consensus direction) and a summary bar below ("N simulations in the last 90 days — N published, avg consensus: X% YES"). For operators who share their MiroShark instance with research teams, the activity page is the "is this thing being used?" signal. For the platform's own growth narrative, it's a public artifact that shows momentum. Pure stdlib: date arithmetic with `datetime`, no new dependencies.

**How:**
1. Add `backend/app/services/activity_service.py` (~120 LoC, pure stdlib). `get_activity(data_dir, days=90)` scans all sim dirs for `config.json`, reads `created_at` (ISO from config) or falls back to the directory's `os.path.getmtime()` if `created_at` is absent. Buckets each sim into its calendar date (UTC). For each day in the `days` range (from `today - days` to `today` inclusive), computes: `run_count` (total sims created that day), `published_count` (sims with `published_at` set to that day — check `published_at` in config.json), `avg_yes_pct` (mean of `final_yes_pct` from `signal.json` for all published sims on that day, null if none). Returns a list of 90 date objects sorted ascending. Caches to `activity_cache.json` in data_dir with a 30-minute TTL. Add `GET /api/activity` accepting `?days=N` (default 90, max 365, min 7). Returns the list. Public endpoint, no auth. Also add `GET /api/activity/summary` (public): returns `{total_runs_30d: int, total_published_30d: int, avg_daily_runs_30d: float, most_active_date: str|null, most_active_count: int}` for use in the status page and stats widget — a quick summary without the full day-by-day list.
2. Add `ActivityView.vue` at `/activity`. Layout: dark-theme page with header "Activity" and subtitle "Simulation runs over the past 90 days". Main content: a 13-column (weeks) × 7-row (days of week) grid. Each cell is a 12×12px square with `rounded-sm` and a background that scales from `bg-gray-800` (0 sims) through `bg-violet-900` (1-2), `bg-violet-700` (3-5), `bg-violet-500` (6-9) to `bg-violet-400` (10+). A tooltip on hover (using `title` attribute or a simple `v-tooltip`-equivalent pattern) shows: `{day}: N runs, N published`. Week labels (Mon/Wed/Fri) on the left axis. Month labels on the top axis (show month name at the first Sunday of each month). Below the grid: a single-row summary strip — "N simulations in the last 90 days · N published · avg YES: X%" (formatted from the `summary` endpoint). A second visual: a simple CSS-only bar chart of run_count by week (sum per week, 13 bars, width proportional to max-week count). The page is public (no admin token). Link from the `/status` page (add an "Activity" link in the status footer) and from the `/explore` toolbar (add "Activity" chip alongside Tags and Leaderboard). Add `/activity` to Vue Router. In `frontend/src/api/simulation.js`, add `getActivity(days)` and `getActivitySummary()` helpers.
3. Add `GET /api/activity` and `GET /api/activity/summary` to the OpenAPI spec under the "Platform" tag (alongside `/api/health` and `/api/stats`) with `ActivityDay` and `ActivitySummary` schemas. Add `avg_daily_runs_30d: float` and `most_active_date: str|null` to `GET /api/stats`. Add 10 unit tests in `backend/tests/test_unit_activity.py`: empty data_dir returns 90 zeros, single sim on day N appears in correct bucket, published_count only includes sims with published_at on that day, avg_yes_pct null when no published sims that day, avg_yes_pct computed from signal.json correctly, mtime fallback when created_at absent, days param respected (max 365 clamped), result sorted ascending by date, cache written on first call, summary endpoint returns correct total_runs_30d. In `docs/FEATURES.md`, add a "Platform Activity Timeline" section: explains the scan source (config.json created_at or dir mtime), the 30-minute cache, and the color scale thresholds (for operators who want to customize the heatmap colors in a fork).

### 3. Ecosystem Partner Health Monitor

**Type:** Integration
**Effort:** Small (hours)
**Impact:** The ecosystem.json endpoint (PR #145) is now live with machine-readable entries for 6 ecosystem partners. But the ecosystem page today is static — it shows partner names and descriptions as they were when they submitted their ECOSYSTEM.md PR. Three months from now, some of those projects may have shipped major versions (or gone quiet). An operator curating MiroShark's ecosystem doesn't know which partners are actively shipping without checking each GitHub repo manually. The health monitor automates this: `GET /api/ecosystem/health` reads ecosystem.json, and for each entry with a GitHub URL runs `gh api repos/<owner>/<repo>` (the gh CLI handles auth internally — no sandbox workaround needed) to fetch `pushed_at`, `stargazers_count`, and `open_issues_count`. A project is "active" if pushed_at is within 30 days, "quiet" within 90 days, and "stale" beyond 90 days. The result is written to `ecosystem_health.json` with a 6-hour TTL. The EcosystemView.vue partner cards gain a live health badge — a green/yellow/red dot plus "Active · N stars" or "Last commit: N days ago" — turning the ecosystem page from a static ECOSYSTEM.md mirror into a living dashboard. For the six partners building on MiroShark, it's a public signal that MiroShark tracks who's shipping. For new operators evaluating which ecosystem tools to adopt, health status is exactly the signal they need before integrating.

**How:**
1. Add `backend/app/services/ecosystem_health_service.py` (~120 LoC, pure stdlib). `check_ecosystem_health(data_dir)` reads the ecosystem cache from `GET /api/ecosystem` (or directly from the in-memory cache in `ecosystem_service.py`). For each project entry that has a non-null `github` URL, extract `owner/repo` from the URL path. Call `subprocess.run(["gh", "api", f"repos/{owner}/{repo}", "--jq", "{pushed_at, stargazers_count, open_issues_count}"], capture_output=True, text=True, timeout=10)` — the gh CLI handles GitHub API auth internally. Parse the JSON result. Compute `days_since_push = (today - pushed_at).days`. Assign `health_status`: `"active"` if `days_since_push <= 30`, `"quiet"` if `<= 90`, `"stale"` if `> 90`. On `subprocess.CalledProcessError` or JSON parse failure (e.g., private repo, renamed repo), set `health_status = "unknown"`. Writes `<data_dir>/ecosystem_health.json = {generated_at: ISO, projects: [{name, github_url, github_stars, days_since_push, open_issues, health_status}]}` with 6-hour TTL (skip re-fetch if file age < 21600s). Add `GET /api/ecosystem/health` (public, returns cached result): if file is fresh, return it immediately; otherwise trigger a background rebuild via `threading.Thread(target=check_ecosystem_health, daemon=True).start()` and return the stale result with `X-Cache: stale` header (never block the request on live GitHub API calls). Add `POST /api/ecosystem/health/refresh` (admin-gated): forces a synchronous rebuild (for admins who just merged a new ecosystem PR and want fresh data immediately).
2. Update `EcosystemView.vue` to fetch `GET /api/ecosystem/health` on mount alongside `GET /api/ecosystem`. For each project card, look up the health entry by `name` and render a health badge in the card's top-right corner: a 10px dot (green for active, yellow for quiet, red for stale, gray for unknown) and a text label: "Active" / "Quiet" / "Stale" / "Unknown". Below the project name, add a small stats row: `⭐ {github_stars}` and `{open_issues} open issues` — only rendered when health data is present and `health_status != "unknown"`. For stale projects, replace the stats row with `Last active {days_since_push}d ago` in `text-amber-400`. In the page header, add a "Ecosystem Health" summary chip: "N active · N quiet · N stale" (counted from the health response). On click, scroll to a filter UI that shows only a given health tier. Add `getEcosystemHealth()` to `frontend/src/api/simulation.js`. In `EcosystemView.vue`, handle the case where health data hasn't loaded yet (render cards without badges, no layout shift — health overlay appears progressively as the second fetch resolves).
3. Add `GET /api/ecosystem/health` and `POST /api/ecosystem/health/refresh` to the OpenAPI spec under the "Community" tag (alongside the existing Ecosystem endpoints). Add `EcosystemHealthResponse` schema: `{generated_at: str, projects: [{name: str, github_url: str, github_stars: int, days_since_push: int|null, open_issues: int|null, health_status: str}]}`. Add 10 unit tests in `backend/tests/test_unit_ecosystem_health.py`: active status when days_since_push <= 30, quiet status 31-90 days, stale status > 90 days, unknown status when gh call fails, unknown status when repo URL is non-GitHub (no subdomain match), health file written with correct TTL, stale file served with X-Cache: stale header, fresh file served without re-fetch, admin token required for POST /refresh, public GET requires no token. In `docs/FEATURES.md`, update the "Ecosystem Registry" section to add a "Partner Health" subsection: explains the 6-hour TTL, the background rebuild pattern (request never blocks), and the three health tiers with their day thresholds.

### 4. Simulation Project Series

**Type:** Feature
**Effort:** Small (hours)
**Impact:** `GET /api/project/<id>/stats` shipped in PR #147 — an aggregated stats view for a named project. But there is no way to create a project, assign simulations to one, or browse a project's simulation gallery. The backend has a stats endpoint with no objects to compute stats over. Project Series fills the gap: an operator creates a named research series ("EU Digital Sovereignty Study Q2 2026", "Weekly Market Pulse") and assigns simulations to it at run time or retroactively. Unlike Collections (editorial curator picks) and Tags (semantic labels), Projects are workflow containers — everything related to one research initiative in one place. The `/projects` page shows all projects with their sim count, latest activity, and average confidence. Clicking into a project shows a gallery filtered to those sims with the aggregate stats banner from #147 at the top. For operators running recurring simulation series — weekly scenarios, per-client research batches, longitudinal studies — Projects is the organizational primitive that makes MiroShark viable as a production research tool rather than a collection of isolated runs.

**How:**
1. Add `backend/app/services/projects_service.py` (~180 LoC, pure stdlib). Stores project metadata in `<data_dir>/projects.json`: `{projects: [{project_id: str, name: str (max 80 chars), description: str|null (max 280 chars), created_at: ISO, created_by_hash: str (sha256[:12] of admin token, same pattern as operator dashboard), color: str|null (hex color for UI, 7 chars)}]}`. `create_project(data_dir, name, description, admin_token)` validates name (1-80 chars, no leading/trailing whitespace, unique within the file), generates `project_id = hashlib.sha256(f"{name}{created_at}".encode()).hexdigest()[:12]`, appends to `projects.json` (atomic write). `assign_simulation(data_dir, sim_id, project_id)` validates both exist, writes `project_id` to `<sim_dir>/config.json` (same pattern as `parent_sim_id` from the Simulation Fork feature). `get_project_simulations(data_dir, project_id)` scans all sim dirs for config entries with matching `project_id`, returns gallery-card payloads. `list_projects(data_dir, admin_token_hash)` returns all projects (admin sees all; public sees only projects that have at least one published simulation). Add `POST /api/project` (admin-gated), `GET /api/projects` (public: projects with ≥1 published sim; admin: all), `POST /api/simulation/:id/project/:project_id` (admin-gated: assigns a sim), `DELETE /api/simulation/:id/project` (admin-gated: removes sim from its current project), `GET /api/project/:id/simulations` (public: returns gallery cards for published sims in the project; admin: all sims). The existing `GET /api/project/:id/stats` (#147) already returns aggregated stats — integrate `get_project_simulations` to feed it if it currently scans on its own.
2. Add `ProjectsView.vue` at `/projects` — a page listing all project series. Admin-gated for the full list; public visitors see projects with published simulations. Page header: "Projects" with a "New Project" button (admin-only). Content: a list of project cards, each showing: a colored left border (project color or default violet), project name, description (truncated at 120 chars), sim count chip, avg confidence score badge, "Last active: N days ago", and a "→" link to the project detail page. A "New Project" button opens a modal with name, description, and color-picker fields. Add `ProjectDetailView.vue` at `/projects/:id`. Header: project name, description, color accent. Stat banner (from `GET /api/project/:id/stats`): total sims, published count, avg confidence, avg yes_pct. Below: a simulation gallery grid (same cards as `/explore`) filtered to this project's sims, sorted by created_at desc. An "Assign simulation" input (admin-only) — a text input for sim_id with an "Add to project" button that calls `POST /api/simulation/:id/project/:project_id`. In `Step1GraphBuild.vue`, add an "Add to project" select dropdown in the "Advanced" section (same collapsible as the completion webhook option): lists all projects from `GET /api/projects`. In `Step3Simulation.vue`, if the sim has a `project_id`, show the project name as a clickable chip in the sim header: "📁 {Project Name} →". Add `/projects` and `/projects/:id` routes to Vue Router. Add a "Projects" chip in the `/explore` toolbar.
3. Add `POST /api/project`, `GET /api/projects`, `POST /api/simulation/:id/project/:project_id`, `DELETE /api/simulation/:id/project`, and `GET /api/project/:id/simulations` to the OpenAPI spec under a new "Projects" tag with `Project`, `ProjectsResponse`, and `ProjectSimulationsResponse` schemas. Add `project_id: string|null` and `project_name: string|null` to the gallery card payload (populated from config.json `project_id` and resolved via `projects.json`). Add 14 unit tests in `backend/tests/test_unit_projects.py`: create_project writes to projects.json, duplicate name rejected, name > 80 chars rejected, assign_simulation writes project_id to config.json, assign_simulation with non-existent project returns 404, reassign overwrites previous project_id (a sim can only belong to one project), get_project_simulations returns correct sims, public GET /projects excludes projects with no published sims, admin GET /projects includes all projects, GET /project/:id/simulations public returns only published sims, GET /project/:id/simulations admin returns all, DELETE /api/simulation/:id/project clears project_id, admin token required for POST and DELETE, `project_id` present in gallery card payload. In `docs/FEATURES.md`, add a "Simulation Projects" section: distinguishes Projects (workflow containers, operator-organized) from Collections (curator-curated bundles), Tags (semantic labels), and the Operator Dashboard (full sim inventory across all projects).

### 5. Simulation Highlights Reel

**Type:** Feature
**Effort:** Small (hours)
**Impact:** A simulation with 15 rounds and 20 agents generates hundreds of actions and stance transitions. The most interesting thing that happened — "the Risk Manager flipped in round 7 and took two neutral agents with it, triggering the biggest single-round consensus shift of the entire run" — is buried in the data. Finding it requires scrubbing through the replay manually. The Highlights Reel auto-extracts the 3-5 most significant moments from each simulation's data: the round with the largest single-round yes_pct delta (from `trajectory.json`), the most-influential post by any agent (highest influence score from `actions.jsonl`), the round where the first coalition crystallized (from `coalitions.json`), and the agent whose stance flip correlated with the largest downstream consensus change. Each highlight has a `type`, a `round`, an optional `agent`, a `description` (plain text, generated from the data — no AI call needed), and a `significance` score. `GET /api/simulation/:id/highlights` returns this list cached to `highlights.json` (computed once on first request, never regenerated unless the sim reruns). On the watch page, a "Highlights" button in the replay toolbar opens a compact 5-item carousel; clicking any highlight jumps the replay to that round. A "Key Moment" chip above the replay player surfaces the single highest-significance moment as a teaser. For sharing in a Telegram group or research note, the highlights answer "what's the most interesting thing about this sim?" in one sentence each — no replay required.

**How:**
1. Add `backend/app/services/highlights_service.py` (~200 LoC, pure stdlib). `generate_highlights(sim_id, data_dir)` reads: `trajectory.json` (array of `{round: int, yes_pct: float, no_pct, neutral_pct}` — compute delta from previous round to find the round with the max `|yes_pct_delta|`); `actions.jsonl` (each line: `{round, agent, action_type, post_text, influence_score: float|null}` — find the post with the highest `influence_score`); `coalitions.json` (array of `{round_detected: int, members: [...], label: str|null}` — take the lowest `round_detected` coalition as the "first coalition forms" moment); `trajectory.json` again for agent-level stance changes if `agent_stances` per round is present (or skip if not available). For each found highlight, generate a plain-English description from the data (no API call — rule-based templates): e.g., `"Round {N}: YES consensus jumped from {prev}% to {curr}% — the single largest shift of the run"` or `"Round {N}: {Agent} posted the most influential message, reaching {score:.1f} influence score"` or `"Round {N}: First coalition formed — {label if label else str(len(members)) + ' agents'} aligned"`. Assign `significance` as a normalized float (0.0–1.0): for consensus_shift, `significance = abs(delta) / 50.0` (capped at 1.0); for influential_post, `significance = min(influence_score / 10.0, 1.0)` (adjust scale to observed data); for coalition_form, `significance = 0.6` (fixed, always notable but rarely the most dramatic). Sort by significance desc. Limit to top 5. Write to `<sim_dir>/highlights.json`. Return the list. Graceful fallback: if any source file is absent, skip that highlight type rather than failing. `get_highlights(sim_id, data_dir)` reads `highlights.json` if present, calls `generate_highlights` if not. Add `GET /api/simulation/:id/highlights` (public): returns the highlights array. If `highlights.json` doesn't exist and the simulation is published, generates synchronously (fast — no external calls). Returns empty array `[]` if source files are all absent (unpublished sims or very old sims).
2. In `Step3Simulation.vue`, in the replay toolbar (wherever round navigation controls are rendered), add a "Highlights" button (🎬 or a film-strip icon). On click, show a `highlights-carousel` component: a compact horizontal strip (max 5 items) with left/right arrow navigation. Each item: a type badge chip (`consensus_shift` = orange, `influential_post` = blue, `coalition_form` = purple), a "Round N" label, and the description text (truncated to 80 chars with ellipsis, full text in a `title` tooltip). Clicking an item sets `currentRound = highlight.round` in the replay player (same reactive state change as manual round navigation — no new mechanism needed). When the carousel is closed, the replay player stays at the selected round. Above the replay player (before the timeline slider), render the single highest-significance highlight as a "Key Moment" chip: `★ {description[:60]}… (Round N)` with an orange left border. This chip is always visible when highlights are available — it's the "hook" for sharing. Fetch `GET /api/simulation/:id/highlights` on page mount alongside the simulation data (parallel fetch). In `EmbedDialog.vue`, add a "Highlights" line in the embed description section: "Key moments auto-detected — jump to round N directly from the embed URL with `?round=N`." In the Archive Bundle (if/when it ships), include `highlights.json`.
3. Add `GET /api/simulation/:id/highlights` to the OpenAPI spec under the "Analytics" tag with `HighlightsResponse` and `Highlight` schemas: `{highlights: [{type: str, round: int, agent: str|null, description: str, significance: float}]}`. Add `has_highlights: bool` to the gallery card payload (true if `highlights.json` exists for the sim). Add a "Most Significant Round" column to the leaderboard if it ever adds an `impact` metric tab — seed from the highest-significance highlight round. ## Selection Rationale

Five angles distinct from all 10 ideas suggested May 28 and May 30, the 5 from June 2, the 31 push-blocked local builds, and any open PRs:

- **Agent Archetype Atlas** (#1) — agents.json (27th surface, PR #137) landed recently and exports per-agent identity data per simulation. Nobody has aggregated it cross-simulation yet. The atlas is the first analytics surface that answers "how does this agent type behave across the entire platform?" — a qualitatively different question from all per-simulation analytics (journey, sparklines, interaction graph). It also feeds back into the simulation creation flow as a reference for operators choosing archetypes.

- **Platform Activity Timeline** (#2) — there is no temporal view of simulation activity anywhere in the platform. The health endpoint tells you the system is running; the leaderboard tells you what's popular; neither tells you whether activity is accelerating or plateauing. A GitHub-style contribution heatmap is the standard temporal summary for developer tools, and MiroShark's scan infrastructure (health_service, leaderboard_service) already demonstrates the scan-and-aggregate pattern. One new service, one new page.

- **Ecosystem Partner Health Monitor** (#3) — ecosystem.json (#145) is the most recent backend addition. Building on it directly — by adding a live health layer using the gh CLI (already available, already authenticated) — turns a static partner directory into a living ecosystem dashboard. No new external APIs, no new secrets, no new dependencies. Six partners are already there; within weeks there could be twelve.

- **Simulation Project Series** (#4) — `GET /api/project/:id/stats` (#147) shipped a backend endpoint for per-project analytics with nothing to serve it. Project creation, assignment, and gallery pages complete the missing half of #147. Distinct from Collections (curator picks, not operator workflow), Tags (semantic labels, not research grouping), and the Operator Dashboard (full inventory, not organized series). The practical gap: an operator running 30 simulations across four research initiatives has no way to organize them.

- **Simulation Highlights Reel** (#5) — the platform has deep per-round, per-agent data across trajectory.json, actions.jsonl, and coalitions.json. The highlights engine reads that data and surfaces the 3-5 most significant moments using rule-based templates — no API call, no AI cost. The result is a shareable summary of "what happened in this simulation" that lives at the round level (links directly to `?round=N`), surfaces as a Key Moment chip on the watch page, and makes the replay player navigable by insight rather than by sequential scrubbing.
