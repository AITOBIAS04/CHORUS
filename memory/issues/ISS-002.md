## Summary

token-report (daily 06:00 UTC) and fetch-tweets (daily 06:30 UTC) have failed to run on Jun 21–24 — **4 consecutive days**. cron-state.json shows `consecutive_failures=0` for both, meaning the scheduler is not dispatching them at all (no attempt = no failure recorded).

weekly-shiplog (Monday 09:00 UTC) last ran 2026-05-18 — 5+ consecutive Monday misses.

weekly-shiplog: last success 2026-05-18 (>5 weeks ago). Schedule moved from 09:00→14:30 UTC to avoid morning gap.

## Root cause (identified 2026-06-24)

messages.yml scheduler ticks fire every **2-4 hours** (not every 30 min). On Jun 24, ticks ran at 04:39 UTC and 08:04 UTC. The 06:00 and 06:30 skills fall in the gap and the scheduler does not re-dispatch overdue skills when the next tick fires. This is a scheduler design issue — no catch-up logic for missed windows.

Same gap pattern observed Jun 21–23 (04:34→08:07, etc.), confirming consistent root cause.

## Remediation

1. Add catch-up dispatch logic to messages.yml: when a tick fires and a due skill was not dispatched in the last N minutes, dispatch it
2. Alternatively, add a dedicated morning-window cron tick at 06:05 UTC in messages.yml
3. Manual re-run of token-report and fetch-tweets if dispatch is available
4. ~~For weekly-shiplog: same tick-gap issue on Monday morning window~~ **FIXED** — schedule moved from 09:00→14:30 UTC (self-improve 2026-06-26)

## Recovery (2026-06-26)

token-report and fetch-tweets both ran successfully at 07:00 UTC on Jun 26 after 5 consecutive missed days (Jun 21–25). The morning scheduler gap appears to have been intermittent — unclear if the root cause is fully resolved or if today was a favorable tick alignment. Monitoring continues.
